# CI/CD 파이프라인 이론

---

## 1. CI/CD란?

### 주요 용어

| 용어                                    | 설명                                                     |
| --------------------------------------- | -------------------------------------------------------- |
| **CI (Continuous Integration)**         | 코드 변경을 자주 통합하고 자동으로 빌드·테스트하는 관행  |
| **CD (Continuous Delivery/Deployment)** | 검증된 코드를 자동으로 스테이징/프로덕션에 배포하는 관행 |
| **파이프라인 (Pipeline)**               | 빌드 → 테스트 → 배포를 자동화한 일련의 단계              |
| **아티팩트 (Artifact)**                 | 빌드 결과물 (JAR, Docker Image, 번들 파일 등)            |

> [!CONCEPT] CI/CD 핵심 개념
> CI(Continuous Integration)는 개발자가 코드를 자주 통합하고 자동으로 빌드·테스트하는 관행이고, CD(Continuous Delivery/Deployment)는 검증된 코드를 자동으로 스테이징/프로덕션에 배포하는 관행입니다. 수동 작업을 자동화하여 배포 속도와 품질을 동시에 높입니다.

### CI/CD 파이프라인 전체 흐름

```
┌──────────────────────────────────────────────────────────────────────┐
│                         CI/CD 파이프라인                             │
│                                                                      │
│  ┌───────┐   ┌───────┐   ┌────────┐   ┌─────────┐   ┌────────────┐   │
│  │ 코드  │──►│ 빌드  │──►│ 테스트 │──►│ 스테이징│──►│ 프로덕션   │   │
│  │ Push  │   │ Build │   │ Test   │   │ Stage   │   │ Production │   │
│  └───────┘   └───────┘   └────────┘   └─────────┘   └────────────┘   │
│                                                                      │
│  ├────── CI (지속적 통합) ──────────┤├──── CD (지속적 배포) ──────┤  │
│                                                                      │
│  자동화: 코드 변경 → 빌드 → 테스트 → 배포 (수 분 내 완료)            │
└──────────────────────────────────────────────────────────────────────┘
```

### CI vs CD 비교

| 항목       | CI (지속적 통합)        | CD (지속적 전달/배포) |
| ---------- | ----------------------- | --------------------- |
| **목적**   | 코드 품질 보장          | 빠른 릴리스           |
| **범위**   | 빌드 + 테스트           | 배포 자동화           |
| **트리거** | 코드 Push/PR            | CI 성공 후            |
| **결과**   | 빌드 아티팩트           | 서비스 업데이트       |
| **도구**   | GitHub Actions, Jenkins | CodeDeploy, ArgoCD    |

### CI/CD 도입 전후 비교

| 항목            | CI/CD 없이       | CI/CD 적용        |
| --------------- | ---------------- | ----------------- |
| **배포 빈도**   | 월 1~2회         | 일 수회~수십 회   |
| **배포 시간**   | 수 시간          | 수 분             |
| **실패율**      | 높음 (수동 실수) | 낮음 (자동 검증)  |
| **복구 시간**   | 수 시간          | 수 분 (자동 롤백) |
| **개발자 경험** | 배포 스트레스    | 코드에 집중       |

## 2. 빌드 / 테스트 / 배포 단계

### 주요 용어

| 용어                               | 설명                                                                              |
| ---------------------------------- | --------------------------------------------------------------------------------- |
| **빌드 (Build)**                   | 소스 코드를 컴파일하고 의존성을 다운로드하여 실행 가능한 아티팩트를 생성하는 과정 |
| **단위 테스트 (Unit Test)**        | 개별 함수/메서드 단위로 동작을 검증하는 테스트                                    |
| **통합 테스트 (Integration Test)** | 여러 모듈이 함께 동작하는지 검증하는 테스트                                       |
| **E2E 테스트 (End-to-End)**        | 사용자 관점에서 전체 시스템 흐름을 검증하는 테스트                                |
| **스모크 테스트 (Smoke Test)**     | 배포 후 핵심 기능이 정상 동작하는지 빠르게 확인하는 테스트                        |

> [!CONCEPT] 빌드, 테스트, 배포 단계
> CI/CD 파이프라인은 **빌드**(Build), **테스트**(Test), **배포**(Deploy) 세 핵심 단계로 구성됩니다. 각 단계는 이전 단계가 성공해야 다음으로 진행되며, 실패 시 즉시 개발자에게 알림을 보냅니다.

### 각 단계의 역할

```
┌─────────────────────────────────────────────────────────┐
│  1. 빌드 (Build)                                        │
│  ├── 소스 코드 컴파일                                   │
│  ├── 의존성 다운로드 (npm install, gradle build)        │
│  ├── 정적 분석 (lint, code style)                       │
│  └── 아티팩트 생성 (JAR, Docker Image)                  │
├─────────────────────────────────────────────────────────┤
│  2. 테스트 (Test)                                       │
│  ├── 단위 테스트 (Unit Test)                            │
│  ├── 통합 테스트 (Integration Test)                     │
│  ├── E2E 테스트 (End-to-End)                            │
│  └── 보안 스캔 (SAST, 의존성 취약점)                    │
├─────────────────────────────────────────────────────────┤
│  3. 배포 (Deploy)                                       │
│  ├── 스테이징 환경 배포                                 │
│  ├── 스모크 테스트 (기본 동작 확인)                     │
│  ├── 승인 (수동 또는 자동)                              │
│  └── 프로덕션 배포                                      │
└─────────────────────────────────────────────────────────┘
```

### 테스트 피라미드

```
                    ┌───────┐
                    │  E2E  │  느림, 비용 높음, 적게
                    │ Tests │
                 ┌──┴───────┴──┐
                 │ Integration │  중간
                 │    Tests    │
              ┌──┴─────────────┴──┐
              │    Unit Tests     │  빠름, 비용 낮음, 많이
              └───────────────────┘

권장 비율: Unit 70% / Integration 20% / E2E 10%
```

## 3. 배포 전략

### 주요 용어

| 용어                    | 설명                                                     |
| ----------------------- | -------------------------------------------------------- |
| **Rolling 배포**        | 서버를 순차적으로 하나씩 새 버전으로 교체하는 방식       |
| **Blue-Green 배포**     | 두 환경을 동시에 유지하고 트래픽을 한 번에 전환하는 방식 |
| **Canary 배포**         | 소수 사용자에게 먼저 배포 후 점진적으로 확대하는 방식    |
| **롤백 (Rollback)**     | 배포 실패 시 이전 버전으로 되돌리는 것                   |
| **다운타임 (Downtime)** | 서비스가 중단되어 사용자가 접근할 수 없는 시간           |

> [!CONCEPT] 배포 전략 비교
> 배포 전략은 새 버전을 프로덕션에 릴리스하는 방법을 정의합니다. 각 전략은 다운타임, 위험도, 롤백 속도에서 트레이드오프가 있으며, 서비스 특성에 맞게 선택해야 합니다.

### Rolling 배포

```
시간 →
서버1: [v1][v1][v2][v2][v2][v2]  ← 먼저 업데이트
서버2: [v1][v1][v1][v2][v2][v2]  ← 다음 업데이트
서버3: [v1][v1][v1][v1][v2][v2]  ← 마지막 업데이트

특징: 순차적으로 하나씩 교체
장점: 리소스 추가 불필요
단점: 배포 중 v1/v2 혼재 (호환성 필요)
```

### Blue-Green 배포

```
배포 전:
┌───────────┐         ┌───────────┐
│  Blue     │ ◄─ LB ─ │  사용자   │
│  (v1)     │         └───────────┘
│  현재 활성│
└───────────┘
┌───────────┐
│  Green    │  ← 대기 (v2 배포 준비)
│  (v2)     │
└───────────┘

배포 후 (트래픽 전환):
┌───────────┐
│  Blue     │  ← 대기 (롤백용 유지)
│  (v1)     │
└───────────┘
┌───────────┐         ┌───────────┐
│  Green    │ ◄─ LB ─ │  사용자   │
│  (v2)     │         └───────────┘
│  현재 활성│
└───────────┘

특징: 두 환경을 동시에 유지, LB로 즉시 전환
장점: 다운타임 제로, 즉시 롤백 가능
단점: 2배 리소스 비용
```

### Canary 배포

```
단계 1: 5% 트래픽을 v2로
┌────────────┐  95%  ┌──────────┐
│  v1 (기존) │ ◄──── │    LB    │
└────────────┘       │          │
┌────────────┐   5%  │          │
│  v2 (신규) │ ◄──── │          │
└────────────┘       └──────────┘

단계 2: 문제 없으면 25% → 50% → 100%

특징: 소수 사용자에게 먼저 배포 후 점진적 확대
장점: 위험 최소화, 실제 트래픽으로 검증
단점: 복잡한 라우팅 설정 필요
```

### 배포 전략 비교표

| 전략           | 다운타임 | 롤백 속도 | 리소스 비용 | 복잡도 | 위험도 |
| -------------- | :------: | :-------: | :---------: | :----: | :----: |
| **Rolling**    |   없음   |   느림    |    동일     |  낮음  |  중간  |
| **Blue-Green** |   없음   |   즉시    |     2배     |  중간  |  낮음  |
| **Canary**     |   없음   |   빠름    |    +소량    |  높음  |  최저  |
| **Recreate**   |   있음   |   느림    |    동일     |  최저  |  높음  |

## 4. 버전 관리 전략

### 주요 용어

| 용어                  | 설명                                                                                     |
| --------------------- | ---------------------------------------------------------------------------------------- |
| **Git Flow**          | main, develop, feature, release, hotfix 브랜치를 사용하는 전략 (릴리스 주기 긴 프로젝트) |
| **GitHub Flow**       | main + feature 브랜치만 사용하는 단순한 전략 (빈번한 배포)                               |
| **PR (Pull Request)** | 코드 변경을 리뷰하고 병합을 요청하는 프로세스                                            |
| **브랜치 (Branch)**   | 독립적인 개발 라인. 다른 브랜치에 영향 없이 작업 가능                                    |

> [!CONCEPT] Git 브랜치 전략
> **브랜치 전략**은 팀이 Git 브랜치를 어떻게 사용할지 정의하는 규칙입니다. 프로젝트 규모와 배포 빈도에 따라 적합한 전략이 다릅니다.

### Git Flow

```
main     ─────●─────────────────●─────  (릴리스만)
              │                 │
release  ─────┼──●──●──────────●┤        (릴리스 준비)
              │  │             ││
develop  ──●──●──●──●──●──●──●─●●──●──  (개발 통합)
           │     │     │     │
feature/A ─┘     │     │     │          (기능 개발)
feature/B ───────┘     │     │
feature/C ─────────────┘     │
hotfix    ───────────────────┘          (긴급 수정)

적합: 릴리스 주기가 긴 프로젝트 (월 1~2회)
```

### GitHub Flow

```
main     ──●──●──●──●──●──●──●──●──●──  (항상 배포 가능)
           │  │  │  │  │  │  │  │  │
feature/A ─┘  │  │  │  │  │  │  │  │    (PR → Review → Merge)
feature/B ────┘  │  │  │  │  │  │  │
feature/C ───────┘  │  │  │  │  │  │
bugfix/D  ──────────┘  │  │  │  │  │
feature/E ─────────────┘  │  │  │  │

적합: CI/CD가 잘 갖춰진 팀, 빈번한 배포
규칙: main은 항상 배포 가능 상태 유지
```

### Git Flow vs GitHub Flow 비교

| 항목           | Git Flow            | GitHub Flow    |
| -------------- | ------------------- | -------------- |
| **브랜치 수**  | 많음 (5종류)        | 적음 (2종류)   |
| **복잡도**     | 높음                | 낮음           |
| **배포 빈도**  | 낮음 (계획적)       | 높음 (수시)    |
| **적합 팀**    | 대규모, 릴리스 관리 | 소규모, 애자일 |
| **CI/CD 의존** | 낮음                | 높음           |

### 리포지토리 전략: 모노레포 vs 멀티레포

브랜치 전략 외에, **프론트엔드와 백엔드를 하나의 레포에 관리할지(모노레포), 분리할지(멀티레포)**도 중요한 결정입니다.

> [!CONCEPT] 모노레포 (Monorepo) vs 멀티레포 (Multi-repo)
>
> **모노레포**: 하나의 Git 리포지토리에 프론트엔드, 백엔드, 인프라 코드를 모두 관리
>
> ```
> my-project/
> ├── frontend/          ← Vue.js
> ├── backend/           ← Spring Boot
> ├── infra/             ← CloudFormation
> └── .github/workflows/
>     ├── deploy-frontend.yml
>     └── deploy-backend.yml
> ```
>
> **멀티레포**: 프론트엔드와 백엔드를 각각 독립된 리포지토리로 관리
>
> ```
> my-frontend/           ← Vue.js 전용 레포
> ├── src/
> ├── .github/workflows/deploy.yml
> └── package.json
>
> my-backend/            ← Spring 전용 레포
> ├── src/
> ├── .github/workflows/deploy.yml
> └── build.gradle
> ```

#### 모노레포의 장점과 단점

| 장점                                        | 단점                                                  |
| ------------------------------------------- | ----------------------------------------------------- |
| 프론트/백엔드 동시 변경 시 하나의 PR로 관리 | 레포 크기가 커지고 clone 속도 저하                    |
| 공유 코드(타입, 유틸)를 쉽게 참조           | CI/CD에 `paths` 필터 설정 필요 (변경된 부분만 배포)   |
| 전체 프로젝트의 히스토리를 한눈에 파악      | 팀 간 PR 충돌 가능성 증가                             |
| 일관된 린트/포맷 규칙 적용                  | 프론트 변경이 백엔드 CI를 불필요하게 트리거할 수 있음 |
| 하나의 Issue 트래커로 관리                  | 권한 분리가 어려움 (레포 레벨 접근 제어 불가)         |

#### 멀티레포의 장점과 단점

| 장점                                        | 단점                                               |
| ------------------------------------------- | -------------------------------------------------- |
| 각 팀이 독립적으로 개발·배포                | 프론트/백엔드 동시 변경 시 각각 PR 필요            |
| 레포 = 배포 단위 (CI/CD 단순)               | 공유 코드 참조가 어려움 (별도 패키지 배포 필요)    |
| 팀별 독립된 권한 관리                       | 전체 프로젝트 히스토리를 파악하려면 여러 레포 확인 |
| 레포가 가볍고 clone 빠름                    | API 계약(인터페이스) 관리에 추가 노력 필요         |
| 프론트만 빈번히 배포해도 백엔드에 영향 없음 | 버전 호환성을 수동으로 관리해야 함                 |

#### 어떤 전략을 선택할까?

| 상황                                 | 권장 전략      |
| ------------------------------------ | -------------- |
| 1~3명 소규모 팀, 풀스택 개발         | 모노레포       |
| 프론트/백엔드 팀 분리 (3명+)         | 멀티레포       |
| 배포 주기가 프론트/백엔드 독립적     | 멀티레포       |
| 공유 타입/유틸이 많고 동시 변경 빈번 | 모노레포       |
| 마이크로서비스 (서비스 3개+)         | 멀티레포       |
| 학습/개인 프로젝트                   | 어느 쪽이든 OK |

#### 모노레포에서 CI/CD 설정 (참고)

모노레포에서는 `paths` 필터로 변경된 디렉토리만 배포합니다:

```yaml
# 프론트엔드 워크플로우 — frontend/ 변경 시에만 실행
on:
  push:
    branches: [main]
    paths:
      - 'frontend/**'

jobs:
  deploy:
    steps:
      - uses: actions/checkout@v4
      - run: npm ci
        working-directory: frontend
      - run: npm run build
        working-directory: frontend
```

> [!NOTE]
> 이 과정에서는 **멀티레포(별도 레포)** 방식을 사용합니다.
> 프론트엔드 레포와 백엔드 레포를 각각 만들고, 각 레포에 독립된 배포 워크플로우를 작성합니다.
> Step 8-1 실습에서 실제로 별도 레포 기반 CI/CD를 구축합니다.

## 5. 환경 분리 (dev/staging/prod)

### 주요 용어

| 용어                           | 설명                                                              |
| ------------------------------ | ----------------------------------------------------------------- |
| **개발 환경 (Development)**    | 개발자가 자유롭게 테스트하는 환경. 불안정해도 OK                  |
| **스테이징 환경 (Staging)**    | 프로덕션과 동일한 구성으로 최종 검증하는 환경                     |
| **프로덕션 환경 (Production)** | 실제 사용자가 접근하는 운영 환경. 최고 안정성 요구                |
| **환경 변수**                  | 환경별로 다른 설정값(DB 주소, API 키 등)을 외부에서 주입하는 방법 |

> [!CONCEPT] 환경 분리의 필요성
> **환경 분리**는 개발(dev), 스테이징(staging), 프로덕션(prod) 환경을 독립적으로 운영하여 코드 변경이 사용자에게 영향을 주기 전에 충분히 검증하는 전략입니다.

### 환경별 역할

```
┌─────────────────────────────────────────────────────────┐
│  개발 (Development)                                     │
│  • 개발자가 자유롭게 테스트                             │
│  • 불안정해도 OK                                        │
│  • 최소 사양 인프라                                     │
├─────────────────────────────────────────────────────────┤
│  스테이징 (Staging)                                     │
│  • 프로덕션과 동일한 구성                               │
│  • QA 팀 테스트                                         │
│  • 프로덕션 배포 전 최종 검증                           │
├─────────────────────────────────────────────────────────┤
│  프로덕션 (Production)                                  │
│  • 실제 사용자 서비스                                   │
│  • 최고 안정성 요구                                     │
│  • 변경 시 승인 프로세스                                │
└─────────────────────────────────────────────────────────┘
```

### 환경별 설정 비교

| 항목              | dev             | staging             | prod           |
| ----------------- | --------------- | ------------------- | -------------- |
| **인스턴스 수**   | 1               | 2                   | 4+             |
| **인스턴스 타입** | t3.micro        | t3.small            | m5.large       |
| **DB**            | 로컬/소형 RDS   | RDS (Single-AZ)     | RDS (Multi-AZ) |
| **도메인**        | dev.example.com | staging.example.com | example.com    |
| **로그 레벨**     | DEBUG           | INFO                | WARN           |
| **배포 승인**     | 자동            | 자동                | 수동 승인      |

## 6. Infrastructure as Code (IaC)

### 주요 용어

| 용어                             | 설명                                                                           |
| -------------------------------- | ------------------------------------------------------------------------------ |
| **IaC (Infrastructure as Code)** | 인프라를 코드로 정의하고 버전 관리하는 방법론                                  |
| **선언적 (Declarative)**         | "원하는 상태"를 정의하면 도구가 알아서 맞추는 방식 (CloudFormation, Terraform) |
| **명령적 (Imperative)**          | "수행할 단계"를 순서대로 지시하는 방식 (스크립트)                              |
| **멱등성 (Idempotency)**         | 같은 작업을 여러 번 실행해도 결과가 동일한 성질                                |
| **State 파일**                   | 현재 인프라 상태를 기록한 파일 (Terraform)                                     |

> [!CONCEPT] Infrastructure as Code란
> **IaC**(Infrastructure as Code)는 인프라를 코드로 정의하고 관리하는 방법론입니다. 수동 콘솔 클릭 대신 코드 파일로 인프라를 선언하여 버전 관리, 재현성, 자동화를 달성합니다.

### IaC 도구 비교

| 도구               | 제공자    | 언어                 | 방식          | 상태 관리      |
| ------------------ | --------- | -------------------- | ------------- | -------------- |
| **CloudFormation** | AWS       | YAML/JSON            | 선언적        | AWS 관리       |
| **Terraform**      | HashiCorp | HCL                  | 선언적        | State 파일     |
| **CDK**            | AWS       | TypeScript/Python    | 명령적→선언적 | CloudFormation |
| **Pulumi**         | Pulumi    | TypeScript/Python/Go | 명령적        | Pulumi Cloud   |
| **Ansible**        | Red Hat   | YAML                 | 명령적        | 없음 (멱등성)  |

### 수동 관리 vs IaC

```
수동 관리 (콘솔 클릭):
• "VPC 만들고... 서브넷 만들고... EC2 만들고..."
• 재현 불가능 (같은 환경 다시 만들기 어려움)
• 변경 이력 추적 불가
• 실수 발생 가능

IaC (코드로 정의):
• infrastructure.yaml에 모든 리소스 정의
• git으로 버전 관리 → 변경 이력 추적
• 동일 환경 반복 생성 가능 (dev/staging/prod)
• 코드 리뷰로 실수 방지
• 자동화 파이프라인 연동
```

## 7. GitOps

### 주요 용어

| 용어                         | 설명                                                                                    |
| ---------------------------- | --------------------------------------------------------------------------------------- |
| **GitOps**                   | Git 저장소를 인프라의 단일 진실 공급원(Single Source of Truth)으로 사용하는 운영 방법론 |
| **단일 진실 공급원 (SSOT)**  | 모든 설정/상태의 유일한 원본이 되는 저장소                                              |
| **자가 치유 (Self-Healing)** | 실제 상태가 선언과 다르면 자동으로 복구하는 기능                                        |
| **Pull 기반 배포**           | 에이전트가 Git을 감시하고 변경을 자동 적용하는 방식 (ArgoCD, Flux)                      |

> [!CONCEPT] GitOps 운영 방법론
> **GitOps**는 Git 저장소를 인프라와 애플리케이션의 "단일 진실 공급원(Single Source of Truth)"으로 사용하는 운영 방법론입니다. Git에 선언된 상태와 실제 인프라 상태를 자동으로 동기화합니다.

### GitOps 원칙

| 원칙          | 설명                                |
| ------------- | ----------------------------------- |
| **선언적**    | 원하는 상태를 코드로 선언           |
| **버전 관리** | 모든 변경은 Git을 통해              |
| **자동 적용** | Git 변경 → 자동으로 인프라 반영     |
| **자가 치유** | 실제 상태가 선언과 다르면 자동 복구 |

### GitOps 워크플로우

```
┌───────────────────────────────────────────────────────────┐
│  GitOps 워크플로우                                        │
│                                                           │
│  ┌──────────┐    PR/Merge    ┌─────────┐                  │
│  │ 개발자   │ ─────────────► │   Git   │                  │
│  │          │                │  Repo   │                  │
│  └──────────┘                └────┬────┘                  │
│                                   │                       │
│                              변경 감지                    │
│                                   │                       │
│                                   ▼                       │
│                             ┌────────────┐                │
│                             │  GitOps    │                │
│                             │  Operator  │                │
│                             │(ArgoCD 등) │                │
│                             └─────┬──────┘                │
│                                   │                       │
│                              자동 동기화                  │
│                                   │                       │
│                                   ▼                       │
│                            ┌──────────────┐               │
│                            │  인프라/앱   │               │
│                            │  (AWS, K8s)  │               │
│                            └──────────────┘               │
│                                                           │
│  Git 상태 = 실제 인프라 상태 (항상 동기화)                │
└───────────────────────────────────────────────────────────┘
```

### Push vs Pull 기반 GitOps

| 방식     | 동작                           | 도구 예시                | 보안                       |
| -------- | ------------------------------ | ------------------------ | -------------------------- |
| **Push** | CI가 변경을 인프라에 Push      | GitHub Actions + kubectl | CI에 인프라 접근 권한 필요 |
| **Pull** | 에이전트가 Git을 감시하고 Pull | ArgoCD, Flux             | 인프라 내부에서만 접근     |
