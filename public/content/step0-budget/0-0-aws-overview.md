## 1. 클라우드 컴퓨팅이란?

> [!CONCEPT]
> **클라우드 컴퓨팅**은 인터넷을 통해 IT 리소스(서버, 스토리지, 데이터베이스, 네트워크 등)를 **온디맨드(On-Demand)**로 제공받는 서비스입니다. 사용한 만큼만 비용을 지불하며, 물리적 하드웨어를 직접 구매하거나 관리할 필요가 없습니다.

### 클라우드 컴퓨팅의 5가지 핵심 특성 (NIST 정의)

| 특성                       | 설명                                              |
| -------------------------- | ------------------------------------------------- |
| **온디맨드 셀프서비스**    | 사람의 개입 없이 필요할 때 즉시 리소스 프로비저닝 |
| **광범위한 네트워크 접근** | 인터넷을 통해 어디서든 접근 가능                  |
| **리소스 풀링**            | 여러 사용자가 물리 리소스를 공유 (멀티테넌시)     |
| **빠른 탄력성**            | 수요에 따라 리소스를 즉시 확장/축소               |
| **측정 가능한 서비스**     | 사용량을 정확히 측정하고 과금                     |

### 클라우드 도입의 이점

```
┌─────────────────────────────────────────────────────┐
│              클라우드 도입 6가지 이점                    │
├─────────────────────────────────────────────────────┤
│  1. 자본 비용 → 가변 비용 전환 (CapEx → OpEx)         │
│  2. 규모의 경제 혜택                                  │
│  3. 용량 추정 불필요                                  │
│  4. 속도와 민첩성 향상                                │
│  5. 데이터센터 운영 비용 절감                          │
│  6. 글로벌 배포 (몇 분 만에)                          │
└─────────────────────────────────────────────────────┘
```

## 2. On-Premise vs Cloud 비교

> [!CONCEPT]
> **On-Premise(온프레미스)**는 기업이 자체 데이터센터에 물리 서버를 직접 구매·설치·운영하는 전통적 방식입니다. **Cloud**는 클라우드 제공자의 인프라를 빌려 사용하는 방식입니다.

| 비교 항목       | On-Premise             | Cloud (AWS)               |
| --------------- | ---------------------- | ------------------------- |
| **초기 비용**   | 높음 (서버 구매, 설치) | 없음 (사용한 만큼 지불)   |
| **확장 시간**   | 수 주 ~ 수 개월        | 수 분                     |
| **유지보수**    | 직접 관리 (인력 필요)  | 클라우드 제공자가 관리    |
| **가용성**      | 직접 이중화 구성       | 다중 AZ로 쉽게 구현       |
| **보안**        | 물리+논리 모두 직접    | 물리는 AWS, 논리는 사용자 |
| **비용 모델**   | CapEx (자본 지출)      | OpEx (운영 지출)          |
| **글로벌 배포** | 각 지역에 DC 구축 필요 | 리전 선택만으로 가능      |
| **기술 부채**   | 하드웨어 노후화        | 항상 최신 인프라          |

### 하이브리드 클라우드

```
┌──────────────┐     VPN/Direct Connect     ┌──────────────┐
│  On-Premise  │ ◄─────────────────────────► │   AWS Cloud  │
│  Data Center │                             │   (VPC)      │
│              │     일관된 네트워크 연결       │              │
└──────────────┘                             └──────────────┘
       │                                            │
  민감 데이터 보관                            탄력적 워크로드 처리
  레거시 시스템                              새로운 서비스 배포
```

## 3. 서비스 모델: IaaS / PaaS / SaaS

> [!CONCEPT]
> 클라우드 서비스는 사용자가 관리하는 범위에 따라 **IaaS**(Infrastructure), **PaaS**(Platform), **SaaS**(Software)로 구분됩니다. 아래로 갈수록 사용자의 관리 부담이 줄어듭니다.

### 관리 책임 비교표

| 계층         | On-Premise |    IaaS    |    PaaS    |    SaaS    |
| ------------ | :--------: | :--------: | :--------: | :--------: |
| 애플리케이션 |   사용자   |   사용자   |   사용자   | **제공자** |
| 데이터       |   사용자   |   사용자   |   사용자   | **제공자** |
| 런타임       |   사용자   |   사용자   | **제공자** | **제공자** |
| 미들웨어     |   사용자   |   사용자   | **제공자** | **제공자** |
| OS           |   사용자   |   사용자   | **제공자** | **제공자** |
| 가상화       |   사용자   | **제공자** | **제공자** | **제공자** |
| 서버         |   사용자   | **제공자** | **제공자** | **제공자** |
| 스토리지     |   사용자   | **제공자** | **제공자** | **제공자** |
| 네트워크     |   사용자   | **제공자** | **제공자** | **제공자** |

### AWS 서비스 예시

| 모델     | AWS 서비스 예시                | 사용 사례                     |
| -------- | ------------------------------ | ----------------------------- |
| **IaaS** | EC2, VPC, EBS                  | 서버 직접 구성, 커스텀 환경   |
| **PaaS** | Elastic Beanstalk, RDS, Lambda | 코드만 배포, 인프라 자동 관리 |
| **SaaS** | WorkMail, Chime, QuickSight    | 즉시 사용 가능한 완성 서비스  |

## 4. AWS 글로벌 인프라

> [!CONCEPT]
> AWS는 전 세계에 분산된 **리전(Region)**, **가용 영역(AZ)**, **엣지 로케이션(Edge Location)**으로 구성된 글로벌 인프라를 운영합니다. 이 구조가 고가용성과 낮은 지연 시간의 핵심입니다.

### 인프라 계층 구조

```
AWS 글로벌 인프라
│
├── 리전 (Region) - 예: ap-northeast-2 (서울)
│   ├── 가용 영역 (AZ) - 예: ap-northeast-2a
│   │   └── 데이터센터 1개 이상
│   ├── 가용 영역 (AZ) - 예: ap-northeast-2b
│   │   └── 데이터센터 1개 이상
│   ├── 가용 영역 (AZ) - 예: ap-northeast-2c
│   │   └── 데이터센터 1개 이상
│   └── 가용 영역 (AZ) - 예: ap-northeast-2d
│       └── 데이터센터 1개 이상
│
├── 엣지 로케이션 (Edge Location) - 400개 이상
│   └── CloudFront CDN, Route 53 DNS
│
└── 로컬 존 (Local Zone) - 특정 도시 근접 배치
```

### 리전 선택 기준

| 기준              | 설명                                       |
| ----------------- | ------------------------------------------ |
| **법적 규정**     | 데이터 주권법에 따라 특정 국가에 저장 필요 |
| **지연 시간**     | 사용자와 가까운 리전 선택                  |
| **서비스 가용성** | 모든 서비스가 모든 리전에 있지 않음        |
| **비용**          | 리전마다 가격이 다름 (서울 > 버지니아)     |

### 가용 영역(AZ)의 특징

- 각 AZ는 **물리적으로 분리**된 데이터센터
- AZ 간 **고속 전용 네트워크**로 연결 (수 ms 이내)
- 자연재해, 정전 등에 대한 **독립적 인프라**
- Multi-AZ 배포로 **고가용성** 확보

## 5. 책임 공유 모델 (Shared Responsibility Model)

> [!CONCEPT]
> AWS의 **책임 공유 모델**은 보안과 규정 준수에 대한 책임을 AWS와 고객이 나누는 프레임워크입니다. AWS는 "클라우드의 보안(Security OF the Cloud)"을, 고객은 "클라우드에서의 보안(Security IN the Cloud)"을 담당합니다.

### 책임 분담 다이어그램

```
┌─────────────────────────────────────────────────────────┐
│                    고객 책임                              │
│              (Security IN the Cloud)                     │
├─────────────────────────────────────────────────────────┤
│  • 고객 데이터                                           │
│  • 플랫폼, 애플리케이션, IAM 관리                         │
│  • 운영체제, 네트워크, 방화벽 설정                         │
│  • 클라이언트 측 데이터 암호화                             │
│  • 서버 측 암호화 (파일 시스템/데이터)                     │
│  • 네트워크 트래픽 보호 (암호화, 무결성, 인증)             │
├─────────────────────────────────────────────────────────┤
│                    AWS 책임                               │
│              (Security OF the Cloud)                     │
├─────────────────────────────────────────────────────────┤
│  • 하드웨어 / AWS 글로벌 인프라                           │
│  • 리전, 가용 영역, 엣지 로케이션                         │
│  • 컴퓨팅, 스토리지, 데이터베이스, 네트워킹               │
│  • 소프트웨어 (가상화 계층, 호스트 OS, 패치)              │
│  • 물리적 보안 (시설, 전력, 냉각)                         │
└─────────────────────────────────────────────────────────┘
```

### 서비스 유형별 책임 차이

| 항목          | IaaS (EC2) | PaaS (RDS) | SaaS (S3) |
| ------------- | :--------: | :--------: | :-------: |
| OS 패치       |    고객    |    AWS     |    AWS    |
| 방화벽 설정   |    고객    |    고객    |    AWS    |
| 데이터 암호화 |    고객    |    고객    |   고객    |
| IAM 관리      |    고객    |    고객    |   고객    |
| 물리 보안     |    AWS     |    AWS     |    AWS    |

## 6. AWS 계정 구조: Root와 IAM

> [!CONCEPT]
> AWS 계정에는 **Root 사용자**(계정 소유자)와 **IAM 사용자**(권한이 부여된 개별 사용자)가 있습니다. Root 사용자는 모든 권한을 가지므로 일상 작업에는 IAM 사용자를 사용해야 합니다.

### 계정 구조 다이어그램

```
AWS 계정 (Account ID: 123456789012)
│
├── Root 사용자 (이메일 + 비밀번호)
│   ├── 모든 권한 보유 (제한 불가)
│   ├── MFA 필수 설정
│   └── 일상 작업에 사용 금지!
│
├── IAM 사용자 (개발자 A)
│   ├── 정책(Policy)으로 권한 제어
│   ├── 프로그래밍 방식 접근 (Access Key)
│   └── 콘솔 접근 (비밀번호)
│
├── IAM 그룹 (Developers)
│   ├── 그룹에 정책 연결
│   └── 그룹 멤버가 정책 상속
│
└── IAM 역할 (Role)
    ├── EC2, Lambda 등 서비스에 부여
    └── 임시 자격 증명 사용
```

### Root vs IAM 사용자 비교

| 항목              | Root 사용자          | IAM 사용자             |
| ----------------- | -------------------- | ---------------------- |
| **생성 시점**     | 계정 생성 시 자동    | 관리자가 수동 생성     |
| **권한 제한**     | 불가능 (모든 권한)   | 정책으로 세밀하게 제어 |
| **MFA**           | 반드시 설정          | 권장                   |
| **Access Key**    | 생성 금지 권장       | 필요 시 생성           |
| **사용 시나리오** | 계정 설정 변경, 결제 | 일상 개발/운영 작업    |

### IAM 정책(Policy) 구조

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Action": "s3:GetObject",
      "Resource": "arn:aws:s3:::my-bucket/*"
    }
  ]
}
```

- **Effect**: Allow 또는 Deny
- **Action**: 허용/거부할 AWS API 작업
- **Resource**: 대상 리소스의 ARN

## 핵심 정리

| 개념                | 한 줄 요약                             |
| ------------------- | -------------------------------------- |
