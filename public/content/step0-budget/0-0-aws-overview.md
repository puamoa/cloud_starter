# AWS 클라우드 기초 개념

---

## 1. 클라우드 컴퓨팅이란?

> [!CONCEPT] 클라우드 컴퓨팅의 정의
> **클라우드 컴퓨팅**은 인터넷을 통해 IT 리소스(서버, 스토리지, 데이터베이스, 네트워크 등)를 **온디맨드**(On-Demand)로 제공받는 서비스입니다.  
> 사용한 만큼만 비용을 지불하며, 물리적 하드웨어를 직접 구매하거나 관리할 필요가 없습니다.

### 클라우드 컴퓨팅의 5가지 핵심 특성 (NIST: National Institute of Standards and Technology 정의)

| 특성                       | 설명                                              |
| -------------------------- | ------------------------------------------------- |
| **온디맨드 셀프서비스**    | 사람의 개입 없이 필요할 때 즉시 리소스 프로비저닝 |
| **광범위한 네트워크 접근** | 인터넷을 통해 어디서든 접근 가능                  |
| **리소스 풀링**            | 여러 사용자가 물리 리소스를 공유 (멀티테넌시)     |
| **빠른 탄력성**            | 수요에 따라 리소스를 즉시 확장/축소               |
| **측정 가능한 서비스**     | 사용량을 정확히 측정하고 과금                     |

### 클라우드 도입의 이점

```
┌────────────────────────────────────────────────┐
│            클라우드 도입 6가지 이점            │
├────────────────────────────────────────────────┤
│  1. 자본 비용 → 가변 비용 전환 (CapEx → OpEx)  │
│  2. 규모의 경제 혜택                           │
│  3. 용량 추정 불필요                           │
│  4. 속도와 민첩성 향상                         │
│  5. 데이터센터 운영 비용 절감                  │
│  6. 글로벌 배포 (몇 분 만에)                   │
└────────────────────────────────────────────────┘
```

## 2. On-Premise vs Cloud 비교

> [!CONCEPT] 온프레미스와 클라우드의 차이
> **On-Premise**(온프레미스)는 기업이 자체 데이터센터에 물리 서버를 직접 구매·설치·운영하는 전통적 방식입니다.  
> **Cloud**는 클라우드 제공자의 인프라를 빌려 사용하는 방식입니다.

| 비교 항목       | On-Premise                             | Cloud (AWS)                               |
| --------------- | -------------------------------------- | ----------------------------------------- |
| **리소스 확보** | 사전 구매/설치 필요                    | 온디맨드 (필요할 때 즉시 생성/삭제)       |
| **초기 비용**   | 높음 (서버 구매, 설치)                 | 없음 (사용한 만큼 지불)                   |
| **확장 시간**   | 수 주 ~ 수 개월                        | 수 분                                     |
| **유지보수**    | 직접 관리 (인력 필요)                  | 클라우드 제공자가 관리                    |
| **가용성**      | 직접 이중화 구성                       | 다중 AZ(Availability Zone)로 쉽게 구현    |
| **보안**        | 물리+논리 모두 직접                    | 물리는 AWS, 논리는 사용자                 |
| **비용 모델**   | CapEx (Capital Expenditure, 자본 지출) | OpEx (Operational Expenditure, 운영 지출) |
| **글로벌 배포** | 각 지역에 DC(Data Center) 구축 필요    | 리전 선택만으로 가능                      |
| **기술 부채**   | 하드웨어 노후화                        | 항상 최신 인프라                          |

### 하이브리드 클라우드 (Hybrid Cloud)

> [!NOTE]
> **VPN** = Virtual Private Network (가상 사설 네트워크),  
> **AWS Direct Connect** = AWS 전용 네트워크 연결 서비스

```
┌──────────────┐      VPN/Direct Connect     ┌──────────────┐
│  On-Premise  │ ◄─────────────────────────► │   AWS Cloud  │
│  Data Center │                             │   (VPC)      │
│              │    일관된 네트워크 연결     │              │
└──────────────┘                             └──────────────┘
       │                                            │
민감 데이터 보관                           탄력적 워크로드 처리
레거시 시스템                              새로운 서비스 배포
```

## 3. 서비스 모델: IaaS / PaaS / SaaS

> [!CONCEPT] IaaS, PaaS, SaaS 서비스 모델
> 클라우드 서비스는 사용자가 관리하는 범위에 따라 **IaaS**(Infrastructure as a Service), **PaaS**(Platform as a Service), **SaaS**(Software as a Service)로 구분됩니다. 아래로 갈수록 사용자의 관리 부담이 줄어듭니다.

### 관리 책임 비교표

| 계층                  | On-Premise |    IaaS    |    PaaS    |    SaaS    |
| --------------------- | :--------: | :--------: | :--------: | :--------: |
| 애플리케이션          |   사용자   |   사용자   |   사용자   | **제공자** |
| 데이터                |   사용자   |   사용자   |   사용자   | **제공자** |
| 런타임                |   사용자   |   사용자   | **제공자** | **제공자** |
| 미들웨어              |   사용자   |   사용자   | **제공자** | **제공자** |
| OS (Operating System) |   사용자   |   사용자   | **제공자** | **제공자** |
| 가상화                |   사용자   | **제공자** | **제공자** | **제공자** |
| 서버                  |   사용자   | **제공자** | **제공자** | **제공자** |
| 스토리지              |   사용자   | **제공자** | **제공자** | **제공자** |
| 네트워크              |   사용자   | **제공자** | **제공자** | **제공자** |

### AWS 서비스 예시

| 모델     | AWS 서비스 예시                                                                                          | 사용 사례                     |
| -------- | -------------------------------------------------------------------------------------------------------- | ----------------------------- |
| **IaaS** | Amazon EC2 (Elastic Compute Cloud), Amazon VPC (Virtual Private Cloud), Amazon EBS (Elastic Block Store) | 서버 직접 구성, 커스텀 환경   |
| **PaaS** | AWS Elastic Beanstalk, Amazon RDS (Relational Database Service), AWS Lambda                              | 코드만 배포, 인프라 자동 관리 |
| **SaaS** | Amazon WorkMail, Amazon Chime, Amazon QuickSight                                                         | 즉시 사용 가능한 완성 서비스  |

## 4. AWS 글로벌 인프라

> [!CONCEPT] AWS 글로벌 인프라 구조
> AWS는 전 세계에 분산된 **리전**(Region), **가용 영역**(Availability Zone, AZ), **엣지 로케이션**(Edge Location)으로 구성된 글로벌 인프라를 운영합니다.  
> 이 구조가 고가용성과 낮은 지연 시간의 핵심입니다.

### 핵심 용어 정리

| 용어                                  | 설명                                                                                                              |
| ------------------------------------- | ----------------------------------------------------------------------------------------------------------------- |
| **리전 (Region)**                     | 지리적으로 독립된 AWS 인프라 영역. 각 리전은 고유 코드를 가짐 (예: `ap-northeast-2` = 서울)                       |
| **가용 영역 (Availability Zone, AZ)** | 리전 내 물리적으로 분리된 데이터센터 그룹. 각 리전에 최소 3개 이상 존재하며, 각 AZ는 1개 이상의 데이터센터로 구성 |
| **데이터센터 (Data Center)**          | 서버, 스토리지, 네트워크 장비가 설치된 물리적 시설. 독립된 전력, 냉각, 네트워크를 갖추고 있으며 AWS가 직접 운영   |
| **엣지 로케이션 (Edge Location)**     | 사용자와 가까운 곳에 위치한 캐시 서버. Amazon CloudFront CDN과 Amazon Route 53 DNS가 사용                         |
| **로컬 존 (Local Zone)**              | 특정 대도시에 근접 배치된 소규모 인프라. 초저지연이 필요한 워크로드용                                             |
| **고가용성 (High Availability, HA)**  | 시스템이 장애 없이 지속적으로 운영되는 능력. Multi-AZ 배포로 달성                                                 |
| **지연 시간 (Latency)**               | 요청을 보내고 응답을 받기까지 걸리는 시간. 사용자와 리전이 가까울수록 낮음                                        |
| **CDN (Content Delivery Network)**    | 전 세계 엣지 로케이션에 콘텐츠를 캐싱하여 빠르게 전달하는 네트워크                                                |
| **DNS (Domain Name System)**          | 도메인 이름(예: example.com)을 IP 주소로 변환하는 시스템                                                          |

### 인프라 계층 구조

```
AWS 글로벌 인프라
│
├── 리전 (Region) - 예: ap-northeast-2 (서울)
│   ├── 가용 영역 (Availability Zone) - 예: ap-northeast-2a
│   │   └── 데이터센터 1개 이상
│   ├── 가용 영역 (Availability Zone) - 예: ap-northeast-2b
│   │   └── 데이터센터 1개 이상
│   ├── 가용 영역 (Availability Zone) - 예: ap-northeast-2c
│   │   └── 데이터센터 1개 이상
│   └── 가용 영역 (Availability Zone) - 예: ap-northeast-2d
│       └── 데이터센터 1개 이상
│
├── 엣지 로케이션 (Edge Location) - 400개 이상
│   └── Amazon CloudFront (CDN), Amazon Route 53 (DNS)
│
└── 로컬 존 (Local Zone) - 특정 도시 근접 배치
```

### 리전 선택 기준

| 기준              | 설명                                       |
| ----------------- | ------------------------------------------ |
| **법적 규정**     | 데이터 주권법에 따라 특정 국가에 저장 필요 |
| **지연 시간**     | 사용자와 가까운 리전 선택                  |
| **서비스 가용성** | 모든 서비스가 모든 리전에 있지 않음        |
| **비용**          | 리전마다 가격이 다름 (서울 > 버지니아)     |

### 가용 영역(AZ)의 특징

- 각 AZ는 **물리적으로 분리**된 데이터센터
- AZ 간 **고속 전용 네트워크**로 연결 (수 ms 이내)
- 자연재해, 정전 등에 대한 **독립적 인프라**
- Multi-AZ 배포로 **고가용성** 확보

## 5. 책임 공유 모델 (Shared Responsibility Model)

> [!CONCEPT] 책임 공유 모델
> AWS의 **책임 공유 모델**은 보안과 규정 준수에 대한 책임을 AWS와 고객이 나누는 프레임워크입니다.  
> AWS는 "클라우드의 보안(Security OF the Cloud)"을, 고객은 "클라우드에서의 보안(Security IN the Cloud)"을 담당합니다.

### 책임 분담 다이어그램

```
┌──────────────────────────────────────────────────┐
│                    고객 책임                     │
│             (Security IN the Cloud)              │
├──────────────────────────────────────────────────┤
│  • 고객 데이터                                   │
│  • 플랫폼, 애플리케이션, IAM 관리                │
│  • 운영체제, 네트워크, 방화벽 설정               │
│  • 클라이언트 측 데이터 암호화                   │
│  • 서버 측 암호화 (파일 시스템/데이터)           │
│  • 네트워크 트래픽 보호 (암호화, 무결성, 인증)   │
├──────────────────────────────────────────────────┤
│                    AWS 책임                      │
│             (Security OF the Cloud)              │
├──────────────────────────────────────────────────┤
│  • 하드웨어 / AWS 글로벌 인프라                  │
│  • 리전, 가용 영역, 엣지 로케이션                │
│  • 컴퓨팅, 스토리지, 데이터베이스, 네트워킹      │
│  • 소프트웨어 (가상화 계층, 호스트 OS, 패치)     │
│  • 물리적 보안 (시설, 전력, 냉각)                │
└──────────────────────────────────────────────────┘
```

### 서비스 유형별 책임 차이

| 항목          | IaaS (Amazon EC2) | PaaS (Amazon RDS) | SaaS (Amazon S3) |
| ------------- | :---------------: | :---------------: | :--------------: |
| OS 패치       |       고객        |        AWS        |       AWS        |
| 방화벽 설정   |       고객        |       고객        |       AWS        |
| 데이터 암호화 |       고객        |       고객        |       고객       |
| IAM 관리      |       고객        |       고객        |       고객       |
| 물리 보안     |        AWS        |        AWS        |       AWS        |

## 6. AWS 계정 구조: Root와 IAM (Identity and Access Management)

> [!CONCEPT] Root 사용자와 IAM 사용자
> AWS 계정에는 **Root 사용자**(계정 소유자)와 **IAM(Identity and Access Management) 사용자**(권한이 부여된 개별 사용자)가 있습니다. Root 사용자는 모든 권한을 가지므로 일상 작업에는 IAM 사용자를 사용해야 합니다.

> [!NOTE]
> **AWS 계정 ID**는 12자리 숫자(예: 123456789012)로 구성됩니다.  
> 이 ID는 리소스의 ARN에 포함되며, 외부에 노출되면 피싱이나 소셜 엔지니어링 공격의 대상이 될 수 있으므로 공개적으로 공유하지 않는 것이 좋습니다.

> [!TIP]
> **왜 IAM 사용자를 사용해야 하나요?**
>
> - Root 사용자는 권한을 제한할 수 없어서, 실수로 리소스를 삭제하거나 과금이 발생해도 막을 방법이 없음
> - Root 자격 증명이 탈취되면 계정 전체가 장악됨 (리소스 삭제, 암호화폐 채굴, 수천만 원 과금 등 실제 사례 다수)
> - IAM 사용자는 필요한 권한만 부여(최소 권한 원칙)하여 해킹 시에도 피해 범위를 제한
> - IAM 사용자별로 활동 로그가 남아 누가 무엇을 했는지 추적 가능 (AWS CloudTrail)
> - 팀원마다 별도 IAM 사용자를 만들면 자격 증명 공유 없이 독립적으로 관리 가능

### 계정 구조 다이어그램

```
AWS 계정 (Account ID: 123456789012)
│
├── Root 사용자 (이메일 + 비밀번호)
│   ├── 모든 권한 보유 (제한 불가)
│   ├── MFA (Multi-Factor Authentication) 필수 설정
│   └── 일상 작업에 사용 금지!
│
├── IAM 사용자 (개발자 A)
│   ├── 정책(Policy)으로 권한 제어
│   ├── 프로그래밍 방식 접근 (Access Key)
│   └── 콘솔 접근 (비밀번호)
│
├── IAM 그룹 (Developers)
│   ├── 그룹에 정책 연결
│   └── 그룹 멤버가 정책 상속
│
└── IAM 역할 (Role)
    ├── Amazon EC2, AWS Lambda 등 서비스에 부여
    └── 임시 자격 증명 사용
```

### Root vs IAM 사용자 비교

| 항목              | Root 사용자          | IAM 사용자             |
| ----------------- | -------------------- | ---------------------- |
| **생성 시점**     | 계정 생성 시 자동    | 관리자가 수동 생성     |
| **권한 제한**     | 불가능 (모든 권한)   | 정책으로 세밀하게 제어 |
| **MFA**           | 반드시 설정          | 권장                   |
| **Access Key**    | 생성 금지 권장       | 필요 시 생성           |
| **사용 시나리오** | 계정 설정 변경, 결제 | 일상 개발/운영 작업    |

### IAM 그룹을 사용하는 이유

> [!CONCEPT] IAM 권한 부여 모범 사례: 그룹 기반 관리
> AWS에서는 사용자에게 직접 정책을 붙이지 않고, **그룹에 정책을 연결한 뒤 사용자를 그룹에 추가**하는 것을 권장합니다.
>
> **직접 부여 방식의 문제점:**
>
> - 사용자가 늘어날수록 정책 관리가 복잡해짐
> - 같은 역할인데 사용자마다 권한이 다를 수 있음
> - 퇴사/이동 시 어떤 권한을 회수해야 하는지 파악 어려움
>
> **그룹 방식의 장점:**
>
> - 팀원 추가 시 그룹에 넣기만 하면 됨
> - 같은 역할의 사용자들이 동일한 권한을 가짐
> - 정책 변경 시 그룹 내 모든 사용자에게 즉시 반영
> - 보안 감사 시 그룹 단위로 권한 파악 가능

### IAM 정책(Policy) 구조

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Action": "s3:GetObject",
      "Resource": "arn:aws:s3:::my-bucket/*"
    }
  ]
}
```

- **Effect**: Allow 또는 Deny
- **Action**: 허용/거부할 AWS API 작업
- **Resource**: 대상 리소스의 ARN (Amazon Resource Name)
- **MFA**: Multi-Factor Authentication (다중 인증)

### 계정 별칭 (Account Alias)

IAM 사용자가 콘솔에 로그인할 때 12자리 계정 ID 대신 기억하기 쉬운 별칭을 사용할 수 있습니다.

- 기본 로그인 URL: `https://123456789012.signin.aws.amazon.com/console`
- 별칭 설정 후: `https://my-company.signin.aws.amazon.com/console`

### IAM 사용자의 Billing 접근

기본적으로 IAM 사용자는 Billing(결제) 정보에 접근할 수 없습니다. Root 계정에서 별도로 활성화해야 합니다.

> [!NOTE]
> **활성화 방법**: Root 계정 로그인 → Account → IAM User and Role Access to Billing Information  
> → Edit → Activate IAM Access → Update
>
> 이 설정을 하지 않으면 IAM 사용자가 AWS Budgets나 Cost Explorer에 접근할 수 없습니다.

## 7. Access Key와 보안

> [!CONCEPT] Access Key와 Secret Access Key
> **Access Key**는 AWS API를 프로그래밍 방식으로 호출할 때 사용하는 자격 증명입니다.  
> **Access Key ID**(공개 식별자)와 **Secret Access Key**(비밀 키) 쌍으로 구성되며, 이 두 값을 알면 해당 사용자의 권한으로 AWS 리소스를 조작할 수 있습니다.
>
> - **Access Key ID**: `AKIAIOSFODNN7EXAMPLE` 형태 (20자)
> - **Secret Access Key**: `wJalrXUtnFEMI/K7MDENG/bPxRfiCYEXAMPLEKEY` 형태 (40자)
> - Secret Access Key는 생성 시 한 번만 표시되며, 분실 시 재발급 필요

> [!WARNING]
> **Access Key를 절대 외부에 노출하지 마세요!**
>
> - GitHub, GitLab 등 공개 저장소에 Access Key를 커밋하면 수 분 내에 봇이 탐지하여 악용
> - 실제 사례: 학생이 GitHub에 키를 올린 후 하룻밤 사이에 수백만 원 과금 (암호화폐 채굴)
> - AWS는 GitHub에 노출된 키를 자동 감지하여 경고 메일을 보내지만, 이미 악용된 후일 수 있음
> - `.gitignore`에 `.env`, `credentials` 파일을 반드시 등록
> - 키가 노출되었다면 즉시 IAM 콘솔에서 해당 키를 비활성화/삭제

## 8. AWS 프리 티어 (Free Tier)와 비용 관리

> [!CONCEPT] AWS 프리 티어 (2025.07.15 이후 신규 가입자)
> 2025년 7월 15일부터 AWS 프리 티어가 **크레딧 기반**으로 변경되었습니다.  
> 가입 시 **$100 크레딧**이 기본 제공되고, AWS 서비스 탐색 활동을 통해 추가로 최대 $100을 더 받을 수 있습니다.  
> 크레딧은 가입일로부터 **6개월간** 유효하며, 소진되면 과금이 시작됩니다.
>
> **무료 플랜 vs 유료 플랜:**
>
> - **무료 플랜 (Free Plan)**: 제한된 서비스 접근, 크레딧 소진 후 과금 없음 (학습/실험용)
> - **유료 플랜 (Paid Plan)**: 150여 개 전체 서비스 접근, 크레딧 소진 후 사용량에 따라 과금

### "항상 무료(Always Free)" 서비스 (한도 내)

크레딧과 무관하게 계정이 존재하는 한 **아래 한도 내에서** 영구적으로 무료인 서비스들 (한도 초과 시 과금):

| 서비스            | 항상 무료 한도 (초과 시 과금)                    |
| ----------------- | ------------------------------------------------ |
| AWS Lambda        | 월 100만 요청 + 40만 GB-초                       |
| Amazon DynamoDB   | 25GB 저장, 월 2억 5천만 읽기/쓰기                |
| Amazon CloudWatch | 기본 모니터링, 알람 10개 (Logs 저장/수집은 유료) |
| Amazon SNS        | 월 100만 게시                                    |
| Amazon SQS        | 월 100만 요청                                    |

### 원래 무료인 리소스 (프리 티어와 무관)

다음 리소스들은 프리 티어 여부와 관계없이 항상 무료입니다:

- **Amazon VPC** (Virtual Private Cloud)
- **Subnet** (서브넷)
- **Internet Gateway** (IGW)
- **Route Table** (라우팅 테이블)
- **Security Group** (보안 그룹)
- **Network ACL** (NACL)

> [!WARNING]
> **크레딧 소진 주의 서비스:**
>
> - **NAT Gateway**: 시간당 과금으로 크레딧이 빠르게 소진됨 → 실습 후 즉시 삭제 필수
> - **Elastic IP**: Amazon EC2에 연결하지 않은 상태로 방치 시 과금
> - **Amazon EBS 볼륨**: EC2를 Stop해도 연결된 EBS는 계속 과금 (금액은 적음)
> - **Amazon RDS Multi-AZ**: Single-AZ 대비 2배 비용

### Billing and Cost Management

AWS 콘솔의 **Billing and Cost Management**에서 비용 관련 모든 정보를 확인할 수 있습니다.

| 카테고리                    | 메뉴                   | 설명                                   |
| --------------------------- | ---------------------- | -------------------------------------- |
| **Billing and Payments**    | Bills                  | 월별 청구서 상세 내역                  |
|                             | Payments               | 결제 내역 및 결제 수단 관리            |
|                             | Credits                | 보유 크레딧 잔액 및 사용 내역 확인     |
| **Cost and Usage Analysis** | Cost Explorer          | 서비스별/일별/리전별 비용 시각화       |
|                             | Free Tier              | 프리 티어 사용량 및 크레딧 소진율 확인 |
|                             | Cost Anomaly Detection | 비정상적 비용 증가 자동 감지           |
|                             | Budgets                | 예산 설정 및 초과 시 알림              |

> [!NOTE]
> IAM 사용자가 Billing에 접근하려면 Root 계정에서 **IAM user and role access to Billing information**을 활성화해야 합니다.  
> 이 설정은 Session 1에서 진행합니다.

### AWS Budgets로 비용 관리

**AWS Budgets**는 예산을 설정하고 초과 시 알림을 받을 수 있는 서비스입니다. 실습 계정에서도 반드시 설정하세요.

| 설정 항목     | 권장 값                          |
| ------------- | -------------------------------- |
| **예산 금액** | $10 (학습용)                     |
| **알림 기준** | 실제 비용 50%, 80%, 예측 100%    |
| **알림 방법** | 이메일                           |
| **확인 주기** | 주 1회 이상 AWS Billing 대시보드 |

> [!TIP]
> AWS Budgets에는 **Zero spend budget** 템플릿이 있습니다.  
> 이 템플릿을 사용하면 프리티어/크레딧 범위를 벗어나 실제 비용이 $0.01이라도 발생하는 즉시 알림을 받을 수 있습니다.  
> 학습용 계정에서는 이 템플릿을 가장 먼저 설정하는 것을 권장합니다.

> [!TIP]
> **비용 사고 예방 체크리스트:**
>
> - 실습 후 사용하지 않는 리소스는 즉시 삭제 또는 중지
> - AWS Budgets에서 $10 예산 알림 설정 (계정 생성 직후)
> - 매주 Billing 대시보드에서 현재 비용 확인
> - Free Tier 사용량 대시보드에서 크레딧 소진율 모니터링
> - NAT Gateway는 실습 후 반드시 삭제 (크레딧 소진 주범)

> [!NOTE]
> 프리 티어와 비용 관리에 대한 상세 실습은 **Session 2: AWS Budget 설정 및 프리티어 이해**에서 직접 진행합니다.

> [!NOTE]
> **기존 계정 (2025.07.15 이전 가입자)**: 기존 "12개월 무료 + 서비스별 한도" 체계가 그대로 적용됩니다.  
> 새 크레딧 기반 체계는 2025년 7월 15일 이후 신규 가입자에게만 해당됩니다.
