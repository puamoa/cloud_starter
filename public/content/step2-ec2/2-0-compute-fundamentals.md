# 컴퓨팅 기초 이론

---

> **이 그룹에서는** AWS에서 가상 서버(EC2)를 생성하고 관리하기 위한 핵심 인프라 개념을 학습합니다.

## 1. 가상화(Virtualization)란?

> [!CONCEPT] 가상화 기술이란
> **가상화**는 하나의 물리 서버 위에 여러 개의 가상 서버(VM)를 실행하는 기술입니다.  
> 물리 하드웨어를 논리적으로 분할하여 각 VM이 독립된 운영체제와 애플리케이션을 실행할 수 있게 합니다.  
> 클라우드 컴퓨팅의 핵심 기반 기술입니다.

### 핵심 용어 정리

| 용어                           | 설명                                                         | AWS 관련                  |
| ------------------------------ | ------------------------------------------------------------ | ------------------------- |
| **가상화 (Virtualization)**    | 하나의 물리 서버에서 여러 가상 서버를 실행하는 기술          | EC2 인스턴스              |
| **하이퍼바이저 (Hypervisor)**  | 물리 하드웨어 위에서 가상 머신을 생성/관리하는 소프트웨어    | AWS Nitro System          |
| **인스턴스 (Instance)**        | 클라우드에서 실행되는 가상 서버 한 대                        | Amazon EC2 인스턴스       |
| **AMI (Amazon Machine Image)** | EC2 인스턴스를 생성하기 위한 템플릿 (OS + 소프트웨어 포함)   | Amazon Linux 2023, Ubuntu |
| **인스턴스 타입**              | CPU, 메모리, 스토리지, 네트워크 조합을 정의한 사양           | t2.micro, m5.large        |
| **EBS (Elastic Block Store)**  | EC2에 연결하는 네트워크 기반 블록 스토리지 (가상 하드디스크) | gp3, io2 볼륨             |
| **키 페어 (Key Pair)**         | EC2에 SSH 접속할 때 사용하는 공개키/개인키 쌍                | .pem 파일                 |
| **User Data**                  | EC2 인스턴스 최초 부팅 시 자동 실행되는 스크립트             | 패키지 설치, 서비스 시작  |
| **Security Group**             | EC2 인스턴스 레벨의 가상 방화벽 (인바운드/아웃바운드 규칙)   | 포트 22, 80, 443 허용     |
| **Elastic IP**                 | 고정 공인 IPv4 주소. 인스턴스에 연결/해제 가능               | 서버 재시작해도 IP 유지   |

### 가상화 이전 vs 이후

```
가상화 이전:                          가상화 이후:
┌──────────────┐                    ┌────────────────────┐
│   App A      │                    │ VM1  │ VM2  │ VM3  │
│   OS         │                    │ App  │ App  │ App  │
│   물리 서버  │                    │ OS   │ OS   │ OS   │
└──────────────┘                    ├──────┴──────┴──────┤
서버 활용률: 10~15%                 │   하이퍼바이저     │
1서버 = 1애플리케이션               │   물리 서버        │
                                    └────────────────────┘
                                    서버 활용률: 60~80%
                                    1서버 = 다수 애플리케이션
```

### 가상화의 이점

| 이점              | 설명                        |
| ----------------- | --------------------------- |
| **리소스 효율성** | 물리 서버 활용률 극대화     |
| **격리**          | VM 간 완전한 독립성 보장    |
| **유연성**        | 필요에 따라 VM 생성/삭제    |
| **스냅샷**        | 특정 시점 상태 저장 및 복원 |
| **마이그레이션**  | VM을 다른 물리 서버로 이동  |

## 2. 하이퍼바이저 (Hypervisor)

> [!CONCEPT] 하이퍼바이저의 역할
> **하이퍼바이저**는 물리 하드웨어 위에서 여러 가상 머신을 생성하고 관리하는 소프트웨어 계층입니다.  
> 각 VM에 CPU, 메모리, 스토리지, 네트워크를 할당하고 격리를 보장합니다.

### 주요 용어

| 용어                          | 설명                                                               |
| ----------------------------- | ------------------------------------------------------------------ |
| **하이퍼바이저 (Hypervisor)** | 물리 하드웨어 위에서 VM을 생성·관리하는 소프트웨어 계층            |
| **Type 1 (Bare-Metal)**       | 하드웨어 위에 직접 설치되는 하이퍼바이저 (예: Xen, KVM)            |
| **Type 2 (Hosted)**           | 호스트 OS 위에서 실행되는 하이퍼바이저 (예: VirtualBox)            |
| **AWS Nitro System**          | AWS 자체 개발 하이퍼바이저. 전용 칩으로 네트워크·스토리지 오프로드 |

### 하이퍼바이저 유형

```
Type 1 (Bare-Metal):              Type 2 (Hosted):
┌──────────────────────┐          ┌───────────────────┐
│  VM1  │  VM2  │  VM3 │          │  VM1  │  VM2      │
├───────┴───────┴──────┤          ├───────┴───────────┤
│    하이퍼바이저      │          │    하이퍼바이저   │
├──────────────────────┤          ├───────────────────┤
│    물리 하드웨어     │          │    호스트 OS      │
└──────────────────────┘          ├───────────────────┤
                                  │    물리 하드웨어  │
예: VMware ESXi, Xen, KVM         └───────────────────┘
    AWS Nitro System              예: VirtualBox, VMware Workstation
```

### AWS Nitro System

| 특징              | 설명                                      |
| ----------------- | ----------------------------------------- |
| **전용 하드웨어** | 네트워크, 스토리지를 전용 칩으로 오프로드 |
| **보안**          | 하이퍼바이저 최소화로 공격 표면 감소      |
| **성능**          | 거의 베어메탈 수준의 성능 제공            |
| **혁신**          | 새로운 인스턴스 타입 빠르게 출시 가능     |

## 3. EC2 인스턴스 타입 체계

> [!CONCEPT] EC2 인스턴스 타입 명명 규칙
> EC2 인스턴스 타입은 **패밀리(용도) + 세대 + 크기**로 구성됩니다.  
> 예: `t3.micro`에서 t=패밀리, 3=세대, micro=크기입니다.  
> 워크로드 특성에 맞는 타입을 선택하는 것이 비용 최적화의 핵심입니다.

### 주요 용어

| 용어                | 설명                                                          |
| ------------------- | ------------------------------------------------------------- |
| **인스턴스 패밀리** | 용도별로 분류된 인스턴스 그룹 (t=버스트, m=범용, c=컴퓨팅 등) |
| **인스턴스 세대**   | 숫자가 클수록 최신 세대 (성능↑, 가격↓)                        |
| **CPU 크레딧**      | t 패밀리에서 버스트 성능을 위해 축적/소모하는 단위            |
| **Graviton**        | AWS 자체 설계 ARM 기반 프로세서 (비용 대비 성능 우수)         |

### 인스턴스 타입 명명 규칙

```
  m  5  a  .  xlarge
  │  │  │     │
  │  │  │     └── 크기: nano < micro < small < medium < large < xlarge < 2xlarge ...
  │  │  │
  │  │  └── 추가 속성: a=AMD, g=Graviton(ARM), n=네트워크 강화, d=로컬 스토리지
  │  │
  │  └── 세대: 숫자가 클수록 최신 (성능↑, 가격↓)
  │
  └── 패밀리: 용도별 분류
```

### 주요 인스턴스 패밀리

| 패밀리                  | 용도             | 특징                    | 사용 사례                   |
| ----------------------- | ---------------- | ----------------------- | --------------------------- |
| **t** (범용-버스트)     | 가변 워크로드    | CPU 크레딧 기반 버스팅  | 개발/테스트, 소규모 웹      |
| **m** (범용)            | 균형 잡힌 리소스 | CPU:메모리 = 1:4        | 일반 웹서버, 앱서버         |
| **c** (컴퓨팅 최적화)   | CPU 집약         | 높은 CPU 대 메모리 비율 | 배치 처리, 과학 계산        |
| **r** (메모리 최적화)   | 메모리 집약      | 높은 메모리 대 CPU 비율 | 인메모리 DB, 캐시           |
| **i** (스토리지 최적화) | I/O 집약         | 높은 로컬 스토리지 성능 | NoSQL DB, 데이터 웨어하우스 |
| **g/p** (가속 컴퓨팅)   | GPU 워크로드     | GPU 탑재                | ML 학습, 그래픽 렌더링      |

### t 패밀리 주요 타입 비교 (서울 리전 기준)

실습과 프로덕션에서 가장 많이 사용하는 t 패밀리 타입입니다:

| 인스턴스 타입 | vCPU | 메모리 | 기준선 CPU | 시간당 비용 | 월 비용 (24/7) | 적합한 용도                |
| ------------- | ---- | ------ | ---------- | ----------- | -------------- | -------------------------- |
| `t3.nano`     | 2    | 0.5 GB | 5%         | ~$0.0052    | ~$3.74         | 경량 에이전트, 테스트      |
| `t3.micro`    | 2    | 1 GB   | 10%        | ~$0.0104    | ~$7.49         | 실습, 소규모 API 서버      |
| `t3.small`    | 2    | 2 GB   | 20%        | ~$0.0208    | ~$14.98        | Spring Boot, Node.js 서버  |
| `t3.medium`   | 2    | 4 GB   | 20%        | ~$0.0416    | ~$29.95        | Spring Boot + DB 동시 운영 |
| `t3.large`    | 2    | 8 GB   | 30%        | ~$0.0832    | ~$59.90        | 메모리 필요한 Java 앱      |
| `t3.xlarge`   | 4    | 16 GB  | 40%        | ~$0.1664    | ~$119.81       | 중규모 프로덕션            |

※ 위 금액은 작성 시점 기준 참고 값이며, 실제 요금은 리전, 환율, AWS 정책 변경에 따라 상이할 수 있습니다.

> [!TIP]
> **인스턴스 타입 선택 가이드:**
>
> - **학습/테스트**: `t3.micro` (1GB) — 가장 저렴하고 실습에 충분
> - **Java/Spring Boot**: `t3.small` (2GB) — JVM이 최소 512MB 사용하므로 여유 필요
> - **DB 동시 운영**: `t3.medium` (4GB) — MySQL + 앱 서버 동시 실행 시
> - **프로덕션 (상시 트래픽)**: `m6i.large` 이상 — 버스트 제한 없는 범용 타입 권장
>
> 기준선 CPU를 초과하면 CPU 크레딧을 소모합니다. 상시 높은 CPU가 필요하면 t 패밀리 대신 m/c 패밀리를 사용하세요.
>
> 최신 요금은 [AWS EC2 요금 페이지](https://aws.amazon.com/ec2/pricing/on-demand/)에서 확인하세요.

### t 패밀리 CPU 크레딧 시스템

```
CPU 사용률
100% ─────────────────────────────────────
     │         ▲ 버스트
     │        ╱ ╲
     │       ╱   ╲ 크레딧 소모
 20% │──────╱─────╲──── 기준선(Baseline) ──
     │     ╱       ╲
     │    ╱         ╲ 크레딧 축적
  0% └───────────────────────────────── 시간

t3.micro: 기준선 10%, 크레딧 축적 → 필요 시 100% 버스트
```

> [!WARNING]
> **t3 패밀리 무제한(Unlimited) 모드 주의**
>
> t3 인스턴스는 AWS 콘솔에서 생성 시 기본적으로 **무제한 모드(Unlimited)**가 활성화되어 있습니다.
> 이 모드에서는 CPU 크레딧이 고갈되어도 성능이 제한되지 않는 대신, 기준선을 초과한 사용량에 대해 **추가 요금**이 청구됩니다.
>
> 학습 환경에서 장시간 고부하 작업(빌드, DB 마이그레이션 등)을 실행할 때 의도치 않은 비용이 발생할 수 있으므로:
>
> - CloudWatch에서 `CPUCreditBalance` 지표를 모니터링하거나
> - 상시 높은 CPU가 필요하면 m 또는 c 패밀리로 전환하세요

## 4. AMI (Amazon Machine Image)

> [!CONCEPT] AMI란 무엇인가
> **AMI**는 EC2 인스턴스를 시작하는 데 필요한 모든 정보를 담은 템플릿입니다.  
> 운영체제, 애플리케이션, 설정이 포함되어 있어 동일한 환경을 빠르게 복제할 수 있습니다.

### 주요 용어

| 용어                           | 설명                                                       |
| ------------------------------ | ---------------------------------------------------------- |
| **AMI (Amazon Machine Image)** | EC2 인스턴스를 시작하기 위한 템플릿 (OS + 소프트웨어 포함) |
| **루트 볼륨**                  | OS가 설치된 기본 EBS 볼륨                                  |
| **블록 디바이스 매핑**         | AMI에서 인스턴스 시작 시 연결할 EBS 볼륨 설정              |
| **아키텍처 (x86_64 / arm64)**  | CPU 명령어 세트. Intel/AMD는 x86_64, Graviton은 arm64      |

### AMI 구성 요소

```
AMI
├── 루트 볼륨 템플릿 (OS + 소프트웨어)
├── 시작 권한 (누가 사용 가능한지)
├── 블록 디바이스 매핑 (EBS 볼륨 설정)
└── 아키텍처 (x86_64 또는 arm64)
```

### AMI 유형 비교

| 유형            | 제공자   | 비용                    | 예시                      |
| --------------- | -------- | ----------------------- | ------------------------- |
| **AWS 제공**    | Amazon   | 무료 (OS 라이선스 별도) | Amazon Linux 2023, Ubuntu |
| **Marketplace** | 서드파티 | 유/무료                 | WordPress AMI, NVIDIA     |
| **커뮤니티**    | 커뮤니티 | 무료                    | 다양한 커스텀 이미지      |
| **커스텀**      | 사용자   | 스토리지 비용만         | 내 서버 스냅샷            |

### AMI 활용 워크플로우

```
1. 기본 AMI로 EC2 시작
2. 소프트웨어 설치 및 설정
3. 커스텀 AMI 생성 (이미지 캡처)
4. 커스텀 AMI로 동일 인스턴스 대량 생성
   └── Auto Scaling, 다른 리전 복제 등
```

## 5. EBS (Elastic Block Store)

> [!CONCEPT] EBS 블록 스토리지
> **EBS**는 EC2 인스턴스에 연결하는 네트워크 기반 블록 스토리지입니다.  
> 인스턴스와 독립적으로 존재하며, 인스턴스를 중지해도 데이터가 유지됩니다.  
> 물리 서버의 하드디스크/SSD에 해당합니다.

### 주요 용어

| 용어                          | 설명                                                  |
| ----------------------------- | ----------------------------------------------------- |
| **EBS (Elastic Block Store)** | EC2에 연결하는 네트워크 기반 블록 스토리지            |
| **IOPS**                      | 초당 입출력 작업 수. 스토리지 성능의 핵심 지표        |
| **gp3**                       | 범용 SSD 볼륨 타입. 대부분의 워크로드에 권장          |
| **스냅샷 (Snapshot)**         | EBS 볼륨의 특정 시점 백업. 다른 AZ/리전으로 복제 가능 |

### EBS 볼륨 타입 비교

| 타입    | 이름              | IOPS         | 처리량          | 사용 사례                 |
| ------- | ----------------- | ------------ | --------------- | ------------------------- |
| **gp3** | 범용 SSD          | 3,000~16,000 | 125~1,000 MB/s  | 일반 워크로드 (기본 추천) |
| **gp2** | 범용 SSD          | 100~16,000   | 최대 250 MB/s   | 레거시 (gp3 권장)         |
| **io2** | 프로비저닝 IOPS   | 최대 64,000  | 최대 1,000 MB/s | 고성능 DB                 |
| **st1** | 처리량 최적화 HDD | 500          | 최대 500 MB/s   | 빅데이터, 로그            |
| **sc1** | 콜드 HDD          | 250          | 최대 250 MB/s   | 아카이브                  |

### EBS 핵심 특성

```
┌─────────────────────────────────────────────────┐
│  EC2 인스턴스 (AZ: ap-northeast-2a)             │
│  ┌───────────┐                                  │
│  │ 인스턴스  │◄──── 네트워크 연결 ──┐           │
│  └───────────┘                      │           │
│                              ┌──────┴──────┐    │
│                              │  EBS 볼륨   │    │
│                              │  (같은 AZ)  │    │
│                              └─────────────┘    │
│                                                 │
│  ⚠️ EBS는 같은 AZ 내에서만 연결 가능            │
│  ✅ 스냅샷으로 다른 AZ/리전으로 복제 가능       │
└─────────────────────────────────────────────────┘
```

### EBS 스냅샷과 IaaS 운영

> [!CONCEPT] EBS 스냅샷이란
> **EBS 스냅샷**은 EBS 볼륨의 특정 시점 백업입니다.  
> S3에 증분(incremental) 방식으로 저장되어 변경된 블록만 추가 저장하므로 비용 효율적입니다.  
> 스냅샷에서 새 볼륨을 생성하면 다른 AZ나 리전에서도 데이터를 복원할 수 있습니다.

| 용어 | 설명 |
| ---- | ---- |
| **EBS 스냅샷** | EBS 볼륨의 특정 시점 백업. S3에 증분 저장 |
| **증분 백업 (Incremental)** | 이전 스냅샷 대비 변경된 블록만 저장. 첫 스냅샷만 전체 복사 |
| **커스텀 AMI** | 스냅샷을 포함하여 인스턴스 전체 상태를 이미지로 저장 |
| **교차 리전 복제** | 스냅샷을 다른 리전으로 복사하여 재해 복구(DR)에 활용 |

### 스냅샷 동작 방식

```
  [EBS 볼륨]     [스냅샷 1 (전체)]   [스냅샷 2 (증분)]   [스냅샷 3 (증분)]
  ┌───────┐      ┌───────┐          ┌───────┐          ┌───────┐
  │ A B C │  →   │ A B C │          │   D   │          │ E     │
  │ D E F │      │ D E F │          │   F'  │          │       │
  └───────┘      └───────┘          └───────┘          └───────┘
                  (전체 데이터)       (변경분만)          (변경분만)
                  비용: 높음         비용: 낮음          비용: 낮음
```

### IaaS(EC2)에서 스냅샷이 필요한 상황

| 상황 | 설명 | 실무 예시 |
| ---- | ---- | --------- |
| **배포 전 백업** | 신규 배포/업데이트 전 현재 상태를 저장 | 앱 업데이트 실패 시 롤백 가능 |
| **서버 복제** | 동일 환경의 서버를 빠르게 생성 | 스냅샷 → AMI → 새 인스턴스 |
| **리전 이전** | 서울 → 도쿄 등 다른 리전으로 서버 이전 | 스냅샷 복사 → 대상 리전에서 복원 |
| **재해 복구 (DR)** | 장애 시 다른 AZ/리전에서 복구 | 정기 스냅샷을 다른 리전에 복제 |
| **개발/테스트 환경** | 프로덕션 데이터로 테스트 환경 구축 | 프로덕션 스냅샷 → 개발 서버에 연결 |

### IaaS 운영의 장단점과 스냅샷의 역할

> [!CONCEPT] IaaS(EC2) 직접 운영의 현실
> EC2를 직접 운영하면 유연성은 높지만, **인프라 관리의 모든 책임**이 사용자에게 있습니다.  
> 스냅샷은 이 부담을 줄여주는 핵심 도구입니다.

**IaaS의 장점:**
- OS, 런타임, 미들웨어를 완전히 제어
- 특수 요구사항(GPU, 커스텀 커널 등) 구현 가능
- 비용 예측이 쉬움 (시간당 고정 요금)

**IaaS의 단점 (= 관리 부담):**
- OS 패치, 보안 업데이트를 직접 수행해야 함
- 서버 장애 시 수동 복구 (자동 복구 미제공)
- 스케일링을 직접 구성해야 함 (Auto Scaling 별도 설정)
- **배포 실패 시 롤백이 어려움** → 스냅샷으로 해결

**스냅샷이 해결하는 IaaS 운영 문제:**

```
[문제] 앱 업데이트 후 서버가 먹통
  └── 스냅샷 없이: OS 재설치 → 패키지 설치 → 설정 → 배포 (수 시간)
  └── 스냅샷 있음: 이전 스냅샷에서 볼륨 복원 (수 분)

[문제] AZ 장애로 서버 접근 불가
  └── 스냅샷 없이: 새 AZ에서 처음부터 서버 구축 (수 시간)
  └── 스냅샷 있음: 다른 AZ에서 스냅샷으로 즉시 복원 (수 분)

[문제] 개발자가 실수로 DB 데이터 삭제
  └── 스냅샷 없이: 복구 불가능 (로그 분석 → 수동 재입력)
  └── 스냅샷 있음: 스냅샷에서 새 볼륨 생성 → 데이터 추출
```

### 실무에서의 스냅샷 운영 패턴

| 패턴 | 주기 | 보관 기간 | 자동화 |
| ---- | ---- | --------- | ------ |
| **배포 전 임시 스냅샷** | 배포마다 | 배포 성공 확인 후 삭제 | CI/CD 파이프라인에 포함 |
| **일일 정기 스냅샷** | 매일 | 7일 보관 후 삭제 | Amazon Data Lifecycle Manager |
| **주간 장기 보관** | 매주 | 30일 | Amazon Data Lifecycle Manager |
| **교차 리전 복제** | 일일 스냅샷과 동시 | 3일 | DLM Cross-Region Copy |

> [!TIP]
> **Amazon Data Lifecycle Manager (DLM)** 을 사용하면 스냅샷 생성·보관·삭제를 자동화할 수 있습니다.  
> 태그 기반으로 대상 볼륨을 지정하고, 스케줄과 보관 규칙을 설정하면 수동 관리가 필요 없습니다.
>
> 이 실습에서는 EC2 스냅샷을 직접 생성하지 않지만, 실무에서 EC2를 운영할 때 배포 전 스냅샷은 필수적인 안전장치입니다.  
> PaaS(RDS, Elastic Beanstalk)로 전환하면 이런 백업이 자동화되는 것이 가장 큰 장점 중 하나입니다.

## 6. 인스턴스 수명 주기

> [!CONCEPT] 인스턴스 수명 주기와 과금
> EC2 인스턴스는 생성부터 종료까지 여러 상태를 거칩니다.  
> 각 상태에서의 과금 여부와 데이터 보존 여부를 이해하는 것이 비용 관리의 핵심입니다.

### 주요 용어

| 용어                | 설명                                                       |
| ------------------- | ---------------------------------------------------------- |
| **running**         | 인스턴스가 실행 중인 상태. EC2 + EBS 모두 과금             |
| **stopped**         | 인스턴스가 중지된 상태. EBS만 과금, 공인 IP 해제           |
| **terminated**      | 인스턴스가 종료된 상태. 복구 불가, 모든 과금 중지          |
| **인스턴스 스토어** | EC2 호스트에 물리적으로 연결된 임시 스토리지. Stop 시 삭제 |
| **Elastic IP**      | 고정 공인 IPv4 주소. 인스턴스 재시작해도 IP 유지           |

### 인스턴스 상태 전이 다이어그램

```
                    ┌───────────┐
         시작 ────► │  pending  │
                    └─────┬─────┘
                          │
                          ▼
                    ┌───────────┐  ◄──── 재시작(reboot)
         ┌───────   │  running  │  ────────┐
         │          └───────────┘          │
         │                │                │
         │ 중지           │ 종료           │ 최대 절전
         ▼                ▼                ▼
    ┌──────────┐  ┌───────────────┐  ┌─────────────┐
    │ stopping │  │ shutting-down │  │ hibernating │
    └────┬─────┘  └───────┬───────┘  └─────────────┘
         │                │
         ▼                ▼
    ┌──────────┐    ┌────────────┐
    │ stopped  │    │ terminated │
    └──────────┘    └────────────┘
    (재시작 가능)    (복구 불가)
```

### 상태별 과금 및 데이터 보존

| 상태           | EC2 과금 | EBS 과금 | 공인 IP | EBS 데이터 | 인스턴스 스토어 |
| -------------- | :------: | :------: | :-----: | :--------: | :-------------: |
| **running**    |    ✅    |    ✅    |  유지   |    유지    |      유지       |
| **stopped**    |    ❌    |    ✅    |  해제   |    유지    |    **삭제**     |
| **terminated** |    ❌    |   ❌\*   |  해제   | **삭제\*** |    **삭제**     |
| **hibernated** |    ❌    |    ✅    |  유지   |    유지    |    **삭제**     |

> \*terminated 시 EBS는 "종료 시 삭제" 설정에 따라 다름 (기본: 루트 볼륨 삭제)

### 실무 팁: 비용 절약

```
┌─────────────────────────────────────────────────────┐
│  💡 비용 절약 전략                                  │
├─────────────────────────────────────────────────────┤
│                                                     │
│  개발/테스트 환경:                                  │
│  • 퇴근 시 Stop → 출근 시 Start                     │
│  • EC2 과금 중지, EBS만 과금                        │
│                                                     │
│  더 이상 불필요:                                    │
│  • Terminate (EBS도 함께 삭제 설정)                 │
│  • 완전히 과금 중지                                 │
│                                                     │
│  ⚠️ 주의: Stop 시 공인 IP 변경됨                    │
│     → Elastic IP 사용으로 해결                      │
│                                                     │
└─────────────────────────────────────────────────────┘
```

## 7. User Data (사용자 데이터)

> [!CONCEPT] User Data 초기화 스크립트
> **User Data**는 EC2 인스턴스가 처음 시작될 때 자동으로 실행되는 스크립트입니다.  
> 소프트웨어 설치, 설정 파일 생성, 서비스 시작 등 초기 구성을 자동화할 수 있습니다.  
> root 권한으로 실행됩니다.

### 주요 용어

| 용어           | 설명                                                       |
| -------------- | ---------------------------------------------------------- |
| **User Data**  | EC2 인스턴스 최초 부팅 시 자동 실행되는 초기화 스크립트    |
| **cloud-init** | EC2 인스턴스 초기 설정을 자동화하는 도구. User Data를 실행 |
| **Base64**     | 바이너리 데이터를 텍스트로 인코딩하는 방식                 |

### User Data 실행 흐름

```
EC2 인스턴스 시작
│
├── 1. 하드웨어 초기화
├── 2. OS 부팅
├── 3. cloud-init 실행
│   └── User Data 스크립트 실행 (root 권한)
│       ├── 패키지 업데이트
│       ├── 소프트웨어 설치
│       ├── 설정 파일 생성
│       └── 서비스 시작
└── 4. 인스턴스 Ready
```

### User Data 특성

| 특성          | 설명                           |
| ------------- | ------------------------------ |
| **실행 시점** | 인스턴스 최초 시작 시 1회      |
| **실행 권한** | root                           |
| **크기 제한** | 16KB                           |
| **로그 위치** | /var/log/cloud-init-output.log |
| **인코딩**    | Base64 (콘솔에서 자동 처리)    |

## 8. CloudFormation 기초

> [!CONCEPT] Infrastructure as Code (IaC)
> **CloudFormation**은 AWS 리소스를 코드(YAML/JSON 템플릿)로 정의하고, 한 번의 실행으로 자동 생성·관리하는 서비스입니다.  
> 수동으로 콘솔에서 하나씩 만드는 대신, 템플릿 파일 하나로 VPC, Subnet, EC2, Security Group 등을 일괄 생성합니다.

**왜 필요한가요?**

Step 1에서 VPC, Subnet, IGW, Route Table을 하나씩 수동으로 만들었습니다.  
도중에 설정을 잘못하면?  
처음부터 다시 해야 합니다.  
팀원에게 동일한 환경을 만들어 달라고 하면?  
같은 과정을 설명해야 합니다.  
CloudFormation은 이 전체 과정을 **파일 하나로 기록**해두고, 버튼 한 번으로 동일한 환경을 언제든 재현할 수 있게 해줍니다.

### 핵심 용어

| 용어         | 설명                                                      |
| ------------ | --------------------------------------------------------- |
| **템플릿**   | 리소스를 정의하는 YAML/JSON 파일                          |
| **스택**     | 템플릿을 실행하여 생성된 리소스 묶음                      |
| **파라미터** | 템플릿 실행 시 외부에서 입력하는 변수 (예: 프로젝트 이름) |
| **Outputs**  | 스택 생성 후 결과 값 (예: VPC ID, Subnet ID)              |
| **롤백**     | 생성 실패 시 이미 만들어진 리소스를 자동 삭제             |

### 수동 생성 vs CloudFormation

```
수동 생성:
VPC 콘솔 → VPC 생성 → Subnet 생성 → IGW 생성 → RT 생성 → SG 생성
(5~10분, 실수 가능, 재현 불가)

CloudFormation:
템플릿 업로드 → 파라미터 입력 → Create Stack
(2분, 동일 환경 재현 가능, 삭제도 한 번에)
```

### 스택 수명 주기

```
Create Stack → CREATE_IN_PROGRESS → CREATE_COMPLETE
                                  → CREATE_FAILED (자동 롤백)

Delete Stack → DELETE_IN_PROGRESS → DELETE_COMPLETE
```

> [!TIP]
> CloudFormation은 생성의 역순으로 리소스를 삭제합니다.  
> 스택을 삭제하면 스택이 생성한 모든 리소스가 자동으로 정리됩니다.

## 9. IAM Role과 Instance Profile

> [!CONCEPT] IAM Role
> **IAM Role**은 AWS 리소스(EC2, Lambda 등)에 권한을 부여하는 방식입니다.  
> 사용자가 Access Key를 직접 관리하는 대신, Role을 통해 **임시 자격 증명**이 자동으로 발급됩니다.  
> 보안상 Access Key 직접 사용보다 훨씬 안전합니다.

**쉬운 비유: 호텔 카드키**

- **Access Key** = 호텔 마스터키를 복사해서 주는 것. 분실하면 모든 방에 들어갈 수 있고, 회수도 어렵습니다.
- **IAM Role** = 특정 방에만 들어갈 수 있는 카드키를 자동 발급하는 것. 시간이 지나면 자동 만료되고, 필요한 방만 열립니다.

### Access Key vs IAM Role

| 항목           | Access Key (비권장)         | IAM Role (권장)               |
| -------------- | --------------------------- | ----------------------------- |
| 자격 증명 방식 | 영구 키 (유출 시 즉시 위험) | 임시 토큰 (자동 갱신, 만료됨) |
| 관리 부담      | 키 파일 보관/교체 필요      | AWS가 자동 관리               |
| EC2 사용 시    | 서버에 키 저장 필요         | Role 연결만 하면 자동 동작    |
| 권한 변경      | 키 재배포 필요              | Role 정책만 수정              |

### EC2에 Role을 연결하는 구조

```
EC2 인스턴스
   └── Instance Profile (컨테이너)
         └── IAM Role
               └── Policy (권한 정의)
                     예: AmazonSSMManagedInstanceCore
```

- **Policy**: "무엇을 할 수 있는가" 정의 (JSON 형식)
- **Role**: Policy를 담는 그릇 + 누가 이 Role을 사용할 수 있는지(Trust Relationship) 정의
- **Instance Profile**: EC2에 Role을 연결하는 중간 매개체 (콘솔에서는 자동 처리)

### 실습에서 사용하는 Role

| Role 이름         | 연결 대상 | 포함 정책                      | 용도                 |
| ----------------- | --------- | ------------------------------ | -------------------- |
| `my-ec2-ssm-role` | EC2       | `AmazonSSMManagedInstanceCore` | Session Manager 접속 |

## 10. 키 페어와 SSH

> [!CONCEPT] 키 페어와 SSH 접속
> **키 페어**는 공개키 암호화를 이용한 인증 방식입니다.  
> AWS에서 EC2 인스턴스에 접속할 때 비밀번호 대신 **프라이빗 키**(.pem)를 사용하여 안전하게 SSH 접속합니다.

### 주요 용어

| 용어                     | 설명                                          |
| ------------------------ | --------------------------------------------- |
| **키 페어 (Key Pair)**   | 공개키와 개인키로 구성된 암호화 키 쌍         |
| **공개키 (Public Key)**  | EC2 인스턴스에 저장되는 키. 인증 요청을 검증  |
| **개인키 (Private Key)** | 사용자가 보관하는 .pem 파일. SSH 접속 시 사용 |
| **SSH (Secure Shell)**   | 암호화된 원격 접속 프로토콜 (포트 22)         |

### 키 페어 동작 원리

```
키 페어 생성 시:
┌──────────────────────────────────────────────┐
│  AWS가 키 쌍 생성                            │
│                                              │
│  공개키 (Public Key)  →  EC2 인스턴스에 저장 │
│  개인키 (Private Key) →  사용자가 다운로드   │
│                          (.pem 파일)         │
└──────────────────────────────────────────────┘

SSH 접속 시:
┌───────────┐         SSH (포트 22)         ┌──────────┐
│  사용자   │ ──── 개인키로 인증 요청 ────► │   EC2    │
│  (로컬)   │                               │ (공개키) │
│  .pem     │ ◄──── 인증 성공, 접속 허용 ── │          │
└───────────┘                               └──────────┘
```

### SSH 접속 명령어 구조

```bash
ssh -i "my-key.pem" ec2-user@<Public-IP>
     │       │           │         │
     │       │           │         └── EC2의 공인 IP
     │       │           └── 기본 사용자 (Amazon Linux)
     │       └── 개인키 파일 경로
     └── 키 파일 지정 옵션
```

### OS별 기본 사용자

| AMI          | 기본 사용자 |
| ------------ | ----------- |
| Amazon Linux | ec2-user    |
| Ubuntu       | ubuntu      |
| CentOS       | centos      |
| Debian       | admin       |
| RHEL         | ec2-user    |

## 11. SSM Session Manager

> [!CONCEPT] AWS Systems Manager Session Manager
> **Session Manager**는 SSH 없이 EC2에 접속할 수 있는 AWS 서비스입니다.  
> 키 페어 불필요, Security Group에 포트를 열 필요 없음, Private Subnet에서도 접속 가능합니다.  
> 모든 접속 기록이 CloudTrail에 자동 기록됩니다.

**왜 SSH 대신 SSM을 쓰나요?**

SSH만 사용하면 이런 문제가 생깁니다:

- `.pem` 키 파일을 분실하면? → 서버에 접속할 수 없습니다
- 팀원이 바뀌면? → 키 파일을 전달해야 하는데 추적·회수가 불가능합니다
- 누가 언제 서버에 접속했는지 알고 싶으면? → SSH 자체로는 기록이 남지 않습니다
- Private Subnet에 있는 서버에 접속하려면? → Bastion Host를 별도로 구성해야 합니다

SSM Session Manager는 이 모든 문제를 해결합니다.

### SSH vs SSM Session Manager

| 비교 항목      | SSH                      | SSM Session Manager      |
| -------------- | ------------------------ | ------------------------ |
| 키 페어        | ✅ 필요 (.pem 파일 관리) | ❌ 불필요                |
| Security Group | 22번 포트 오픈 필수      | 포트 오픈 불필요         |
| Public IP      | ✅ 필요                  | ❌ Private Subnet도 가능 |
| 접속 기록      | 별도 설정 필요           | CloudTrail에 자동 기록   |
| 기본 사용자    | `ec2-user`               | `ssm-user`               |
| 필요 조건      | 키 + SG + Public IP      | IAM Role만               |

### SSM 동작 원리

```
[사용자] → AWS API → Systems Manager 서비스 → SSM Agent (EC2 내부)
                                                     ↑
                                              EC2가 IAM Role로
                                              SSM 서비스에 등록
```

SSM Agent는 Amazon Linux 2023에 기본 설치·자동 실행됩니다.  
EC2에 `AmazonSSMManagedInstanceCore` 정책이 포함된 IAM Role만 연결하면 동작합니다.

### ssm-user vs ec2-user

| 항목        | ec2-user (SSH)   | ssm-user (Session Manager) |
| ----------- | ---------------- | -------------------------- |
| 홈 디렉토리 | `/home/ec2-user` | 없음 또는 `/home/ssm-user` |
| sudo 권한   | ✅ 있음          | ✅ 있음                    |
| 전환 방법   | -                | `sudo su - ec2-user`       |

> [!TIP]
> SSH로 배포한 앱 파일은 `/home/ec2-user`에 있으므로, SSM으로 접속 시 `sudo su - ec2-user`로 전환해야 합니다.

## 12. 파일 경로와 주요 리눅스 명령어

> [!CONCEPT] 절대경로와 상대경로
> 파일이나 폴더의 위치를 지정하는 방법은 두 가지입니다.  
> EC2에서 명령어를 실행할 때, 파일을 전송할 때 모두 경로를 정확히 지정해야 합니다.

### 절대경로 vs 상대경로

| 구분         | 의미                              | 시작          | 예시                                    |
| ------------ | --------------------------------- | ------------- | --------------------------------------- |
| **절대경로** | 시스템 최상위(루트)부터 전체 경로 | `/`           | `/home/ec2-user/my-app/dist/index.html` |
| **상대경로** | 현재 위치 기준으로의 경로         | `.` 또는 `..` | `./my-app/dist/index.html`              |

### 경로 특수 기호

| 기호 | 의미                          | 예시                                   |
| ---- | ----------------------------- | -------------------------------------- |
| `/`  | 루트 디렉토리 (시스템 최상위) | `cd /` → 시스템 루트로 이동            |
| `~`  | 홈 디렉토리 (사용자 폴더)     | `cd ~` → `/home/ec2-user/`로 이동      |
| `.`  | 현재 디렉토리                 | `./gradlew` → 현재 폴더의 gradlew 실행 |
| `..` | 상위 디렉토리 (한 단계 위)    | `cd ..` → 부모 폴더로 이동             |
| `-`  | 이전 디렉토리                 | `cd -` → 직전에 있던 폴더로 이동       |

### 경로 예시로 이해하기

```
현재 위치: /home/ec2-user/my-app/

절대경로:
  /home/ec2-user/my-app/dist/index.html  ← 루트부터 전체 경로
  /usr/share/nginx/html/                 ← 어디서든 이 경로 그대로 사용

상대경로:
  ./dist/index.html         ← 현재 폴더(my-app) 안의 dist/index.html
  ../                       ← 한 단계 위 (/home/ec2-user/)
  ../../                    ← 두 단계 위 (/home/)

홈 디렉토리:
  ~/my-app/                 ← /home/ec2-user/my-app/ 와 동일
  ~/Downloads/my-key.pem    ← /home/ec2-user/Downloads/my-key.pem
```

### OS별 경로 차이

| 항목        | Linux / Mac                                             | Windows                            |
| ----------- | ------------------------------------------------------- | ---------------------------------- |
| 경로 구분자 | `/` (슬래시)                                            | `\` (역슬래시)                     |
| 루트        | `/`                                                     | `C:\`                              |
| 홈 디렉토리 | `/home/사용자명/` (Linux) <br> `/Users/사용자명/` (Mac) | `C:\Users\사용자명\`               |
| 홈 단축키   | `~`                                                     | `%USERPROFILE%`                    |
| 예시        | `~/Downloads/key.pem`                                   | `C:\Users\yhkim\Downloads\key.pem` |

> [!TIP]
> **Windows PowerShell에서는 `/`도 사용 가능합니다.**  
> `C:/Users/yhkim/Downloads/key.pem` 형태로 써도 동작합니다.  
> 하지만 `scp` 등 SSH 관련 명령어에서는 Linux 스타일(`/`)을 사용하는 것이 안전합니다.

### EC2에서 자주 쓰는 리눅스 명령어

#### 디렉토리 이동 및 확인

| 명령어    | 설명                      | 예시                       |
| --------- | ------------------------- | -------------------------- |
| `pwd`     | 현재 위치 확인 (절대경로) | `pwd` → `/home/ec2-user`   |
| `cd 경로` | 디렉토리 이동             | `cd /opt/app`              |
| `cd ~`    | 홈으로 이동               | `cd ~` → `/home/ec2-user/` |
| `cd ..`   | 상위 폴더로 이동          | `cd ..`                    |
| `ls`      | 현재 폴더 내용 목록       | `ls -la` (숨김 파일 포함)  |

#### 파일/폴더 관리

| 명령어            | 설명                            | 예시                                  |
| ----------------- | ------------------------------- | ------------------------------------- |
| `mkdir -p 경로`   | 폴더 생성 (중간 폴더 자동 생성) | `mkdir -p /opt/app/logs`              |
| `cp 원본 대상`    | 파일 복사                       | `cp app.jar /opt/app/`                |
| `cp -r 원본 대상` | 폴더 전체 복사                  | `cp -r dist/* /usr/share/nginx/html/` |
| `mv 원본 대상`    | 파일 이동 또는 이름 변경        | `mv app.jar /opt/app/app.jar`         |
| `rm 파일`         | 파일 삭제                       | `rm old-app.jar`                      |
| `rm -rf 폴더`     | 폴더 전체 삭제 (확인 없이)      | `rm -rf /tmp/deploy/`                 |

> [!WARNING]
> `rm -rf`는 확인 없이 즉시 삭제됩니다. 경로를 잘못 입력하면 복구 불가능합니다.  
> 특히 `sudo rm -rf /`는 **시스템 전체를 삭제**하므로 절대 실행하지 마세요.

#### 파일 내용 확인

| 명령어              | 설명                             | 예시                                |
| ------------------- | -------------------------------- | ----------------------------------- |
| `cat 파일`          | 파일 전체 출력                   | `cat /etc/nginx/nginx.conf`         |
| `head -n 숫자 파일` | 앞에서 n줄만 출력                | `head -20 app.log`                  |
| `tail -n 숫자 파일` | 뒤에서 n줄만 출력                | `tail -50 app.log`                  |
| `tail -f 파일`      | 실시간 로그 확인 (Ctrl+C로 종료) | `tail -f /var/log/nginx/access.log` |

#### 권한과 소유자

| 명령어                   | 설명             | 예시                                            |
| ------------------------ | ---------------- | ----------------------------------------------- |
| `chmod 400 파일`         | 소유자만 읽기    | `chmod 400 ~/.ssh/my-key.pem`                   |
| `chmod +x 파일`          | 실행 권한 추가   | `chmod +x gradlew`                              |
| `chown 사용자:그룹 파일` | 소유자 변경      | `sudo chown nginx:nginx /usr/share/nginx/html/` |

| --------------------------------- | ---------------------- |

| 명령어        | 설명                    | 예시                             |

| `curl URL`    | HTTP 요청 보내기        | `curl http://localhost:8080`     |

| `kill -9 PID` | 프로세스 강제 종료      | `kill -9 12345`                  |

#### 파일 전송 (로컬 ↔ EC2)

| 명령어                          | 방향          | 예시                                      |
| ------------------------------- | ------------- | ----------------------------------------- |
| `scp -i 키 로컬파일 EC2:경로`   | 로컬 → EC2    | `scp -i key.pem app.jar ec2-user@IP:~/`   |
| `scp -i 키 EC2:경로 로컬경로`   | EC2 → 로컬    | `scp -i key.pem ec2-user@IP:~/app.log ./` |
| `rsync --exclude 'X' 원본 대상` | 제외하고 전송 | `rsync --exclude 'node_modules' ...`      |

#### SSH, SCP, rsync의 관계와 차이

이 세 가지 명령어는 모두 **SSH 프로토콜 기반**으로 동작하지만, 역할이 다릅니다:

```
SSH (Secure Shell)
├── ssh    → 원격 서버에 "접속"하여 명령어 실행
├── scp    → SSH를 이용하여 "파일 전송" (단순 복사)
└── rsync  → SSH를 이용하여 "파일 동기화" (증분 전송, 제외 옵션)
```

| 명령어    | 역할        | 특징                           | 주 사용처                 |
| --------- | ----------- | ------------------------------ | ------------------------- |
| **ssh**   | 원격 접속   | 서버에 로그인하여 터미널 사용  | EC2 접속, 명령어 실행     |
| **scp**   | 파일 복사   | 단순하지만 제외 옵션 없음      | 단일 파일/JAR/WAR 전송    |
| **rsync** | 파일 동기화 | `--exclude`, 증분 전송, 재시도 | 프로젝트 폴더 통째로 전송 |

> [!CONCEPT] 세 명령어의 핵심 차이
>
> **ssh** — "EC2에 들어가서 작업한다"
>
> ```bash
> ssh -i key.pem ec2-user@3.35.xxx.xxx
> # → EC2 내부에서 명령어 실행 가능 (exit로 나옴)
> ```
>
> **scp** — "파일을 보낸다 (단순 복사)"
>
> ```bash
> scp -i key.pem app.jar ec2-user@3.35.xxx.xxx:~/
> # → 로컬의 app.jar를 EC2의 홈 디렉토리에 복사
> # → 접속하지 않고 파일만 전송 후 즉시 종료
> ```
>
> **rsync** — "폴더를 동기화한다 (똑똑한 전송)"
>
> ```bash
> rsync -avz --exclude 'node_modules' ./my-app/ ec2-user@IP:~/my-app/
> # → node_modules 제외하고 전체 폴더 동기화
> # → 이미 같은 파일은 건너뜀 (변경분만 전송 = 빠름)
> ```

#### 언제 뭘 쓰나요?

| 상황                                          | 사용할 명령어 |
| --------------------------------------------- | ------------- |
| EC2에 접속해서 명령어 실행 (설치, 설정 등)    | `ssh`         |
| JAR/WAR 파일 1개를 EC2에 전송                 | `scp`         |
| 빌드 결과물(dist/) 폴더를 EC2에 전송          | `scp -r`      |
| 프로젝트 폴더 전체를 전송 (node_modules 제외) | `rsync`       |
| 두 번째부터 업데이트 (변경된 파일만 전송)     | `rsync`       |

#### OS별 사용 가능 여부

| 명령어  | Mac     | Linux   | Windows PowerShell | Windows Git Bash  |
| ------- | ------- | ------- | ------------------ | ----------------- |
| `ssh`   | ✅ 기본 | ✅ 기본 | ✅ 기본 (Win10+)   | ✅                |
| `scp`   | ✅ 기본 | ✅ 기본 | ✅ 기본 (Win10+)   | ✅                |
| `rsync` | ✅ 기본 | ✅ 기본 | ❌ 없음            | ⚠️ 별도 설치 필요 |

> [!NOTE]
> Windows 사용자는 `ssh`와 `scp`는 PowerShell에서 바로 사용 가능하지만,  
> `rsync`는 사용할 수 없습니다. 프로젝트 폴더 전체를 보내야 하는 경우:
>
> - **Git Bash** 설치 후 rsync 사용
> - **WSL (Windows Subsystem for Linux)** 터미널에서 사용
> - 또는 **로컬에서 빌드 후 결과물만 `scp`로 전송** (Step 2-2 방법 ③ 참고)

> [!TIP]

## 13. 터미널 환경 구분하기

> [!CONCEPT] 지금 어디에서 작업하고 있는지 항상 확인하세요
> 실습을 진행하다 보면 **로컬 PC의 터미널**, **EC2 서버의 쉘**, **MySQL 프롬프트**를 오가게 됩니다.  
> 명령어를 잘못된 환경에서 실행하면 에러가 나거나, 최악의 경우 의도치 않은 서버에서 작업할 수 있습니다.  
> **프롬프트(커서 앞 문자열)를 보고 현재 위치를 판단하는 습관**을 들이세요.

### 프롬프트로 현재 환경 구분하기

```
┌──────────────────────────────────────────────────────────────┐
│  1. 로컬 PC (내 컴퓨터)                                      │
│  ─────────────────────────────────────────                   │
│  프롬프트 예시:                                              │
│    username@MacBook ~ %              ← Mac (zsh)             │
│    username@DESKTOP-XXX:~$           ← Linux/WSL             │
│    C:\Users\username>                ← Windows CMD           │
│    PS C:\Users\username>             ← Windows PowerShell    │
│                                                              │
│  여기서 하는 작업: ssh 접속, 로컬 파일 관리, aws cli 실행    │
├──────────────────────────────────────────────────────────────┤
│  2. EC2 서버 (원격 서버 — SSH 접속 후)                       │
│  ─────────────────────────────────────────                   │
│  프롬프트 예시:                                              │
│    [ec2-user@ip-10-0-1-217 ~]$       ← SSH 접속 시           │
│    sh-4.2$                           ← SSM 접속 시           │
│    ssm-user@ip-10-0-1-217:~$         ← SSM 접속 시           │
│    [root@ip-10-0-1-217 ~]#           ← root 전환 시 (# 표시) │
│                                                              │
│  여기서 하는 작업: 패키지 설치, 서비스 관리, 설정 파일 편집  │
├──────────────────────────────────────────────────────────────┤
│  3. MySQL 프롬프트 (DB 내부 — mysql 명령어 실행 후)          │
│  ─────────────────────────────────────────                   │
│  프롬프트 예시:                                              │
│    mysql>                            ← 일반 상태             │
│    mysql> _                          ← 입력 대기             │
│        ->                            ← 여러 줄 입력 중       │
│                                                              │
│  여기서 하는 작업: SQL 실행 (CREATE, SELECT, GRANT 등)       │
│  ⚠️ 리눅스 명령어(sudo, dnf 등)는 여기서 실행 안 됩니다!     │
└──────────────────────────────────────────────────────────────┘
```

### 환경 간 이동 명령어

```
로컬 PC
  │
  │  ssh -i key.pem ec2-user@<IP>     ← EC2 접속
  │  (또는 SSM Session Manager)
  ▼
EC2 서버 [ec2-user@ip-10-0-1-xxx ~]$
  │
  │  mysql -u root -p                  ← MySQL 접속
  ▼
MySQL 프롬프트 mysql>
  │
  │  EXIT;  (또는 \q, quit)            ← MySQL 종료
  ▼
EC2 서버 [ec2-user@ip-10-0-1-xxx ~]$
  │
  │  exit                              ← SSH 세션 종료
  ▼
로컬 PC
```

### 흔한 실수와 구분법

| 실수                                           | 원인                                  | 해결                                 |
| ---------------------------------------------- | ------------------------------------- | ------------------------------------ |
| `sudo: command not found` (MySQL 프롬프트에서) | MySQL 안에서 리눅스 명령어 실행       | `EXIT;`로 MySQL 먼저 종료 후 실행    |
| `ERROR 1064 (42000)` (EC2 쉘에서)              | 리눅스 쉘에서 SQL 문법 실행           | `mysql -u root -p`로 MySQL 먼저 접속 |
| 로컬에서 `sudo dnf install` 실행               | EC2에 접속하지 않고 로컬에서 실행     | SSH로 EC2에 접속한 뒤 실행           |
| `Connection refused` (로컬에서 mysql 접속)     | EC2의 MySQL에 로컬에서 직접 접속 시도 | `-h <EC2-Public-IP>` 옵션 추가 필요  |

> [!TIP]
> **지금 어디인지 모르겠을 때:**
>
> ```bash
> # EC2 쉘인지 확인
> hostname       # ip-10-0-1-xxx 형태면 EC2
> whoami         # ec2-user 또는 ssm-user면 EC2
>
> # MySQL 안인지 확인
> # 프롬프트가 mysql> 이면 MySQL 내부
> # EXIT; 입력하면 EC2 쉘로 돌아감
> ```
>
> **프롬프트의 `$` vs `#` 차이:**
>
> - `$`: 일반 사용자 (ec2-user, ssm-user)
> - `#`: root 사용자 (`sudo su -`로 전환한 상태)

## 14. MySQL 데이터베이스 기초

> [!CONCEPT] MySQL과 관계형 데이터베이스
> **MySQL**은 세계에서 가장 많이 사용되는 오픈소스 관계형 데이터베이스(RDBMS)입니다.  
> 데이터를 테이블(행과 열)로 구조화하여 저장하며, SQL(Structured Query Language)로 조회·수정합니다.  
> AWS에서는 EC2에 직접 설치하거나, 관리형 서비스인 RDS를 사용할 수 있습니다.

### 주요 용어

| 용어                         | 설명                                                         |
| ---------------------------- | ------------------------------------------------------------ |
| **RDBMS**                    | 관계형 데이터베이스 관리 시스템. 데이터를 테이블 형태로 저장 |
| **스키마 (Schema/Database)** | 테이블, 뷰, 인덱스 등의 논리적 컨테이너                      |
| **사용자 (User)**            | 데이터베이스에 접속하고 작업하는 계정                        |
| **권한 (Privileges)**        | 사용자가 수행할 수 있는 작업의 범위                          |
| **bind-address**             | MySQL이 네트워크 연결을 수신하는 IP 주소 설정                |
| **포트 (Port)**              | MySQL의 기본 통신 포트는 3306                                |

### EC2 직접 설치 vs RDS (관리형 서비스)

| 비교 항목     | EC2에 직접 설치             | Amazon RDS                 |
| ------------- | --------------------------- | -------------------------- |
| OS 접근       | ✅ SSH로 풀 접근            | ❌ OS 접근 불가            |
| 패치/업데이트 | 수동 관리                   | AWS가 자동 관리            |
| 백업          | 수동 설정                   | 자동 백업 (35일)           |
| 고가용성      | 직접 구성 필요              | Multi-AZ 옵션              |
| 비용          | EC2 요금만                  | EC2보다 20~30% 높음        |
| 커스터마이징  | 자유로움 (my.cnf 직접 편집) | Parameter Group으로 제한적 |
| 적합한 상황   | 학습, 특수 설정 필요 시     | 프로덕션, 운영 편의 중시   |

이 실습(Step 2)에서는 EC2에 직접 설치하여 MySQL의 동작 원리를 이해하고, Step 4에서 RDS를 사용합니다.

## 15. MySQL 사용자와 권한 관리

> [!CONCEPT] 최소 권한 원칙 (Principle of Least Privilege)
> 데이터베이스 사용자에게는 **업무에 필요한 최소한의 권한**만 부여해야 합니다.  
> 과도한 권한은 실수나 보안 침해 시 피해를 확대시킵니다.  
> 이는 AWS IAM의 권한 관리와 동일한 원칙입니다.

### 사용자 생성과 호스트 제한

```sql
-- 기본 구조
CREATE USER '사용자명'@'호스트' IDENTIFIED BY '비밀번호';
```

| 호스트 지정          | 의미                  | 사용 사례            |
| -------------------- | --------------------- | -------------------- |
| `'user'@'%'`         | 모든 IP에서 접속 허용 | 개발/테스트 환경     |
| `'user'@'10.0.%'`    | 10.0.x.x 대역만 허용  | VPC 내부 접속만 허용 |
| `'user'@'10.0.1.50'` | 특정 IP만 허용        | 앱 서버 IP로 제한    |
| `'user'@'localhost'` | 같은 서버에서만 접속  | 로컬 앱만 사용       |

### 권한 부여 (GRANT) 구문

```sql
GRANT <권한 목록> ON <데이터베이스>.<테이블> TO '사용자'@'호스트';
FLUSH PRIVILEGES;  -- 변경사항 즉시 적용
```

### 권한의 종류와 위험도

| 권한             | 설명                      | 위험도   | 용도                 |
| ---------------- | ------------------------- | -------- | -------------------- |
| `SELECT`         | 데이터 조회               | 낮음     | 읽기 전용 앱, 리포팅 |
| `INSERT`         | 데이터 추가               | 낮음     | 일반 CRUD 작업       |
| `UPDATE`         | 데이터 수정               | 낮음     | 일반 CRUD 작업       |
| `DELETE`         | 데이터 삭제               | 중간     | 일반 CRUD 작업       |
| `CREATE`         | 테이블/DB 생성            | 중간     | 배포, 마이그레이션   |
| `ALTER`          | 테이블 구조 변경          | 중간     | 배포, 마이그레이션   |
| `DROP`           | 테이블/DB 삭제            | **높음** | DBA만 사용           |
| `GRANT`          | 다른 사용자에게 권한 부여 | **높음** | DBA만 사용           |
| `ALL PRIVILEGES` | 위 모든 권한 포함         | **높음** | 개발 환경에서만      |

### 실무 계정 분리 패턴

```
┌─────────────────────────────────────────────────────────────┐
│  프로덕션 DB 계정 구성 예시                                 │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  appuser (애플리케이션 계정)                                │
│  ├── 권한: SELECT, INSERT, UPDATE, DELETE                   │
│  ├── 호스트: '10.0.1.%' (App Subnet만)                      │
│  └── 용도: 앱 서버의 일반 CRUD 작업                         │
│                                                             │
│  deploy_user (배포 계정)                                    │
│  ├── 권한: CREATE, ALTER, INDEX, SELECT                     │
│  ├── 호스트: CI/CD 서버 IP만                                │
│  └── 용도: 스키마 마이그레이션, 테이블 변경                 │
│                                                             │
│  admin (DBA 계정)                                           │
│  ├── 권한: ALL PRIVILEGES                                   │
│  ├── 호스트: 'localhost' 또는 VPN IP만                      │
│  └── 용도: 긴급 대응, 백업, 사용자 관리                     │
│                                                             │
│  ⚠️ root 계정은 원격 접속 차단 (localhost만 허용)           │
└─────────────────────────────────────────────────────────────┘
```

### 권한 확인 및 회수

```sql
-- 사용자 권한 확인
SHOW GRANTS FOR 'appuser'@'%';

-- 불필요한 권한 회수
REVOKE DROP, ALTER ON appdb.* FROM 'appuser'@'%';
FLUSH PRIVILEGES;

-- 사용자 삭제
DROP USER 'appuser'@'%';
```

## 16. MySQL 네트워크 접속 구조

> [!CONCEPT] MySQL 외부 접속의 3가지 조건
> EC2에 설치된 MySQL에 외부에서 접속하려면 **3가지 계층의 허용**이 모두 필요합니다.  
> 하나라도 막혀 있으면 접속할 수 없습니다.

### 접속을 위한 3단계 허용

```
외부 클라이언트 ─────► AWS Security Group ─────► MySQL bind-address ─────► MySQL 사용자 호스트

  ① 네트워크 방화벽    ② 서비스 리스닝         ③ DB 접근 제어
  (AWS 레벨)            (OS/서비스 레벨)         (MySQL 레벨)
```

| 계층             | 설정 위치                           | 확인 방법                                | 막혔을 때 증상                       |
| ---------------- | ----------------------------------- | ---------------------------------------- | ------------------------------------ |
| ① Security Group | AWS 콘솔 → EC2 → SG → Inbound rules | 포트 3306 규칙 존재 여부                 | `Connection timed out`               |
| ② bind-address   | `/etc/my.cnf` → `bind-address`      | `ss -tlnp \| grep 3306` → `0.0.0.0` 확인 | `Connection refused`                 |
| ③ MySQL 사용자   | MySQL 내부 → `mysql.user` 테이블    | `SELECT user, host FROM mysql.user;`     | `Access denied` / `Host not allowed` |

### bind-address 설정값

| 설정값       | 의미                                            | 사용 사례                         |
| ------------ | ----------------------------------------------- | --------------------------------- |
| `*` (기본값) | 모든 네트워크 인터페이스에서 수신 (IPv4 + IPv6) | MySQL 공식 RPM 설치 시 기본값     |
| `0.0.0.0`    | 모든 IPv4 인터페이스에서 수신                   | 외부 접속 허용 (SG로 제어)        |
| `127.0.0.1`  | localhost에서만 접속 가능                       | 일부 Linux 배포판의 패키지 기본값 |
| 특정 IP      | 해당 IP에서만 수신                              | 멀티 NIC 환경                     |

## 17. 웹 애플리케이션 배포 아키텍처

> [!CONCEPT] 모놀리식 배포 vs 독립 배포
> 웹 애플리케이션을 서버에 배포하는 방식은 크게 두 가지입니다.  
> **모놀리식 배포**는 프론트엔드를 백엔드 안에 포함시키는 방식이고, **독립 배포**는 프론트엔드와 백엔드를 분리하여 각각 배포하는 방식입니다.  
> 이 과정에서는 독립 배포 방식을 사용합니다.

### 모놀리식 배포 (프론트엔드 내장)

```
[브라우저] → [Spring Boot (Tomcat)]
                 ├── /api/** → REST Controller (백엔드 로직)
                 └── 그 외   → index.html (Vue/React 빌드 파일 내장)
```

- Vue/React 빌드 결과물을 Spring Boot의 `webapp/resources/`에 넣음
- 서버 하나로 프론트엔드 + 백엔드 모두 서빙
- 빌드 시 `vite.config.js`의 `outDir`을 백엔드 폴더로 설정

```js
// vite.config.js (모놀리식 - 빌드 결과를 백엔드에 넣음)
export default defineConfig({
  build: {
    outDir: '../backend/src/main/webapp/resources',
    emptyOutDir: true,
  },
  server: {
    proxy: { '/api': { target: 'http://localhost:8080' } },
  },
});
```

### 독립 배포 (프론트/백엔드 분리)

```
[브라우저] → [Nginx (프론트엔드)]     ← 정적 파일 서빙 + SPA fallback
                 │
                 └── /api/** 프록시 → [Spring Boot (백엔드)]  ← API만 담당
```

- 프론트엔드: Nginx (또는 S3 + CloudFront)에서 정적 파일 서빙
- 백엔드: API 서버만 담당
- 각각 독립적으로 배포, 스케일링 가능

### 비교 요약

| 항목            | 모놀리식                 | 독립 배포              |
| --------------- | ------------------------ | ---------------------- |
| 배포 단위       | 하나 (백엔드에 포함)     | 프론트/백엔드 각각     |
| 프론트 업데이트 | 백엔드 재배포 필요       | 프론트만 재배포        |
| 스케일링        | 함께 스케일              | 독립 스케일            |
| CI/CD           | 단일 파이프라인          | 각각 파이프라인        |
| SPA fallback    | 백엔드 컨트롤러에서 처리 | 웹서버(Nginx)에서 처리 |
| 적합한 상황     | 소규모/학습/프로토타입   | 중대규모 실무 서비스   |

## 18. SPA 라우팅 이론

> [!CONCEPT] SPA (Single Page Application)
> **SPA**는 하나의 HTML 파일(`index.html`)로 모든 페이지를 처리하는 웹 애플리케이션입니다.  
> 페이지 이동 시 서버에 새로운 HTML을 요청하는 것이 아니라, JavaScript가 브라우저 내에서 화면을 전환합니다.  
> Vue Router, React Router 등이 이 역할을 합니다.

### 전통적 웹 (MPA) vs SPA

| 항목        | MPA (Multi Page Application)    | SPA (Single Page Application)    |
| ----------- | ------------------------------- | -------------------------------- |
| 페이지 이동 | 서버에서 새 HTML 응답           | JavaScript가 화면 교체           |
| 서버 파일   | `/about.html`, `/board.html` 등 | `index.html` 하나만 존재         |
| URL 변경    | 실제 파일 경로와 매칭           | 가상 경로 (Router가 처리)        |
| 사용자 경험 | 깜박임 (페이지 새로 로드)       | 부드러운 전환 (필요 부분만 변경) |
| 프레임워크  | JSP, Thymeleaf, PHP             | Vue, React, Angular              |

### 렌더링 방식: CSR, SSR, SSG

웹 페이지의 HTML을 **어디서, 언제** 만드느냐에 따라 렌더링 방식이 나뉩니다.

> [!CONCEPT] 렌더링(Rendering)이란?
> 렌더링이란 **데이터를 실제 화면(HTML)으로 변환하는 과정**입니다.
> "누가 HTML을 만드느냐"에 따라 웹 애플리케이션의 특성이 크게 달라집니다.

#### CSR (Client Side Rendering) — 클라이언트 측 렌더링

```
[서버] → 빈 HTML + JS 번들 전송 → [브라우저에서 JS 실행] → 화면 생성
```

- 서버는 거의 빈 `index.html`과 JavaScript 파일만 전송
- 브라우저가 JS를 실행해서 화면을 그림
- **SPA = CSR 방식** (Vue, React, Angular의 기본 동작)
- 첫 로딩 후에는 페이지 이동이 빠름 (서버 요청 없이 화면 교체)

```html
<!-- CSR: 서버가 보내는 HTML (거의 비어있음) -->
<html>
  <body>
    <div id="app"></div>
    <!-- 여기에 JS가 화면을 그림 -->
    <script src="/assets/app.js"></script>
    <!-- 이 JS가 모든 걸 처리 -->
  </body>
</html>
```

**장점:**

- 페이지 이동이 빠름 (새로고침 없음)
- 서버 부하 적음 (정적 파일만 서빙)
- 사용자 인터랙션이 부드러움

**단점:**

- 첫 로딩이 느림 (JS 다운로드 + 실행 대기)
- SEO 불리 (검색엔진이 빈 HTML을 보게 됨)
- JavaScript가 비활성화되면 아무것도 안 보임

#### SSR (Server Side Rendering) — 서버 측 렌더링

```
[서버에서 HTML 완성] → 완성된 HTML 전송 → [브라우저에서 바로 표시] → JS로 인터랙션 추가
```

- 서버가 매 요청마다 데이터를 조회하고 HTML을 완성해서 전송
- 브라우저는 받은 HTML을 바로 표시 (빠른 첫 화면)
- 이후 JS가 로드되면 인터랙션 추가 (**Hydration**)
- Next.js(React), Nuxt(Vue)가 대표적

```html
<!-- SSR: 서버가 보내는 HTML (내용이 이미 채워져 있음) -->
<html>
  <body>
    <div id="app">
      <h1>게시판</h1>
      <ul>
        <li>첫 번째 글</li>
        <!-- 서버에서 DB 조회하여 미리 채움 -->
        <li>두 번째 글</li>
      </ul>
    </div>
    <script src="/assets/app.js"></script>
    <!-- Hydration: 인터랙션 추가 -->
  </body>
</html>
```

**장점:**

- 첫 화면 표시 빠름 (HTML이 이미 완성됨)
- SEO 우수 (검색엔진이 완성된 HTML을 읽을 수 있음)
- 소셜 미디어 공유 시 미리보기 정상 동작

**단점:**

- 서버 부하 높음 (매 요청마다 렌더링)
- TTFB(Time To First Byte) 지연 (서버 처리 시간만큼)
- 서버 인프라가 필요 (정적 호스팅만으로는 불가)

#### SSG (Static Site Generation) — 정적 사이트 생성

```
[빌드 시 HTML 미리 생성] → 완성된 HTML 파일 배포 → [브라우저에서 바로 표시]
```

- **빌드 시점에** 모든 페이지의 HTML을 미리 생성
- 런타임에는 이미 만들어진 HTML을 그냥 전송 (서버 렌더링 없음)
- 블로그, 문서 사이트, 마케팅 페이지에 적합
- Gatsby, Astro, Next.js(정적 export), VitePress가 대표적

**장점:**

- 가장 빠른 로딩 (미리 만들어진 정적 파일)
- CDN에 최적화 (전 세계 캐싱)
- 서버 불필요 (S3, GitHub Pages로 호스팅 가능)
- SEO 우수

**단점:**

- 콘텐츠 변경 시 재빌드 필요
- 사용자별 동적 콘텐츠 처리 어려움
- 페이지 수가 많으면 빌드 시간 증가

#### 렌더링 방식 비교 요약

| 항목            | CSR                     | SSR                 | SSG                       |
| --------------- | ----------------------- | ------------------- | ------------------------- |
| HTML 생성 시점  | 브라우저 (런타임)       | 서버 (요청 시)      | 빌드 시 (미리)            |
| 첫 화면 속도    | 느림 (JS 로딩 후)       | 보통 (서버 처리 후) | 빠름 (정적 파일)          |
| SEO             | ❌ 불리                 | ✅ 우수             | ✅ 우수                   |
| 서버 부하       | 낮음                    | 높음                | 없음                      |
| 인터랙션        | 풍부                    | 풍부                | 제한적 (JS 추가 시 가능)  |
| 실시간 데이터   | ✅                      | ✅                  | ❌ (빌드 시점 데이터)     |
| 배포 방식       | 정적 호스팅 (Nginx, S3) | Node.js 서버 필요   | 정적 호스팅 (Nginx, S3)   |
| 대표 프레임워크 | Vue, React (기본)       | Next.js, Nuxt       | Gatsby, Astro, VitePress  |
| 적합한 서비스   | 대시보드, 관리자 페이지 | 뉴스, 쇼핑몰, SNS   | 블로그, 문서, 랜딩 페이지 |

#### 하이브리드 방식

현대 프레임워크(Next.js, Nuxt 3)는 **페이지별로 다른 렌더링 방식을 혼합**할 수 있습니다:

```
Next.js 프로젝트 예시:
├── /             → SSG (마케팅 랜딩 페이지)
├── /blog         → SSG (블로그 글)
├── /dashboard    → CSR (로그인 후 관리자 페이지)
└── /products/:id → SSR (실시간 재고 표시가 필요)
```

#### 이 과정에서의 선택

이 과정에서는 **CSR (SPA)** 방식을 사용합니다:

- Vue 3 + Vite로 빌드 → `dist/` 폴더에 정적 파일 생성
- Nginx 또는 S3+CloudFront에서 정적 파일 서빙
- API는 별도 백엔드(Spring)에서 처리

CSR을 선택한 이유:

1. 배포가 단순 (정적 파일만 서빙하면 됨)
2. AWS 인프라 학습에 집중 (Node.js 서버 운영 불필요)
3. 프론트/백엔드 완전 분리 가능 (독립 배포)

> [!TIP]
> **"SEO가 중요한 서비스(블로그, 쇼핑몰)인데 SPA를 쓰면 안 되나요?"**
>
> SPA(CSR)도 SEO를 개선할 수 있습니다:
>
> - **Prerendering**: 빌드 시 주요 페이지만 미리 렌더링 (vite-plugin-ssr)
> - **Dynamic Rendering**: 검색엔진 봇에게만 SSR 결과 제공 (Puppeteer)
> - **Meta 태그 동적 설정**: vue-meta, react-helmet으로 페이지별 메타 태그 추가
>
> 하지만 SEO가 핵심인 서비스라면 처음부터 Next.js(SSR/SSG)를 선택하는 것이 효율적입니다.

### SPA 라우팅 문제: URL 직접 입력 시 404

```
앱 내 링크 클릭:
[브라우저] /about 클릭 → Vue Router가 화면 교체 (서버 요청 없음) → ✅ 동작

URL 직접 입력 또는 새로고침:
[브라우저] /about 입력 → 서버에 GET /about 요청 → 서버에 /about 파일 없음 → ❌ 404
```

SPA에서는 모든 라우팅을 `index.html` 하나가 처리합니다.  
하지만 브라우저 주소창에 `/about`을 직접 입력하면 서버는 실제 `/about` 파일을 찾으려 합니다.  
파일이 없으니 404가 발생합니다. **이것이 SPA 라우팅 문제의 본질입니다.**

### 언제 서버 설정이 필요한가?

모든 URL 이동이 문제인 것은 아닙니다. **서버에 실제 요청이 가는 경우**에만 문제가 됩니다:

| 사용자 동작                       | 누가 처리           | 서버에 요청 | 서버 설정 필요? |
| --------------------------------- | ------------------- | ----------- | --------------- |
| 앱 내 링크 클릭 (`<RouterLink>`)  | Router (클라이언트) | ❌          | ❌              |
| 브라우저 뒤로가기 / 앞으로가기    | Router (클라이언트) | ❌          | ❌              |
| **주소창에 URL 직접 입력 + 엔터** | **서버**            | ✅          | ✅              |
| **현재 페이지에서 새로고침 (F5)** | **서버**            | ✅          | ✅              |
| **다른 사이트에서 링크 클릭**     | **서버**            | ✅          | ✅              |
| **북마크에서 접속**               | **서버**            | ✅          | ✅              |

Router는 앱이 이미 로드된 상태에서만 동작합니다. 앱이 로드되지 않은 상태(직접 입력, 새로고침, 외부 링크)에서는 서버가 먼저 `index.html`을 반환해야 Router가 작동할 수 있습니다.

### SPA 라우팅 문제 해결 방법

| 방법                   | 설정 위치            | 원리                                   | URL 형태            |
| ---------------------- | -------------------- | -------------------------------------- | ------------------- |
| **Nginx `try_files`**  | 서버 설정            | 파일 없으면 `index.html` 반환          | `/about` (깔끔)     |
| **Hash Mode**          | 프로젝트 Router 설정 | `#` 뒤는 서버로 전송 안 됨             | `/#/about` (지저분) |
| **Spring Boot 포워딩** | 백엔드 컨트롤러      | API 외 경로를 `index.html`로 포워딩    | `/about` (깔끔)     |
| **S3 + CloudFront**    | AWS 인프라 설정      | Error Page를 `index.html`로 리다이렉트 | `/about` (깔끔)     |

### 방법 1: Nginx `try_files` (독립 배포 — 이 과정에서 사용)

```nginx
location / {
    try_files $uri $uri/ /index.html;
}
```

- 요청된 파일이 있으면 → 그 파일 반환 (JS, CSS, 이미지 등)
- 없으면 → `index.html` 반환 → Vue Router가 경로 처리

### 방법 2: Vue Router Hash Mode (서버 설정 불필요)

```js
import { createRouter, createWebHashHistory } from 'vue-router'

const router = createRouter({
  history: createWebHashHistory(),  // URL: http://example.com/#/about
  routes: [...]
})
```

- `#` 뒤의 내용은 브라우저가 서버로 보내지 않음
- 서버는 항상 `index.html`만 응답하면 됨
- 단점: URL이 `/#/about` 형태로 지저분, SEO 불리

### 방법 3: Spring Boot 포워딩 (모놀리식 배포)

```java
@Controller
public class SpaForwardController {
    // /api 외의 모든 경로를 index.html로 포워딩
    @GetMapping(value = {"/{path:[^\\.]*}", "/{path:[^\\.]*}/**"})
    public String forward() {
        return "forward:/resources/index.html";
    }
}
```

- `/api/**`는 REST Controller가 처리
- 나머지 경로는 `index.html`로 포워딩 → Vue Router가 처리
- 이 설정이 없으면 `/board/list` 직접 접속 시 404 발생

### 방법 4: S3 + CloudFront Error Page (Step 5에서 다룸)

```
CloudFront 동작 설정:
  403/404 에러 → /index.html 반환 (HTTP 200)
```

- S3에 정적 파일을 업로드하고 CloudFront로 서빙
- 존재하지 않는 경로 → S3가 403/404 → CloudFront가 index.html로 대체
- 서버리스 배포에서 사용

### Vite `base` 경로 설정

서브 디렉토리(`/app/`)에 배포하는 경우, 리소스 경로를 맞춰야 합니다:

```js
// vite.config.js
export default defineConfig({
  base: '/app/', // 서브 경로에 배포하는 경우
});
```

```js
// src/router/index.js
const router = createRouter({
  history: createWebHistory('/app/'),  // base와 동일하게
  routes: [...]
})
```

| 배포 위치    | `base` 설정     | 예시                                |
| ------------ | --------------- | ----------------------------------- |
| 루트 (`/`)   | `'/'` (기본값)  | `http://example.com/`               |
| 서브 경로    | `'/app/'`       | `http://example.com/app/`           |
| GitHub Pages | `'/repo-name/'` | `https://user.github.io/repo-name/` |

루트에 배포하면 `base` 설정을 변경할 필요가 없습니다.

### 이 과정에서의 배포 전략

| Step         | 배포 방식                      | SPA fallback        |
| ------------ | ------------------------------ | ------------------- |
| **Step 2-2** | EC2 + Nginx (독립)             | Nginx `try_files`   |
| **Step 5-3** | S3 + CloudFront (서버리스)     | Error Page redirect |
| **Step 8**   | GitHub Actions → EC2 자동 배포 | Nginx `try_files`   |
| **Step 9**   | 3-Tier (ALB + EC2)             | Nginx `try_files`   |

## 19. Nginx 웹 서버

> [!CONCEPT] Nginx란
> **Nginx**는 고성능 웹 서버이자 리버스 프록시 서버입니다.  
> 정적 파일 서빙, SPA 라우팅(try_files), 리버스 프록시, 로드 밸런싱 등 다양한 역할을 수행합니다.  
> Apache보다 메모리 사용량이 적고 동시 접속 처리 성능이 뛰어나 현대 웹 서비스에서 널리 사용됩니다.

### Nginx의 역할 (이 과정에서)

```
[브라우저]
    │
    ├── /            → 정적 파일 서빙 (Vue 빌드 결과)
    ├── /about       → SPA fallback (try_files → index.html)
    └── /api/**      → 리버스 프록시 (→ Spring 8080)
```

### 설정 파일 구조

```
/etc/nginx/
├── nginx.conf           ← 메인 설정 (기본 server 블록 포함)
├── conf.d/              ← 추가 설정 파일 (*.conf 자동 로드)
│   └── vue-app.conf     ← 우리가 만드는 설정
└── mime.types           ← 파일 확장자별 Content-Type 매핑
```

### 핵심 지시어

| 지시어       | 역할                                    | 예시                                |
| ------------ | --------------------------------------- | ----------------------------------- |
| `listen`     | 수신할 포트                             | `listen 80;`                        |
| `root`       | 정적 파일의 기본 경로                   | `root /usr/share/nginx/html;`       |
| `try_files`  | 파일이 없으면 fallback                  | `try_files $uri $uri/ /index.html;` |
| `proxy_pass` | 요청을 다른 서버로 전달 (리버스 프록시) | `proxy_pass http://localhost:8080;` |
| `expires`    | 브라우저 캐시 기간 설정                 | `expires 1y;`                       |
| `gzip on`    | 응답 압축                               | 전송 크기 60~80% 감소               |

### 리버스 프록시란

**왜 브라우저에서 직접 Spring(8080)으로 안 보내고 Nginx를 거칠까요?**

Spring은 API 처리(비즈니스 로직, DB 조회 등)에 최적화된 서버입니다.  
하지만 이미지, CSS, JS 같은 정적 파일을 빠르게 전달하는 데는 비효율적입니다.  
Nginx가 정적 파일은 직접 처리하고 API 요청만 Spring에 넘기면, Spring은 진짜 해야 할 일에만 집중할 수 있습니다.

```
리버스 프록시 없이:
[브라우저] ──── :80 (프론트) ──── 직접 :8080 호출 (CORS 문제 발생)

리버스 프록시 사용:
[브라우저] ──── :80 (Nginx) ─┬── 정적 파일 → Vue 앱
                             └── /api/* → proxy_pass → :8080 (Spring)
                                 (같은 origin이므로 CORS 없음)
```

- 외부에 백엔드 포트(8080)를 노출하지 않아도 됨
- 프론트와 백엔드가 같은 도메인/포트에서 동작 → CORS 문제 없음
- Nginx가 정적 파일 캐싱, 압축, SSL 종료 등을 처리

## 20. Java / JVM 기초

> [!CONCEPT] Java와 JVM
> **Java**는 "Write Once, Run Anywhere" 철학의 프로그래밍 언어이며, **JVM(Java Virtual Machine)**에서 실행됩니다.  
> Spring Boot/Spring MVC 애플리케이션을 EC2에서 실행하려면 JDK(Java Development Kit)가 설치되어 있어야 합니다.

### 주요 용어

| 용어                | 설명                                                      |
| ------------------- | --------------------------------------------------------- |
| **JDK**             | Java 개발 도구 (컴파일러 + JRE)                           |
| **JRE**             | Java 실행 환경 (JVM 포함)                                 |
| **JVM**             | Java 바이트코드를 실행하는 가상 머신                      |
| **Amazon Corretto** | AWS가 제공하는 무료 OpenJDK 배포판 (장기 지원)            |
| **JAR**             | Java Archive — 컴파일된 클래스 + 리소스를 담은 실행 파일  |
| **WAR**             | Web Archive — 웹 애플리케이션용 패키징 (외부 Tomcat 필요) |
| **JAVA_HOME**       | JDK 설치 경로를 가리키는 환경변수                         |

### JVM 메모리 옵션

Java는 시작하자마자 메모리를 미리 확보합니다.  
t3.micro(1GB RAM)에서 MySQL(~300MB) + Spring(~512MB)을 동시에 실행하면, OS 자체가 동작할 메모리가 부족해집니다.  
따라서 EC2 인스턴스 크기에 맞게 JVM 메모리를 제한하는 것이 중요합니다.

```bash
java -Xms256m -Xmx512m -jar app.jar
      │         │
      │         └── 최대 힙 메모리 (이 이상 사용하면 OOM 에러)
      └── 초기 힙 메모리 (시작 시 할당)
```

| 인스턴스 타입 | RAM  | 권장 -Xmx  | 이유                      |
| ------------- | ---- | ---------- | ------------------------- |
| t3.micro      | 1 GB | 256~384m   | OS + MySQL 여유 공간 필요 |
| t3.small      | 2 GB | 512~768m   | Spring 안정 운영 가능     |
| t3.medium     | 4 GB | 1024~1536m | Spring + MySQL 여유 운영  |

| 명령어                    | 역할                                     |
| ------------------------- | ---------------------------------------- |
| `./gradlew build`         | 컴파일 + 테스트 + JAR/WAR 생성           |
| `./gradlew build -x test` | 테스트 건너뛰고 빌드 (DB 미연결 시 유용) |

- 빌드 결과: `build/libs/<이름>.jar` 또는 `build/libs/<이름>.war`
- Windows에서는 `./gradlew` 대신 `gradlew.bat` 사용

## 21. Spring Boot / Spring MVC와 Tomcat

> [!CONCEPT] Servlet 컨테이너와 애플리케이션 서버
> Java 웹 애플리케이션은 **Servlet 컨테이너** 위에서 실행됩니다.  
> Servlet 컨테이너는 HTTP 요청을 받아 Java 코드(Servlet)로 전달하고, 응답을 반환하는 역할을 합니다.  
> Apache Tomcat이 가장 대표적인 Servlet 컨테이너입니다.

### Spring Boot vs Spring MVC

| 항목        | Spring Boot (JAR)                 | Spring MVC (WAR)              |
| ----------- | --------------------------------- | ----------------------------- |
| 실행 방식   | `java -jar app.jar`               | 외부 Tomcat에 WAR 배포        |
| Tomcat      | 내장 (앱 안에 포함)               | 외부 설치 필요                |
| 빌드 결과물 | `.jar` (20~50MB, Tomcat 포함)     | `.war` (수 MB, Tomcat 미포함) |
| 설정 방식   | `application.yml` / `.properties` | XML + properties              |
| 독립 실행   | ✅ (단독 실행 가능)               | ❌ (Tomcat이 있어야 실행)     |
| 배포 단순성 | ✅ 파일 하나로 완결               | Tomcat 설치·설정·관리 필요    |
| 실무 추세   | ✅ 현재 주류                      | 레거시 프로젝트에서 사용      |

### 내장 Tomcat vs 외부 Tomcat

```
Spring Boot (내장 Tomcat):
┌─────────────────────────┐
│  app.jar                │
│  ├── 내 코드            │
│  ├── Spring Framework   │
│  └── Tomcat (내장)      │  ← java -jar 하나로 실행
└─────────────────────────┘

Spring MVC (외부 Tomcat):
┌─────────────────────────┐
│  Tomcat (/opt/tomcat)   │  ← 별도 설치·관리
│  └── webapps/           │
│      └── ROOT.war       │  ← WAR 파일만 배포
│          └── 내 코드    │
└─────────────────────────┘
```

### Tomcat의 역할

| 기능               | 설명                                               |
| ------------------ | -------------------------------------------------- |
| HTTP 수신          | 클라이언트 요청을 받아 Servlet으로 전달            |
| 스레드 관리        | 동시 요청을 멀티 스레드로 처리                     |
| WAR 자동 배포      | `webapps/`에 WAR 파일을 넣으면 자동 압축 해제·배포 |
| JSP 컴파일         | JSP 파일을 Servlet으로 변환                        |
| 컨텍스트 경로 결정 | WAR 파일명으로 URL 경로 결정 (ROOT.war → `/`)      |

### Spring 버전과 Tomcat 호환성

| Spring 버전           | Servlet API           | Tomcat 버전 | 최소 Java 버전   | 패키지            |
| --------------------- | --------------------- | ----------- | ---------------- | ----------------- |
| Spring 5.x / Boot 2.x | javax.servlet (4.0)   | Tomcat 9    | Java 8 / 11      | `javax.servlet`   |
| Spring 6.x / Boot 3.x | jakarta.servlet (6.0) | Tomcat 10+  | **Java 17 필수** | `jakarta.servlet` |
| Spring 7.x / Boot 4.x | jakarta.servlet (6.1) | Tomcat 10+  | **Java 21 필수** | `jakarta.servlet` |

> [!WARNING]
> Spring 5.x 프로젝트에 Tomcat 10을 사용하면 `ClassNotFoundException: javax.servlet`이 발생합니다.  
> 반드시 Spring 버전에 맞는 Tomcat을 선택하세요.

### WAR 파일명과 접속 경로

| WAR 파일명    | 접속 경로                   |
| ------------- | --------------------------- |
| `ROOT.war`    | `http://host:8080/`         |
| `app.war`     | `http://host:8080/app/`     |
| `backend.war` | `http://host:8080/backend/` |

> [!TIP]
> Spring Boot를 사용하면 Tomcat 버전 관리, WAR 배포, 컨텍스트 경로 설정 등을 신경 쓸 필요가 없습니다.  
> 새 프로젝트라면 Spring Boot(JAR)를 선택하세요. WAR 배포는 기존 인프라 제약이 있는 경우에만 사용합니다.

## 22. systemd 서비스 관리

> [!CONCEPT] systemd란
> **systemd**는 Linux의 시스템/서비스 관리자입니다.  
> 부팅 시 서비스 자동 시작, 비정상 종료 시 자동 재시작, 로그 관리 등을 담당합니다.  
> EC2에서 Spring Boot나 Tomcat을 "서비스"로 등록하면 서버 재부팅 시 자동으로 시작되고, 장애 시 자동 복구됩니다.

**왜 서비스 등록이 필요한가요?**

`java -jar app.jar`로 직접 실행하면:

- 터미널을 닫으면 앱도 함께 종료됩니다
- 서버가 재부팅되면 아무도 앱을 다시 켜주지 않습니다
- 새벽에 메모리 부족으로 앱이 죽으면 아침에 출근할 때까지 서비스가 중단됩니다

systemd 서비스로 등록하면 이 모든 상황에서 **자동으로 앱이 복구**됩니다.

### 서비스 관리 명령어

| 명령어                            | 역할                          |
| --------------------------------- | ----------------------------- |
| `sudo systemctl start <서비스>`   | 서비스 시작                   |
| `sudo systemctl stop <서비스>`    | 서비스 중지                   |
| `sudo systemctl restart <서비스>` | 중지 + 시작                   |
| `sudo systemctl status <서비스>`  | 현재 상태 확인                |
| `sudo systemctl enable <서비스>`  | 부팅 시 자동 시작 등록        |
| `sudo systemctl disable <서비스>` | 부팅 시 자동 시작 해제        |
| `sudo systemctl daemon-reload`    | 서비스 파일 변경 후 다시 읽기 |

### 서비스 파일 구조 (.service)

```ini
[Unit]
Description=서비스 설명
After=network.target          # 네트워크 활성화 후 시작

[Service]
Type=simple                   # 프로세스 유형 (simple: 포그라운드)
User=ec2-user                 # 실행 사용자
ExecStart=/usr/bin/java -jar /opt/app/app.jar   # 실행 명령
WorkingDirectory=/opt/app     # 작업 디렉토리
Restart=on-failure            # 실패 시 자동 재시작
RestartSec=10                 # 재시작 대기 시간 (초)
SuccessExitStatus=143         # Java SIGTERM 정상 종료 코드를 성공으로 인정
Environment=KEY=VALUE         # 환경변수 설정

[Install]
WantedBy=multi-user.target    # enable 시 연결할 타겟
```

### 자동 재시작 동작

```
앱 정상 실행 중 → 예외/OOM으로 프로세스 종료 → systemd 감지
                                                    │
                                              RestartSec=10 대기
                                                    │
                                              ExecStart 다시 실행 → 앱 복구
```

### journalctl 로그 확인

| 명령어                                        | 용도             |
| --------------------------------------------- | ---------------- |
| `sudo journalctl -u spring-app -f`            | 실시간 로그 감시 |
| `sudo journalctl -u spring-app -n 50`         | 최근 50줄        |
| `sudo journalctl -u spring-app --since today` | 오늘 로그만      |
| `sudo journalctl -u spring-app -p err`        | 에러만 필터링    |

## 23. 애플리케이션 로그 레벨

> [!CONCEPT] 로그 레벨이란
> **로그 레벨**은 애플리케이션이 출력하는 로그의 상세도를 제어하는 설정입니다.  
> 레벨이 높을수록(TRACE에 가까울수록) 더 많은 정보가 출력됩니다.  
> 문제 해결 시에는 상세하게, 운영 시에는 간결하게 설정합니다.

### 로그 레벨 체계

```
TRACE → DEBUG → INFO → WARN → ERROR
(가장 상세)                    (가장 간결)
```

설정한 레벨 **이상**의 로그만 출력됩니다.  
예: `INFO`로 설정하면 INFO, WARN, ERROR만 출력되고 DEBUG, TRACE는 무시됩니다.

### 로그 레벨별 용도

| 레벨      | 출력 내용                                | 사용 상황           |
| --------- | ---------------------------------------- | ------------------- |
| **TRACE** | 모든 내부 동작, 메서드 진입/종료         | 라이브러리 디버깅   |
| **DEBUG** | 요청 URL, 파라미터, SQL 쿼리, 변수 값    | 개발/테스트 환경    |
| **INFO**  | 서비스 시작/종료, 배포 정보, 주요 이벤트 | **프로덕션 기본값** |
| **WARN**  | 잠재적 문제, 더 이상 사용되지 않는 API   | 모니터링 알림 대상  |
| **ERROR** | 예외 발생, 처리 불가능한 오류            | 즉시 대응 필요      |

### Spring Boot에서의 로그 설정

```properties
# application.properties

# 전체 애플리케이션 로그 레벨
logging.level.root=INFO

# 특정 패키지만 DEBUG (요청 처리 과정 확인)
logging.level.org.springframework.web=DEBUG

# 내 프로젝트 코드만 DEBUG
logging.level.com.example.demo=DEBUG

# SQL 쿼리 확인 (JPA 사용 시)
logging.level.org.hibernate.SQL=DEBUG
```

### 실무 가이드

| 환경         | 권장 레벨                              | 이유                            |
| ------------ | -------------------------------------- | ------------------------------- |
| 개발/로컬    | `DEBUG`                                | 요청 흐름과 데이터 확인         |
| 스테이징     | `INFO` + 특정 패키지만 `DEBUG`         | 문제 재현 시 상세 로그 확인     |
| **프로덕션** | **`INFO`** 또는 `WARN`                 | 로그량 최소화, 디스크/성능 보호 |
| 장애 대응    | 일시적으로 `DEBUG`로 변경 후 원인 파악 | 해결 후 반드시 `INFO`로 복원    |

> [!WARNING]
> **프로덕션에서 `DEBUG`를 켜두면:**
>
> - 로그 파일이 수 GB로 빠르게 증가 → 디스크 풀
> - 매 요청마다 수십 줄 출력 → 성능 저하
> - 민감한 정보(파라미터, 헤더)가 로그에 노출 가능
>
> 장애 분석이 끝나면 반드시 `INFO`로 되돌리세요.
