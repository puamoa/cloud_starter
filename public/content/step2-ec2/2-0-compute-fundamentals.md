## 1. 가상화(Virtualization)란?

> [!CONCEPT] 가상화 기술이란
> **가상화**는 하나의 물리 서버 위에 여러 개의 가상 서버(VM)를 실행하는 기술입니다. 물리 하드웨어를 논리적으로 분할하여 각 VM이 독립된 운영체제와 애플리케이션을 실행할 수 있게 합니다. 클라우드 컴퓨팅의 핵심 기반 기술입니다.

### 가상화 이전 vs 이후

```
가상화 이전:                          가상화 이후:
┌──────────────┐                    ┌────────────────────┐
│   App A      │                    │ VM1  │ VM2  │ VM3  │
│   OS         │                    │ App  │ App  │ App  │
│   물리 서버  │                    │ OS   │ OS   │ OS   │
└──────────────┘                    ├──────┴──────┴──────┤
서버 활용률: 10~15%                 │   하이퍼바이저     │
1서버 = 1애플리케이션               │   물리 서버        │
                                    └────────────────────┘
                                    서버 활용률: 60~80%
                                    1서버 = 다수 애플리케이션
```

### 가상화의 이점

| 이점              | 설명                        |
| ----------------- | --------------------------- |
| **리소스 효율성** | 물리 서버 활용률 극대화     |
| **격리**          | VM 간 완전한 독립성 보장    |
| **유연성**        | 필요에 따라 VM 생성/삭제    |
| **스냅샷**        | 특정 시점 상태 저장 및 복원 |
| **마이그레이션**  | VM을 다른 물리 서버로 이동  |

## 2. 하이퍼바이저 (Hypervisor)

> [!CONCEPT] 하이퍼바이저의 역할
> **하이퍼바이저**는 물리 하드웨어 위에서 여러 가상 머신을 생성하고 관리하는 소프트웨어 계층입니다. 각 VM에 CPU, 메모리, 스토리지, 네트워크를 할당하고 격리를 보장합니다.

### 하이퍼바이저 유형

```
Type 1 (Bare-Metal):              Type 2 (Hosted):
┌──────────────────────┐          ┌───────────────────┐
│  VM1  │  VM2  │  VM3 │          │  VM1  │  VM2      │
├───────┴───────┴──────┤          ├───────┴───────────┤
│    하이퍼바이저      │          │    하이퍼바이저   │
├──────────────────────┤          ├───────────────────┤
│    물리 하드웨어     │          │    호스트 OS      │
└──────────────────────┘          ├───────────────────┤
                                  │    물리 하드웨어  │
예: VMware ESXi, Xen, KVM         └───────────────────┘
    AWS Nitro System              예: VirtualBox, VMware Workstation
```

### AWS Nitro System

| 특징              | 설명                                      |
| ----------------- | ----------------------------------------- |
| **전용 하드웨어** | 네트워크, 스토리지를 전용 칩으로 오프로드 |
| **보안**          | 하이퍼바이저 최소화로 공격 표면 감소      |
| **성능**          | 거의 베어메탈 수준의 성능 제공            |
| **혁신**          | 새로운 인스턴스 타입 빠르게 출시 가능     |

## 3. EC2 인스턴스 타입 체계

> [!CONCEPT] EC2 인스턴스 타입 명명 규칙
> EC2 인스턴스 타입은 **패밀리(용도) + 세대 + 크기**로 구성됩니다. 예: `t3.micro`에서 t=패밀리, 3=세대, micro=크기입니다. 워크로드 특성에 맞는 타입을 선택하는 것이 비용 최적화의 핵심입니다.

### 인스턴스 타입 명명 규칙

```
  m  5  a  .  xlarge
  │  │  │     │
  │  │  │     └── 크기: nano < micro < small < medium < large < xlarge < 2xlarge ...
  │  │  │
  │  │  └── 추가 속성: a=AMD, g=Graviton(ARM), n=네트워크 강화, d=로컬 스토리지
  │  │
  │  └── 세대: 숫자가 클수록 최신 (성능↑, 가격↓)
  │
  └── 패밀리: 용도별 분류
```

### 주요 인스턴스 패밀리

| 패밀리                  | 용도             | 특징                    | 사용 사례                   |
| ----------------------- | ---------------- | ----------------------- | --------------------------- |
| **t** (범용-버스트)     | 가변 워크로드    | CPU 크레딧 기반 버스팅  | 개발/테스트, 소규모 웹      |
| **m** (범용)            | 균형 잡힌 리소스 | CPU:메모리 = 1:4        | 일반 웹서버, 앱서버         |
| **c** (컴퓨팅 최적화)   | CPU 집약         | 높은 CPU 대 메모리 비율 | 배치 처리, 과학 계산        |
| **r** (메모리 최적화)   | 메모리 집약      | 높은 메모리 대 CPU 비율 | 인메모리 DB, 캐시           |
| **i** (스토리지 최적화) | I/O 집약         | 높은 로컬 스토리지 성능 | NoSQL DB, 데이터 웨어하우스 |
| **g/p** (가속 컴퓨팅)   | GPU 워크로드     | GPU 탑재                | ML 학습, 그래픽 렌더링      |

### t 패밀리 CPU 크레딧 시스템

```
CPU 사용률
100% ─────────────────────────────────────
     │         ▲ 버스트
     │        ╱ ╲
     │       ╱   ╲ 크레딧 소모
 20% │──────╱─────╲──── 기준선(Baseline) ──
     │     ╱       ╲
     │    ╱         ╲ 크레딧 축적
  0% └───────────────────────────────── 시간

t3.micro: 기준선 10%, 크레딧 축적 → 필요 시 100% 버스트
```

## 4. AMI (Amazon Machine Image)

> [!CONCEPT] AMI란 무엇인가
> **AMI**는 EC2 인스턴스를 시작하는 데 필요한 모든 정보를 담은 템플릿입니다. 운영체제, 애플리케이션, 설정이 포함되어 있어 동일한 환경을 빠르게 복제할 수 있습니다.

### AMI 구성 요소

```
AMI
├── 루트 볼륨 템플릿 (OS + 소프트웨어)
├── 시작 권한 (누가 사용 가능한지)
├── 블록 디바이스 매핑 (EBS 볼륨 설정)
└── 아키텍처 (x86_64 또는 arm64)
```

### AMI 유형 비교

| 유형            | 제공자   | 비용                    | 예시                      |
| --------------- | -------- | ----------------------- | ------------------------- |
| **AWS 제공**    | Amazon   | 무료 (OS 라이선스 별도) | Amazon Linux 2023, Ubuntu |
| **Marketplace** | 서드파티 | 유/무료                 | WordPress AMI, NVIDIA     |
| **커뮤니티**    | 커뮤니티 | 무료                    | 다양한 커스텀 이미지      |
| **커스텀**      | 사용자   | 스토리지 비용만         | 내 서버 스냅샷            |

### AMI 활용 워크플로우

```
1. 기본 AMI로 EC2 시작
2. 소프트웨어 설치 및 설정
3. 커스텀 AMI 생성 (이미지 캡처)
4. 커스텀 AMI로 동일 인스턴스 대량 생성
   └── Auto Scaling, 다른 리전 복제 등
```

## 5. EBS (Elastic Block Store)

> [!CONCEPT] EBS 블록 스토리지
> **EBS**는 EC2 인스턴스에 연결하는 네트워크 기반 블록 스토리지입니다. 인스턴스와 독립적으로 존재하며, 인스턴스를 중지해도 데이터가 유지됩니다. 물리 서버의 하드디스크/SSD에 해당합니다.

### EBS 볼륨 타입 비교

| 타입    | 이름              | IOPS         | 처리량          | 사용 사례                 |
| ------- | ----------------- | ------------ | --------------- | ------------------------- |
| **gp3** | 범용 SSD          | 3,000~16,000 | 125~1,000 MB/s  | 일반 워크로드 (기본 추천) |
| **gp2** | 범용 SSD          | 100~16,000   | 최대 250 MB/s   | 레거시 (gp3 권장)         |
| **io2** | 프로비저닝 IOPS   | 최대 64,000  | 최대 1,000 MB/s | 고성능 DB                 |
| **st1** | 처리량 최적화 HDD | 500          | 최대 500 MB/s   | 빅데이터, 로그            |
| **sc1** | 콜드 HDD          | 250          | 최대 250 MB/s   | 아카이브                  |

### EBS 핵심 특성

```
┌─────────────────────────────────────────────────┐
│  EC2 인스턴스 (AZ: ap-northeast-2a)             │
│  ┌───────────┐                                  │
│  │ 인스턴스  │◄──── 네트워크 연결 ──┐           │
│  └───────────┘                      │           │
│                              ┌──────┴──────┐    │
│                              │  EBS 볼륨   │    │
│                              │  (같은 AZ)  │    │
│                              └─────────────┘    │
│                                                 │
│  ⚠️ EBS는 같은 AZ 내에서만 연결 가능            │
│  ✅ 스냅샷으로 다른 AZ/리전으로 복제 가능       │
└─────────────────────────────────────────────────┘
```

## 6. 키 페어와 SSH

> [!CONCEPT] 키 페어와 SSH 접속
> **키 페어**는 공개키 암호화를 이용한 인증 방식입니다. AWS에서 EC2 인스턴스에 접속할 때 비밀번호 대신 **프라이빗 키**(.pem)를 사용하여 안전하게 SSH 접속합니다.

### 키 페어 동작 원리

```
키 페어 생성 시:
┌──────────────────────────────────────────────┐
│  AWS가 키 쌍 생성                            │
│                                              │
│  공개키 (Public Key)  →  EC2 인스턴스에 저장 │
│  개인키 (Private Key) →  사용자가 다운로드   │
│                          (.pem 파일)         │
└──────────────────────────────────────────────┘

SSH 접속 시:
┌───────────┐         SSH (포트 22)         ┌──────────┐
│  사용자   │ ──── 개인키로 인증 요청 ────► │   EC2    │
│  (로컬)   │                               │ (공개키) │
│  .pem     │ ◄──── 인증 성공, 접속 허용 ── │          │
└───────────┘                               └──────────┘
```

### SSH 접속 명령어 구조

```bash
ssh -i "my-key.pem" ec2-user@<Public-IP>
     │       │           │         │
     │       │           │         └── EC2의 공인 IP
     │       │           └── 기본 사용자 (Amazon Linux)
     │       └── 개인키 파일 경로
     └── 키 파일 지정 옵션
```

### OS별 기본 사용자

| AMI          | 기본 사용자 |
| ------------ | ----------- |
| Amazon Linux | ec2-user    |
| Ubuntu       | ubuntu      |
| CentOS       | centos      |
| Debian       | admin       |
| RHEL         | ec2-user    |

## 7. User Data (사용자 데이터)

> [!CONCEPT] User Data 초기화 스크립트
> **User Data**는 EC2 인스턴스가 처음 시작될 때 자동으로 실행되는 스크립트입니다. 소프트웨어 설치, 설정 파일 생성, 서비스 시작 등 초기 구성을 자동화할 수 있습니다. root 권한으로 실행됩니다.

### User Data 실행 흐름

```
EC2 인스턴스 시작
│
├── 1. 하드웨어 초기화
├── 2. OS 부팅
├── 3. cloud-init 실행
│   └── User Data 스크립트 실행 (root 권한)
│       ├── 패키지 업데이트
│       ├── 소프트웨어 설치
│       ├── 설정 파일 생성
│       └── 서비스 시작
└── 4. 인스턴스 Ready
```

### User Data 특성

| 특성          | 설명                           |
| ------------- | ------------------------------ |
| **실행 시점** | 인스턴스 최초 시작 시 1회      |
| **실행 권한** | root                           |
| **크기 제한** | 16KB                           |
| **로그 위치** | /var/log/cloud-init-output.log |
| **인코딩**    | Base64 (콘솔에서 자동 처리)    |

## 8. 인스턴스 수명 주기

> [!CONCEPT] 인스턴스 수명 주기와 과금
> EC2 인스턴스는 생성부터 종료까지 여러 상태를 거칩니다. 각 상태에서의 과금 여부와 데이터 보존 여부를 이해하는 것이 비용 관리의 핵심입니다.

### 인스턴스 상태 전이 다이어그램

```
                    ┌─────────┐
         시작 ────► │ pending │
                    └────┬────┘
                         │
                         ▼
                    ┌──────────┐ ◄──── 재시작(reboot)
         ┌───────   │ running  │ ───────┐
         │          └──────────┘        │
         │               │              │
         │ 중지          │ 종료         │ 최대 절전
         ▼               ▼              ▼
    ┌──────────┐  ┌─────────────┐  ┌────────────┐
    │ stopping │  │shutting-down│  │ hibernating│
    └────┬─────┘  └──────┬──────┘  └────────────┘
         │               │
         ▼               ▼
    ┌──────────┐  ┌────────────┐
    │ stopped  │  │ terminated │
    └──────────┘  └────────────┘
    (재시작 가능)   (복구 불가)
```

### 상태별 과금 및 데이터 보존

| 상태           | EC2 과금 | EBS 과금 | 공인 IP | EBS 데이터 | 인스턴스 스토어 |
| -------------- | :------: | :------: | :-----: | :--------: | :-------------: |
| **running**    |    ✅    |    ✅    |  유지   |    유지    |      유지       |
| **stopped**    |    ❌    |    ✅    |  해제   |    유지    |    **삭제**     |
| **terminated** |    ❌    |   ❌\*   |  해제   | **삭제\*** |    **삭제**     |
| **hibernated** |    ❌    |    ✅    |  유지   |    유지    |    **삭제**     |

> \*terminated 시 EBS는 "종료 시 삭제" 설정에 따라 다름 (기본: 루트 볼륨 삭제)

### 실무 팁: 비용 절약

```
┌─────────────────────────────────────────────────────┐
│  💡 비용 절약 전략                                  │
├─────────────────────────────────────────────────────┤
│                                                     │
│  개발/테스트 환경:                                  │
│  • 퇴근 시 Stop → 출근 시 Start                     │
│  • EC2 과금 중지, EBS만 과금                        │
│                                                     │
│  더 이상 불필요:                                    │
│  • Terminate (EBS도 함께 삭제 설정)                 │
│  • 완전히 과금 중지                                 │
│                                                     │
│  ⚠️ 주의: Stop 시 공인 IP 변경됨                    │
│     → Elastic IP 사용으로 해결                      │
│                                                     │
└─────────────────────────────────────────────────────┘
```

## 핵심 정리

| 개념          | 한 줄 요약                                    |
| ------------- | --------------------------------------------- |
| 가상화        | 물리 서버 위에 여러 가상 서버를 실행하는 기술 |
| 하이퍼바이저  | VM을 생성·관리하는 소프트웨어 (AWS: Nitro)    |
| 인스턴스 타입 | 패밀리+세대+크기로 워크로드에 맞게 선택       |
| AMI           | 인스턴스 시작 템플릿 (OS + 소프트웨어)        |
| EBS           | 네트워크 블록 스토리지 (인스턴스와 독립)      |
| 키 페어       | 공개키 암호화 기반 SSH 인증                   |
| User Data     | 인스턴스 최초 시작 시 자동 실행 스크립트      |
| 수명 주기     | pending → running → stopping/terminated       |
