## 1. 고가용성(HA)이란?

> [!CONCEPT] 고가용성의 정의와 측정
> **고가용성**(High Availability)은 시스템이 장애 상황에서도 지속적으로 서비스를 제공할 수 있는 능력입니다. "다운타임 최소화"가 핵심 목표이며, 일반적으로 가용성을 백분율(%)로 측정합니다.

### 가용성 등급 (Nines)

| 가용성            | 연간 다운타임 | 월간 다운타임 | 등급      |
| ----------------- | ------------- | ------------- | --------- |
| 99% (2 nines)     | 3.65일        | 7.3시간       | 낮음      |
| 99.9% (3 nines)   | 8.76시간      | 43.8분        | 보통      |
| 99.95%            | 4.38시간      | 21.9분        | 높음      |
| 99.99% (4 nines)  | 52.6분        | 4.38분        | 매우 높음 |
| 99.999% (5 nines) | 5.26분        | 26.3초        | 극히 높음 |

### 가용성 계산

```
가용성(%) = (총 시간 - 다운타임) / 총 시간 × 100

예시: 한 달(30일) 동안 2시간 장애 발생
가용성 = (720 - 2) / 720 × 100 = 99.72%

직렬 구성 (A → B):
전체 가용성 = A × B = 99.9% × 99.9% = 99.8%

병렬 구성 (A || B):
전체 가용성 = 1 - (1-A) × (1-B) = 1 - 0.001 × 0.001 = 99.9999%
```

## 2. SLA / SLO / SLI

> [!CONCEPT] SLA, SLO, SLI의 관계
> **SLA**는 서비스 제공자와 고객 간의 계약, **SLO**는 내부 목표, **SLI**는 실제 측정값입니다. SLI로 현재 상태를 측정하고, SLO 목표를 설정하며, SLA로 고객에게 보장합니다.

### 세 개념의 관계

```
┌─────────────────────────────────────────────────────┐
│                                                     │
│  SLI (Service Level Indicator) - 실제 측정값        │
│  "현재 가용성이 99.95%입니다"                       │
│       │                                             │
│       ▼                                             │
│  SLO (Service Level Objective) - 내부 목표          │
│  "가용성 99.9% 이상을 유지하겠다"                   │
│       │                                             │
│       ▼                                             │
│  SLA (Service Level Agreement) - 고객 계약          │
│  "99.9% 미달 시 크레딧 보상"                        │
│                                                     │
│  일반적으로: SLO > SLA (내부 목표가 더 엄격)        │
└─────────────────────────────────────────────────────┘
```

### AWS 주요 서비스 SLA

| 서비스       | SLA           | 미달 시 보상  |
| ------------ | ------------- | ------------- |
| EC2          | 99.99% (리전) | 10~30% 크레딧 |
| RDS Multi-AZ | 99.95%        | 10~25% 크레딧 |
| S3           | 99.9%         | 10~25% 크레딧 |
| Lambda       | 99.95%        | 10~25% 크레딧 |
| CloudFront   | 99.9%         | 10~25% 크레딧 |

## 3. 단일 장애점 (SPOF)

> [!CONCEPT] 단일 장애점 식별과 제거
> **SPOF**(Single Point of Failure)는 해당 구성 요소가 실패하면 전체 시스템이 중단되는 지점입니다. 고가용성 설계의 핵심은 모든 SPOF를 식별하고 이중화(Redundancy)로 제거하는 것입니다.

### SPOF 식별과 제거

```
SPOF가 있는 아키텍처:
┌───────┐      ┌──────┐     ┌──────┐
│사용자 │────► │ EC2  │────►│ RDS  │
└───────┘      │(1대) │     │(1대) │
               └──────┘     └──────┘
               ⚠️ SPOF      ⚠️ SPOF

SPOF를 제거한 아키텍처:
┌───────┐     ┌────────┐     ┌────────┐     ┌───────┐
│사용자 │────►│ ALB    │────►│ EC2 x2 │────►│ RDS   │
└───────┘     │(관리형)│     │ (Multi │     │ Multi │
              └────────┘     │  AZ)   │     │ AZ    │
              ✅ 이중화      └────────┘     └───────┘
                             ✅ 이중화    ✅ 이중화
```

### AWS에서 SPOF 제거 방법

| SPOF      | 해결 방법               | AWS 서비스          |
| --------- | ----------------------- | ------------------- |
| 단일 EC2  | 다중 인스턴스 + LB      | ALB + ASG           |
| 단일 AZ   | Multi-AZ 배포           | 서브넷 분산         |
| 단일 DB   | Multi-AZ + Read Replica | RDS Multi-AZ        |
| 단일 리전 | 멀티 리전 배포          | Route 53 Failover   |
| DNS       | 관리형 DNS 사용         | Route 53 (100% SLA) |
| NAT       | AZ별 NAT Gateway        | NAT Gateway per AZ  |

## 4. 수직 확장 vs 수평 확장

> [!CONCEPT] 수직 확장과 수평 확장
> **수직 확장**(Scale Up)은 단일 서버의 사양을 높이는 것이고, **수평 확장**(Scale Out)은 서버 수를 늘리는 것입니다. 수평 확장이 클라우드 환경에서 더 적합하며, 고가용성과 비용 효율성을 동시에 달성할 수 있습니다.

### 비교 다이어그램

```
수직 확장 (Scale Up):                수평 확장 (Scale Out):
┌───────────────────┐                 ┌─────┐ ┌─────┐ ┌─────┐
│                   │                 │ EC2 │ │ EC2 │ │ EC2 │
│    t3.micro       │                 │  A  │ │  B  │ │  C  │
│       ↓           │                 └──┬──┘ └──┬──┘ └──┬──┘
│    t3.large       │                    │       │       │
│       ↓           │                 ┌──┴───────┴───────┴──┐
│    m5.xlarge      │                 │        ALB          │
│       ↓           │                 └─────────────────────┘
│    m5.4xlarge     │
│                   │                 서버 추가로 처리량 증가
│ 한계: 물리적 최대 │                 이론적으로 무한 확장 가능
└───────────────────┘
```

### 상세 비교

| 항목          | 수직 확장 (Scale Up)  | 수평 확장 (Scale Out) |
| ------------- | --------------------- | --------------------- |
| **방법**      | 인스턴스 타입 변경    | 인스턴스 수 증가      |
| **다운타임**  | 필요 (재시작)         | 불필요                |
| **한계**      | 물리적 최대 사양      | 이론적 무한           |
| **비용**      | 비선형 증가           | 선형 증가             |
| **복잡도**    | 낮음 (코드 변경 없음) | 높음 (상태 관리 필요) |
| **가용성**    | SPOF 유지             | 이중화 자동 확보      |
| **적합 사례** | RDS, 단일 DB          | 웹서버, API 서버      |

## 5. 로드밸런싱 알고리즘

> [!CONCEPT] 로드밸런싱 알고리즘
> **로드밸런싱**은 들어오는 트래픽을 여러 서버에 분배하는 기술입니다. 분배 알고리즘에 따라 각 서버의 부하를 균등하게 유지하며, 특정 서버 장애 시 자동으로 트래픽을 우회합니다.

### 주요 알고리즘

| 알고리즘                 | 동작 방식                  | 장점                 | 단점                |
| ------------------------ | -------------------------- | -------------------- | ------------------- |
| **Round Robin**          | 순서대로 분배              | 구현 간단, 균등 분배 | 서버 성능 차이 무시 |
| **Weighted Round Robin** | 가중치에 따라 분배         | 서버 성능 반영       | 가중치 수동 설정    |
| **Least Connections**    | 연결 수 가장 적은 서버로   | 실시간 부하 반영     | 연결 추적 오버헤드  |
| **IP Hash**              | 클라이언트 IP 기반         | 세션 유지 (Sticky)   | 분배 불균형 가능    |
| **Least Response Time**  | 응답 시간 가장 빠른 서버로 | 최적 성능            | 측정 오버헤드       |

### AWS ALB의 라우팅 방식

```
┌─────────────────────────────────────────────────────┐
│  ALB (Application Load Balancer)                    │
│                                                     │
│  라우팅 규칙:                                       │
│  ┌─────────────────────────────────────────┐        │
│  │ IF path = /api/*     → Target Group A   │        │
│  │ IF path = /admin/*   → Target Group B   │        │
│  │ IF host = api.ex.com → Target Group C   │        │
│  │ DEFAULT              → Target Group D   │        │
│  └─────────────────────────────────────────┘        │
│                                                     │
│  Target Group 내 분배: Round Robin (기본)           │
│  + Slow Start Mode (새 인스턴스 점진적 트래픽 증가) │
└─────────────────────────────────────────────────────┘
```

## 6. Health Check

> [!CONCEPT] Health Check의 동작 원리
> **Health Check**는 로드밸런서가 대상 서버의 정상 여부를 주기적으로 확인하는 메커니즘입니다. 비정상 서버를 자동으로 트래픽 분배에서 제외하여 사용자에게 항상 정상 서버로 요청을 전달합니다.

### Health Check 동작 흐름

```
ALB Health Check 설정:
- 경로: /health
- 간격: 30초
- 임계값: 연속 3회 실패 시 Unhealthy
- 복구: 연속 3회 성공 시 Healthy

┌─────┐    GET /health (30초마다)    ┌───────────┐
│ ALB │ ──────────────────────────►  │  EC2-A    │
│     │ ◄─── 200 OK ──────────────   │ (Healthy) │ ✅ 트래픽 전달
│     │                              └───────────┘
│     │    GET /health (30초마다)    ┌───────────┐
│     │ ──────────────────────────►  │  EC2-B    │
│     │ ◄─── 503 Error ───────────   │(Unhealthy)│ ❌ 트래픽 제외
└─────┘                              └───────────┘
```

### Health Check 설정 항목

| 항목                    | 설명             | 권장값                        |
| ----------------------- | ---------------- | ----------------------------- |
| **Protocol**            | 확인 프로토콜    | HTTP                          |
| **Path**                | 확인 경로        | /health 또는 /actuator/health |
| **Interval**            | 확인 주기        | 30초                          |
| **Timeout**             | 응답 대기 시간   | 5초                           |
| **Healthy Threshold**   | 정상 판정 횟수   | 3회                           |
| **Unhealthy Threshold** | 비정상 판정 횟수 | 3회                           |
| **Success Code**        | 정상 응답 코드   | 200                           |

## 7. Auto Scaling 전략

> [!CONCEPT] Auto Scaling의 동작 방식
> **Auto Scaling**은 트래픽 변화에 따라 서버 수를 자동으로 조절하는 서비스입니다. 트래픽 증가 시 서버를 추가(Scale Out)하고, 감소 시 제거(Scale In)하여 비용과 성능을 최적화합니다.

### Auto Scaling 구성 요소

```
┌─────────────────────────────────────────────────────┐
│  Auto Scaling Group (ASG)                           │
│                                                     │
│  ┌─────────────────────────────────────────┐        │
│  │  Launch Template                        │        │
│  │  • AMI: ami-12345                       │        │
│  │  • Instance Type: t3.medium             │        │
│  │  • Security Group: sg-web               │        │
│  │  • User Data: 앱 시작 스크립트          │        │
│  └─────────────────────────────────────────┘        │
│                                                     │
│  용량 설정:                                         │
│  • Minimum: 2 (최소 인스턴스 수)                    │
│  • Desired: 2 (현재 원하는 수)                      │
│  • Maximum: 10 (최대 인스턴스 수)                   │
│                                                     │
│  스케일링 정책:                                     │
│  • CPU > 70% → +2 인스턴스                          │
│  • CPU < 30% → -1 인스턴스                          │
└─────────────────────────────────────────────────────┘
```

### 스케일링 정책 유형

| 정책                | 동작                          | 사용 사례                |
| ------------------- | ----------------------------- | ------------------------ |
| **Target Tracking** | 지표를 목표값으로 유지        | CPU 50% 유지             |
| **Step Scaling**    | 지표 범위별 다른 조정         | CPU 70%→+1, 90%→+3       |
| **Scheduled**       | 예약된 시간에 조정            | 매일 9시 확장, 18시 축소 |
| **Predictive**      | ML로 트래픽 예측 후 선제 조정 | 패턴이 있는 워크로드     |

### 스케일링 시나리오

```
인스턴스 수
    10 ─────────────────────────────────── Maximum
     │
     │              ┌──────┐
     │         ┌────┘      └────┐
     │    ┌────┘                └────┐
     │────┘                          └──── Desired
     2 ─────────────────────────────────── Minimum
     └──────────────────────────────────── 시간
     06:00  09:00  12:00  15:00  18:00  21:00
            출근    점심    피크    퇴근
```

## 8. DNS와 도메인 구조

> [!CONCEPT] DNS와 도메인 계층 구조
> **DNS**(Domain Name System)는 도메인 이름을 IP 주소로 변환하는 분산 시스템입니다. 도메인은 계층적 구조를 가지며, 오른쪽에서 왼쪽으로 상위→하위 도메인 순서입니다.

### 도메인 계층 구조

```
www.shop.example.com.
 │    │      │     │ │
 │    │      │     │ └── Root Domain (.)
 │    │      │     └──── TLD (Top Level Domain)
 │    │      └────────── SLD (Second Level Domain)
 │    └───────────────── Subdomain
 └────────────────────── Subdomain (호스트)

예시:
api.v2.myservice.co.kr.
 │   │      │      │  │
 │   │      │      │  └── Root
 │   │      │      └──── ccTLD (국가 코드)
 │   │      └────────── SLD
 │   └───────────────── Subdomain
 └────────────────────── Subdomain
```

### Route 53 라우팅 정책

| 정책            | 설명                        | 사용 사례               |
| --------------- | --------------------------- | ----------------------- |
| **Simple**      | 단일 리소스로 라우팅        | 단일 서버               |
| **Weighted**    | 가중치 비율로 분배          | A/B 테스트, 점진적 배포 |
| **Latency**     | 지연 시간 가장 낮은 리전    | 글로벌 서비스           |
| **Failover**    | Primary 장애 시 Secondary   | DR (재해 복구)          |
| **Geolocation** | 사용자 위치 기반            | 지역별 콘텐츠           |
| **Multi-Value** | 여러 IP 반환 + Health Check | 간단한 LB               |

## 9. SSL/TLS 인증서 동작 원리

> [!CONCEPT] SSL/TLS 인증서와 HTTPS
> **SSL/TLS**는 클라이언트와 서버 간 통신을 암호화하는 프로토콜입니다. **인증서**는 서버의 신원을 보증하며, CA(Certificate Authority)가 발급합니다. AWS ACM은 무료로 인증서를 발급하고 자동 갱신합니다.

### TLS Handshake 과정

```
클라이언트 (브라우저)              서버 (ALB)
        │                            │
        │──── ClientHello ──────────►│  지원 암호화 방식 전달
        │                            │
        │◄─── ServerHello ───────────│  선택된 암호화 방식
        │◄─── Certificate ───────────│  서버 인증서 전달
        │◄─── ServerHelloDone ───────│
        │                            │
        │   인증서 검증 (CA 확인)    │
        │   Pre-Master Secret 생성   │
        │                            │
        │──── ClientKeyExchange ────►│  공개키로 암호화된 비밀
        │──── ChangeCipherSpec ─────►│  암호화 시작 알림
        │──── Finished ─────────────►│
        │                            │
        │◄─── ChangeCipherSpec ──────│
        │◄─── Finished ──────────────│
        │                            │
        │◄═══ 암호화된 통신 시작 ═══►│  대칭키로 데이터 암호화
```

### AWS ACM (Certificate Manager)

```
┌─────────────────────────────────────────────────────┐
│  ACM 인증서 발급 및 적용 흐름                       │
│                                                     │
│  1. ACM에서 인증서 요청                             │
│     └── 도메인: *.example.com                       │
│                                                     │
│  2. 도메인 소유권 검증                              │
│     ├── DNS 검증 (CNAME 레코드 추가) ← 권장         │
│     └── 이메일 검증                                 │
│                                                     │
│  3. 인증서 발급 완료                                │
│                                                     │
│  4. ALB/CloudFront에 연결                           │
│     └── HTTPS 리스너에 인증서 지정                  │
│                                                     │
│  5. 자동 갱신 (만료 60일 전)                        │
│     └── DNS 검증 사용 시 완전 자동                  │
└─────────────────────────────────────────────────────┘
```

### HTTP vs HTTPS 비교

| 항목         | HTTP                 | HTTPS                |
| ------------ | -------------------- | -------------------- |

## 핵심 정리

| 개념         | 한 줄 요약                                     |
| ------------ | ---------------------------------------------- |
| 고가용성     | 장애에도 서비스 지속 (99.9%+ 가용성)           |
| SLA/SLO/SLI  | 계약/목표/측정값의 계층 구조                   |
| SPOF         | 전체 장애를 유발하는 단일 지점 → 이중화로 제거 |
| 수평 확장    | 서버 수 증가 (클라우드에 적합)                 |
| 로드밸런싱   | 트래픽을 여러 서버에 분배                      |
| Health Check | 비정상 서버 자동 제외                          |
| Auto Scaling | 트래픽에 따라 서버 수 자동 조절                |
| DNS          | 도메인 → IP 변환 (계층적 구조)                 |
| SSL/TLS      | 통신 암호화 + 서버 인증                        |
