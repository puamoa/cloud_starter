# 고가용성과 확장 전략 이론

---

## 1. 고가용성(HA)이란?

> [!CONCEPT] 고가용성의 정의와 측정
> **고가용성**(High Availability)은 시스템이 장애 상황에서도 지속적으로 서비스를 제공할 수 있는 능력입니다.  
> "다운타임 최소화"가 핵심 목표이며, 일반적으로 가용성을 백분율(%)로 측정합니다.
>
> - 쉽게 말해, 편의점처럼 "24시간 언제 와도 열려있는 상태"를 유지하는 것이 고가용성입니다.
> - 가용성 99.99%는 한 달에 약 4분만 멈출 수 있다는 뜻입니다. 점심시간에 컵라면 끓이는 시간도 안 됩니다.

### 핵심 용어 정리

| 용어                                 | 설명                                                             | AWS 관련                      |
| ------------------------------------ | ---------------------------------------------------------------- | ----------------------------- |
| **고가용성 (High Availability, HA)** | 시스템이 장애 없이 지속적으로 운영되는 능력                      | Multi-AZ 배포                 |
| **SLA (Service Level Agreement)**    | 서비스 제공자가 보장하는 가용성 수준 계약                        | AWS SLA 99.99%                |
| **SLO (Service Level Objective)**    | 내부적으로 설정하는 가용성 목표                                  | 월 99.95% 가동                |
| **SPOF (Single Point of Failure)**   | 장애 시 전체 시스템이 중단되는 단일 지점                         | 단일 AZ, 단일 인스턴스        |
| **로드밸런서 (Load Balancer)**       | 트래픽을 여러 서버에 분산하여 부하를 균등하게 배분하는 장치      | ALB, NLB                      |
| **Health Check**                     | 로드밸런서가 대상 서버의 정상 동작 여부를 주기적으로 확인하는 것 | Target Group Health Check     |
| **Auto Scaling**                     | 트래픽에 따라 서버 수를 자동으로 늘리거나 줄이는 기능            | Amazon EC2 Auto Scaling Group        |
| **수평 확장 (Scale Out)**            | 서버 수를 늘려 처리 능력을 확장                                  | 인스턴스 추가                 |
| **수직 확장 (Scale Up)**             | 서버 사양(CPU, 메모리)을 높여 처리 능력을 확장                   | 인스턴스 타입 변경            |
| **DNS (Domain Name System)**         | 도메인 이름을 IP 주소로 변환하는 시스템                          | Amazon Route 53               |
| **SSL/TLS 인증서**                   | HTTPS 통신을 위한 암호화 인증서                                  | AWS Certificate Manager (ACM) |

### 가용성 등급 (Nines)

| 가용성            | 연간 다운타임 | 월간 다운타임 | 등급      |
| ----------------- | ------------- | ------------- | --------- |
| 99% (2 nines)     | 3.65일        | 7.3시간       | 낮음      |
| 99.9% (3 nines)   | 8.76시간      | 43.8분        | 보통      |
| 99.95%            | 4.38시간      | 21.9분        | 높음      |
| 99.99% (4 nines)  | 52.6분        | 4.38분        | 매우 높음 |
| 99.999% (5 nines) | 5.26분        | 26.3초        | 극히 높음 |

### 가용성 계산

```
가용성(%) = (총 시간 - 다운타임) / 총 시간 × 100

예시: 한 달(30일) 동안 2시간 장애 발생
가용성 = (720 - 2) / 720 × 100 = 99.72%

직렬 구성 (A → B):
전체 가용성 = A × B = 99.9% × 99.9% = 99.8%

병렬 구성 (A || B):
전체 가용성 = 1 - (1-A) × (1-B) = 1 - 0.001 × 0.001 = 99.9999%
```

## 2. SLA / SLO / SLI

> [!CONCEPT] SLA, SLO, SLI의 관계
> **SLA**는 서비스 제공자와 고객 간의 계약, **SLO**는 내부 목표, **SLI**는 실제 측정값입니다.  
> SLI로 현재 상태를 측정하고, SLO 목표를 설정하며, SLA로 고객에게 보장합니다.
>
> - 식당으로 비유하면:  
>   SLA는 "30분 안에 음식 안 나오면 무료" 같은 고객 약속,  
>   SLO는 "20분 안에 나가자"는 주방 내부 목표,  
>   SLI는 실제 측정한 평균 배달 시간입니다.

### 주요 용어

| 용어                              | 설명                                                |
| --------------------------------- | --------------------------------------------------- |
| **SLA (Service Level Agreement)** | 서비스 제공자가 고객에게 보장하는 가용성 수준 계약  |
| **SLO (Service Level Objective)** | 내부적으로 설정하는 가용성 목표 (보통 SLA보다 엄격) |
| **SLI (Service Level Indicator)** | 실제 측정된 가용성 지표 값                          |
| **에러 버짓 (Error Budget)**      | SLO 목표 내에서 허용되는 장애 시간                  |

### 세 개념의 관계

```
┌─────────────────────────────────────────────────────┐
│                                                     │
│  SLI (Service Level Indicator) - 실제 측정값        │
│  "현재 가용성이 99.95%입니다"                       │
│       │                                             │
│       ▼                                             │
│  SLO (Service Level Objective) - 내부 목표          │
│  "가용성 99.9% 이상을 유지하겠다"                   │
│       │                                             │
│       ▼                                             │
│  SLA (Service Level Agreement) - 고객 계약          │
│  "99.9% 미달 시 크레딧 보상"                        │
│                                                     │
│  일반적으로: SLO > SLA (내부 목표가 더 엄격)        │
└─────────────────────────────────────────────────────┘
```

### AWS 주요 서비스 SLA

| 서비스       | SLA           | 미달 시 보상  |
| ------------ | ------------- | ------------- |
| Amazon EC2          | 99.99% (리전) | 10~30% 크레딧 |
| Amazon Amazon RDS Multi-AZ | 99.95%        | 10~25% 크레딧 |
| Amazon S3           | 99.9%         | 10~25% 크레딧 |
| AWS Lambda       | 99.95%        | 10~25% 크레딧 |
| Amazon CloudFront   | 99.9%         | 10~25% 크레딧 |

## 3. 단일 장애점 (SPOF)

> [!CONCEPT] 단일 장애점 식별과 제거
> **SPOF**(Single Point of Failure)는 해당 구성 요소가 실패하면 전체 시스템이 중단되는 지점입니다.  
> 고가용성 설계의 핵심은 모든 SPOF를 식별하고 이중화(Redundancy)로 제거하는 것입니다.
>
> - 엘리베이터가 1대뿐인 고층 건물을 떠올려 보세요. 그 엘리베이터가 고장 나면 모든 사람이 계단을 걸어야 합니다. 이것이 SPOF입니다.
> - 해결책은 간단합니다. 엘리베이터를 2대 이상 설치하는 것(이중화)처럼, 서버도 여러 대를 두는 것입니다.

### 주요 용어

| 용어                               | 설명                                                  |
| ---------------------------------- | ----------------------------------------------------- |
| **SPOF (Single Point of Failure)** | 장애 시 전체 시스템이 중단되는 단일 지점              |
| **이중화 (Redundancy)**            | 동일 구성 요소를 복수로 배치하여 장애에 대비하는 설계 |
| **Failover**                       | 장애 발생 시 대기 시스템으로 자동 전환되는 과정       |

### SPOF 식별과 제거

```
SPOF가 있는 아키텍처:
┌───────┐      ┌──────┐     ┌──────┐
│사용자 │────► │ EC2  │────►│ RDS  │
└───────┘      │(1대) │     │(1대) │
               └──────┘     └──────┘
               ⚠️ SPOF      ⚠️ SPOF

SPOF를 제거한 아키텍처:
┌───────┐     ┌────────┐     ┌────────┐     ┌───────┐
│사용자 │────►│ ALB    │────►│ EC2 x2 │────►│ RDS   │
└───────┘     │(관리형)│     │ (Multi │     │ Multi │
              └────────┘     │  AZ)   │     │ AZ    │
              ✅ 이중화      └────────┘     └───────┘
                             ✅ 이중화    ✅ 이중화
```

### AWS에서 SPOF 제거 방법

| SPOF      | 해결 방법               | AWS 서비스          |
| --------- | ----------------------- | ------------------- |
| 단일 Amazon EC2  | 다중 인스턴스 + LB      | ALB + ASG           |
| 단일 AZ   | Multi-AZ 배포           | 서브넷 분산         |
| 단일 DB   | Multi-AZ + Read Replica | Amazon Amazon RDS Multi-AZ        |
| 단일 리전 | 멀티 리전 배포          | Amazon Route 53 Failover   |
| DNS       | 관리형 DNS 사용         | Amazon Route 53 (100% SLA) |
| NAT       | AZ별 NAT Gateway        | NAT Gateway per AZ  |

## 4. 수직 확장 vs 수평 확장

> [!CONCEPT] 수직 확장과 수평 확장
> **수직 확장**(Scale Up)은 단일 서버의 사양을 높이는 것이고, **수평 확장**(Scale Out)은 서버 수를 늘리는 것입니다.  
> 수평 확장이 클라우드 환경에서 더 적합하며, 고가용성과 비용 효율성을 동시에 달성할 수 있습니다.
>
> - 수직 확장은 혼자 일하는 직원에게 더 빠른 컴퓨터를 사주는 것이고, 수평 확장은 직원을 더 채용하는 것입니다.
> - 직원 1명이 아무리 유능해도 동시에 전화 10통을 받을 수는 없습니다. 그래서 클라우드에서는 수평 확장을 선호합니다.

### 주요 용어

| 용어                      | 설명                                             |
| ------------------------- | ------------------------------------------------ |
| **수직 확장 (Scale Up)**  | 단일 서버의 사양(CPU, 메모리)을 높이는 확장 방식 |
| **수평 확장 (Scale Out)** | 서버 수를 늘려 처리 능력을 확장하는 방식         |
| **Scale In**              | 서버 수를 줄이는 축소 (수평 확장의 반대)         |

### 비교 다이어그램

```
수직 확장 (Scale Up):                수평 확장 (Scale Out):
┌───────────────────┐                 ┌─────┐ ┌─────┐ ┌─────┐
│                   │                 │ EC2 │ │ EC2 │ │ EC2 │
│    t3.micro       │                 │  A  │ │  B  │ │  C  │
│       ↓           │                 └──┬──┘ └──┬──┘ └──┬──┘
│    t3.large       │                    │       │       │
│       ↓           │                 ┌──┴───────┴───────┴──┐
│    m5.xlarge      │                 │        ALB          │
│       ↓           │                 └─────────────────────┘
│    m5.4xlarge     │
│                   │                 서버 추가로 처리량 증가
│ 한계: 물리적 최대 │                 이론적으로 무한 확장 가능
└───────────────────┘
```

### 상세 비교

| 항목          | 수직 확장 (Scale Up)  | 수평 확장 (Scale Out) |
| ------------- | --------------------- | --------------------- |
| **방법**      | 인스턴스 타입 변경    | 인스턴스 수 증가      |
| **다운타임**  | 필요 (재시작)         | 불필요                |
| **한계**      | 물리적 최대 사양      | 이론적 무한           |
| **비용**      | 비선형 증가           | 선형 증가             |
| **복잡도**    | 낮음 (코드 변경 없음) | 높음 (상태 관리 필요) |
| **가용성**    | SPOF 유지             | 이중화 자동 확보      |
| **적합 사례** | Amazon RDS, 단일 DB          | 웹서버, API 서버      |

## 5. DNS와 도메인 구조

> [!CONCEPT] DNS와 도메인 계층 구조
> **DNS**(Domain Name System)는 도메인 이름을 IP 주소로 변환하는 분산 시스템입니다.  
> 도메인은 계층적 구조를 가지며, 오른쪽에서 왼쪽으로 상위→하위 도메인 순서입니다.
>
> - DNS는 인터넷의 전화번호부입니다. "google.com"이라는 이름을 "142.250.196.110"이라는 실제 주소로 찾아주는 역할을 합니다.
> - 우리가 친구 이름으로 전화를 걸 수 있는 것처럼, DNS 덕분에 IP 숫자를 외우지 않아도 웹사이트에 접속할 수 있습니다.

### 주요 용어

| 용어                          | 설명                                                      |
| ----------------------------- | --------------------------------------------------------- |
| **TLD (Top Level Domain)**    | 최상위 도메인 (.com, .kr, .org 등)                        |
| **SLD (Second Level Domain)** | TLD 바로 아래 도메인 (example.com에서 example)            |
| **Subdomain**                 | SLD 아래에 추가하는 하위 도메인 (api.example.com)         |
| **Route 53**                  | AWS의 관리형 DNS 서비스. 100% SLA 보장                    |
| **라우팅 정책**               | DNS 쿼리에 대한 응답 방식 (Simple, Weighted, Failover 등) |

### 도메인 계층 구조

```
www.shop.example.com.
 │    │      │    │ │
 │    │      │    │ └── Root Domain (.)
 │    │      │    └──── TLD (Top Level Domain)
 │    │      └────────── SLD (Second Level Domain)
 │    └───────────────── Subdomain
 └────────────────────── Subdomain (호스트)

예시:
api.v2.myservice.co.kr.
 │   │      │       │ │
 │   │      │       │ └── Root
 │   │      │       └──── ccTLD (국가 코드)
 │   │      └────────── SLD
 │   └───────────────── Subdomain
 └────────────────────── Subdomain
```

### Amazon Route 53 라우팅 정책

| 정책            | 설명                        | 사용 사례               |
| --------------- | --------------------------- | ----------------------- |
| **Simple**      | 단일 리소스로 라우팅        | 단일 서버               |
| **Weighted**    | 가중치 비율로 분배          | A/B 테스트, 점진적 배포 |
| **Latency**     | 지연 시간 가장 낮은 리전    | 글로벌 서비스           |
| **Failover**    | Primary 장애 시 Secondary   | DR (재해 복구)          |
| **Geolocation** | 사용자 위치 기반            | 지역별 콘텐츠           |
| **Multi-Value** | 여러 IP 반환 + Health Check | 간단한 LB               |

## 6. SSL/TLS 인증서 동작 원리

> [!CONCEPT] SSL/TLS 인증서와 HTTPS
> **SSL/TLS**는 클라이언트와 서버 간 통신을 암호화하는 프로토콜입니다.  
> **인증서**는 서버의 신원을 보증하며, CA(Certificate Authority)가 발급합니다.  
> AWS ACM은 무료로 인증서를 발급하고 자동 갱신합니다.
>
> - TLS Handshake는 택배 수령 시 본인확인 과정과 비슷합니다.  
>    택배기사(브라우저)가 "이 집이 맞나요?"(인증서 확인)  
>   → 수령인이 신분증 보여주고(서버 인증)  
>   → 확인 후 물건 전달(암호화 통신 시작)하는 흐름입니다.
> - HTTP는 엽서(누구나 내용을 볼 수 있음)이고, HTTPS는 밀봉된 편지(암호화됨)라고 생각하면 됩니다.

### 주요 용어

| 용어                              | 설명                                                      |
| --------------------------------- | --------------------------------------------------------- |
| **SSL/TLS**                       | 클라이언트와 서버 간 통신을 암호화하는 프로토콜           |
| **인증서 (Certificate)**          | 서버의 신원을 보증하는 디지털 문서. CA가 발급             |
| **CA (Certificate Authority)**    | 인증서를 발급하는 신뢰된 기관                             |
| **ACM (AWS Certificate Manager)** | AWS에서 무료로 SSL/TLS 인증서를 발급·관리하는 서비스      |
| **TLS Handshake**                 | 클라이언트와 서버가 암호화 통신을 시작하기 위한 협상 과정 |

### TLS Handshake 과정

```
클라이언트 (브라우저)              서버 (ALB)
        │                            │
        │──── ClientHello ──────────►│  지원 암호화 방식 전달
        │                            │
        │◄─── ServerHello ───────────│  선택된 암호화 방식
        │◄─── Certificate ───────────│  서버 인증서 전달
        │◄─── ServerHelloDone ───────│
        │                            │
        │   인증서 검증 (CA 확인)    │
        │   Pre-Master Secret 생성   │
        │                            │
        │──── ClientKeyExchange ────►│  공개키로 암호화된 비밀
        │──── ChangeCipherSpec ─────►│  암호화 시작 알림
        │──── Finished ─────────────►│
        │                            │
        │◄─── ChangeCipherSpec ──────│
        │◄─── Finished ──────────────│
        │                            │
        │◄═══ 암호화된 통신 시작 ═══►│  대칭키로 데이터 암호화
```

### AWS ACM (Certificate Manager)

```
┌─────────────────────────────────────────────────────┐
│  ACM 인증서 발급 및 적용 흐름                       │
│                                                     │
│  1. ACM에서 인증서 요청                             │
│     └── 도메인: *.example.com                       │
│                                                     │
│  2. 도메인 소유권 검증                              │
│     ├── DNS 검증 (CNAME 레코드 추가) ← 권장         │
│     └── 이메일 검증                                 │
│                                                     │
│  3. 인증서 발급 완료                                │
│                                                     │
│  4. ALB/CloudFront에 연결                           │
│     └── HTTPS 리스너에 인증서 지정                  │
│                                                     │
│  5. 자동 갱신 (만료 60일 전)                        │
│     └── DNS 검증 사용 시 완전 자동                  │
└─────────────────────────────────────────────────────┘
```

### HTTP vs HTTPS 비교

| 항목         | HTTP                 | HTTPS                |
| ------------ | -------------------- | -------------------- |
| **포트**     | 80                   | 443                  |
| **암호화**   | ❌ 평문 전송         | ✅ TLS 암호화        |
| **인증서**   | 불필요               | 필요 (ACM 무료)      |
| **SEO**      | 불이익               | 가산점               |
| **브라우저** | "안전하지 않음" 경고 | 🔒 자물쇠 표시       |
| **성능**     | 약간 빠름            | HTTP/2로 오히려 빠름 |

## 7. 로드밸런싱 알고리즘

> [!CONCEPT] 로드밸런싱 알고리즘
> **로드밸런싱**은 들어오는 트래픽을 여러 서버에 분배하는 기술입니다.  
> 분배 알고리즘에 따라 각 서버의 부하를 균등하게 유지하며, 특정 서버 장애 시 자동으로 트래픽을 우회합니다.
>
> - 은행 창구를 떠올려 보세요. "다음 번호 고객님~" 하고 비어있는 창구로 안내하는 직원이 바로 로드밸런서입니다.
> - 한 창구에만 줄을 서면 느려지지만, 여러 창구에 골고루 분배하면 모두가 빠르게 처리됩니다.

### 주요 용어

| 용어                            | 설명                                              |
| ------------------------------- | ------------------------------------------------- |
| **로드밸런싱 (Load Balancing)** | 들어오는 트래픽을 여러 서버에 분배하는 기술       |
| **Round Robin**                 | 서버에 순서대로 요청을 분배하는 알고리즘          |
| **Least Connections**           | 현재 연결 수가 가장 적은 서버로 분배하는 알고리즘 |
| **Sticky Session**              | 같은 클라이언트를 항상 같은 서버로 보내는 설정    |
| **Target Group**                | ALB가 트래픽을 분배하는 대상 서버 그룹            |

### 주요 알고리즘

| 알고리즘                 | 동작 방식                  | 장점                 | 단점                |
| ------------------------ | -------------------------- | -------------------- | ------------------- |
| **Round Robin**          | 순서대로 분배              | 구현 간단, 균등 분배 | 서버 성능 차이 무시 |
| **Weighted Round Robin** | 가중치에 따라 분배         | 서버 성능 반영       | 가중치 수동 설정    |
| **Least Connections**    | 연결 수 가장 적은 서버로   | 실시간 부하 반영     | 연결 추적 오버헤드  |
| **IP Hash**              | 클라이언트 IP 기반         | 세션 유지 (Sticky)   | 분배 불균형 가능    |
| **Least Response Time**  | 응답 시간 가장 빠른 서버로 | 최적 성능            | 측정 오버헤드       |

### AWS ALB의 라우팅 방식

```
┌─────────────────────────────────────────────────────┐
│  ALB (Application Load Balancer)                    │
│                                                     │
│  라우팅 규칙:                                       │
│  ┌─────────────────────────────────────────┐        │
│  │ IF path = /api/*     → Target Group A   │        │
│  │ IF path = /admin/*   → Target Group B   │        │
│  │ IF host = api.ex.com → Target Group C   │        │
│  │ DEFAULT              → Target Group D   │        │
│  └─────────────────────────────────────────┘        │
│                                                     │
│  Target Group 내 분배: Round Robin (기본)           │
│  + Slow Start Mode (새 인스턴스 점진적 트래픽 증가) │
└─────────────────────────────────────────────────────┘
```

## 8. Health Check

> [!CONCEPT] Health Check의 동작 원리
> **Health Check**는 로드밸런서가 대상 서버의 정상 여부를 주기적으로 확인하는 메커니즘입니다.  
> 비정상 서버를 자동으로 트래픽 분배에서 제외하여 사용자에게 항상 정상 서버로 요청을 전달합니다.
>
> - 식당 매니저가 30초마다 각 주방 직원에게 "괜찮아? 주문 받을 수 있어?"라고 확인하는 것과 같습니다.
> - 3번 연속 "못해요"라고 답하면 그 직원에게는 더 이상 주문을 보내지 않고, 다시 "할 수 있어요"가 3번 나와야 복귀시킵니다.

### 주요 용어

| 용어                    | 설명                                                              |
| ----------------------- | ----------------------------------------------------------------- |
| **Health Check**        | 로드밸런서가 대상 서버의 정상 여부를 주기적으로 확인하는 메커니즘 |
| **Healthy / Unhealthy** | Health Check 결과에 따른 서버 상태 (정상/비정상)                  |
| **Healthy Threshold**   | 정상 판정에 필요한 연속 성공 횟수                                 |
| **Unhealthy Threshold** | 비정상 판정에 필요한 연속 실패 횟수                               |

### Health Check 동작 흐름

```
ALB Health Check 설정:
- 경로: /health
- 간격: 30초
- 임계값: 연속 3회 실패 시 Unhealthy
- 복구: 연속 3회 성공 시 Healthy

┌─────┐    GET /health (30초마다)    ┌───────────┐
│ ALB │ ──────────────────────────►  │  EC2-A    │
│     │ ◄─── 200 OK ──────────────   │ (Healthy) │ ✅ 트래픽 전달
│     │                              └───────────┘
│     │    GET /health (30초마다)    ┌───────────┐
│     │ ──────────────────────────►  │  EC2-B    │
│     │ ◄─── 503 Error ───────────   │(Unhealthy)│ ❌ 트래픽 제외
└─────┘                              └───────────┘
```

### Health Check 설정 항목

| 항목                    | 설명             | 권장값                        |
| ----------------------- | ---------------- | ----------------------------- |
| **Protocol**            | 확인 프로토콜    | HTTP                          |
| **Path**                | 확인 경로        | /health 또는 /actuator/health |
| **Interval**            | 확인 주기        | 30초                          |
| **Timeout**             | 응답 대기 시간   | 5초                           |
| **Healthy Threshold**   | 정상 판정 횟수   | 3회                           |
| **Unhealthy Threshold** | 비정상 판정 횟수 | 3회                           |
| **Success Code**        | 정상 응답 코드   | 200                           |

## 9. Auto Scaling 전략

> [!CONCEPT] Auto Scaling의 동작 방식
> **Auto Scaling**은 트래픽 변화에 따라 서버 수를 자동으로 조절하는 서비스입니다.  
> 트래픽 증가 시 서버를 추가(Scale Out)하고, 감소 시 제거(Scale In)하여 비용과 성능을 최적화합니다.
>
> - 식당 알바를 생각해 보세요. 점심시간에 손님이 몰리면 알바생을 더 부르고, 한가한 오후에는 일부를 퇴근시킵니다.  
>   Auto Scaling은 이것을 자동으로 해주는 "스마트 매니저"입니다.
> - Minimum=2는 "최소 2명은 항상 근무", Maximum=10은 "아무리 바빠도 10명까지만 부를 수 있다"는 뜻입니다.

### 주요 용어

| 용어                         | 설명                                                       |
| ---------------------------- | ---------------------------------------------------------- |
| **Auto Scaling Group (ASG)** | 인스턴스 수를 자동으로 조절하는 그룹                       |
| **Launch Template**          | Auto Scaling에서 인스턴스를 생성할 때 사용하는 설정 템플릿 |
| **Desired Capacity**         | 현재 유지하고자 하는 인스턴스 수                           |
| **Target Tracking**          | 지표(CPU 등)를 목표값으로 유지하는 스케일링 정책           |
| **Predictive Scaling**       | ML로 트래픽을 예측하여 선제적으로 확장하는 정책            |

### Auto Scaling 구성 요소

```
┌─────────────────────────────────────────────────────┐
│  Auto Scaling Group (ASG)                           │
│                                                     │
│  ┌─────────────────────────────────────────┐        │
│  │  Launch Template                        │        │
│  │  • AMI: ami-12345                       │        │
│  │  • Instance Type: t3.medium             │        │
│  │  • Security Group: sg-web               │        │
│  │  • User Data: 앱 시작 스크립트          │        │
│  └─────────────────────────────────────────┘        │
│                                                     │
│  용량 설정:                                         │
│  • Minimum: 2 (최소 인스턴스 수)                    │
│  • Desired: 2 (현재 원하는 수)                      │
│  • Maximum: 10 (최대 인스턴스 수)                   │
│                                                     │
│  스케일링 정책:                                     │
│  • CPU > 70% → +2 인스턴스                          │
│  • CPU < 30% → -1 인스턴스                          │
└─────────────────────────────────────────────────────┘
```

### 스케일링 정책 유형

| 정책                | 동작                          | 사용 사례                |
| ------------------- | ----------------------------- | ------------------------ |
| **Target Tracking** | 지표를 목표값으로 유지        | CPU 50% 유지             |
| **Step Scaling**    | 지표 범위별 다른 조정         | CPU 70%→+1, 90%→+3       |
| **Scheduled**       | 예약된 시간에 조정            | 매일 9시 확장, 18시 축소 |
| **Predictive**      | ML로 트래픽 예측 후 선제 조정 | 패턴이 있는 워크로드     |

### 스케일링 시나리오

```
인스턴스 수
    10 ─────────────────────────────────── Maximum
     │
     │              ┌──────┐
     │         ┌────┘      └────┐
     │    ┌────┘                └────┐
     │────┘                          └──── Desired
     2 ─────────────────────────────────── Minimum
     └──────────────────────────────────── 시간
     06:00  09:00  12:00  15:00  18:00  21:00
            출근    점심    피크    퇴근
```
