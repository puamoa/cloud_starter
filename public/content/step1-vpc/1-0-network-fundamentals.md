# 네트워크 기초 이론

---

## 1. IP 주소 (IPv4)

> [!CONCEPT] IP 주소의 개념
> **IP 주소**는 네트워크에서 각 장치를 식별하는 고유한 논리적 주소입니다. IPv4는 32비트로 구성되며, 점(.)으로 구분된 4개의 옥텟(0~255)으로 표현합니다. 예: `192.168.1.100`

### IPv4 주소 구조

```
    192    .   168    .    1     .   100
 ┌────────┐ ┌────────┐ ┌────────┐ ┌────────┐
 │11000000│ │10101000│ │00000001│ │01100100│
 └────────┘ └────────┘ └────────┘ └────────┘
   8비트      8비트      8비트      8비트
          총 32비트 = 약 43억 개 주소
```

### 공인 IP vs 사설 IP

| 구분            | 공인 IP (Public)       | 사설 IP (Private)                     |
| --------------- | ---------------------- | ------------------------------------- |
| **용도**        | 인터넷에서 직접 통신   | 내부 네트워크에서만 사용              |
| **유일성**      | 전 세계에서 유일       | 같은 대역이 여러 네트워크에 존재 가능 |
| **할당**        | ISP/IANA가 관리        | 조직 내부에서 자유롭게 할당           |
| **비용**        | 유료 (AWS: Elastic IP) | 무료                                  |
| **인터넷 접근** | 직접 가능              | NAT 필요                              |

### 사설 IP 대역 (RFC 1918)

| 클래스 | 대역                          | CIDR           | 사용 가능 IP 수 |
| ------ | ----------------------------- | -------------- | --------------- |
| A      | 10.0.0.0 ~ 10.255.255.255     | 10.0.0.0/8     | 약 1,677만 개   |
| B      | 172.16.0.0 ~ 172.31.255.255   | 172.16.0.0/12  | 약 104만 개     |
| C      | 192.168.0.0 ~ 192.168.255.255 | 192.168.0.0/16 | 약 6.5만 개     |

> AWS VPC에서는 주로 **10.0.0.0/16** 대역을 사용합니다.

## 2. CIDR 표기법

> [!CONCEPT] CIDR 표기법
> **CIDR**(Classless Inter-Domain Routing)은 IP 주소와 서브넷 마스크를 슬래시(/) 뒤의 숫자로 간결하게 표현하는 방법입니다. `/숫자`는 네트워크 부분의 비트 수를 의미합니다.

### CIDR 계산 방법

```
IP: 10.0.0.0/16

10.0.0.0 =   00001010   .   00000000   .   00000000   .   00000000
          ├── 네트워크 부분 (16비트) ──┤├── 호스트 부분 (16비트) ──┤

네트워크 부분: 변경 불가 (고정)
호스트 부분:   자유롭게 할당 가능

사용 가능 IP 수 = 2^(32-16) - 2 = 65,534개
  (네트워크 주소 1개 + 브로드캐스트 주소 1개 제외)
```

### 자주 사용하는 CIDR 블록

| CIDR | 서브넷 마스크   | 사용 가능 IP | AWS 사용 사례 |
| ---- | --------------- | ------------ | ------------- |
| /8   | 255.0.0.0       | 16,777,214   | 대규모 조직   |
| /16  | 255.255.0.0     | 65,534       | VPC 기본      |
| /20  | 255.255.240.0   | 4,094        | 대형 서브넷   |
| /24  | 255.255.255.0   | 254          | 일반 서브넷   |
| /28  | 255.255.255.240 | 14           | 소형 서브넷   |
| /32  | 255.255.255.255 | 1            | 단일 호스트   |

### AWS VPC에서의 CIDR 제한

- VPC CIDR: **/16** (최대) ~ **/28** (최소)
- 각 서브넷에서 AWS가 **5개 IP를 예약**:
  - `.0` - 네트워크 주소
  - `.1` - VPC 라우터
  - `.2` - DNS 서버
  - `.3` - 향후 사용 예약
  - `.255` - 브로드캐스트 (VPC에서 미지원이나 예약)

## 3. 서브넷 마스크와 서브넷팅

> [!CONCEPT] 서브넷팅의 원리
> **서브넷팅**은 하나의 큰 네트워크를 여러 개의 작은 네트워크(서브넷)로 분할하는 기술입니다. 이를 통해 네트워크를 논리적으로 분리하고 보안과 관리 효율성을 높입니다.

### 서브넷팅 예시: VPC 10.0.0.0/16 분할

```
VPC: 10.0.0.0/16 (65,534 IPs)
│
├── Public Subnet A:  10.0.1.0/24  (254 IPs) - AZ-a
├── Public Subnet B:  10.0.2.0/24  (254 IPs) - AZ-b
├── Private Subnet A: 10.0.10.0/24 (254 IPs) - AZ-a
├── Private Subnet B: 10.0.20.0/24 (254 IPs) - AZ-b
├── DB Subnet A:      10.0.100.0/24 (254 IPs) - AZ-a
└── DB Subnet B:      10.0.200.0/24 (254 IPs) - AZ-b
```

## 4. 라우팅 개념

> [!CONCEPT] 라우팅의 동작 방식
> **라우팅**은 네트워크 패킷이 출발지에서 목적지까지 도달하는 경로를 결정하는 과정입니다. 라우터는 **라우팅 테이블**을 참조하여 패킷을 다음 홉(Next Hop)으로 전달합니다.

### 라우팅 테이블 예시 (AWS VPC)

| Destination | Target  | 설명                          |
| ----------- | ------- | ----------------------------- |
| 10.0.0.0/16 | local   | VPC 내부 통신 (자동 생성)     |
| 0.0.0.0/0   | igw-xxx | 인터넷으로 나가는 모든 트래픽 |

### 라우팅 동작 원리

```
패킷 목적지: 8.8.8.8 (Google DNS)

1. 라우팅 테이블 검색 (가장 구체적인 경로 우선)
2. 10.0.0.0/16에 해당? → NO
3. 0.0.0.0/0에 해당? → YES → Internet Gateway로 전달

패킷 목적지: 10.0.20.5 (Private Subnet의 EC2)

1. 라우팅 테이블 검색
2. 10.0.0.0/16에 해당? → YES → local (VPC 내부 직접 전달)
```

### Longest Prefix Match (최장 접두사 매칭)

라우팅 테이블에 여러 경로가 매칭될 때, **가장 구체적인(CIDR이 큰) 경로**가 우선합니다:

- `10.0.0.0/16` vs `10.0.1.0/24` → **/24가 우선** (더 구체적)

## 5. DNS 기초

> [!CONCEPT] DNS의 동작 원리
> **DNS**(Domain Name System)는 사람이 읽을 수 있는 도메인 이름(예: www.example.com)을 컴퓨터가 이해하는 IP 주소(예: 93.184.216.34)로 변환하는 시스템입니다.

### DNS 조회 과정

```
사용자 → "www.example.com 접속"
│
├─① 로컬 DNS 캐시 확인
│
├─② ISP DNS 서버 (Recursive Resolver)
│   │
│   ├─③ Root DNS 서버 (.com 담당 서버 알려줌)
│   │
│   ├─④ TLD DNS 서버 (.com → example.com NS 알려줌)
│   │
│   └─⑤ Authoritative DNS 서버 (example.com → IP 반환)
│
└─⑥ IP 주소 93.184.216.34 반환 → 웹서버 접속
```

### 주요 DNS 레코드 타입

| 레코드    | 용도                 | 예시                      |
| --------- | -------------------- | ------------------------- |
| **A**     | 도메인 → IPv4        | example.com → 1.2.3.4     |
| **AAAA**  | 도메인 → IPv6        | example.com → 2001:db8::1 |
| **CNAME** | 도메인 → 다른 도메인 | www → example.com         |
| **MX**    | 메일 서버 지정       | mail.example.com          |
| **NS**    | 네임서버 지정        | ns1.example.com           |
| **TXT**   | 텍스트 정보          | SPF, DKIM 인증            |

## 6. TCP/UDP와 포트

> [!CONCEPT] TCP/UDP와 포트 번호
> **TCP**(Transmission Control Protocol)는 신뢰성 있는 연결 지향 프로토콜이고, **UDP**(User Datagram Protocol)는 빠르지만 신뢰성을 보장하지 않는 비연결 프로토콜입니다. **포트**는 하나의 IP에서 여러 서비스를 구분하는 논리적 번호입니다.

### TCP vs UDP 비교

| 특성          | TCP                         | UDP                 |
| ------------- | --------------------------- | ------------------- |
| **연결 방식** | 연결 지향 (3-way handshake) | 비연결              |
| **신뢰성**    | 보장 (재전송, 순서 보장)    | 미보장              |
| **속도**      | 상대적으로 느림             | 빠름                |
| **사용 사례** | HTTP, SSH, DB 연결          | DNS, 스트리밍, 게임 |
| **헤더 크기** | 20바이트                    | 8바이트             |

### 주요 포트 번호

| 포트 | 프로토콜 | 서비스     | AWS 관련       |
| ---- | -------- | ---------- | -------------- |
| 22   | TCP      | SSH        | EC2 접속       |
| 80   | TCP      | HTTP       | ALB, 웹서버    |
| 443  | TCP      | HTTPS      | ALB + ACM      |
| 3306 | TCP      | MySQL      | RDS MySQL      |
| 5432 | TCP      | PostgreSQL | RDS PostgreSQL |
| 6379 | TCP      | Redis      | ElastiCache    |
| 8080 | TCP      | 대체 HTTP  | Spring Boot    |

## 7. OSI 7계층 모델

> [!CONCEPT] OSI 7계층 모델
> **OSI(Open Systems Interconnection) 모델**은 네트워크 통신을 7개 계층으로 나누어 설명하는 참조 모델입니다. 각 계층은 독립적으로 동작하며, 상위 계층은 하위 계층의 서비스를 이용합니다.

### 7계층 구조와 AWS 매핑

| 계층 | 이름                       | 프로토콜/기술     | AWS 서비스                 |
| :--: | -------------------------- | ----------------- | -------------------------- |
|  7   | **응용** (Application)     | HTTP, HTTPS, DNS  | ALB, API Gateway, Route 53 |
|  6   | **표현** (Presentation)    | SSL/TLS, 암호화   | ACM, KMS                   |
|  5   | **세션** (Session)         | 세션 관리         | -                          |
|  4   | **전송** (Transport)       | TCP, UDP          | NLB, Security Group        |
|  3   | **네트워크** (Network)     | IP, ICMP, 라우팅  | VPC, Subnet, Route Table   |
|  2   | **데이터링크** (Data Link) | MAC, ARP          | ENI                        |
|  1   | **물리** (Physical)        | 케이블, 전기 신호 | AWS 물리 인프라            |

### AWS 로드밸런서와 OSI 계층

```
┌─────────────────────────────────────────────┐
│  ALB (Application Load Balancer) - Layer 7  │
│  • HTTP/HTTPS 헤더 기반 라우팅              │
│  • URL 경로, 호스트 기반 분배               │
│  • WebSocket 지원                           │
├─────────────────────────────────────────────┤
│  NLB (Network Load Balancer) - Layer 4      │
│  • TCP/UDP 포트 기반 라우팅                 │
│  • 초고성능, 고정 IP                        │
│  • 수백만 요청/초 처리                      │
└─────────────────────────────────────────────┘
```

## 8. VPC가 필요한 이유

> [!CONCEPT] VPC란 무엇인가
> **VPC**(Virtual Private Cloud)는 AWS 클라우드 내에서 논리적으로 격리된 가상 네트워크입니다. 다른 AWS 고객의 리소스와 완전히 분리되며, IP 대역, 서브넷, 라우팅, 보안을 직접 제어할 수 있습니다.

### VPC 없이 vs VPC 사용

```
VPC 없는 환경 (Classic EC2 - 과거):
┌──────────────────────────────────────────────┐
│  AWS 공유 네트워크                           │
│  ┌───┐ ┌───┐ ┌───┐ ┌───┐                     │
│  │ A │ │ B │ │ C │ │ D │ ← 모두 같은 네트워크│
│  └───┘ └───┘ └───┘ └───┘                     │
│  (격리 불가, 보안 취약)                      │
└──────────────────────────────────────────────┘

VPC 사용 환경 (현재):
┌─────────────────────────────────────┐
│  ┌─── VPC A ───┐  ┌─── VPC B ───┐   │
│  │ ┌───┐ ┌───┐ │  │ ┌───┐ ┌───┐ │   │
│  │ │EC2│ │RDS│ │  │ │EC2│ │S3 │ │   │
│  │ └───┘ └───┘ │  │ └───┘ └───┘ │   │
│  │  완전 격리  │  │  완전 격리  │   │
│  └─────────────┘  └─────────────┘   │
└─────────────────────────────────────┘
```

### VPC의 핵심 구성 요소

| 구성 요소            | 역할                            |
| -------------------- | ------------------------------- |
| **CIDR Block**       | VPC의 IP 주소 범위 정의         |
| **Subnet**           | VPC를 AZ별로 분할               |
| **Internet Gateway** | VPC ↔ 인터넷 연결               |
| **NAT Gateway**      | Private → 인터넷 (단방향)       |
| **Route Table**      | 트래픽 경로 결정                |
| **Security Group**   | 인스턴스 레벨 방화벽 (Stateful) |
| **NACL**             | 서브넷 레벨 방화벽 (Stateless)  |

## 핵심 정리

| 개념      | 한 줄 요약                            |
| --------- | ------------------------------------- |
| IP 주소   | 네트워크 장치의 고유 식별자 (32비트)  |
| CIDR      | IP 범위를 슬래시 표기로 간결하게 표현 |
| 서브넷팅  | 큰 네트워크를 작은 단위로 분할        |
| 라우팅    | 패킷의 목적지까지 경로 결정           |
| DNS       | 도메인 이름 → IP 주소 변환            |
| TCP/UDP   | 신뢰성(TCP) vs 속도(UDP) 트레이드오프 |
| OSI 7계층 | 네트워크 통신의 표준 참조 모델        |
| VPC       | AWS에서 격리된 가상 네트워크          |
