## 1. IP 주소 (IPv4)

> [!CONCEPT] IP 주소의 개념
> **IP 주소**는 네트워크에서 각 장치를 식별하는 고유한 논리적 주소입니다.  
> IPv4는 32비트로 구성되며, 점(.)으로 구분된 4개의 옥텟(0~255)으로 표현합니다. 예: `192.168.1.100`

### 주요 용어

| 용어                     | 설명                                                     |
| ------------------------ | -------------------------------------------------------- |
| **IP 주소 (IP Address)** | 네트워크에서 각 장치를 식별하는 고유한 논리적 주소       |
| **IPv4**                 | 32비트(4옥텟)로 구성된 IP 주소 체계. 약 43억 개 주소     |
| **공인 IP (Public IP)**  | 인터넷에서 직접 통신 가능한 전 세계 유일 주소            |
| **사설 IP (Private IP)** | 내부 네트워크에서만 유효한 주소. 인터넷 접근 시 NAT 필요 |
| **옥텟 (Octet)**         | IP 주소를 구성하는 8비트 단위 (0~255 범위)               |
| **RFC 1918**             | 사설 IP 대역을 정의한 국제 표준 문서                     |

### IPv4 주소 구조

```
       192     .      168     .       1      .     100
 ┌────────────┐ ┌────────────┐ ┌────────────┐ ┌────────────┐
 │  11000000  │ │  10101000  │ │  00000001  │ │  01100100  │
 └────────────┘ └────────────┘ └────────────┘ └────────────┘
      8비트          8비트          8비트         8비트
          총 32비트 = 약 43억 개 주소
```

### IPv4 vs IPv6 비교

| 항목          | IPv4                              | IPv6                                     |
| ------------- | --------------------------------- | ---------------------------------------- |
| **주소 길이** | 32비트 (4바이트)                  | 128비트 (16바이트)                       |
| **표기법**    | 10진수 4개 (192.168.1.1)          | 16진수 8그룹 (2001:0db8::1)              |
| **주소 수**   | 약 43억 개                        | 약 340간 개 (사실상 무한)                |
| **NAT 필요**  | 필요 (주소 부족으로 사설 IP 사용) | 불필요 (모든 장치에 고유 주소 할당 가능) |
| **AWS VPC**   | 기본 지원 (실습에서 사용)         | 선택적 지원 (듀얼 스택)                  |

> [!NOTE]
> 이 가이드의 모든 실습은 **IPv4**를 사용합니다. AWS VPC는 IPv4와 IPv6를 동시에 사용하는 듀얼 스택을 지원하지만, 학습 단계에서는 IPv4만으로 충분합니다.

> [!TIP]
> **실무에서 IPv4와 IPv6를 혼용하는 이유:**
>
> - IPv4 주소는 이미 고갈되었지만, 대부분의 기존 시스템과 네트워크 장비가 IPv4 기반으로 동작
> - IPv6로 완전 전환하려면 모든 장비·소프트웨어·ISP가 지원해야 하므로 수십 년이 걸림
> - 그래서 실무에서는 **듀얼 스택**(IPv4 + IPv6 동시 운영)으로 점진적 전환 중
> - AWS에서는 ALB, CloudFront 등이 IPv6를 지원하여 모바일/최신 클라이언트에 대응하면서, 내부 VPC 통신은 IPv4로 유지하는 패턴이 일반적

### 공인 IP vs 사설 IP

| 구분            | 공인 IP (Public)       | 사설 IP (Private)                     |
| --------------- | ---------------------- | ------------------------------------- |
| **용도**        | 인터넷에서 직접 통신   | 내부 네트워크에서만 사용              |
| **유일성**      | 전 세계에서 유일       | 같은 대역이 여러 네트워크에 존재 가능 |
| **할당**        | ISP/IANA가 관리        | 조직 내부에서 자유롭게 할당           |
| **비용**        | 유료 (AWS: Elastic IP) | 무료                                  |
| **인터넷 접근** | 직접 가능              | NAT 필요                              |

### 사설 IP 대역 (RFC 1918)

| 클래스 | 대역                          | CIDR           | 사용 가능 IP 수 |
| ------ | ----------------------------- | -------------- | --------------- |
| A      | 10.0.0.0 ~ 10.255.255.255     | 10.0.0.0/8     | 약 1,677만 개   |
| B      | 172.16.0.0 ~ 172.31.255.255   | 172.16.0.0/12  | 약 104만 개     |
| C      | 192.168.0.0 ~ 192.168.255.255 | 192.168.0.0/16 | 약 6.5만 개     |

> AWS VPC에서는 주로 **10.0.0.0/16** 대역을 사용합니다.

> [!NOTE]
> **왜 사설 IP를 사용할까?**
>
> IPv4 주소는 약 43억 개뿐인데, 전 세계 인터넷 연결 장치는 이미 수백억 대를 넘었습니다.  
> 모든 장치에 공인 IP를 부여하는 것은 물리적으로 불가능합니다.  
> 이 문제를 해결하기 위해 RFC 1918에서 사설 IP 대역을 정의했고, 핵심 이유는 다음과 같습니다:
>
> 1. **주소 절약**: 하나의 공인 IP 뒤에 수천 대의 장치를 사설 IP로 연결할 수 있어 공인 IP 소모를 극적으로 줄입니다.
> 2. **보안 강화**: 사설 IP는 인터넷에서 직접 접근할 수 없으므로, 외부 공격으로부터 내부 장치를 자연스럽게 보호합니다.
> 3. **자유로운 설계**: 조직 내부에서 IP 대역을 자유롭게 할당·변경할 수 있어 네트워크 설계 유연성이 높습니다.
> 4. **비용 절감**: 공인 IP는 유료(AWS Elastic IP: 약 $3.6/월)이지만, 사설 IP는 무료로 무제한 사용 가능합니다.
>
> AWS VPC도 이 원리를 그대로 활용합니다.  
> EC2 인스턴스에는 사설 IP를 부여하고, 인터넷 통신이 필요한 경우에만 NAT Gateway나 Elastic IP를 통해 공인 IP로 변환합니다.

> [!TIP]
> **일상 속 사설 IP와 NAT: 가정용 공유기**
>
> 집에서 사용하는 Wi-Fi 공유기가 바로 NAT 장치입니다.  
> ISP(인터넷 서비스 제공자)가 공유기에 공인 IP 1개를 할당하면, 공유기는 연결된 스마트폰, 노트북, TV 등에 사설 IP(예: 192.168.0.2, 192.168.0.3...)를 부여합니다.  
> 내부 기기가 인터넷에 접속할 때 공유기가 사설 IP를 공인 IP로 변환(NAT)해주므로, 공인 IP 1개로 여러 기기가 동시에 인터넷을 사용할 수 있습니다.
>
> AWS VPC에서도 동일한 원리입니다.  
> Private Subnet의 인스턴스는 사설 IP만 가지고 있고, 인터넷에 접근하려면 NAT Gateway가 공유기 역할을 합니다.

## 2. CIDR 표기법

> [!CONCEPT] CIDR 표기법
> **CIDR**(Classless Inter-Domain Routing)은 IP 주소와 서브넷 마스크를 슬래시(/) 뒤의 숫자로 간결하게 표현하는 방법입니다.  
> `/숫자`는 네트워크 부분의 비트 수를 의미합니다.

### Classful vs Classless: IT에서 ~ful과 ~less의 의미

IT 용어에서 **~ful**은 "~이 있는/~을 따르는", **~less**는 "~이 없는/~에서 자유로운"을 의미합니다.

| 용어 패턴 | 의미                      | 네트워크 예시                      | 다른 IT 예시                     |
| --------- | ------------------------- | ---------------------------------- | -------------------------------- |
| **~ful**  | 해당 개념이 존재/적용됨   | Classful = 클래스 체계를 따름      | Stateful = 상태를 기억함         |
| **~less** | 해당 개념이 없음/자유로움 | Classless = 클래스 체계에서 벗어남 | Stateless = 상태를 기억하지 않음 |

```
IT에서 자주 보는 ~ful / ~less 패턴:

  Stateful  (상태 있음) ↔ Stateless  (상태 없음)
  • Security Group (Stateful): 요청을 기억 → 응답 자동 허용
  • NACL (Stateless): 각 패킷을 독립 판단 → 인/아웃 각각 설정

  Classful  (클래스 있음) ↔ Classless  (클래스 없음)
  • Classful: IP를 A/B/C 클래스로 고정 분류
  • Classless (CIDR): 클래스 무시, 자유롭게 네트워크 크기 지정

  Serverful (서버 있음) ↔ Serverless (서버 없음)
  • EC2 (Serverful): 서버를 직접 관리
  • Lambda (Serverless): 서버 관리 없이 코드만 실행

  Connectionful ↔ Connectionless
  • TCP (Connection-oriented): 연결 수립 후 통신
  • UDP (Connectionless): 연결 없이 바로 전송
```

### Classful 네트워크 (과거 방식)

초기 인터넷에서는 IP 주소를 **클래스(A, B, C)**로 고정 분류했습니다:

| 클래스 | 첫 옥텟 범위 | 기본 서브넷 마스크  | 네트워크 수 | 호스트 수/네트워크 |
| ------ | ------------ | ------------------- | ----------- | ------------------ |
| A      | 1 ~ 126      | 255.0.0.0 (/8)      | 126개       | 약 1,677만 개      |
| B      | 128 ~ 191    | 255.255.0.0 (/16)   | 16,384개    | 약 6.5만 개        |
| C      | 192 ~ 223    | 255.255.255.0 (/24) | 약 209만 개 | 254개              |

```
Classful의 문제점:

  회사에 500대의 장치가 있다면?

  • Class C (/24) = 254개 → 부족!
  • Class B (/16) = 65,534개 → 너무 많음! (65,034개 낭비)

  → 딱 맞는 크기를 선택할 수 없어 IP 주소가 대량 낭비됨
  → 이 문제를 해결하기 위해 Classless(CIDR)가 등장
```

### Classless 네트워크 (현재 방식 = CIDR)

CIDR은 클래스의 고정된 경계를 무시하고, **/숫자**로 네트워크 크기를 자유롭게 지정합니다:

```
Classless(CIDR)의 장점:

  회사에 500대의 장치가 있다면?

  • /23 = 510개 → 딱 맞음! (낭비 최소화)
  • /22 = 1,022개 → 여유 있게 사용

  → 필요한 만큼만 할당하여 IP 주소 절약
  → AWS VPC도 CIDR 방식을 사용 (/16 ~ /28 자유 선택)
```

> [!NOTE]
> AWS VPC는 **Classless(CIDR)** 방식만 사용합니다.  
> 클래스 개념은 사설 IP 대역 이름(Class A: 10.x, Class B: 172.16.x, Class C: 192.168.x)에 흔적이 남아있지만, 실제 네트워크 크기는 CIDR로 자유롭게 지정합니다.  
> 예: `10.0.0.0/16`은 Class A 대역이지만 /16(Class B 크기)으로 사용 — 이것이 Classless의 핵심입니다.

### 핵심 용어 정리

> [!NOTE]
> 이 섹션의 모든 내용은 **IPv4** 기준입니다. IPv6는 128비트 주소 체계로 표기법과 구조가 다르지만, AWS VPC 실습에서는 IPv4를 주로 사용합니다.

| 용어                  | 설명                                                                                 | 예시 (10.0.1.0/24 기준)     |
| --------------------- | ------------------------------------------------------------------------------------ | --------------------------- |
| **IP 주소**           | 네트워크에서 장치를 식별하는 32비트 주소. 네트워크 부분 + 호스트 부분으로 구성       | `10.0.1.50`                 |
| **서브넷 마스크**     | IP에서 네트워크/호스트 부분을 구분하는 값. 1 = 네트워크, 0 = 호스트                  | `255.255.255.0` (/24)       |
| **네트워크 주소**     | 서브넷의 첫 번째 IP. 호스트 부분이 모두 0. 네트워크 자체를 식별 (장치에 할당 불가)   | `10.0.1.0`                  |
| **브로드캐스트 주소** | 서브넷의 마지막 IP. 호스트 부분이 모두 1. 네트워크 내 전체 전송용 (장치에 할당 불가) | `10.0.1.255`                |
| **게이트웨이**        | 다른 네트워크로 나가는 출구 장치. 보통 서브넷의 첫 번째 사용 가능 IP                 | `10.0.1.1` (AWS VPC 라우터) |
| **사용 가능 IP 범위** | 네트워크 주소와 브로드캐스트 주소를 제외한 나머지                                    | `10.0.1.1` ~ `10.0.1.254`   |

```
10.0.1.0/24 서브넷 구조:

  10.0.1.0   → 네트워크 주소 (할당 불가)
  10.0.1.1   → AWS VPC 라우터 (게이트웨이)
  10.0.1.2   → AWS DNS 서버
  10.0.1.3   → AWS 예약
  10.0.1.4   → 첫 번째 사용 가능 IP ← EC2 등에 할당 가능
  ...
  10.0.1.254 → 마지막 사용 가능 IP
  10.0.1.255 → 브로드캐스트 주소 (할당 불가)

  실제 사용 가능: 10.0.1.4 ~ 10.0.1.254 = 251개 (AWS 예약 5개 제외)
```

### CIDR 계산 방법

```
IP: 10.0.0.0/16

/16의 의미: 서브넷 마스크의 왼쪽부터 1이 16개 연속

서브넷 마스크: 11111111.11111111.00000000.00000000
             = 255.255.0.0

10.0.0.0 =   00001010   .   00000000   .   00000000   .   00000000
마스크   =   11111111   .   11111111   .   00000000   .   00000000
          ├── 네트워크 부분 (16비트) ──┤├── 호스트 부분 (16비트) ──┤

네트워크 부분 (1인 비트): 변경 불가 (고정)   → 네트워크 식별
호스트 부분 (0인 비트):   자유롭게 할당 가능 → 개별 장치 식별

사용 가능 IP 수 = 2^(32-16) - 2 = 65,534개
    (네트워크 주소 1개 + 브로드캐스트 주소 1개 제외)
```

### AND(&) 연산으로 네트워크 주소 구하기

라우터가 "이 IP가 같은 네트워크에 속하는가?"를 판단할 때 **IP 주소와 서브넷 마스크를 AND 연산**합니다. 결과가 같으면 같은 네트워크입니다.

```
예: 10.0.1.50이 10.0.1.0/24 네트워크에 속하는가?

  IP 주소:       00001010 . 00000000 . 00000001 . 00110010  (10.0.1.50)
  서브넷 마스크: 11111111 . 11111111 . 11111111 . 00000000  (255.255.255.0)
  ─────────────────────────────────────────────────────────
  AND 결과:      00001010 . 00000000 . 00000001 . 00000000  (10.0.1.0) ← 네트워크 주소
```

> [!CONCEPT] AND 연산 규칙
>
> - `1 AND 1 = 1` (마스크가 1인 비트는 IP 값 그대로 유지)
> - `1 AND 0 = 0`
> - `0 AND 1 = 0` (마스크가 0인 비트는 무조건 0으로 → 호스트 부분 제거)
> - `0 AND 0 = 0`
>
> 즉, 서브넷 마스크의 1인 부분만 남기고 나머지를 0으로 만들어 **네트워크 주소**를 추출합니다.

```
같은 네트워크 판별 예시:

  10.0.1.50  AND 255.255.255.0 = 10.0.1.0
  10.0.1.200 AND 255.255.255.0 = 10.0.1.0  ← 결과 동일 → 같은 네트워크!

  10.0.1.50  AND 255.255.255.0 = 10.0.1.0
  10.0.2.10  AND 255.255.255.0 = 10.0.2.0  ← 결과 다름 → 다른 네트워크 → 라우터 필요
```

> [!TIP]
> AWS VPC에서 이 원리가 적용되는 곳:
>
> - **라우팅 테이블**: 패킷의 목적지 IP와 각 경로의 CIDR을 AND 연산하여 매칭 여부 판단
> - **Security Group**: Source IP가 허용된 CIDR 범위에 속하는지 판단
> - **서브넷 설계**: 같은 서브넷에 속하는 IP 범위를 결정

> [!NOTE]
> `/숫자`가 클수록 네트워크 부분(1)이 많아지고 호스트 부분(0)이 줄어들어 사용 가능 IP가 적어집니다.
>
> - `/16` = 1이 16개 → 호스트 16비트 → 2^16 - 2 = **65,534개**
> - `/24` = 1이 24개 → 호스트 8비트 → 2^8 - 2 = **254개**
> - `/28` = 1이 28개 → 호스트 4비트 → 2^4 - 2 = **14개**

### 자주 사용하는 CIDR 블록

| CIDR | 서브넷 마스크   | 사용 가능 IP | AWS 사용 사례 |
| ---- | --------------- | ------------ | ------------- |
| /8   | 255.0.0.0       | 16,777,214   | 대규모 조직   |
| /16  | 255.255.0.0     | 65,534       | VPC 기본      |
| /20  | 255.255.240.0   | 4,094        | 대형 서브넷   |
| /24  | 255.255.255.0   | 254          | 일반 서브넷   |
| /28  | 255.255.255.240 | 14           | 소형 서브넷   |
| /32  | 255.255.255.255 | 1            | 단일 호스트   |

### AWS VPC에서의 CIDR 제한

- VPC CIDR: **/16** (최대) ~ **/28** (최소)
- 각 서브넷에서 AWS가 **5개 IP를 예약** (서브넷 크기와 무관):
  - 첫 번째 IP (예: 10.0.1.0) - 네트워크 주소
  - 두 번째 IP (예: 10.0.1.1) - VPC 라우터
  - 세 번째 IP (예: 10.0.1.2) - DNS 서버
  - 네 번째 IP (예: 10.0.1.3) - 향후 사용 예약
  - 마지막 IP (예: 10.0.1.255 for /24) - 브로드캐스트 (VPC에서 미지원이나 예약)

> [!NOTE]
> **/16이 최대**인 이유: VPC 하나에 65,534개 IP면 대부분의 워크로드에 충분합니다.  
> 더 큰 네트워크가 필요하면 VPC를 여러 개 만들고 VPC Peering이나 Transit Gateway로 연결합니다.
>
> **/28이 최소**인 이유: AWS 예약 IP 5개를 제외하면 /28은 실제 사용 가능 IP가 11개뿐입니다. 이보다 작으면 실용성이 없습니다.

> [!TIP]
> **실습에서 권장하는 CIDR 설계:**
>
> | 리소스         | CIDR 예시   | 사용 가능 IP | 용도          |
> | -------------- | ----------- | ------------ | ------------- |
> | VPC            | 10.0.0.0/16 | 65,534       | 전체 네트워크 |
> | Public Subnet  | 10.0.1.0/24 | 251 (256-5)  | 웹 서버, ALB  |
> | Private Subnet | 10.0.2.0/24 | 251 (256-5)  | 앱 서버, DB   |
>
> /24 서브넷이면 251개 IP를 사용할 수 있어 학습 및 소규모 서비스에 충분합니다.

## 3. 서브넷 마스크와 서브넷팅

> [!CONCEPT] 서브넷팅의 원리
> **서브넷팅**은 하나의 큰 네트워크를 여러 개의 작은 네트워크(서브넷)로 분할하는 기술입니다.  
> 이를 통해 네트워크를 논리적으로 분리하고 보안과 관리 효율성을 높입니다.

### 주요 용어

| 용어                       | 설명                                                           |
| -------------------------- | -------------------------------------------------------------- |
| **서브넷팅 (Subnetting)**  | 하나의 큰 네트워크를 여러 작은 서브넷으로 분할하는 기술        |
| **서브넷 (Subnet)**        | 분할된 작은 네트워크 단위. VPC 내에서 AZ별로 생성              |
| **Public Subnet**          | 인터넷 접근이 가능한 서브넷 (Route Table에 IGW 경로 존재)      |
| **Private Subnet**         | 인터넷에서 직접 접근 불가한 서브넷 (보안이 필요한 리소스 배치) |
| **AZ (Availability Zone)** | 하나의 리전 내 물리적으로 분리된 데이터센터 그룹               |

### 왜 서브넷팅을 하는가?

하나의 큰 네트워크를 그대로 사용하면 다음과 같은 문제가 발생합니다:

1. **보안 분리 불가**: 웹 서버와 데이터베이스가 같은 네트워크에 있으면, 웹 서버가 해킹당했을 때 DB에 직접 접근 가능
2. **트래픽 관리 어려움**: 모든 리소스가 같은 네트워크에 있으면 트래픽 경로를 세밀하게 제어할 수 없음
3. **장애 격리 불가**: 네트워크 문제가 전체 시스템에 영향을 미침
4. **규정 준수 어려움**: 금융·의료 등 규제 산업에서는 데이터 계층의 네트워크 격리를 요구

서브넷팅으로 이 문제들을 해결합니다:

```
서브넷팅의 효과:

┌─── VPC 10.0.0.0/16 ─────────────────────────────────────┐
│                                                         │
│  ┌─ Public Subnet ──┐     ┌─ Private Subnet ─┐          │
│  │  웹 서버, ALB    │     │  앱 서버, API    │          │
│  │  (인터넷 노출)   │────→│  (내부만 접근)   │          │
│  └──────────────────┘     └────────┬─────────┘          │
│                                    │                    │
│                           ┌────────▼─────────┐          │
│                           │  DB Subnet       │          │
│                           │  RDS, ElastiCache│          │
│                           │  (앱 서버만 접근)│          │
│                           └──────────────────┘          │
└─────────────────────────────────────────────────────────┘

→ 각 계층이 분리되어 있어 웹 서버가 해킹당해도 DB에 직접 접근 불가
→ Security Group으로 계층 간 허용 트래픽을 정밀하게 제어
```

### 서브넷 마스크의 역할

서브넷 마스크는 IP 주소에서 **어디까지가 네트워크이고 어디부터가 호스트인지** 경계를 정합니다.

```
서브넷 마스크에 따른 네트워크 분할 차이:

10.0.0.0/16 (마스크: 255.255.0.0)
→ 네트워크: 10.0 (고정) | 호스트: 0.0~255.255 (65,534개)
→ 하나의 큰 네트워크

10.0.0.0/24 (마스크: 255.255.255.0)
→ 네트워크: 10.0.0 (고정) | 호스트: 0~255 (254개)
→ 256개의 작은 네트워크로 분할 가능

비유: 아파트 주소
/16 = "서울시 강남구" (큰 범위, 수만 가구)
/24 = "서울시 강남구 역삼동 123번지 A아파트" (작은 범위, 254가구)
/28 = "A아파트 3층" (아주 작은 범위, 14가구)
```

### 서브넷팅 계산 실습

```
문제: 10.0.0.0/16 VPC를 /24 서브넷으로 나누면 몇 개?

풀이:
  VPC 호스트 비트:      32 - 16 = 16비트
  서브넷 호스트 비트:   32 - 24 = 8비트
  서브넷용 비트:        16 - 8  = 8비트
  생성 가능 서브넷 수:  2^8     = 256개

  각 서브넷의 IP 범위:
  10.0.0.0/24   → 10.0.0.0 ~ 10.0.0.255
  10.0.1.0/24   → 10.0.1.0 ~ 10.0.1.255
  10.0.2.0/24   → 10.0.2.0 ~ 10.0.2.255
  ...
  10.0.255.0/24 → 10.0.255.0 ~ 10.0.255.255
```

### 서브넷팅 예시: VPC 10.0.0.0/16 분할

```
VPC: 10.0.0.0/16 (65,534 IPs)
│
├── Public Subnet A:  10.0.1.0/24   (251 IPs) - AZ-a  ← 웹서버, ALB
├── Public Subnet B:  10.0.2.0/24   (251 IPs) - AZ-b  ← 웹서버, ALB (이중화)
├── Private Subnet A: 10.0.10.0/24  (251 IPs) - AZ-a  ← 앱서버, API
├── Private Subnet B: 10.0.20.0/24  (251 IPs) - AZ-b  ← 앱서버, API (이중화)
├── DB Subnet A:      10.0.100.0/24 (251 IPs) - AZ-a  ← RDS Primary
└── DB Subnet B:      10.0.200.0/24 (251 IPs) - AZ-b  ← RDS Standby
```

> [!NOTE]
> **왜 2개 AZ에 걸쳐 서브넷을 만드는가?**
>
> AWS에서 고가용성(HA)을 확보하려면 최소 2개 AZ에 리소스를 분산해야 합니다.  
> 하나의 AZ에 장애가 발생해도 다른 AZ의 서브넷에서 서비스를 계속할 수 있습니다.  
> ALB는 최소 2개 AZ의 서브넷을 요구하며, RDS Multi-AZ도 2개 서브넷이 필요합니다.

> [!TIP]
> **서브넷 CIDR 설계 시 실무 팁:**
>
> - 서브넷 번호를 **역할별로 그룹화**하면 관리가 편합니다 (1~9: Public, 10~99: Private, 100~: DB)
> - 나중에 서브넷을 추가할 여유 공간을 남겨두세요. /16 VPC에 /24 서브넷이면 256개까지 가능하므로 넉넉합니다.
> - 서브넷 크기는 예상 리소스 수의 **2~3배** 여유를 두는 것이 좋습니다 (Auto Scaling, 배포 시 임시 인스턴스 등 고려)
> - 서브넷은 한번 생성하면 CIDR을 변경할 수 없으므로, 처음 설계가 중요합니다

## 4. 라우팅 개념

> [!CONCEPT] 라우팅의 동작 방식
> **라우팅**은 네트워크 패킷이 출발지에서 목적지까지 도달하는 경로를 결정하는 과정입니다.  
> 라우터는 **라우팅 테이블**을 참조하여 패킷을 다음 홉(Next Hop)으로 전달합니다.

### 주요 용어

| 용어                            | 설명                                                            |
| ------------------------------- | --------------------------------------------------------------- |
| **라우팅 (Routing)**            | 패킷이 목적지까지 도달하는 경로를 결정하는 과정                 |
| **라우팅 테이블 (Route Table)** | 목적지별 다음 홉을 정의한 규칙 목록                             |
| **Next Hop**                    | 패킷이 다음으로 전달될 네트워크 장치                            |
| **Default Route (0.0.0.0/0)**   | 다른 경로에 매칭되지 않는 모든 트래픽의 기본 경로               |
| **Longest Prefix Match**        | 여러 경로 매칭 시 가장 구체적인(CIDR이 큰) 경로를 우선하는 규칙 |
| **IGW (Internet Gateway)**      | VPC와 인터넷 간 양방향 통신을 가능하게 하는 게이트웨이          |
| **NAT Gateway**                 | Private Subnet → 인터넷 단방향 통신을 중계하는 서비스           |

### 라우팅의 기본 원리

<img src="/images/architecture/step1-0-vpc-routing.png" alt="VPC 라우팅 동작 흐름" class="guide-img-sm" />

라우팅은 우편 배달과 비슷합니다. 우체국(라우터)이 편지(패킷)의 주소(목적지 IP)를 보고, 배달 규칙표(라우팅 테이블)에 따라 다음 배달소(Next Hop)로 전달합니다.

```
라우팅의 핵심 질문: "이 패킷을 어디로 보내야 하는가?"

┌─────────────────────────────────────────────────────────┐
│  라우터의 판단 과정                                     │
│                                                         │
│  1. 패킷의 목적지 IP 확인                               │
│  2. 라우팅 테이블의 각 경로와 비교                      │
│  3. 매칭되는 경로 중 가장 구체적인 것 선택              │
│  4. 해당 경로의 Target(Next Hop)으로 패킷 전달          │
└─────────────────────────────────────────────────────────┘
```

### AWS VPC 라우팅 테이블

AWS VPC에서는 서브넷마다 하나의 라우팅 테이블을 연결합니다. 서브넷 내 모든 리소스는 이 테이블의 규칙에 따라 트래픽을 전달합니다.

**Public Subnet 라우팅 테이블:**

| Destination | Target   | 설명                                 |
| ----------- | -------- | ------------------------------------ |
| 10.0.0.0/16 | local    | VPC 내부 통신 (자동 생성, 삭제 불가) |
| 0.0.0.0/0   | igw-xxxx | 인터넷으로 나가는 모든 트래픽        |

**Private Subnet 라우팅 테이블:**

| Destination | Target   | 설명                                   |
| ----------- | -------- | -------------------------------------- |
| 10.0.0.0/16 | local    | VPC 내부 통신 (자동 생성, 삭제 불가)   |
| 0.0.0.0/0   | nat-xxxx | 인터넷 접근 시 NAT Gateway 경유 (선택) |

> [!NOTE]
> `10.0.0.0/16 → local` 경로는 VPC 생성 시 자동으로 추가되며 삭제할 수 없습니다.  
> 이 경로 덕분에 같은 VPC 내의 모든 서브넷은 별도 설정 없이 서로 통신할 수 있습니다.

### 라우팅 동작 원리 (단계별)

```
예시 1: EC2(10.0.1.50)에서 Google(8.8.8.8)로 요청

  ① EC2가 패킷 생성: src=10.0.1.50, dst=8.8.8.8
  ② 라우팅 테이블 검색:
     - 10.0.0.0/16 매칭? → 8.8.8.8은 10.0.x.x 범위 아님 → NO
     - 0.0.0.0/0 매칭?   → 모든 IP 포함 → YES
  ③ Target: igw-xxxx → Internet Gateway로 전달
  ④ IGW가 패킷을 인터넷으로 전송


예시 2: EC2(10.0.1.50)에서 RDS(10.0.100.20)로 요청

  ① EC2가 패킷 생성: src=10.0.1.50, dst=10.0.100.20
  ② 라우팅 테이블 검색:
     - 10.0.0.0/16 매칭? → 10.0.100.20은 10.0.x.x 범위 → YES
     - 0.0.0.0/0 매칭?   → YES (하지만 /16이 더 구체적)
  ③ Target: local → VPC 내부에서 직접 전달
  ④ 패킷이 RDS 서브넷으로 도달
```

### Longest Prefix Match (최장 접두사 매칭)

라우팅 테이블에 여러 경로가 매칭될 때, **가장 구체적인(CIDR 숫자가 큰) 경로**가 우선합니다:

```
라우팅 테이블:
  10.0.0.0/16  → local
  10.0.1.0/24  → some-target
  0.0.0.0/0    → igw-xxxx

패킷 목적지: 10.0.1.50

매칭되는 경로:
  ✓ 10.0.0.0/16  (16비트 매칭)
  ✓ 10.0.1.0/24  (24비트 매칭) ← 가장 구체적 → 이것이 선택됨
  ✓ 0.0.0.0/0    (0비트 매칭)

규칙: /24 > /16 > /0 (숫자가 클수록 구체적 = 우선순위 높음)
```

> [!TIP]
> **AWS VPC 라우팅 설계 시 주의사항:**
>
> - `local` 경로는 항상 VPC CIDR 전체에 대해 존재하므로, VPC 내부 통신은 별도 설정 불필요
> - Public Subnet과 Private Subnet의 차이는 **오직 라우팅 테이블**에 의해 결정됨
> - 라우팅 테이블에 없는 목적지로의 패킷은 **자동으로 폐기**(drop)됨
> - 하나의 라우팅 테이블을 여러 서브넷에 연결할 수 있지만, 하나의 서브넷에는 하나의 라우팅 테이블만 연결 가능
> - VPC Peering, Transit Gateway, VPN 연결 시 해당 CIDR에 대한 경로를 라우팅 테이블에 추가해야 함

### NAT Gateway의 라우팅 동작

```
Private Subnet의 EC2가 인터넷에 접근하는 과정:

  EC2 (10.0.10.50)
       │
       ▼  dst=8.8.8.8
  [Private RT: 0.0.0.0/0 → nat-xxxx]
       │
       ▼
  NAT Gateway (Public Subnet에 위치, 공인 IP 보유)
       │  src를 NAT GW의 공인 IP로 변환
       ▼
  [Public RT: 0.0.0.0/0 → igw-xxxx]
       │
       ▼
  Internet Gateway → 인터넷

  응답은 역순으로: 인터넷 → IGW → NAT GW → EC2
  (NAT GW가 공인 IP를 다시 사설 IP로 변환)
```

> [!NOTE]
> NAT Gateway는 **단방향**입니다.  
> Private Subnet의 EC2가 인터넷으로 요청을 보낼 수는 있지만, 인터넷에서 Private Subnet의 EC2로 직접 접근하는 것은 불가능합니다.  
> 이것이 Private Subnet의 보안 이점입니다.

## 5. DNS 기초

> [!CONCEPT] DNS의 동작 원리
> **DNS**(Domain Name System)는 사람이 읽을 수 있는 도메인 이름(예: www.example.com)을 컴퓨터가 이해하는 IP 주소(예: 93.184.216.34)로 변환하는 시스템입니다.  
> 인터넷의 "전화번호부"라고 생각하면 됩니다.

### 주요 용어

| 용어                         | 설명                                                      |
| ---------------------------- | --------------------------------------------------------- |
| **DNS (Domain Name System)** | 도메인 이름을 IP 주소로 변환하는 분산 시스템              |
| **Recursive Resolver**       | 클라이언트 대신 DNS 조회를 반복 수행하는 서버 (ISP DNS)   |
| **Authoritative DNS**        | 특정 도메인의 최종 IP 정보를 보유한 네임서버              |
| **TLD (Top Level Domain)**   | 최상위 도메인 (.com, .kr, .org 등)                        |
| **DNS 레코드**               | 도메인과 IP/다른 도메인 간 매핑 정보 (A, CNAME, MX 등)    |
| **TTL (Time To Live)**       | DNS 캐시 유지 시간. 짧으면 변경 반영 빠름, 길면 성능 좋음 |
| **Route 53**                 | AWS의 관리형 DNS 서비스 (Authoritative DNS)               |

### DNS가 필요한 이유

```
DNS 없는 세상:
  "93.184.216.34로 접속해주세요" → 사람이 IP를 외워야 함
  서버 IP가 바뀌면?              → 모든 사용자에게 새 IP를 알려야 함

DNS 있는 세상:
  "www.example.com으로 접속해주세요" → 사람이 읽기 쉬운 이름 사용
  서버 IP가 바뀌면?                  → DNS 레코드만 수정하면 자동 반영
```

### DNS 조회 과정 (상세)

<img src="/images/architecture/step1-0-dns-lookup.png" alt="DNS 조회 과정" class="guide-img-md" />

```
사용자가 브라우저에 "www.example.com" 입력

① 브라우저 DNS 캐시 확인  → 없으면
② OS DNS 캐시 확인        → 없으면
③ /etc/hosts 파일 확인    → 없으면
④ Recursive Resolver (ISP DNS 또는 8.8.8.8)에 질의
   │
   ├─⑤ Root DNS 서버에 질의
   │    "example.com의 IP는?"
   │        → ".com을 담당하는 TLD 서버는 a.gtld-servers.net"
   │
   ├─⑥ TLD DNS 서버(.com)에 질의
   │    "example.com의 IP는?"
   │        → "example.com의 네임서버는 ns1.example.com"
   │
   └─⑦ Authoritative DNS 서버에 질의
        "www.example.com의 IP는?"
            → "93.184.216.34" (A 레코드)

⑧ Recursive Resolver가 결과를 캐시하고 클라이언트에 반환
⑨ 브라우저가 93.184.216.34로 HTTP 요청 전송
```

> [!NOTE]
> 이 전체 과정은 보통 **수십 밀리초** 내에 완료됩니다. 캐시가 있으면 ①~③ 단계에서 즉시 반환되어 더 빠릅니다.

### 주요 DNS 레코드 타입

| 레코드    | 용도                 | 예시                               | AWS 사용 사례                 |
| --------- | -------------------- | ---------------------------------- | ----------------------------- |
| **A**     | 도메인 → IPv4        | example.com → 1.2.3.4              | EC2, ALB의 IP 매핑            |
| **AAAA**  | 도메인 → IPv6        | example.com → 2001:db8::1          | IPv6 지원 리소스              |
| **CNAME** | 도메인 → 다른 도메인 | www.example.com → example.com      | CloudFront, ALB DNS 이름 연결 |
| **ALIAS** | 도메인 → AWS 리소스  | example.com → d1234.cloudfront.net | Route 53 전용, Zone Apex 지원 |
| **MX**    | 메일 서버 지정       | mail.example.com                   | Amazon SES, WorkMail          |
| **NS**    | 네임서버 지정        | ns1.example.com                    | Route 53 호스팅 영역          |
| **TXT**   | 텍스트 정보          | "v=spf1 include:..."               | 도메인 소유권 검증, SPF/DKIM  |

### AWS Route 53과 VPC DNS

```
AWS VPC 내부 DNS 동작:

┌─── VPC (10.0.0.0/16) ──────────────────────────────┐
│                                                    │
│  DNS 서버: 10.0.0.2 (VPC CIDR + 2)                 │
│  ┌─────────────────────────────────────────────┐   │
│  │  EC2가 "mydb.abc123.ap-northeast-2.rds..."  │   │
│  │  → VPC DNS 서버(10.0.0.2)에 질의            │   │
│  │  → RDS의 사설 IP 반환 (10.0.100.20)         │   │
│  └─────────────────────────────────────────────┘   │
│                                                    │
│  Private Hosted Zone:                              │
│  "api.internal" → 10.0.10.50 (내부 전용 도메인)    │
│                                                    │
└────────────────────────────────────────────────────┘
```

> [!TIP]
> **AWS에서 DNS 관련 알아야 할 것들:**
>
> - VPC의 DNS 서버는 항상 **VPC CIDR의 두 번째 IP** (예: 10.0.0.0/16 → 10.0.0.2)
> - RDS, ElastiCache 등은 IP가 아닌 **DNS 엔드포인트**로 접근 (IP가 변경될 수 있으므로)
> - Route 53의 **ALIAS 레코드**는 CNAME과 달리 Zone Apex(example.com 자체)에 사용 가능
> - Private Hosted Zone을 사용하면 VPC 내부에서만 유효한 커스텀 도메인을 만들 수 있음
> - `enableDnsHostnames`와 `enableDnsSupport`가 VPC에서 활성화되어야 DNS가 정상 동작

### VPC DNS 설정: enableDnsSupport와 enableDnsHostnames

VPC에는 DNS 동작을 제어하는 두 가지 설정이 있습니다. 이 설정이 올바르지 않으면 VPC 내부 DNS 조회가 실패합니다.

| 설정                   | VPC only (API/CLI) | VPC and more (콘솔 마법사) | Default VPC | 역할                                              |
| ---------------------- | :----------------: | :------------------------: | :---------: | ------------------------------------------------- |
| **enableDnsSupport**   |        true        |            true            |    true     | VPC 내부 DNS 서버(10.0.0.2) 활성화 여부           |
| **enableDnsHostnames** |     **false**      |          **true**          |    true     | EC2 인스턴스에 퍼블릭 DNS 호스트명 자동 할당 여부 |

> [!NOTE]
> 콘솔에서 **"VPC and more"** 옵션으로 생성하면 `enableDnsHostnames`가 기본 체크(활성화)되어 있으므로 별도 설정이 필요 없습니다.  
> 하지만 **"VPC only"**로 생성하거나 CLI/CloudFormation으로 생성하면 기본 비활성화이므로 수동으로 켜야 합니다.  
> 실습에서는 "VPC and more"를 사용하므로 자동으로 둘 다 활성화됩니다.

```
enableDnsSupport = true (기본값)
  → VPC의 DNS 서버(CIDR+2, 예: 10.0.0.2)가 활성화됨
  → EC2 인스턴스가 DNS 조회 가능 (nslookup, dig 등)
  → RDS 엔드포인트, S3 엔드포인트 등의 이름 해석 가능

enableDnsSupport = false
  → VPC 내부 DNS 서버 비활성화
  → 인스턴스에서 도메인 이름으로 아무것도 접근 불가
  → 커스텀 DNS 서버를 직접 구성해야 함 (거의 사용하지 않음)
```

```
enableDnsHostnames = true
  → 공인 IP가 있는 EC2에 퍼블릭 DNS 이름 자동 할당
     예: ec2-54-123-45-67.ap-northeast-2.compute.amazonaws.com
  → Private Hosted Zone 연결 시 필수
  → VPC Endpoint(Interface 타입) 사용 시 필수

enableDnsHostnames = false (기본값)
  → EC2에 퍼블릭 DNS 이름이 할당되지 않음
  → IP 주소로만 접근 가능
```

> [!NOTE]
> **이 설정이 필요한 실제 상황들:**
>
> | 상황                        | enableDnsSupport | enableDnsHostnames |
> | --------------------------- | :--------------: | :----------------: |
> | RDS 엔드포인트로 DB 접속    |     ✅ 필수      |         -          |
> | EC2에 퍼블릭 DNS 이름 부여  |     ✅ 필수      |      ✅ 필수       |
> | Private Hosted Zone 사용    |     ✅ 필수      |      ✅ 필수       |
> | VPC Interface Endpoint 사용 |     ✅ 필수      |      ✅ 필수       |
> | ACM 인증서 DNS 검증         |     ✅ 필수      |         -          |
> | ECS/EKS 서비스 디스커버리   |     ✅ 필수      |      ✅ 필수       |
>
> 실습에서는 **둘 다 true로 설정**하는 것을 권장합니다.  
> VPC 생성 시 `enableDnsSupport`는 기본 true이지만, `enableDnsHostnames`는 기본 false이므로 직접 활성화해야 합니다.

> [!TIP]
> **VPC 콘솔에서 설정 변경하는 방법:**
>
> 1. VPC 콘솔 → "Your VPCs" → 해당 VPC 선택
> 2. "Actions" → "Edit VPC settings"
> 3. "DNS settings" 섹션에서 두 옵션 모두 체크
>
> 또는 AWS CLI:
>
> ```
> aws ec2 modify-vpc-attribute --vpc-id vpc-xxxx --enable-dns-support
> aws ec2 modify-vpc-attribute --vpc-id vpc-xxxx --enable-dns-hostnames
> ```

## 6. TCP/UDP와 포트

> [!CONCEPT] TCP/UDP와 포트 번호
> **TCP**(Transmission Control Protocol)는 신뢰성 있는 연결 지향 프로토콜이고, **UDP**(User Datagram Protocol)는 빠르지만 신뢰성을 보장하지 않는 비연결 프로토콜입니다. **포트**는 하나의 IP에서 여러 서비스를 구분하는 논리적 번호입니다.

### 주요 용어

| 용어                                    | 설명                                                       |
| --------------------------------------- | ---------------------------------------------------------- |
| **TCP (Transmission Control Protocol)** | 신뢰성 있는 연결 지향 전송 프로토콜. 재전송·순서 보장      |
| **UDP (User Datagram Protocol)**        | 빠르지만 신뢰성을 보장하지 않는 비연결 전송 프로토콜       |
| **포트 (Port)**                         | 하나의 IP에서 여러 서비스를 구분하는 논리적 번호 (0~65535) |
| **3-way Handshake**                     | TCP 연결 수립 과정 (SYN → SYN-ACK → ACK)                   |
| **Stateful**                            | 연결 상태를 기억하여 응답 트래픽을 자동 허용               |
| **Stateless**                           | 각 패킷을 독립적으로 판단, 요청/응답 규칙을 각각 설정      |

### IP 주소와 포트의 관계

```
비유: IP 주소 = 아파트 건물 주소, 포트 = 호수(동·호)

  서버 IP: 10.0.1.50
  ├── 포트 22   → SSH 서비스 (원격 접속)
  ├── 포트 80   → HTTP 웹 서비스
  ├── 포트 443  → HTTPS 웹 서비스
  └── 포트 3306 → MySQL 데이터베이스

  하나의 서버(IP)에서 여러 서비스를 동시에 운영할 수 있는 이유:
  각 서비스가 서로 다른 포트를 사용하기 때문
```

### TCP 3-way Handshake (연결 수립)

TCP는 데이터를 보내기 전에 먼저 연결을 수립합니다:

```
클라이언트                          서버
    │                                │
    │──── SYN (연결 요청) ──────────→│  ① "연결하고 싶어요"
    │                                │
    │←─── SYN-ACK (요청 수락) ───────│  ② "좋아요, 나도 연결할게요"
    │                                │
    │──── ACK (확인) ───────────────→│  ③ "확인, 이제 데이터 보낼게요"
    │                                │
    │←──── 데이터 전송 시작 ────────→│
    │                                │

연결 종료: 4-way Handshake (FIN → ACK → FIN → ACK)
```

> [!NOTE]
> AWS Security Group은 **Stateful**이므로, 인바운드 규칙에서 허용한 트래픽의 응답은 아웃바운드 규칙과 무관하게 자동 허용됩니다.  
> 반면 NACL은 **Stateless**이므로, 인바운드와 아웃바운드 규칙을 각각 설정해야 합니다.

### TCP vs UDP 비교

| 특성          | TCP                         | UDP                 |
| ------------- | --------------------------- | ------------------- |
| **연결 방식** | 연결 지향 (3-way handshake) | 비연결              |
| **신뢰성**    | 보장 (재전송, 순서 보장)    | 미보장              |
| **속도**      | 상대적으로 느림             | 빠름                |
| **사용 사례** | HTTP, SSH, DB 연결          | DNS, 스트리밍, 게임 |
| **헤더 크기** | 20바이트                    | 8바이트             |
| **흐름 제어** | 있음 (윈도우 크기 조절)     | 없음                |

```
언제 TCP를 쓰고 언제 UDP를 쓰는가?

TCP (신뢰성 중요):
  • 웹 페이지 로딩 (HTTP/HTTPS)   → 데이터 누락 시 페이지 깨짐
  • 파일 다운로드 (FTP)           → 1바이트라도 빠지면 파일 손상
  • 이메일 전송 (SMTP)            → 내용이 빠지면 안 됨
  • DB 쿼리 (MySQL, PostgreSQL)   → 결과가 정확해야 함

UDP (속도 중요):
  • DNS 조회          → 작은 데이터, 빠른 응답 필요
  • 실시간 스트리밍   → 프레임 하나 빠져도 계속 재생
  • 온라인 게임       → 지연보다 빠른 전송이 중요
  • VoIP 통화         → 약간의 끊김은 허용, 지연은 불가
```

### 포트 번호 범위

| 범위          | 이름              | 설명                                   |
| ------------- | ----------------- | -------------------------------------- |
| 0 ~ 1023      | Well-Known Ports  | 표준 서비스용 (HTTP:80, SSH:22 등)     |
| 1024 ~ 49151  | Registered Ports  | 특정 애플리케이션 등록 (MySQL:3306 등) |
| 49152 ~ 65535 | Dynamic/Ephemeral | 클라이언트가 임시로 사용하는 포트      |

### 주요 포트 번호

| 포트 | 프로토콜 | 서비스     | AWS 관련                | Security Group 설정 예시 |
| ---- | -------- | ---------- | ----------------------- | ------------------------ |
| 22   | TCP      | SSH        | Amazon EC2 접속         | 내 IP만 허용 (/32)       |
| 80   | TCP      | HTTP       | ALB, 웹서버             | 0.0.0.0/0 (전체 공개)    |
| 443  | TCP      | HTTPS      | ALB + ACM (인증서)      | 0.0.0.0/0 (전체 공개)    |
| 3306 | TCP      | MySQL      | Amazon RDS (MySQL)      | 앱 서버 SG만 허용        |
| 5432 | TCP      | PostgreSQL | Amazon RDS (PostgreSQL) | 앱 서버 SG만 허용        |
| 6379 | TCP      | Redis      | Amazon ElastiCache      | 앱 서버 SG만 허용        |
| 8080 | TCP      | 대체 HTTP  | Spring Boot 기본 포트   | ALB SG만 허용            |

> [!TIP]
> **Security Group 설정 모범 사례:**
>
> - **최소 권한 원칙**: 필요한 포트만, 필요한 소스에서만 허용
> - SSH(22)는 절대 `0.0.0.0/0`으로 열지 마세요. 내 IP(/32) 또는 VPN에서만 허용
> - DB 포트(3306, 5432)는 인터넷에 노출하지 말고, 앱 서버의 Security Group ID를 소스로 지정
> - Security Group은 **허용 규칙만** 설정 가능 (거부 규칙 없음). 명시적으로 허용하지 않은 트래픽은 자동 거부
> - Ephemeral Port(임시 포트) 범위: NACL 아웃바운드에서 1024-65535를 허용해야 응답 트래픽이 돌아옴

## 7. OSI 7계층 모델

> [!CONCEPT] OSI 7계층 모델
> **OSI(Open Systems Interconnection) 모델**은 네트워크 통신을 7개 계층으로 나누어 설명하는 참조 모델입니다.  
> 각 계층은 독립적으로 동작하며, 상위 계층은 하위 계층의 서비스를 이용합니다.  
> 실무에서는 OSI 모델보다 TCP/IP 4계층 모델을 더 많이 사용하지만, AWS 서비스를 이해하려면 OSI 계층 개념이 필수입니다.

### 주요 용어

| 용어                                | 설명                                                                                                    |
| ----------------------------------- | ------------------------------------------------------------------------------------------------------- |
| **OSI 7계층 모델**                  | 네트워크 통신을 7개 계층으로 나누어 설명하는 참조 모델                                                  |
| **ALB (Application Load Balancer)** | Layer 7(HTTP/HTTPS)에서 동작하는 로드밸런서                                                             |
| **NLB (Network Load Balancer)**     | Layer 4(TCP/UDP)에서 동작하는 초고성능 로드밸런서                                                       |
| **ENI (Elastic Network Interface)** | EC2 인스턴스에 연결되는 가상 네트워크 카드. 사설 IP, Public IP, MAC 주소, Security Group이 ENI에 할당됨 |
| **WAF (Web Application Firewall)**  | Layer 7에서 동작하는 웹 애플리케이션 방화벽                                                             |

> [!TIP]
> **ENI vs NIC: 가상 네트워크 카드와 물리 네트워크 카드**
>
> | 구분 | NIC (Network Interface Card)  | ENI (Elastic Network Interface) |
> | ---- | ----------------------------- | ------------------------------- |
> | 형태 | 물리적 하드웨어 (랜카드)      | 소프트웨어로 정의된 가상 카드   |
> | 위치 | 물리 서버에 장착              | AWS VPC 내 인스턴스에 연결      |
> | 이동 | 서버 간 이동 불가 (물리 장착) | 인스턴스 간 Detach/Attach 가능  |
> | 개수 | 서버당 슬롯 수에 제한         | 인스턴스 타입별 최대 개수 다름  |
> | 관리 | 드라이버 설치, 케이블 연결    | 콘솔/CLI에서 생성·삭제·이동     |
>
> 쉽게 말해, ENI는 **클라우드 버전의 NIC**입니다. 물리 서버에 랜카드를 꽂는 것처럼, EC2 인스턴스에 ENI를 연결합니다.  
> 차이점은 ENI는 소프트웨어이므로 다른 인스턴스로 옮길 수 있고, IP와 Security Group이 ENI에 붙어서 함께 이동한다는 것입니다.

### 왜 계층을 나누는가?

```
계층 분리의 장점:

1. 독립적 개발: 각 계층을 독립적으로 개발·수정 가능
   예: HTTP/2 → HTTP/3 변경 시 물리 계층은 변경 불필요

2. 문제 격리: 장애 발생 시 어느 계층 문제인지 빠르게 파악
   예: ping은 되는데 웹 접속 안 됨 → L3(네트워크)는 정상, L7(응용) 문제

3. 표준화: 각 계층의 프로토콜을 표준화하여 호환성 확보
   예: 어떤 브라우저든 HTTP 표준을 따르면 웹서버와 통신 가능
```

### 7계층 구조와 AWS 매핑

| 계층 | 이름                       | 역할                     | 프로토콜/기술     | AWS 서비스                             |
| :--: | -------------------------- | ------------------------ | ----------------- | -------------------------------------- |
|  7   | **응용** (Application)     | 사용자 인터페이스        | HTTP, HTTPS, DNS  | ALB, API Gateway, Route 53, CloudFront |
|  6   | **표현** (Presentation)    | 데이터 형식 변환, 암호화 | SSL/TLS, JSON     | ACM, KMS                               |
|  5   | **세션** (Session)         | 연결 세션 관리           | 세션 관리         | -                                      |
|  4   | **전송** (Transport)       | 포트 기반 통신, 신뢰성   | TCP, UDP          | NLB, Security Group                    |
|  3   | **네트워크** (Network)     | IP 라우팅, 경로 결정     | IP, ICMP          | VPC, Subnet, Route Table, NAT GW, IGW  |
|  2   | **데이터링크** (Data Link) | MAC 주소 기반 전달       | Ethernet, ARP     | ENI                                    |
|  1   | **물리** (Physical)        | 전기/광 신호 전송        | 케이블, 전기 신호 | AWS 물리 인프라, Direct Connect        |

### 데이터 캡슐화 (Encapsulation)

데이터가 각 계층을 지나면서 헤더가 추가되는 과정:

```
송신 측 (위 → 아래):

  L7: [HTTP 데이터]
  L4: [TCP 헤더 | HTTP 데이터]                  ← 포트 번호 추가
  L3: [IP 헤더 | TCP 헤더 | HTTP 데이터]        ← IP 주소 추가
  L2: [MAC 헤더 | IP | TCP | HTTP | MAC 꼬리]   ← MAC 주소 추가
  L1: 01101001... (전기 신호로 변환)

수신 측 (아래 → 위):
  각 계층에서 자기 헤더를 벗기고(역캡슐화) 상위 계층으로 전달
```

### AWS 로드밸런서와 OSI 계층

```
┌─────────────────────────────────────────────────────────────┐
│  ALB (Application Load Balancer) - Layer 7                  │
│  • HTTP/HTTPS 헤더, URL 경로, 호스트 기반 라우팅            │
│  • /api/* → API 서버,   /static/* → S3                      │
│  • WebSocket, gRPC 지원                                     │
│  • WAF 연동 가능 (SQL Injection, XSS 차단)                  │
│  • 사용 사례: 웹 애플리케이션, 마이크로서비스               │
├─────────────────────────────────────────────────────────────┤
│  NLB (Network Load Balancer) - Layer 4                      │
│  • TCP/UDP 포트 기반 라우팅 (HTTP 내용은 보지 않음)         │
│  • 초고성능: 수백만 요청/초, 극도로 낮은 지연               │
│  • 고정 IP / Elastic IP 할당 가능                           │
│  • 사용 사례: 게임 서버, IoT, 실시간 스트리밍               │
└─────────────────────────────────────────────────────────────┘
```

> [!TIP]
> **OSI 계층으로 AWS 트러블슈팅하기:**
>
> | 증상                      | 의심 계층 | 확인 방법                         |
> | ------------------------- | --------- | --------------------------------- |
> | ping 안 됨                | L3        | Security Group ICMP 허용 확인     |
> | SSH 연결 거부             | L4        | 포트 22 인바운드 규칙 확인        |
> | 웹 페이지 502 Bad Gateway | L7        | ALB Target Group 헬스체크 확인    |
> | HTTPS 인증서 오류         | L6        | ACM 인증서 상태, 도메인 매칭 확인 |
> | 같은 VPC인데 통신 안 됨   | L3        | 라우팅 테이블, NACL 확인          |
> | 연결은 되는데 응답이 느림 | L4/L7     | NLB/ALB 타겟 헬스, 앱 로그 확인   |
>
> 아래 계층부터 위로 올라가며 확인하는 것이 체계적인 트러블슈팅 방법입니다.

## 8. VPC가 필요한 이유

> [!CONCEPT] VPC란 무엇인가
> **VPC**(Virtual Private Cloud)는 AWS 클라우드 내에서 논리적으로 격리된 가상 네트워크입니다.  
> 다른 AWS 고객의 리소스와 완전히 분리되며, IP 대역, 서브넷, 라우팅, 보안을 직접 제어할 수 있습니다.  
> 쉽게 말해, AWS 안에 나만의 사설 네트워크를 구축하는 것입니다.

### 주요 용어

| 용어                            | 설명                                                        |
| ------------------------------- | ----------------------------------------------------------- |
| **VPC (Virtual Private Cloud)** | AWS 클라우드 내에서 논리적으로 격리된 가상 네트워크         |
| **Internet Gateway (IGW)**      | VPC와 인터넷을 연결하는 양방향 게이트웨이                   |
| **NAT Gateway**                 | Private Subnet에서 인터넷으로 단방향 접근을 허용하는 서비스 |
| **Security Group**              | 인스턴스 레벨의 가상 방화벽 (Stateful)                      |
| **NACL (Network ACL)**          | 서브넷 레벨의 방화벽 (Stateless)                            |
| **VPC Peering**                 | 두 VPC 간 사설 네트워크 연결                                |
| **VPC Endpoint**                | VPC에서 AWS 서비스로 인터넷 경유 없이 직접 연결             |

### VPC 없이 vs VPC 사용

```
VPC 없는 환경 (Classic EC2 - 과거):
┌────────────────────────────────────────────────┐
│  AWS 공유 네트워크                             │
│  ┌───┐ ┌───┐ ┌───┐ ┌───┐                       │
│  │ A │ │ B │ │ C │ │ D │ ← 모두 같은 네트워크  │
│  └───┘ └───┘ └───┘ └───┘                       │
│  (격리 불가, 보안 취약)                        │
│  • 다른 고객의 인스턴스와 같은 네트워크        │
│  • IP 대역 선택 불가                           │
│  • 네트워크 수준 보안 제어 제한적              │
└────────────────────────────────────────────────┘

VPC 사용 환경 (현재):
┌──────────────────────────────────────┐
│  ┌─── VPC A ───┐  ┌─── VPC B ───┐    │
│  │ ┌───┐ ┌───┐ │  │ ┌───┐ ┌───┐ │    │
│  │ │EC2│ │RDS│ │  │ │EC2│ │S3 │ │    │
│  │ └───┘ └───┘ │  │ └───┘ └───┘ │    │
│  │  완전 격리  │  │  완전 격리  │    │
│  └─────────────┘  └─────────────┘    │
│  • 각 VPC는 독립된 네트워크          │
│  • IP 대역 자유 선택                 │
│  • 서브넷, 라우팅, 방화벽 완전 제어  │
└──────────────────────────────────────┘
```

### VPC의 핵심 구성 요소

| 구성 요소            | 역할                            | 비유                |
| -------------------- | ------------------------------- | ------------------- |
| **VPC**              | 격리된 가상 네트워크            | 건물 전체           |
| **CIDR Block**       | VPC의 IP 주소 범위 정의         | 건물의 주소 체계    |
| **Subnet**           | VPC를 AZ별로 분할               | 건물의 각 층        |
| **Internet Gateway** | VPC ↔ 인터넷 연결               | 건물의 정문         |
| **NAT Gateway**      | Private → 인터넷 (단방향)       | 내부 전용 출구      |
| **Route Table**      | 트래픽 경로 결정                | 건물 내 안내 표지판 |
| **Security Group**   | 인스턴스 레벨 방화벽 (Stateful) | 각 방의 잠금장치    |
| **NACL**             | 서브넷 레벨 방화벽 (Stateless)  | 각 층의 출입 게이트 |

### Security Group vs NACL 비교

| 특성          | Security Group                   | NACL (Network ACL)                                           |
| ------------- | -------------------------------- | ------------------------------------------------------------ |
| **적용 대상** | 인스턴스 (ENI) 레벨              | 서브넷 레벨                                                  |
| **상태 추적** | Stateful (응답 자동 허용)        | Stateless (인/아웃 각각 설정)                                |
| **규칙 타입** | 허용 규칙만                      | 허용 + 거부 규칙                                             |
| **규칙 평가** | 모든 규칙 평가 후 허용 여부 결정 | 번호 순서대로 평가, 첫 매칭 적용 (낮은 번호 = 높은 우선순위) |
| **기본 동작** | 모든 인바운드 거부               | 모든 트래픽 허용 (기본 NACL)                                 |
| **사용 빈도** | 거의 항상 사용                   | 추가 보안 계층이 필요할 때                                   |

> [!TIP]
> **NACL 규칙 번호는 왜 100, 110, 120처럼 띄엄띄엄 설정하는가?**
>
> NACL 규칙은 번호를 나중에 변경할 수 없습니다. 1, 2, 3으로 붙여서 만들면 사이에 새 규칙을 끼워넣을 수 없어 전체를 삭제하고 재생성해야 합니다.  
> 10 단위(100, 110, 120...)로 간격을 두면 긴급 상황에서 50번에 DENY 규칙을 추가하거나, 105번에 예외 규칙을 끼워넣을 수 있습니다.

```
Security Group (Stateful) 동작:

  인바운드 규칙: 포트 80 허용 (0.0.0.0/0)
  아웃바운드 규칙: 별도 설정 불필요

  ① 클라이언트 → 서버 포트 80으로 요청 (인바운드)
     → Security Group이 인바운드 규칙 검사 → 포트 80 허용됨 ✓
     → 이 연결 정보를 "상태 테이블"에 기록 (출발지 IP, 포트, 프로토콜)

  ② 서버 → 클라이언트로 응답 (아웃바운드, 임시 포트 사용)
     → Security Group이 상태 테이블 확인
     → "이 패킷은 ①에서 이미 허용된 인바운드 요청에 대한 응답이다"
     → 아웃바운드 규칙을 검사하지 않고 자동 허용 ✓

  핵심: 한쪽 방향을 허용하면, 그 응답은 반대쪽 규칙과 무관하게 통과
  → 그래서 "Stateful" (상태를 기억한다)


NACL (Stateless) 동작:

  인바운드 규칙: 포트 80 허용
  아웃바운드 규칙: 포트 1024-65535 허용 (← 이것도 반드시 설정해야 함!)

  ① 클라이언트 → 서버 포트 80으로 요청 (인바운드)
     → NACL이 인바운드 규칙 검사 → 포트 80 허용됨 ✓
     → 연결 정보를 기록하지 않음 (상태 없음)

  ② 서버 → 클라이언트로 응답 (아웃바운드, 임시 포트 49152 사용)
     → NACL은 ①의 요청을 기억하지 못함
     → 아웃바운드 규칙을 처음부터 다시 검사
     → 포트 1024-65535 허용 규칙에 매칭 → 허용 ✓
     → (이 규칙이 없으면 응답이 차단되어 통신 실패!)

  핵심: 매 패킷을 독립적으로 판단, 요청과 응답을 별개로 취급
  → 그래서 "Stateless" (상태를 기억하지 않는다)
```

> [!TIP]
> **NACL의 실무 활용:**
>
> - **대부분의 경우 기본 NACL(모든 트래픽 허용)을 그대로 사용합니다.** 세밀한 접근 제어는 Security Group으로 충분합니다.
> - NACL은 **특정 IP 대역을 서브넷 레벨에서 차단**해야 할 때 사용합니다. (예: DDoS 공격 IP 차단, 특정 국가 IP 차단)
> - Security Group은 거부(DENY) 규칙을 만들 수 없으므로, "이 IP만 차단"하려면 NACL을 사용해야 합니다.
> - NACL 규칙 변경은 즉시 적용되므로, 긴급 차단 시 빠르게 대응할 수 있습니다.
>
> Step 1-3에서 NACL 규칙을 설계하고, Step 2-1에서 EC2를 생성한 뒤 실제 차단/허용 테스트를 진행합니다.

### Security Group 체이닝: 리소스 간 트래픽 흐름

Security Group은 **각 리소스(ENI) 단위**로 적용됩니다. ENI(Elastic Network Interface)는 인스턴스에 연결된 가상 네트워크 카드이며, Security Group은 실제로 이 ENI에 연결됩니다.  
트래픽이 리소스를 떠날 때 **출발지의 아웃바운드 규칙**을 검사하고, 도착할 때 **목적지의 인바운드 규칙**을 검사합니다.

```
EC2 → RDS 통신 시 Security Group 검사 흐름:

  ┌──── EC2 (sg-app) ───┐          ┌──── RDS (sg-db) ────┐
  │                     │          │                     │
  │  아웃바운드 규칙:   │          │  인바운드 규칙:     │
  │  All traffic 허용   │──────────│  3306 from sg-app   │
  │  (기본값)           │          │                     │
  └─────────────────────┘          └─────────────────────┘

  ① EC2가 RDS 포트 3306으로 요청 전송
     → EC2의 SG(sg-app) 아웃바운드 규칙 검사
     → 기본 아웃바운드는 All traffic 허용 → 통과 ✓

  ② 패킷이 RDS에 도착
     → RDS의 SG(sg-db) 인바운드 규칙 검사
     → "포트 3306, 소스: sg-app" 규칙에 매칭 → 허용 ✓

  ③ RDS가 EC2에 응답 반환
     → RDS의 SG(sg-db):  Stateful이므로 ②의 응답 자동 허용 ✓
     → EC2의 SG(sg-app): Stateful이므로 ①의 응답 자동 허용 ✓
```

> [!NOTE]
> **핵심 포인트:**
>
> - 아웃바운드 검사는 **출발하는 리소스의 SG**에서, 인바운드 검사는 **도착하는 리소스의 SG**에서 수행
> - 양쪽 SG 모두 통과해야 통신 성공 (한쪽이라도 거부하면 실패)
> - Stateful이므로 응답은 양쪽 SG 모두에서 자동 허용

### 실제 3-Tier 아키텍처의 Security Group 설계

```
인터넷 → ALB → EC2 App → RDS 전체 흐름:

┌────────────┐     ┌──────────────┐     ┌──────────────┐     ┌──────────────┐
│ 클라이언트 │     │  ALB         │     │  EC2 (App)   │     │  RDS (DB)    │
│ (인터넷)   │────→│  sg-alb      │────→│  sg-app      │────→│  sg-db       │
└────────────┘     └──────────────┘     └──────────────┘     └──────────────┘

sg-alb (ALB용):
  인바운드:  포트 80, 443 from 0.0.0.0/0   ← 인터넷에서 접근 허용
  아웃바운드: All traffic (기본값)

sg-app (EC2 App 서버용):
  인바운드:  포트 8080 from sg-alb         ← ALB에서만 접근 허용
  인바운드:  포트 22 from 내 IP (/32)      ← SSH는 내 IP에서만
  아웃바운드: All traffic (기본값)

sg-db (RDS용):
  인바운드:  포트 3306 from sg-app         ← App 서버에서만 접근 허용
  아웃바운드: All traffic (기본값)
```

```
이 설계의 보안 효과:

  ✗ 인터넷 → RDS 직접 접근: 불가능 (sg-db 인바운드에 0.0.0.0/0 없음)
  ✗ 인터넷 → EC2 직접 접근: 불가능 (sg-app 인바운드에 0.0.0.0/0 없음, 8080은 sg-alb만)
  ✓ 인터넷 → ALB → EC2 → RDS: 정상 흐름만 허용

  → 각 계층이 바로 앞 계층의 SG만 소스로 허용
  → 중간 계층이 뚫려도 다음 계층은 보호됨
```

> [!TIP]
> **Security Group에서 소스로 "SG ID"를 지정하는 이유:**
>
> - IP 주소 대신 SG ID(예: `sg-app`)를 소스로 지정하면, 해당 SG가 연결된 **모든 인스턴스**에서의 접근을 허용
> - Auto Scaling으로 인스턴스가 추가/제거되어도 규칙 변경 불필요 (SG가 같으면 자동 허용)
> - IP가 바뀌어도 SG 기반이므로 영향 없음
> - 이것이 AWS에서 권장하는 **Security Group 체이닝** 패턴입니다

> [!TIP]
> **Public Subnet vs Private Subnet의 차이는 라우팅 테이블에 있습니다:**
>
> - **Public Subnet**: Route Table에 `0.0.0.0/0 → Internet Gateway` 경로가 있음
> - **Private Subnet**: Route Table에 `0.0.0.0/0 → NAT Gateway` 경로가 있거나, 인터넷 경로가 없음
>
> 서브넷 자체에 "Public/Private" 속성이 있는 것이 아니라, 연결된 Route Table이 인터넷 접근 여부를 결정합니다.

> [!NOTE]
> **VPC 설계 시 자주 하는 실수:**
>
> - CIDR을 너무 작게 잡아서 나중에 IP가 부족해지는 경우 (VPC CIDR은 추가는 가능하지만 축소/변경 불가)
> - 모든 리소스를 Public Subnet에 배치하는 경우 (DB는 반드시 Private Subnet에)
> - Security Group에서 SSH(22)를 0.0.0.0/0으로 여는 경우 (보안 사고의 주요 원인)
> - NAT Gateway 없이 Private Subnet의 인스턴스에서 패키지 설치를 시도하는 경우
> - 하나의 AZ에만 서브넷을 만들어 고가용성을 확보하지 못하는 경우

## 9. AWS 태그 (Tags)

> [!CONCEPT] AWS 태그란
> **태그(Tag)**는 AWS 리소스에 붙이는 **키-값(Key-Value) 쌍 라벨**입니다. 리소스의 동작에는 영향을 주지 않지만, 리소스 식별·비용 추적·접근 제어·자동화에 핵심적인 역할을 합니다.

### 주요 용어

| 용어                             | 설명                                                            |
| -------------------------------- | --------------------------------------------------------------- |
| **태그 (Tag)**                   | AWS 리소스에 붙이는 키-값 쌍 메타데이터                         |
| **태그 키 (Tag Key)**            | 태그의 이름. 대소문자 구분, 최대 128자                          |
| **태그 값 (Tag Value)**          | 태그의 값. 최대 256자, 빈 문자열 가능                           |
| **Tag Editor**                   | AWS 콘솔에서 모든 리전의 리소스를 태그 기반으로 조회하는 도구   |
| **태그 정책 (Tag Policy)**       | AWS Organizations에서 태그 규칙을 강제하는 정책                 |
| **SCP (Service Control Policy)** | 특정 조건(태그 포함)을 만족하지 않으면 API 호출을 거부하는 정책 |

### 태그를 사용하는 이유

```
태그가 없는 환경:

  EC2 인스턴스 50개가 실행 중...
  "이거 누가 만들었지?" "어떤 프로젝트용이지?" "삭제해도 되나?"
  → 파악 불가, 비용 추적 불가, 정리 불가

태그가 있는 환경:

  CreatedBy: admin-user  → "아, 내가 만든 거구나"
  Step: step3            → "3주차 NAT 실습용이구나"
  Session: 3-1           → "3-1 세션에서 만든 거니까 실습 끝나면 삭제"
  → 즉시 파악, 비용 추적, 정리 가능
```

### 이 실습에서 사용하는 태그 체계

| 태그 키     | 값 예시          | 용도                                         |
| ----------- | ---------------- | -------------------------------------------- |
| `CreatedBy` | `admin-user`     | 누가 만들었는지 (수동 생성 시)               |
| `CreatedBy` | `cloudformation` | CloudFormation으로 자동 생성된 리소스 구분   |
| `Step`      | `step1`          | 주차별 묶음. Tag Editor에서 주차 단위로 조회 |
| `Session`   | `1-1`            | 세션별 구분. 특정 세션 리소스만 필터링       |

### 실무에서의 태그 활용

```
실무 태그 체계 예시:

  Environment: production / staging / development
  Team:        backend / frontend / devops / data
  Service:     user-api / payment / notification
  CostCenter:  CC-1234 (비용 부서 코드)
  Owner:       john@company.com
  ManagedBy:   terraform / cloudformation / manual
```

### 태그 기반 접근 제어 (실무)

실무에서는 **특정 태그가 없으면 리소스를 생성할 수 없도록** IAM 정책이나 SCP로 강제합니다.

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Deny",
      "Action": "ec2:RunInstances",
      "Resource": "arn:aws:ec2:*:*:instance/*",
      "Condition": {
        "StringNotLike": {
          "aws:RequestTag/Environment": ["production", "staging", "development"]
        }
      }
    }
  ]
}
```

> 위 정책은 `Environment` 태그 없이 EC2 인스턴스를 생성하려고 하면 **거부**합니다.  
> 이렇게 하면 모든 리소스에 태그가 강제되어 비용 추적과 관리가 가능해집니다.

### 태그 기반 비용 추적

AWS Cost Explorer에서 태그별로 비용을 분석할 수 있습니다:

- **Step별 비용**: "step3(NAT 실습)에서 얼마나 썼지?" → `Step: step3`으로 필터링.
- **생성자별 비용**: "내가 만든 리소스 비용은?" → `CreatedBy: admin-user`로 필터링.
- **실무 예시**: "backend 팀의 production 환경 비용은?" → `Team: backend` + `Environment: production`.

> [!NOTE]
> Cost Explorer에서 태그별 비용을 보려면 **Billing → Cost Allocation Tags**에서 해당 태그를 활성화해야 합니다. 활성화 후 24시간 뒤부터 데이터가 표시됩니다.

### Tag Editor로 리소스 관리

[Tag Editor](https://console.aws.amazon.com/resource-groups/tag-editor)를 사용하면 모든 리전의 리소스를 태그 기반으로 한번에 조회할 수 있습니다.

- `Step: step3`으로 검색 → 3주차에 만든 모든 리소스 목록.
- `CreatedBy: cloudformation`으로 검색 → CloudFormation으로 만든 리소스만 조회.
- 태그가 없는 리소스 검색 → 정리가 필요한 미관리 리소스 파악.

> [!TIP]
> **태그 모범 사례:**
>
> - 리소스 생성 시 태그를 **즉시** 붙이세요. 나중에 하려면 잊어버립니다.
> - 태그 키는 팀 내에서 **일관된 규칙**을 정하세요. (`createdBy` vs `CreatedBy` vs `created_by`가 섞이면 필터링 불가)
> - CloudFormation/Terraform으로 생성하면 태그가 자동으로 붙으므로 누락 걱정이 없습니다.
> - 실습이 끝나면 Tag Editor에서 `Session` 태그로 검색하여 해당 세션의 리소스를 한번에 확인하고 정리하세요.

## 📚 참고: 네트워크 진단 명령어

실습이나 트러블슈팅 시 유용한 터미널 명령어입니다.

| 용도             | macOS / Linux                   | Windows                            | 설명                               |
| ---------------- | ------------------------------- | ---------------------------------- | ---------------------------------- |
| IP 주소 확인     | `ifconfig` 또는 `ip addr`       | `ipconfig`                         | 현재 네트워크 인터페이스 정보      |
| 공인 IP 확인     | `curl ifconfig.me`              | `curl ifconfig.me`                 | 외부에서 보이는 공인 IP 확인       |
| 연결 테스트      | `ping 8.8.8.8`                  | `ping 8.8.8.8`                     | 대상 호스트 도달 가능 여부         |
| DNS 조회         | `nslookup example.com`          | `nslookup example.com`             | 도메인의 IP 주소 확인              |
| DNS 상세 조회    | `dig example.com`               | -                                  | DNS 레코드 상세 정보 (A, CNAME 등) |
| 경로 추적        | `traceroute 8.8.8.8`            | `tracert 8.8.8.8`                  | 패킷이 거치는 라우터 경로 확인     |
| 포트 연결 테스트 | `nc -zv host 22`                | `Test-NetConnection host -Port 22` | 특정 포트 열려있는지 확인          |
| 라우팅 테이블    | `netstat -rn` 또는 `ip route`   | `route print`                      | 현재 라우팅 테이블 확인            |
| 열린 포트 확인   | `ss -tlnp` 또는 `netstat -tlnp` | `netstat -an`                      | 현재 리스닝 중인 포트 목록         |

> [!TIP]
> EC2 인스턴스에 SSH 접속 후 연결 문제를 진단할 때 이 명령어들을 활용하세요:
>
> - `ping`이 안 되면 → Security Group 또는 NACL에서 ICMP가 차단되었을 수 있음
> - `nc -zv`로 포트 테스트 실패 → Security Group 인바운드 규칙 확인
> - `curl ifconfig.me`로 공인 IP 확인 → NAT Gateway 또는 Elastic IP 동작 확인
>
> 일부 명령어는 기본 설치되어 있지 않을 수 있습니다:
>
> - **Amazon Linux 2023**: `sudo dnf install -y nc traceroute`
> - **Ubuntu**: `sudo apt install -y netcat traceroute`
> - **macOS**: `brew install nmap` (nc는 기본 포함)
> - **Windows**: `nslookup`, `tracert`, `ping`은 기본 포함. `curl`은 Windows 10 이상 기본 포함, `Test-NetConnection`은 PowerShell에서 사용

## 📚 참고: AWS 공인 IPv4 주소 과금 정책

> [!TIP]
> **AWS 공인 IPv4 주소에 비용이 발생하는 이유 (2024년 2월~)**
>
> 2024년 2월 1일부터 AWS는 모든 공인 IPv4 주소에 대해 시간당 $0.005(IP 1개당 월 약 $3.6)를 과금합니다.  
> 리소스에 연결되어 있든 아니든 동일하게 부과됩니다.
>
> **왜 유료화되었나?**
>
> - IPv4 주소는 전 세계적으로 **고갈** 상태이며, 공인 IPv4 1개를 확보하는 비용이 지난 5년간 300% 이상 상승했습니다.
> - AWS가 고객에게 공인 IP를 제공하려면 시장에서 높은 가격에 주소 블록을 매입해야 하므로, 그 비용을 반영한 것입니다.
> - 유료화를 통해 불필요하게 할당된 공인 IP를 회수하고, IPv6 전환을 장려하려는 목적도 있습니다.
>
> **과금 대상 리소스:**
>
> - EC2 인스턴스의 공인 IP / Elastic IP
> - NAT Gateway의 공인 IP
> - ALB/NLB/CLB의 공인 IP
> - RDS 인스턴스의 공인 IP
> - VPN 터널, Global Accelerator 등
>
> **학습 시 참고할 점:**
>
> - **Free Tier (크레딧 기반 모델)**: 2025년 7월 15일 이후 생성된 계정은 기존 12개월 무료 대신 **크레딧 기반** 모델이 적용됩니다.  
>    가입 시 $100 크레딧이 지급되고, 온보딩 활동 완료 시 추가 $100을 받을 수 있습니다.  
>    공인 IPv4 비용도 이 크레딧에서 차감되며, **6개월 또는 크레딧 소진 시** 무료 플랜이 종료됩니다.  
>   (2025년 7월 15일 이전에 생성된 계정은 기존 12개월 무료 정책 적용)
> - **미사용 Elastic IP 주의**: 할당만 하고 인스턴스에 연결하지 않아도 과금(크레딧 차감)됩니다. 실습 후 반드시 릴리스하세요.
> - **비용 절감 방법**: Private Subnet + NAT Gateway 구성으로 공인 IP 사용을 최소화하거나, IPv6 듀얼 스택을 활용하면 공인 IPv4 의존도를 줄일 수 있습니다.
> - **Public IP Insights**: VPC 콘솔의 "IP Addresses > Public IPv4" 메뉴에서 현재 사용 중인 공인 IP를 모니터링할 수 있습니다.
>
> 참고: [AWS 공식 블로그 - New AWS Public IPv4 Address Charge](https://aws.amazon.com/blogs/aws/new-aws-public-ipv4-address-charge-public-ip-insights/)
>
> ※ 위 금액은 작성 시점 기준 참고 값이며, 실제 요금은 리전, 환율, AWS 정책 변경에 따라 상이할 수 있습니다.
