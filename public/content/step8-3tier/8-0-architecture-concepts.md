## 1. 모놀리식 vs 마이크로서비스

> [!CONCEPT] 모놀리식과 마이크로서비스 비교
> **모놀리식**(Monolithic)은 모든 기능이 하나의 애플리케이션에 통합된 구조이고, **마이크로서비스**(Microservices)는 각 기능을 독립된 서비스로 분리한 구조입니다. 프로젝트 규모와 팀 구성에 따라 적합한 아키텍처가 다릅니다.

### 핵심 용어 정리

| 용어                          | 설명                                                                  | AWS 관련                |
| ----------------------------- | --------------------------------------------------------------------- | ----------------------- |
| **모놀리식 (Monolithic)**     | 모든 기능이 하나의 애플리케이션에 통합된 아키텍처                     | 단일 EC2                |
| **마이크로서비스**            | 기능별로 독립된 작은 서비스로 분리한 아키텍처                         | ECS, Lambda             |
| **3-Tier 아키텍처**           | 프레젠테이션(웹) + 비즈니스 로직(앱) + 데이터(DB) 3계층 분리          | CloudFront + EC2 + RDS  |
| **REST API**                  | HTTP 메서드(GET, POST, PUT, DELETE)로 리소스를 조작하는 API 설계 방식 | API Gateway, ALB        |
| **Stateless**                 | 서버가 클라이언트의 상태를 저장하지 않음. 요청마다 독립적 처리        | Auto Scaling 가능       |
| **Stateful**                  | 서버가 클라이언트의 상태(세션)를 유지. 특정 서버에 종속               | Sticky Session          |
| **캐싱 (Caching)**            | 자주 요청되는 데이터를 빠른 저장소에 임시 보관하여 응답 속도 향상     | ElastiCache, CloudFront |
| **메시지 큐 (Message Queue)** | 서비스 간 비동기 통신을 위한 중간 버퍼. 생산자-소비자 패턴            | Amazon SQS              |

### 구조 비교

```
모놀리식 (Monolithic):
┌─────────────────────────────────────┐
│          하나의 애플리케이션        │
│  ┌──────┐ ┌─────┐ ┌─────┐ ┌─────┐   │
│  │사용자│ │상품 │ │주문 │ │결제 │   │
│  │관리  │ │관리 │ │관리 │ │처리 │   │
│  └──┬───┘ └──┬──┘ └──┬──┘ └──┬──┘   │
│     └────────┴───────┴───────┘      │
│              공유 DB                │
│         ┌──────────────┐            │
│         │   MySQL      │            │
│         └──────────────┘            │
└─────────────────────────────────────┘

마이크로서비스 (Microservices):
┌─────────┐  ┌─────────┐  ┌─────────┐  ┌─────────┐
│ 사용자  │  │ 상품    │  │ 주문    │  │ 결제    │
│ 서비스  │  │ 서비스  │  │ 서비스  │  │ 서비스  │
│         │  │         │  │         │  │         │
│┌───────┐│  │┌───────┐│  │┌───────┐│  │┌───────┐│
││ DB-1  ││  ││ DB-2  ││  ││ DB-3  ││  ││ DB-4  ││
│└───────┘│  │└───────┘│  │└───────┘│  │└───────┘│
└─────────┘  └─────────┘  └─────────┘  └─────────┘
  독립 배포    독립 배포    독립 배포    독립 배포
```

### 상세 비교표

| 항목          | 모놀리식             | 마이크로서비스           |
| ------------- | -------------------- | ------------------------ |
| **배포**      | 전체 재배포          | 서비스별 독립 배포       |
| **확장**      | 전체 확장            | 서비스별 개별 확장       |
| **기술 스택** | 단일                 | 서비스별 다양 가능       |
| **장애 영향** | 전체 서비스 중단     | 해당 서비스만 영향       |
| **개발 속도** | 초기 빠름, 후기 느림 | 초기 느림, 후기 빠름     |
| **복잡도**    | 코드 복잡            | 운영 복잡 (네트워크)     |
| **팀 구성**   | 하나의 큰 팀         | 서비스별 소규모 팀       |
| **적합 시점** | 초기 스타트업, MVP   | 성장한 서비스, 대규모 팀 |

### 선택 가이드

```
프로젝트 시작
    │
    ├── 팀 5명 이하, MVP 단계
    │   └── 모놀리식 추천 ✅
    │       (빠른 개발, 단순한 운영)
    │
    ├── 팀 10명+, 서비스 성장
    │   └── 모듈러 모놀리식 고려
    │       (내부적으로 모듈 분리)
    │
    └── 팀 50명+, 독립 배포 필요
        └── 마이크로서비스 전환 고려
            (조직 구조에 맞춰 분리)
```

## 2. 1-Tier / 2-Tier / 3-Tier 아키텍처

> [!CONCEPT] N-Tier 아키텍처 패턴
> **N-Tier 아키텍처**는 애플리케이션을 논리적 계층(Tier)으로 분리하는 설계 패턴입니다. 각 계층은 특정 역할을 담당하며, 계층 간 명확한 인터페이스를 통해 통신합니다. 분리를 통해 독립적 확장과 유지보수가 가능해집니다.

### 주요 용어

| 용어                  | 설명                                                       |
| --------------------- | ---------------------------------------------------------- |
| **1-Tier**            | 프론트엔드, 백엔드, DB가 모두 하나의 서버에 있는 구조      |
| **2-Tier**            | 클라이언트(UI)와 서버(로직+DB)로 분리된 구조               |
| **3-Tier**            | 프레젠테이션 + 애플리케이션 + 데이터 3계층으로 분리된 구조 |
| **Presentation Tier** | 사용자 인터페이스를 담당하는 계층 (프론트엔드)             |
| **Application Tier**  | 비즈니스 로직을 처리하는 계층 (백엔드 API)                 |
| **Data Tier**         | 데이터 저장·관리를 담당하는 계층 (데이터베이스)            |

### 아키텍처 진화

```
1-Tier (단일 계층):
┌─────────────────────────────────┐
│  하나의 서버에 모든 것          │
│  • 프론트엔드 (HTML)            │
│  • 백엔드 (비즈니스 로직)       │
│  • 데이터베이스                 │
│                                 │
│  예: 로컬 데스크톱 앱           │
└─────────────────────────────────┘

2-Tier (클라이언트-서버):
┌────────────┐         ┌───────────────────┐
│ 클라이언트 │ ◄─────► │     서버          │
│ (UI)       │         │ • 비즈니스 로직   │
│            │         │ • 데이터베이스    │
└────────────┘         └───────────────────┘

3-Tier (프레젠테이션-로직-데이터):
┌────────────┐    ┌───────────┐    ┌───────────┐
│프론트엔드  │───►│ 백엔드    │───►│ 데이터    │
│(Vue, React)│    │(Spring)   │    │(MySQL)    │
│            │    │           │    │           │
│ 정적 파일  │    │ API 서버  │    │ DB 서버   │
└────────────┘    └───────────┘    └───────────┘
 Presentation      Application       Data
    Tier              Tier           Tier
```

### AWS에서의 3-Tier 구현


## 3. 프론트엔드/백엔드 분리의 장점

> [!CONCEPT] 프론트엔드/백엔드 분리 아키텍처
> **프론트엔드/백엔드 분리**는 UI(프론트엔드)와 비즈니스 로직(백엔드)을 독립된 프로젝트로 개발·배포하는 아키텍처입니다. API를 통해 통신하며, 각각 최적의 기술과 인프라를 선택할 수 있습니다.

### 주요 용어

| 용어                                     | 설명                                                    |
| ---------------------------------------- | ------------------------------------------------------- |
| **SPA (Single Page Application)**        | 하나의 HTML 페이지에서 동적으로 콘텐츠를 갱신하는 웹 앱 |
| **SSR (Server Side Rendering)**          | 서버에서 HTML을 생성하여 클라이언트에 전달하는 방식     |
| **CORS (Cross-Origin Resource Sharing)** | 다른 도메인 간 API 호출을 허용하는 보안 메커니즘        |

### 분리 전후 비교

| 항목       | 통합 (SSR)         | 분리 (SPA + API)         |
| ---------- | ------------------ | ------------------------ |
| **배포**   | 함께 배포          | 독립 배포                |
| **확장**   | 함께 확장          | 개별 확장                |
| **기술**   | 같은 프레임워크    | 각각 최적 기술           |
| **팀**     | 풀스택 필요        | 프론트/백엔드 전문화     |
| **캐싱**   | 페이지 단위        | 정적 파일 CDN + API 캐시 |
| **모바일** | 별도 API 필요      | 같은 API 공유            |
| **SEO**    | 유리 (서버 렌더링) | 추가 작업 필요 (SSR/SSG) |

### 분리 아키텍처의 통신 흐름

```
┌───────────────┐                    ┌──────────────┐
│  프론트엔드   │   REST API 호출    │   백엔드     │
│  (Vue.js)     │ ──────────────────►│   (Spring)   │
│               │                    │              │
│  S3 +         │  JSON 응답         │   EC2 +      │
│  CloudFront   │ ◄──────────────────│   ALB        │
└───────────────┘                    └──────────────┘
│                                    │
│ 정적 파일 서빙                     │ 비즈니스 로직
│ • HTML, CSS, JS                    │ • 인증/인가
│ • 이미지, 폰트                     │ • DB CRUD
│ • CDN으로 빠른 전송                │ • 외부 API 연동
```

## 4. REST API 설계 원칙

> [!CONCEPT] REST API 설계 원칙
> **REST**(Representational State Transfer)는 HTTP 프로토콜을 활용한 API 설계 아키텍처 스타일입니다. 리소스를 URL로 식별하고, HTTP 메서드로 행위를 표현하며, 상태를 서버에 저장하지 않는(Stateless) 것이 핵심입니다.

### 주요 용어

| 용어                                       | 설명                                                      |
| ------------------------------------------ | --------------------------------------------------------- |
| **REST (Representational State Transfer)** | HTTP 메서드로 리소스를 조작하는 API 설계 스타일           |
| **리소스 (Resource)**                      | REST에서 URL로 식별되는 데이터 단위 (예: /users, /orders) |
| **HTTP 메서드**                            | GET(조회), POST(생성), PUT(수정), DELETE(삭제) 등의 동작  |
| **페이지네이션 (Pagination)**              | 대량 데이터를 페이지 단위로 나누어 응답하는 기법          |

### REST API 설계 규칙

| 규칙             | 좋은 예             | 나쁜 예                 |
| ---------------- | ------------------- | ----------------------- |
| 명사 사용        | GET /users          | GET /getUsers           |
| 복수형           | GET /products       | GET /product            |
| 계층 표현        | GET /users/1/orders | GET /getUserOrders?id=1 |
| HTTP 메서드 활용 | DELETE /users/1     | POST /deleteUser        |
| 소문자           | /api/user-profiles  | /api/UserProfiles       |

### HTTP 메서드와 CRUD 매핑

| 메서드     | 용도      | 예시            | 응답 코드      |
| ---------- | --------- | --------------- | -------------- |
| **GET**    | 조회      | GET /users/1    | 200 OK         |
| **POST**   | 생성      | POST /users     | 201 Created    |
| **PUT**    | 전체 수정 | PUT /users/1    | 200 OK         |
| **PATCH**  | 부분 수정 | PATCH /users/1  | 200 OK         |
| **DELETE** | 삭제      | DELETE /users/1 | 204 No Content |

### REST API 응답 구조 예시

```json
// 성공 응답
{
  "status": 200,
  "data": {
    "id": 1,
    "name": "김철수",
    "email": "kim@example.com"
  }
}

// 목록 응답 (페이지네이션)
{
  "status": 200,
  "data": [...],
  "pagination": {
    "page": 1,
    "size": 20,
    "totalPages": 5,
    "totalElements": 100
  }
}

// 에러 응답
{
  "status": 404,
  "error": "NOT_FOUND",
  "message": "사용자를 찾을 수 없습니다."
}
```

## 5. 상태 관리: Stateless vs Stateful

> [!CONCEPT] Stateless와 Stateful의 차이
> **Stateless**(무상태)는 서버가 클라이언트의 상태를 저장하지 않는 방식이고, **Stateful**(상태 유지)은 서버가 세션 정보를 유지하는 방식입니다. 수평 확장을 위해서는 Stateless 설계가 필수적입니다.

### 주요 용어

| 용어                     | 설명                                             |
| ------------------------ | ------------------------------------------------ |
| **Stateless (무상태)**   | 서버가 클라이언트의 상태를 저장하지 않는 방식    |
| **Stateful (상태 유지)** | 서버가 세션 정보를 메모리에 유지하는 방식        |
| **세션 (Session)**       | 사용자의 로그인 상태 등을 유지하는 정보          |
| **JWT (JSON Web Token)** | 클라이언트 측에서 인증 정보를 보관하는 토큰 방식 |

### Stateful의 문제점

```
Stateful (세션을 서버 메모리에 저장):

사용자A ──► EC2-1 (세션 저장) ✅ 정상
사용자A ──► EC2-2 (세션 없음) ❌ 로그인 풀림!

문제: 로드밸런서가 다른 서버로 보내면 세션 유실
해결1: Sticky Session (같은 서버로 고정) → 부하 불균형
해결2: 세션 외부 저장소 사용 → Stateless 전환
```

### Stateless 설계

```
Stateless (세션을 외부 저장소에 저장):

┌────────┐     ┌─────┐     ┌───────┐
│ 사용자 │────►│ ALB │────►│ EC2-1 │──┐
└────────┘     └─────┘     │ EC2-2 │  │  세션 조회
                           │ EC2-3 │  │
                           └───────┘  │
                                      ▼
                           ┌───────────────┐
                           │ Redis/        │
                           │ DynamoDB      │
                           │ (세션 저장소) │
                           └───────────────┘

어떤 서버로 가든 동일한 세션 정보 접근 가능
→ 자유로운 수평 확장 가능
```

### Stateless 달성 방법

| 방법           | 설명                   | 장점                 | 단점                     |
| -------------- | ---------------------- | -------------------- | ------------------------ |
| **JWT 토큰**   | 클라이언트가 토큰 보관 | 서버 부담 없음       | 토큰 크기, 무효화 어려움 |
| **Redis 세션** | 외부 캐시에 세션 저장  | 빠른 접근, 만료 관리 | 추가 인프라 필요         |
| **DynamoDB**   | NoSQL에 세션 저장      | 관리형, 확장성       | 비용                     |

## 6. 캐싱 전략 (CDN, Redis)

> [!CONCEPT] 다계층 캐싱 전략
> **캐싱**은 자주 요청되는 데이터를 빠른 저장소에 보관하여 응답 시간을 단축하는 전략입니다. 웹 아키텍처에서는 CDN(정적 콘텐츠), Redis(동적 데이터), 브라우저 캐시 등 여러 계층에서 캐싱을 적용합니다.

### 주요 용어

| 용어                   | 설명                                                  |
| ---------------------- | ----------------------------------------------------- |
| **Redis**              | 인메모리 키-값 저장소. 캐싱, 세션 저장에 사용         |
| **ElastiCache**        | AWS의 관리형 Redis/Memcached 서비스                   |
| **Cache-Aside 패턴**   | 앱이 캐시 확인 → 미스 시 DB 조회 → 캐시 저장하는 패턴 |
| **TTL (Time To Live)** | 캐시 데이터의 유효 시간. 만료 후 자동 삭제            |

### 캐싱 계층별 역할

```
요청 흐름과 캐시 계층:

┌─────────┐  ┌──────────┐  ┌──────────┐  ┌───────┐  ┌────┐
│브라우저 │─►│CloudFront│─►│  Spring  │─►│ Redis │─►│ DB │
│ 캐시    │  │  (CDN)   │  │  Boot    │  │(캐시) │  │    │
└─────────┘  └──────────┘  └──────────┘  └───────┘  └────┘

  정적 자산    정적 파일     API 응답      DB 쿼리    원본
  (JS, CSS)   (이미지)     캐시          결과 캐시   데이터

  <1ms        5~50ms       ~100ms       1~5ms      50ms+
```

### CDN vs Redis 캐싱 비교

| 항목          | CDN (CloudFront)        | Redis (ElastiCache)   |
| ------------- | ----------------------- | --------------------- |
| **대상**      | 정적 콘텐츠             | 동적 데이터           |
| **위치**      | 엣지 로케이션 (전 세계) | Amazon VPC 내부       |
| **프로토콜**  | HTTP/HTTPS              | TCP (Redis 프로토콜)  |
| **TTL**       | 초~일 단위              | 초~시간 단위          |
| **무효화**    | Invalidation API        | DEL 명령              |
| **사용 사례** | 이미지, JS, CSS, HTML   | 세션, 쿼리 결과, 랭킹 |

### Redis 캐싱 패턴 예시

```
// Cache-Aside 패턴 (Spring Boot)

public User getUser(Long id) {
    // 1. Redis에서 먼저 조회
    String key = "user:" + id;
    User cached = redis.get(key);

    if (cached != null) {
        return cached;  // Cache Hit → 빠른 응답
    }

    // 2. Cache Miss → DB 조회
    User user = userRepository.findById(id);

    // 3. Redis에 저장 (TTL 5분)
    redis.set(key, user, Duration.ofMinutes(5));

    return user;
}
```

## 7. 비동기 처리와 메시지 큐

> [!CONCEPT] 비동기 처리와 메시지 큐
> **비동기 처리**는 시간이 오래 걸리는 작업을 즉시 응답하고 백그라운드에서 처리하는 패턴입니다. **메시지 큐**는 생산자(Producer)와 소비자(Consumer) 사이에서 메시지를 임시 저장하여 비동기 통신을 가능하게 합니다.

### 주요 용어

| 용어                                  | 설명                                                                 |
| ------------------------------------- | -------------------------------------------------------------------- |
| **비동기 처리 (Asynchronous)**        | 시간이 오래 걸리는 작업을 백그라운드에서 처리하고 즉시 응답하는 패턴 |
| **메시지 큐 (Message Queue)**         | 생산자와 소비자 사이에서 메시지를 임시 저장하는 버퍼                 |
| **Producer (생산자)**                 | 메시지를 큐에 넣는 주체 (API 서버 등)                                |
| **Consumer (소비자)**                 | 큐에서 메시지를 꺼내 처리하는 주체 (Worker 등)                       |
| **Amazon SQS**                        | AWS의 완전 관리형 메시지 큐 서비스                                   |
| **SNS (Simple Notification Service)** | 1:N 팬아웃 알림을 위한 Pub/Sub 서비스                                |

### 동기 vs 비동기 처리

```
동기 처리 (사용자 대기):
사용자 ──► API ──► 이메일 발송 (3초) ──► 응답
                   총 응답 시간: 3초+ 😰

비동기 처리 (즉시 응답):
사용자 ──► API ──► 큐에 메시지 추가 ──► 즉시 응답 (50ms) 😊
                        │
                        ▼ (백그라운드)
                   Worker ──► 이메일 발송 (3초)
```

### 메시지 큐 아키텍처

```
┌────────────┐     ┌─────────────┐     ┌───────────┐
│ Producer   │────►│  Message    │────►│ Consumer  │
│ (API 서버) │     │  Queue      │     │ (Worker)  │
│            │     │  (SQS)      │     │           │
│ 주문 접수  │     │ ┌──┬──┬──┐  │     │ 주문 처리 │
│            │     │ │M1│M2│M3│  │     │ 이메일    │
└────────────┘     │ └──┴──┴──┘  │     │ 알림 발송 │
                   └─────────────┘     └───────────┘

특징:
• Producer와 Consumer가 독립적으로 동작
• 큐가 버퍼 역할 → 트래픽 급증 시 안정적
• Consumer 장애 시 메시지 유실 없음 (큐에 보관)
```

### AWS 메시지 서비스 비교

| 서비스           | 유형        |   순서 보장    | 사용 사례            |
| ---------------- | ----------- | :------------: | -------------------- |
| **SQS Standard** | 큐          |   ❌ (최선)    | 일반 비동기 처리     |
| **SQS FIFO**     | 큐          |       ✅       | 순서 중요한 처리     |
| **SNS**          | Pub/Sub     |       ❌       | 팬아웃 (1:N 알림)    |
| **EventBridge**  | 이벤트 버스 |       ❌       | 이벤트 기반 아키텍처 |
| **Kinesis**      | 스트림      | ✅ (파티션 내) | 실시간 데이터 처리   |

### 비동기 처리 적합 사례

| 사례            | 이유                        |
| --------------- | --------------------------- |
| 이메일/SMS 발송 | 외부 서비스 호출, 지연 가능 |
| 이미지 리사이징 | CPU 집약적, 시간 소요       |
| 결제 처리       | 외부 PG사 응답 대기         |
| 로그 수집       | 대량 데이터, 실시간 불필요  |
| 알림 발송       | 다수 사용자에게 동시 발송   |

## 8. IaC 스택 분리 전략

> [!CONCEPT] 왜 AWS CloudFormation 스택을 나누는가?
> 하나의 거대한 스택으로 모든 인프라를 관리하면 편해 보이지만, 실무에서는 반드시 문제가 발생합니다.  
> 스택을 **라이프사이클과 팀 단위**로 분리하면 안전하고 효율적인 인프라 관리가 가능합니다.
>
> - 하나의 스택에 모든 것을 넣으면 "파스타 한 그릇"과 같습니다. 면(네트워크) 하나를 빼려고 건드리면 소스(DB)가 쏟아집니다.
> - 스택을 나누면 "도시락 칸"처럼 밥(네트워크), 반찬(데이터), 국(앱)을 독립적으로 꺼내고 넣을 수 있습니다.

### 스택을 분리하는 3가지 기준

| 기준             | 설명                              | 예시                                         |
| ---------------- | --------------------------------- | -------------------------------------------- |
| **라이프사이클** | 변경 빈도가 비슷한 것끼리 묶기    | VPC는 거의 안 바뀜, 앱은 자주 배포           |
| **팀 단위**      | 각 팀이 독립적으로 관리할 수 있게 | 프론트팀 → S3/CloudFront, 백엔드팀 → ALB/EC2 |
| **위험도**       | 삭제 시 영향이 큰 것을 격리       | RDS를 별도 스택으로 분리 → 실수 삭제 방지    |

### 실무에서의 스택 분리 예시

```
┌─────────────────────────────────────────────────────────────┐
│  스택 1: Network (기반 인프라)                              │
│  VPC, Subnet, IGW, NAT, Route Table, Security Groups        │
│  변경 빈도: 거의 없음 | 담당: 인프라팀                      │
├─────────────────────────────────────────────────────────────┤
│  스택 2: Data (데이터 계층)                                 │
│  RDS, DB Subnet Group                                       │
│  변경 빈도: 거의 없음 | 담당: DBA / 백엔드팀                │
│  ⚠️ DeletionPolicy: Snapshot (실수 삭제 시 복구 가능)       │
├─────────────────────────────────────────────────────────────┤
│  스택 3: Frontend (프론트엔드 인프라)                       │
│  S3 Bucket                                                  │
│  변경 빈도: 배포마다 | 담당: 프론트엔드팀                   │
├─────────────────────────────────────────────────────────────┤
│  스택 4: Backend (백엔드 인프라)                            │
│  ALB, Target Group, Listener                                │
│  변경 빈도: 배포마다 | 담당: 백엔드팀                       │
└─────────────────────────────────────────────────────────────┘
```

### 의존성과 생성 순서

스택 간에는 **의존 관계**가 있습니다. 네트워크가 없으면 RDS도 ALB도 만들 수 없습니다.

```
생성 순서 (의존성 순):

  ① Network  ─────────────────────────┐
       │                               │
       ▼                               ▼
  ② Data (RDS)                  ④ Backend (ALB)
  (VPC, Subnet 참조)            (VPC, Subnet 참조)

  ③ Frontend (S3) ← 독립 (①과 동시에 생성 가능)
```

| 스택     | 의존하는 값                          | 출처    |
| -------- | ------------------------------------ | ------- |
| Network  | 없음 (기반)                          | —       |
| Data     | VPC ID, Private Subnet ID, EC2 SG ID | Network |
| Frontend | 없음 (Amazon S3는 VPC 외부 서비스)   | —       |
| Backend  | VPC ID, Public Subnet ID, ALB SG ID  | Network |

### Cross-stack Reference (스택 간 값 참조)

AWS CloudFormation은 `Export`/`ImportValue`로 스택 간 값을 주고받습니다:

```yaml
# Network 스택 - 값을 내보냄 (Export)
Outputs:
  VPCId:
    Value: !Ref VPC
    Export:
      Name: !Sub '${ProjectName}-vpc-id'

# Data 스택 - 값을 가져옴 (ImportValue)
Resources:
  DBSubnetGroup:
    Type: AWS::RDS::DBSubnetGroup
    Properties:
      SubnetIds:
        - Fn::ImportValue: !Sub '${ProjectName}-private-subnet-1-id'
        - Fn::ImportValue: !Sub '${ProjectName}-private-subnet-2-id'
```

### 삭제 순서 (의존성 역순)

생성의 반대 순서로 삭제해야 합니다. 참조되는 스택을 먼저 삭제하면 에러가 발생합니다.

```
삭제 순서:

  ④ Backend (ALB)  ──┐
                      ├──→  ① Network (마지막)
  ② Data (RDS)     ──┘

  ③ Frontend (S3)  ──→  언제든 삭제 가능 (독립)
```

> Backend와 Data가 Network의 Export를 Import하므로, 이 두 스택을 먼저 삭제해야 Network를 삭제할 수 있습니다.  
> Frontend는 다른 스택에 의존하지 않으므로 순서 무관합니다.

> [!TIP]
> **실무에서의 삭제 실수 방지:**
>
> - Amazon RDS 스택에 `DeletionPolicy: Snapshot`을 설정하면 스택 삭제 시 자동으로 스냅샷이 생성됩니다.
> - Network 스택에 `TerminationProtection`을 활성화하면 실수로 삭제할 수 없습니다.
> - Tag Editor로 모든 스택 리소스를 한눈에 확인하고 누락 없이 정리할 수 있습니다.

### 하나의 스택 vs 분리된 스택 비교

| 항목            | 단일 스택                  | 분리된 스택                             |
| --------------- | -------------------------- | --------------------------------------- |
| **초기 설정**   | 간단 (파일 1개)            | 복잡 (Export/Import 설정)               |
| **배포 속도**   | 전체 업데이트 (느림)       | 변경된 스택만 업데이트 (빠름)           |
| **장애 범위**   | 하나 실패 → 전체 롤백      | 해당 스택만 영향                        |
| **팀 협업**     | 충돌 발생 (같은 파일 수정) | 팀별 독립 관리                          |
| **삭제 안전성** | RDS 포함 전체 삭제 위험    | 데이터 스택 별도 보호 가능              |
| **재사용성**    | 불가 (프로젝트 종속)       | Network 스택을 다른 프로젝트에서 재사용 |

> [!NOTE]
> 이 실습에서는 4개의 스택으로 분리하여 구축합니다:
>
> - `step8-network` → 네트워크 기반 (VPC, 서브넷, IGW, NAT, RT, SG)
> - `step8-data` → 데이터 계층 (DB Subnet Group, Amazon RDS)
> - `step8-frontend` → 프론트엔드 인프라 (Amazon S3 버킷)
> - `step8-backend` → 백엔드 인프라 (ALB, Target Group, Listener)
>
> Session 8-1에서 이 4개 스택을 순서대로 생성하고, Cross-stack Reference로 연결합니다.
