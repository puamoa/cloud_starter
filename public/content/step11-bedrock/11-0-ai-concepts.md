## 1. LLM(Large Language Model)이란?

> [!CONCEPT] LLM의 정의와 동작 원리
> **LLM**(대규모 언어 모델)은 방대한 텍스트 데이터로 학습된 딥러닝 모델로, 자연어를 이해하고 생성할 수 있습니다. "다음에 올 가장 적절한 단어를 예측"하는 방식으로 동작하며, 수십억~수조 개의 파라미터를 가집니다.

### 핵심 용어 정리

| 용어                                     | 설명                                                                | AWS 관련                   |
| ---------------------------------------- | ------------------------------------------------------------------- | -------------------------- |
| **LLM (Large Language Model)**           | 대규모 텍스트 데이터로 학습된 언어 생성 AI 모델                     | Claude, Llama, Titan       |
| **Foundation Model**                     | 범용 목적으로 사전 학습된 대규모 AI 모델. 다양한 태스크에 활용 가능 | Amazon Bedrock 모델        |
| **토큰 (Token)**                         | LLM이 텍스트를 처리하는 최소 단위. 비용 계산의 기준                 | 입력/출력 토큰 수 × 단가   |
| **프롬프트 (Prompt)**                    | AI 모델에 전달하는 입력 텍스트. 질문, 지시, 맥락 등을 포함          | System/User/Assistant      |
| **Temperature**                          | 응답의 무작위성을 조절하는 파라미터. 0=결정적, 1=창의적             | 0.0 ~ 1.0                  |
| **Top-P**                                | 누적 확률 기준으로 후보 토큰을 제한하는 파라미터                    | 0.0 ~ 1.0                  |
| **RAG (Retrieval Augmented Generation)** | 외부 지식을 검색하여 LLM 응답에 반영하는 패턴. 환각 감소            | Bedrock Knowledge Base     |
| **Fine-tuning**                          | 사전 학습된 모델을 특정 도메인 데이터로 추가 학습하는 방법          | Bedrock Custom Model       |
| **Prompt Engineering**                   | Fine-tuning 없이 프롬프트 설계만으로 원하는 결과를 얻는 기법        | Few-shot, Chain-of-Thought |
| **AI Agent**                             | 자율적으로 계획·도구 사용·검증을 반복하는 AI 시스템                  | Bedrock Agents             |
| **MCP (Model Context Protocol)**         | AI 모델이 외부 시스템과 상호작용하기 위한 표준 프로토콜              | Anthropic 제안             |
| **Extended Thinking**                    | AI가 답변 전에 내부적으로 긴 추론 과정을 수행하는 기능              | Claude 4.x 지원            |

### LLM의 동작 원리 (간략)

```
입력: "오늘 날씨가"

LLM 내부 처리:
┌─────────────────────────────────────────────────────┐
│  "오늘 날씨가" → 다음 단어 확률 계산                │
│                                                     │
│  "좋다"    → 35% ████████████                       │
│  "맑다"    → 25% ████████                           │
│  "흐리다"  → 15% █████                              │
│  "춥다"    → 10% ███                                │
│  "덥다"    → 8%  ██                                 │
│  ...       → 7%  ██                                 │
│                                                     │
│  선택: "좋다" (확률 기반 샘플링)                    │
└─────────────────────────────────────────────────────┘

출력: "오늘 날씨가 좋다"
→ 이 과정을 반복하여 문장/문단 생성
```

### LLM 규모의 진화

| 모델          | 출시    | 파라미터 수 | 특징                             |
| ------------- | ------- | ----------- | -------------------------------- |
| GPT-2         | 2019    | 15억        | 텍스트 생성 가능                 |
| GPT-3         | 2020    | 1,750억     | Few-shot 학습                    |
| GPT-4         | 2023    | 추정 1조+   | 멀티모달 (텍스트+이미지)         |
| Claude 3      | 2024.03 | 비공개      | 200K 토큰 컨텍스트               |
| Llama 3       | 2024.04 | 700억       | 오픈소스                         |
| Claude 3.5    | 2024.06 | 비공개      | Artifacts, Computer Use          |
| Claude 4      | 2025.05 | 비공개      | 1M 토큰 컨텍스트, 에이전트 코딩  |
| Claude 4.5/4.6| 2025~26 | 비공개      | 장시간 자율 작업, 코딩 성능 향상 |
| Claude Opus 4.8| 2026.06| 비공개      | 최고 성능, 복잡한 추론           |

### LLM이 할 수 있는 것과 없는 것

| 할 수 있는 것         | 할 수 없는 것                |
| --------------------- | ---------------------------- |
| 텍스트 생성/요약/번역 | 실시간 정보 접근 (학습 이후) |
| 코드 작성/설명        | 수학적 정확한 계산 보장      |
| 질문 답변             | 사실 여부 100% 보장          |
| 분류/감정 분석        | 물리적 행동 수행             |
| 창작 (시, 소설)       | 학습 데이터에 없는 지식      |

## 2. Transformer 아키텍처 간단 설명

> [!CONCEPT] Transformer와 Self-Attention
> **Transformer**는 2017년 "Attention Is All You Need" 논문에서 제안된 딥러닝 아키텍처입니다. **Self-Attention** 메커니즘을 통해 입력 시퀀스의 모든 위치 간 관계를 동시에 파악하며, 이전의 RNN/LSTM보다 병렬 처리에 유리합니다.

### 주요 용어

| 용어                                  | 설명                                                       |
| ------------------------------------- | ---------------------------------------------------------- |
| **Transformer**                       | Self-Attention 기반의 딥러닝 아키텍처. 현대 LLM의 핵심     |
| **Self-Attention**                    | 입력 시퀀스의 모든 위치 간 관계를 동시에 파악하는 메커니즘 |
| **임베딩 (Embedding)**                | 단어/토큰을 고차원 벡터로 변환하는 과정                    |
| **위치 인코딩 (Positional Encoding)** | 토큰의 순서 정보를 벡터에 추가하는 기법                    |

### Transformer 구조 (간략)

```
┌──────────────────────────────────────────────────────┐
│  Transformer 아키텍처                                │
│                                                      │
│  입력: "AWS는 클라우드 서비스입니다"                 │
│         │                                            │
│         ▼                                            │
│  ┌──────────────────┐                                │
│  │  토큰 임베딩     │  단어 → 벡터 변환              │
│  │  + 위치 인코딩   │  순서 정보 추가                │
│  └────────┬─────────┘                                │
│           │                                          │
│           ▼                                          │
│  ┌──────────────────┐                                │
│  │  Self-Attention  │  각 단어가 다른 모든 단어와의  │
│  │  (핵심!)         │  관계를 계산                   │
│  │                  │                                │
│  │  "AWS" ←→ "클라우드" (높은 관련성)                │
│  │  "AWS" ←→ "서비스" (높은 관련성)                  │
│  │  "는"  ←→ "입니다" (문법적 관계)                  │
│  └────────┬─────────┘                                │
│           │                                          │
│           ▼                                          │
│  ┌──────────────────┐                                │
│  │  Feed Forward    │  비선형 변환                   │
│  │  Network         │                                │
│  └────────┬─────────┘                                │
│           │                                          │
│           ▼ (여러 층 반복)                           │
│                                                      │
│  출력: 다음 토큰 확률 분포                           │
└──────────────────────────────────────────────────────┘
```

### Self-Attention의 핵심 아이디어

```
문장: "그 동물은 길을 건너지 않았다. 왜냐하면 그것은 너무 피곤했기 때문이다."

질문: "그것"이 가리키는 것은?

Self-Attention:
"그것" → "동물" (높은 attention score)
"그것" → "길" (낮은 attention score)

→ 문맥을 파악하여 "그것 = 동물"임을 이해
→ 이것이 LLM이 문맥을 이해하는 핵심 메커니즘
```

## 3. 토큰 개념

> [!CONCEPT] 토큰과 비용 계산
> **토큰**(Token)은 LLM이 텍스트를 처리하는 기본 단위입니다. 단어, 단어의 일부, 또는 문자가 될 수 있습니다. 영어는 약 1단어 ≈ 1.3토큰, 한국어는 약 1글자 ≈ 1~2토큰입니다. 토큰 수가 비용과 처리 시간을 결정합니다.

### 주요 용어

| 용어                                 | 설명                                                          |
| ------------------------------------ | ------------------------------------------------------------- |
| **토큰 (Token)**                     | LLM이 텍스트를 처리하는 기본 단위. 단어, 단어 일부, 또는 문자 |
| **토크나이저 (Tokenizer)**           | 텍스트를 토큰으로 분할하는 알고리즘                           |
| **컨텍스트 윈도우 (Context Window)** | LLM이 한 번에 처리할 수 있는 최대 토큰 수                     |
| **입력 토큰 / 출력 토큰**            | 프롬프트의 토큰 수 / 생성된 응답의 토큰 수. 각각 별도 과금    |

### 토큰화 예시

```
영어:
"Hello, how are you?"    → ["Hello", ",", " how", " are", " you", "?"]
                           6 토큰

한국어:
"안녕하세요, 반갑습니다" → ["안녕", "하세요", ",", " 반", "갑", "습니다"]
                           약 6~8 토큰 (토크나이저에 따라 다름)

코드:
"console.log('hello')"   → ["console", ".", "log", "('", "hello", "')"]
                           약 6 토큰
```

| Claude Haiku 4.5  | $1.00               | $5.00               | 200K            |
| Claude Sonnet 4.6 | $3.00               | $15.00              | 1M              |
| Claude Opus 4.8   | $5.00               | $25.00              | 1M              |
| GPT-4o            | $2.50               | $10.00              | 128K            |
| GPT-4o mini       | $0.15               | $0.60               | 128K            |

### 컨텍스트 윈도우

```

│  컨텍스트 윈도우 (예: 200K 토큰)                    │

│  • 비용 증가                                        │

```

## 4. Foundation Model

> [!CONCEPT] Foundation Model이란
> **Foundation Model**(기반 모델)은 대규모 데이터로 사전 학습된 범용 AI 모델입니다. 특정 작업에 특화되지 않고, 다양한 작업에 적용할 수 있는 기반을 제공합니다. 이 위에 Fine-tuning이나 Prompt Engineering으로 특정 용도에 맞게 활용합니다.

### 주요 용어

| 용어                             | 설명                                                       |
| -------------------------------- | ---------------------------------------------------------- |
| **Foundation Model (기반 모델)** | 대규모 데이터로 사전 학습된 범용 AI 모델                   |
| **사전 학습 (Pre-training)**     | 대량의 텍스트 데이터로 모델의 기본 능력을 학습시키는 과정  |
| **Zero-shot**                    | 예시 없이 바로 질문하는 방식                               |
| **Few-shot**                     | 몇 가지 예시를 제공한 후 질문하는 방식                     |
| **멀티모달 (Multimodal)**        | 텍스트, 이미지, 음성 등 여러 형태의 데이터를 처리하는 능력 |

| 방법                   | 설명                      | 비용 |  정확도   |
| ---------------------- | ------------------------- | :--: | :-------: |

| **Fine-tuning**        | 모델 자체를 추가 학습     | 높음 |   최고    |

## 5. Fine-tuning vs Prompt Engineering

> [!CONCEPT] Fine-tuning과 Prompt Engineering
> **Fine-tuning**은 기반 모델을 특정 도메인 데이터로 추가 학습시키는 것이고, **Prompt Engineering**은 모델을 변경하지 않고 입력(프롬프트)을 최적화하여 원하는 출력을 얻는 기술입니다.

### 주요 용어

| 용어                   | 설명                                                         |
| ---------------------- | ------------------------------------------------------------ |
| **Fine-tuning**        | 기반 모델을 특정 도메인 데이터로 추가 학습시키는 방법        |
| **Prompt Engineering** | 모델을 변경하지 않고 프롬프트 설계로 원하는 출력을 얻는 기법 |
| **Chain-of-Thought**   | 단계별 추론 과정을 유도하는 프롬프트 기법                    |
| **System Prompt**      | AI의 역할·규칙·맥락을 정의하는 시스템 수준 지시문            |

### 비교 다이어그램

```
Fine-tuning:
┌──────────────┐     추가 학습 데이터     ┌──────────────┐
│ Foundation   │ ──────────────────────►  │ Fine-tuned   │
│ Model        │     (수천~수만 예시)     │ Model        │
│ (범용)       │     시간: 수 시간~수 일  │ (특화)       │
└──────────────┘     비용: $$$            └──────────────┘

Prompt Engineering:
┌──────────────┐     정교한 프롬프트      ┌──────────────┐
│ Foundation   │ ◄─────────────────────   │ 사용자 입력  │
│ Model        │     (지시, 예시, 맥락)   │ + 프롬프트   │
│ (변경 없음)  │     시간: 즉시           │              │
└──────────────┘     비용: $              └──────────────┘
```

### 상세 비교

| 항목            | Fine-tuning              | Prompt Engineering         |
| --------------- | ------------------------ | -------------------------- |
| **모델 변경**   | ✅ 모델 가중치 수정      | ❌ 모델 그대로             |
| **데이터 필요** | 수천~수만 예시           | 0~수십 예시                |
| **비용**        | 높음 (학습 비용)         | 낮음 (API 호출만)          |
| **시간**        | 수 시간~수 일            | 즉시                       |
| **유연성**      | 낮음 (재학습 필요)       | 높음 (프롬프트만 수정)     |
| **적합 사례**   | 특수 도메인, 고유 스타일 | 일반 작업, 빠른 프로토타입 |
| **예시**        | 의료 용어 전문 모델      | "의사처럼 답변해주세요"    |

## 6. RAG (Retrieval Augmented Generation)

> [!CONCEPT] RAG 패턴의 동작 방식
> **RAG**는 LLM의 답변 생성 전에 외부 지식 저장소에서 관련 정보를 검색(Retrieve)하여 프롬프트에 포함시키는 패턴입니다. LLM의 학습 데이터에 없는 최신 정보나 기업 내부 문서를 활용할 수 있게 합니다.

### 주요 용어

| 용어                                     | 설명                                                           |
| ---------------------------------------- | -------------------------------------------------------------- |
| **RAG (Retrieval Augmented Generation)** | 외부 지식을 검색하여 LLM 응답에 반영하는 패턴                  |
| **벡터 임베딩 (Vector Embedding)**       | 텍스트를 수치 벡터로 변환하여 유사도 검색을 가능하게 하는 기법 |
| **벡터 DB (Vector Database)**            | 벡터 임베딩을 저장하고 유사도 검색을 수행하는 데이터베이스     |
| **청크 (Chunk)**                         | 문서를 검색 단위로 분할한 조각 (보통 500~1000자)               |
| **환각 (Hallucination)**                 | LLM이 사실이 아닌 내용을 그럴듯하게 생성하는 현상              |

### RAG 동작 흐름

```

┌───────────────────────────────────────────────────────────┐
│  RAG 파이프라인                                           │
│                                                           │
│  1. 사전 준비 (Indexing):                                 │
│  ┌───────────┐    ┌───────────┐    ┌──────────────┐       │
│  │ 문서 수집 │───►│ 청크 분할 │───►│ 벡터 임베딩  │       │
│  │ (PDF,     │    │ (500자    │    │ + 벡터 DB    │       │
│  │  문서)    │    │  단위)    │    │ 저장         │       │
│  └───────────┘    └───────────┘    └──────────────┘       │
│                                                           │
│  2. 질의 시 (Query):                                      │
│  ┌───────┐    ┌───────────┐    ┌───────────┐    ┌──────┐  │
│  │ 질문  │───►│ 벡터 검색 │───►│ 관련 문서 │───►│ LLM  │  │
│  │       │    │ (유사도)  │    │ + 질문    │    │      │  │
│  └───────┘    └───────────┘    │ 합쳐서    │    │ 답변 │  │
│                                │ 프롬프트  │    │ 생성 │  │
│                                └───────────┘    └──────┘  │
└───────────────────────────────────────────────────────────┘

```

### RAG vs Fine-tuning 비교

| 항목              | RAG                     | Fine-tuning               |
| ----------------- | ----------------------- | ------------------------- |
| **지식 업데이트** | 문서 추가만으로 즉시    | 재학습 필요               |
| **출처 제공**     | ✅ (검색된 문서 참조)   | ❌                        |
| **환각 감소**     | 높음 (근거 기반 답변)   | 중간                      |
| **비용**          | 벡터 DB + 검색 비용     | 학습 비용 (1회)           |
| **적합 사례**     | FAQ, 문서 QA, 최신 정보 | 스타일/톤 변경, 특수 작업 |

### AWS에서의 RAG 구현

```

┌──────────────────────────────────────────────────────┐
│  AWS RAG 아키텍처                                    │
│                                                      │
│  문서 → S3 → Bedrock Knowledge Base → OpenSearch     │
│                                         (벡터 DB)    │
│                                            │         │
│  사용자 질문 → Bedrock (Claude) ◄──── 검색 결과      │
│                    │                                 │
│                    ▼                                 │
│              근거 기반 답변 + 출처                   │
└──────────────────────────────────────────────────────┘

```

## 7. Temperature / Top-P 파라미터

> [!CONCEPT] Temperature와 Top-P 파라미터
> **Temperature**와 **Top-P**는 LLM의 출력 다양성(창의성)을 제어하는 파라미터입니다. Temperature가 높으면 더 창의적이고 다양한 답변을, 낮으면 더 일관되고 예측 가능한 답변을 생성합니다.

### 주요 용어

| 용어                         | 설명                                                    |
| ---------------------------- | ------------------------------------------------------- |
| **Temperature**              | 응답의 무작위성을 조절하는 파라미터. 0=결정적, 1=창의적 |
| **Top-P (Nucleus Sampling)** | 누적 확률 기준으로 후보 토큰을 제한하는 파라미터        |
| **Top-K**                    | 확률 상위 K개 토큰만 후보로 고려하는 파라미터           |
| **Max Tokens**               | 생성할 최대 토큰 수를 제한하는 파라미터                 |
| **샘플링 (Sampling)**        | 확률 분포에서 다음 토큰을 선택하는 과정                 |

### Temperature 효과

```

Temperature = 0 (결정적):
입력: "AWS의 장점은"
출력: "비용 효율성, 확장성, 글로벌 인프라입니다." (항상 동일)

Temperature = 0.7 (균형):
입력: "AWS의 장점은"
출력: "다양한 서비스와 유연한 과금 모델이 큰 장점입니다." (적당히 다양)

Temperature = 1.0 (창의적):
입력: "AWS의 장점은"
출력: "마치 레고 블록처럼 원하는 인프라를 조립할 수 있다는 점이죠." (매우 다양)

```

### 파라미터 비교

| 파라미터        | 범위        | 낮은 값          | 높은 값           |
| --------------- | ----------- | ---------------- | ----------------- |
| **Temperature** | 0~1         | 결정적, 일관됨   | 창의적, 다양함    |
| **Top-P**       | 0~1         | 상위 소수 토큰만 | 더 많은 토큰 고려 |
| **Top-K**       | 1~∞         | 상위 K개만 고려  | 더 많은 후보      |
| **Max Tokens**  | 1~모델 한계 | 짧은 응답        | 긴 응답           |

### 용도별 권장 설정

| 용도         | Temperature | Top-P | 이유             |
| ------------ | :---------: | :---: | ---------------- |
| 코드 생성    |    0~0.2    |  0.1  | 정확성 중요      |
| 데이터 추출  |      0      |  0.1  | 일관된 형식 필요 |
| 일반 대화    |     0.7     |  0.9  | 자연스러운 응답  |
| 창작 글쓰기  |   0.9~1.0   | 0.95  | 다양성과 창의성  |
| 브레인스토밍 |     1.0     |  1.0  | 최대 다양성      |

## 8. AI 에이전트와 도구 사용 (Agentic AI)

> [!CONCEPT] AI 에이전트란
> **AI 에이전트**는 단순히 질문에 답하는 것을 넘어, 목표를 달성하기 위해 스스로 계획을 세우고, 도구를 사용하고, 반복적으로 행동하는 AI 시스템입니다. 2025~2026년 AI 발전의 핵심 방향입니다.

### 주요 용어

| 용어 | 설명 |
| ---- | ---- |
| **AI Agent (에이전트)** | 자율적으로 계획·실행·검증을 반복하는 AI 시스템 |
| **Tool Use (도구 사용)** | 외부 API, 데이터베이스, 코드 실행 등을 AI가 직접 호출하는 기능 |
| **Computer Use** | AI가 마우스·키보드를 조작하여 GUI 작업을 수행하는 기능 |
| **MCP (Model Context Protocol)** | AI 모델이 외부 시스템과 상호작용하기 위한 표준 프로토콜 (Anthropic 제안) |
| **Extended Thinking** | AI가 답변 전에 내부적으로 긴 추론 과정을 수행하는 기능 |

### 에이전트 동작 방식

```
┌─────────────────────────────────────────────────────────┐
│  AI Agent 동작 흐름                                     │
│                                                         │
│  [사용자 목표] → [계획 수립] → [도구 선택] → [실행]     │
│                       ↑                         │       │
│                       └── [결과 검증] ←─────────┘       │
│                                                         │
│  예시: "이 프로젝트의 버그를 찾아서 수정해줘"           │
│                                                         │
│  1. 계획: 코드 분석 → 테스트 실행 → 원인 파악 → 수정   │
│  2. 도구: 파일 읽기, 코드 검색, 터미널 실행             │
│  3. 실행: 파일을 읽고, 에러를 찾고, 수정 코드 작성      │
│  4. 검증: 테스트 재실행으로 수정 확인                   │
└─────────────────────────────────────────────────────────┘
```

### Claude의 도구 사용 예시 (Amazon Bedrock)

```
사용자: "서울 날씨를 확인하고, 비가 오면 우산 리마인더를 설정해줘"

Claude의 처리 과정:
1. [도구 호출] get_weather(location="Seoul") → "비, 70%"
2. [판단] 비 확률 높음 → 리마인더 필요
3. [도구 호출] set_reminder(message="우산 챙기기", time="08:00")
4. [응답] "서울에 비가 70% 예보되어 있어서 오전 8시 우산 리마인더를 설정했습니다."
```

### 에이전트 vs 기본 LLM 비교

| 항목 | 기본 LLM | AI Agent |
| ---- | -------- | -------- |
| 동작 | 입력 → 출력 (1회) | 계획 → 실행 → 검증 (반복) |
| 외부 시스템 | 접근 불가 | 도구로 접근 가능 |
| 자율성 | 수동 (매번 지시 필요) | 자율적 (목표만 제시) |
| 사용 사례 | Q&A, 요약, 번역 | 코딩, 리서치, 워크플로우 자동화 |
| AWS 서비스 | Bedrock Playground | Bedrock Agents |

## 9. 프롬프트 캐싱과 비용 최적화

> [!CONCEPT] 프롬프트 캐싱
> **프롬프트 캐싱**은 반복되는 시스템 프롬프트나 긴 컨텍스트를 캐시하여 비용을 최대 90% 절감하는 기술입니다. Amazon Bedrock과 Anthropic API 모두 지원합니다.

### 비용 최적화 전략

| 전략 | 절감률 | 설명 |
| ---- | ------ | ---- |
| **프롬프트 캐싱** | 최대 90% | 반복 사용되는 긴 프롬프트를 캐시 |
| **Batch Inference** | 50% | 실시간이 아닌 비동기 대량 처리 |
| **모델 선택 최적화** | 60~80% | 작업 복잡도에 맞는 모델 선택 |
| **토큰 최적화** | 20~40% | 불필요한 입력 제거, 간결한 프롬프트 |

### 모델 선택 기준 (2026년 6월 기준)

```
┌─────────────────────────────────────────────────────────┐

│  빠른 응답           균형                최고 품질      │
└─────────────────────────────────────────────────────────┘
```

## 10. AI 서비스 비교 (Bedrock vs SageMaker vs OpenAI API)

> [!CONCEPT] Bedrock, SageMaker, OpenAI 비교
> AWS에서 AI를 활용하는 방법은 여러 가지입니다. **Bedrock**은 다양한 Foundation Model을 API로 즉시 사용하는 서비스이고, **SageMaker**는 ML 모델을 직접 학습·배포하는 플랫폼입니다. **OpenAI API**는 외부 서비스로 GPT 모델을 사용합니다.

### 주요 용어

| 용어                      | 설명                                                                 |
| ------------------------- | -------------------------------------------------------------------- |
| **Amazon Bedrock**        | 다양한 Foundation Model을 API로 즉시 사용하는 AWS 서버리스 AI 서비스 |
| **Amazon SageMaker**      | ML 모델을 직접 학습·배포하는 AWS 플랫폼                              |
| **PrivateLink**           | VPC 내에서 인터넷 경유 없이 AWS 서비스에 접근하는 프라이빗 연결      |
| **엔드포인트 (Endpoint)** | 모델을 호출할 수 있는 API 접점                                       |

### 서비스 포지셔닝

```

관리 부담
  높음 │  SageMaker
       │  ├── 모델 직접 학습
       │  ├── 인프라 관리 (인스턴스 선택)
       │  ├── 엔드포인트 배포/관리
       │  └── 완전한 커스터마이징
       │
       │  Bedrock
       │  ├── Foundation Model API 호출
       │  ├── Fine-tuning 옵션
       │  ├── RAG (Knowledge Base)
       │  └── 서버리스 (인프라 관리 없음)
       │
  낮음 │  OpenAI API
       │  ├── GPT 모델 API 호출
       │  ├── 간단한 통합
       │  └── AWS 외부 서비스
       └──────────────────────────────── 커스터마이징
          낮음                        높음
```

### 상세 비교표

| 항목            | Bedrock                 | SageMaker              | OpenAI API      |
| --------------- | ----------------------- | ---------------------- | --------------- |
| **유형**        | 관리형 FM API           | ML 플랫폼              | 외부 API        |
| **모델**        | Claude, Llama, Titan 등 | 커스텀 + 허브          | GPT-4, DALL-E   |
| **학습**        | Fine-tuning (제한적)    | 완전한 학습 파이프라인 | Fine-tuning API |
| **인프라**      | 서버리스                | 인스턴스 관리          | 없음 (외부)     |
| **데이터 위치** | AWS 내부                | AWS 내부               | 외부 (미국)     |
| **비용 모델**   | 토큰당 과금             | 인스턴스 시간          | 토큰당 과금     |
| **VPC 통합**    | PrivateLink             | VPC 내 배포            | 인터넷 필요     |
| **적합 사례**   | 생성형 AI 앱            | 커스텀 ML 모델         | 빠른 프로토타입 |

### 선택 가이드

| 요구사항                         | 추천 서비스                    |
| -------------------------------- | ------------------------------ |
| FM을 빠르게 앱에 통합            | **Bedrock**                    |
| 데이터가 AWS 밖으로 나가면 안 됨 | **Bedrock** 또는 **SageMaker** |
| 자체 ML 모델 학습 필요           | **SageMaker**                  |
| 이미지/음성 등 커스텀 모델       | **SageMaker**                  |
| 가장 빠른 프로토타입             | **OpenAI API**                 |
| GPT-4 특정 기능 필요             | **OpenAI API**                 |
| 여러 FM 비교 후 선택             | **Bedrock** (다양한 모델 제공) |
