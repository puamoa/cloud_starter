이 실습에서는 Amazon Bedrock Playground를 사용하여 생성형 AI 모델(Claude)과 대화하고, 프롬프트 엔지니어링의 핵심 기법을 체험합니다. 이 실습을 완료하면 AWS 프리티어 추가 크레딧 활동 중 하나(Bedrock 사용)를 달성하여 $20 크레딧을 추가로 받을 수 있습니다.

> [!NOTE]
> 이 실습은 독립적으로 진행할 수 있습니다. AWS 계정만 있으면 바로 시작할 수 있습니다.

> [!WARNING]
> Amazon Bedrock은 토큰(입력/출력 텍스트 양) 기반으로 과금됩니다. 이 실습 수준의 사용량은 매우 적어 크레딧에서 소액만 차감됩니다. 하지만 대량의 텍스트를 반복 생성하면 비용이 증가할 수 있으니 주의하세요.

## 태스크 1: Amazon Bedrock 접속 및 모델 활성화

1. AWS Management Console에 로그인합니다.
2. 리전을 **Asia Pacific (Seoul) ap-northeast-2**로 설정합니다.
3. 상단 검색창에 `Bedrock`을 입력하고 **Amazon Bedrock**을 선택합니다.
4. 왼쪽 메뉴에서 **Playgrounds** → **Chat**을 선택합니다.

> [!NOTE]
> Amazon Bedrock에서는 서버리스 Foundation Model에 대한 액세스가 자동으로 활성화되어 있습니다. 단, Anthropic Claude 모델을 처음 사용하는 경우 일회성 **Use case details** 양식을 제출해야 합니다.

5. 모델 선택 드롭다운에서 **Anthropic** → **Claude 3.5 Sonnet** (또는 최신 Claude 모델)을 선택합니다.

6. Use case details 양식이 표시되면 다음과 같이 입력합니다:
   - **Company name**: `Student` (또는 본인 이름)
   - **Company website URL**: `https://example.com`
   - **Industry**: `Education`
   - **Intended users**: `Internal users` 체크
   - **Use cases**: `Learning and testing prompt engineering`

7. [[Submit use case details]] 버튼을 클릭합니다.

> [!NOTE]
> 양식 제출 후 즉시 승인됩니다. 이 양식은 계정당 한 번만 제출하면 됩니다.

8. Chat Playground 화면이 표시되는지 확인합니다.

✅ **태스크 완료**: Amazon Bedrock Playground에 접속하고 Claude 모델을 활성화했습니다.

## 태스크 2: Zero-shot 프롬프팅

예시 없이 작업을 직접 요청하는 가장 기본적인 프롬프팅 기법입니다.

> [!CONCEPT] Zero-shot 프롬프팅
> 모델에게 예시를 제공하지 않고 바로 작업을 요청합니다. 간단하고 명확한 작업에 적합합니다.

9. Chat Playground의 입력창에 다음 프롬프트를 입력합니다:

```
다음 문장의 감정을 분석해주세요: "오늘 날씨가 정말 좋아서 기분이 최고예요!"
```

10. [[Run]] 버튼을 클릭합니다.

> [!OUTPUT]
> 모델이 "긍정적" 또는 "매우 긍정적인 감정"이라고 응답합니다.

11. 다른 Zero-shot 프롬프트를 시도합니다:

```
AWS EC2와 Lambda의 차이점을 3줄로 요약해주세요.
```

12. 응답을 확인합니다.

> [!TIP]
> Zero-shot은 간단한 질문, 요약, 번역 등에 효과적입니다. 하지만 특정 형식이나 스타일을 원할 때는 Few-shot이 더 적합합니다.

✅ **태스크 완료**: Zero-shot 프롬프팅을 실습했습니다.

## 태스크 3: Few-shot 프롬프팅

몇 가지 예시를 제공하여 모델이 패턴을 학습하도록 합니다.

> [!CONCEPT] Few-shot 프롬프팅
> 원하는 입력-출력 패턴의 예시를 2~3개 제공한 후 새로운 입력을 주면, 모델이 동일한 패턴으로 응답합니다. 일관된 형식이 필요할 때 유용합니다.

13. 새 대화를 시작합니다 (기존 대화 아래에 입력하거나 [[Clear]] 클릭).
14. 다음 프롬프트를 입력합니다:

```
다음 형식으로 AWS 서비스를 분류해주세요.

입력: Amazon S3
출력: [스토리지] Amazon S3 - 오브젝트 스토리지 서비스

입력: AWS Lambda
출력: [컴퓨팅] AWS Lambda - 서버리스 함수 실행 서비스

입력: Amazon RDS
출력:
```

15. [[Run]] 버튼을 클릭합니다.

> [!OUTPUT]
> 모델이 동일한 형식으로 응답합니다:
> `[데이터베이스] Amazon RDS - 관리형 관계형 데이터베이스 서비스`

16. 추가로 다른 서비스를 입력하여 패턴이 유지되는지 확인합니다:

```
입력: Amazon CloudFront
출력:
```

✅ **태스크 완료**: Few-shot 프롬프팅으로 일관된 형식의 응답을 얻었습니다.

## 태스크 4: Chain-of-Thought (CoT) 프롬프팅

단계별 사고 과정을 유도하여 복잡한 문제를 해결합니다.

> [!CONCEPT] Chain-of-Thought 프롬프팅
> "단계별로 생각해보세요" 같은 지시를 추가하면 모델이 중간 추론 과정을 보여주며 더 정확한 답을 도출합니다. 수학 문제, 논리 추론, 복잡한 분석에 효과적입니다.

17. 다음 프롬프트를 입력합니다:

```
AWS에서 웹 애플리케이션을 배포하려고 합니다.
요구사항:
- Vue.js 프론트엔드
- Spring Boot 백엔드
- MySQL 데이터베이스
- 월 비용 최소화
- 가용성 확보

단계별로 생각하여 최적의 아키텍처를 설계해주세요.
```

18. [[Run]] 버튼을 클릭합니다.

> [!OUTPUT]
> 모델이 단계별로 사고 과정을 보여주며 아키텍처를 제안합니다. 프론트엔드 배포 방식, 백엔드 배포 방식, 데이터베이스 선택, 비용 최적화, 최종 아키텍처 순서로 분석 결과를 보여줍니다.

19. CoT 없이 같은 질문을 해보고 응답 품질을 비교합니다:

```
AWS에서 Vue.js + Spring Boot + MySQL 웹앱을 배포하는 최적의 아키텍처를 알려주세요.
```

> [!TIP]
> CoT를 사용하면 모델이 "왜 그런 선택을 했는지" 근거를 함께 제시하므로, 결과를 검증하기 쉽습니다.

✅ **태스크 완료**: Chain-of-Thought 프롬프팅으로 복잡한 문제를 단계별로 해결했습니다.

## 태스크 5: 시스템 프롬프트 설정

시스템 프롬프트로 모델의 역할과 응답 스타일을 제어합니다.

20. Chat Playground 왼쪽의 **System prompt** 영역을 찾습니다.
21. 다음 시스템 프롬프트를 입력합니다:

```
당신은 AWS 솔루션 아키텍트입니다.
다음 규칙을 따르세요:
- 모든 답변은 한국어로 합니다.
- 비용 최적화를 항상 고려합니다.
- 답변은 3문장 이내로 간결하게 합니다.
- 프리티어/크레딧 범위 내 사용 가능 여부를 항상 언급합니다.
```

22. 사용자 메시지로 다음을 입력합니다:

```
EC2 인스턴스 타입을 추천해주세요.
```

23. [[Run]] 버튼을 클릭합니다.

> [!OUTPUT]
> 모델이 시스템 프롬프트의 규칙(한국어, 3문장 이내, 비용 언급)을 따라 응답합니다.

24. 시스템 프롬프트를 변경하여 다른 스타일을 시도합니다:

```
당신은 초보자에게 AWS를 가르치는 친절한 튜터입니다.
비유와 예시를 많이 사용하여 쉽게 설명합니다.
```

25. 같은 질문을 다시 입력하고 응답 스타일이 달라지는지 확인합니다.

✅ **태스크 완료**: 시스템 프롬프트로 모델의 역할과 응답 스타일을 제어했습니다.

## 태스크 6: 파라미터 조정 (Temperature, Top P)

> [!CONCEPT] Temperature와 Top P
>
> - **Temperature** (0~1): 응답의 창의성/무작위성을 제어합니다.
>   - 0에 가까울수록: 일관되고 예측 가능한 응답 (사실 기반 질문에 적합)
>   - 1에 가까울수록: 다양하고 창의적인 응답 (브레인스토밍에 적합)
> - **Top P** (0~1): 응답 생성 시 고려하는 토큰의 범위를 제어합니다.
>   - 낮을수록: 가장 확률 높은 토큰만 선택 (보수적)
>   - 높을수록: 더 많은 토큰을 고려 (다양한 표현)

26. Playground 오른쪽의 **Configuration** 패널에서 파라미터를 확인합니다.
27. **Temperature**를 `0.1`로 설정합니다.
28. 다음 프롬프트를 입력합니다:

```
"클라우드 컴퓨팅"을 한 문장으로 정의해주세요.
```

29. [[Run]] 버튼을 클릭하고 응답을 메모합니다.
30. 같은 프롬프트를 2~3번 더 실행합니다.

> [!OUTPUT]
> Temperature 0.1에서는 매번 거의 동일한 응답이 나옵니다.

31. **Temperature**를 `0.9`로 변경합니다.
32. 같은 프롬프트를 3번 실행하고 응답을 비교합니다.

> [!OUTPUT]
> Temperature 0.9에서는 매번 다른 표현과 관점의 응답이 나옵니다.

### 파라미터 사용 가이드

| 용도              | Temperature | Top P   | 예시             |
| ----------------- | ----------- | ------- | ---------------- |
| 코드 생성         | 0~0.2       | 0.1~0.3 | 정확한 코드 필요 |
| 사실 기반 Q&A     | 0.1~0.3     | 0.3~0.5 | AWS 서비스 설명  |
| 일반 대화         | 0.5~0.7     | 0.7~0.9 | 자연스러운 대화  |
| 창작/브레인스토밍 | 0.8~1.0     | 0.9~1.0 | 아이디어 생성    |

33. 원하는 용도에 맞게 파라미터를 조정하여 몇 가지 프롬프트를 더 시도합니다.

✅ **태스크 완료**: Temperature와 Top P 파라미터를 조정하여 응답 특성을 제어했습니다.

# 🗑️ 리소스 정리

> [!NOTE]
> Amazon Bedrock은 사용하지 않으면 비용이 발생하지 않습니다. EC2나 RDS처럼 "실행 중"인 리소스가 아니므로, 별도의 정리 작업이 필요 없습니다.

---

### 단계 1: 별도 삭제할 리소스 없음

이 실습에서는 별도로 생성하거나 삭제해야 할 리소스가 없습니다.

- **Bedrock Playground**: 사용한 만큼만 토큰 비용이 크레딧에서 차감됩니다. 사용을 중단하면 추가 비용이 발생하지 않습니다.
- **모델 액세스**: 비활성화할 필요 없습니다. 호출하지 않으면 과금되지 않습니다.
- **대화 기록**: Playground의 대화 기록은 세션 종료 시 자동으로 사라집니다.

✅ **실습 종료**: 별도로 삭제할 리소스가 없으며, Bedrock은 사용하지 않으면 비용이 발생하지 않습니다.
