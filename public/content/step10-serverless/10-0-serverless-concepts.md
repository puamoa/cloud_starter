# 서버리스와 NoSQL 이론

---

## 1. 서버리스란? (FaaS, BaaS)

> [!CONCEPT] 서버리스란 무엇인가
> **서버리스**(Serverless)는 서버 관리 없이 코드를 실행하는 클라우드 실행 모델입니다. 서버가 없는 것이 아니라, 서버 관리를 클라우드 제공자에게 완전히 위임하는 것입니다. **FaaS**(Function as a Service)와 **BaaS**(Backend as a Service)로 구분됩니다.

### 핵심 용어 정리

| 용어                             | 설명                                                                 | AWS 관련                |
| -------------------------------- | -------------------------------------------------------------------- | ----------------------- |
| **서버리스 (Serverless)**        | 서버 관리 없이 코드만 배포하면 자동으로 실행/확장되는 컴퓨팅 모델    | AWS Lambda              |
| **FaaS (Function as a Service)** | 함수 단위로 코드를 실행하는 서버리스 컴퓨팅                          | AWS Lambda              |
| **BaaS (Backend as a Service)**  | 인증, DB, 스토리지 등 백엔드 기능을 API로 제공하는 서비스            | Cognito, DynamoDB, S3   |
| **이벤트 드리븐 (Event-Driven)** | 이벤트 발생 시 함수가 트리거되어 실행되는 아키텍처                   | S3 이벤트 → Lambda      |
| **Cold Start**                   | Lambda 함수가 처음 호출될 때 실행 환경을 생성하느라 발생하는 지연    | 100ms ~ 수 초           |
| **NoSQL**                        | 고정 스키마 없이 유연한 데이터 모델을 지원하는 비관계형 데이터베이스 | Amazon DynamoDB         |
| **파티션 키 (Partition Key)**    | DynamoDB에서 데이터를 분산 저장하기 위한 기본 키                     | 고유 식별자 (userId 등) |
| **CAP 정리**                     | 분산 시스템에서 일관성, 가용성, 분할 내성 중 2개만 동시 만족 가능    | DynamoDB = AP           |
| **Eventually Consistent**        | 일시적 불일치를 허용하고 최종적으로 모든 노드가 동기화되는 모델      | DynamoDB 기본 읽기      |

### 서버리스 구분

```
┌─────────────────────────────────────────────────────────┐
│  서버리스 (Serverless)                                  │
│                                                         │
│  ┌─── FaaS (Function as a Service) ─────────────────┐   │
│  │  • 함수 단위로 코드 실행                         │   │
│  │  • 이벤트에 의해 트리거                          │   │
│  │  • 실행 시간만 과금                              │   │
│  │  • 예: AWS Lambda, Azure Functions               │   │
│  └──────────────────────────────────────────────────┘   │
│                                                         │
│  ┌─── BaaS (Backend as a Service) ──────────────────┐   │
│  │  • 완성된 백엔드 기능 제공                       │   │
│  │  • API로 즉시 사용                               │   │
│  │  • 예: DynamoDB, S3, Cognito, API Gateway        │   │
│  └──────────────────────────────────────────────────┘   │
└─────────────────────────────────────────────────────────┘
```

### 서버리스 vs 서버 기반 비교

| 항목              | 서버 기반 (EC2)       | 서버리스 (Lambda)   |
| ----------------- | --------------------- | ------------------- |
| **서버 관리**     | 직접 (OS, 패치, 확장) | 없음 (AWS 관리)     |
| **과금**          | 시간당 (유휴 시에도)  | 요청당 + 실행 시간  |
| **확장**          | 수동/ASG 설정         | 자동 (동시 실행)    |
| **가용성**        | 직접 Multi-AZ 구성    | 기본 내장           |
| **실행 시간**     | 무제한                | 최대 15분           |
| **상태**          | Stateful 가능         | Stateless           |
| **시작 시간**     | 수 분 (인스턴스 부팅) | 수 ms ~ 수 초       |
| **적합 워크로드** | 장시간, 상태 유지     | 이벤트 기반, 간헐적 |

### 서버리스 비용 모델

```
EC2 (항상 실행):
비용 ████████████████████████████████ 24시간 과금
     0시                            24시
     (트래픽 없어도 과금)

Lambda (실행 시만):
비용 █ █ ███ █ ██ █                   요청 시만 과금
     0시                            24시
     (트래픽 없으면 $0)

Lambda 무료 티어: 월 100만 요청 + 40만 GB-초 무료
```

## 2. 이벤트 드리븐 아키텍처

> [!CONCEPT] 이벤트 드리븐 아키텍처
> **이벤트 드리븐 아키텍처**(EDA)는 이벤트의 발생, 감지, 처리를 중심으로 설계된 아키텍처입니다. 컴포넌트 간 직접 호출 대신 이벤트를 통해 느슨하게 결합(Loose Coupling)되며, Lambda와 같은 서버리스 서비스와 자연스럽게 결합됩니다.

### 주요 용어

| 용어                             | 설명                                                 |
| -------------------------------- | ---------------------------------------------------- |
| **이벤트 드리븐 아키텍처 (EDA)** | 이벤트의 발생·감지·처리를 중심으로 설계된 아키텍처   |
| **트리거 (Trigger)**             | Lambda 함수를 실행시키는 이벤트 소스                 |
| **느슨한 결합 (Loose Coupling)** | 컴포넌트 간 직접 호출 대신 이벤트로 통신하는 설계    |
| **팬아웃 (Fan-out)**             | 하나의 이벤트를 여러 소비자에게 동시에 전달하는 패턴 |

### 이벤트 드리븐 패턴

```
┌──────────────────────────────────────────────────────────┐
│  이벤트 소스 (Event Source)                              │
│                                                          │
│  S3 파일 업로드 ───┐                                     │
│  API Gateway ──────┼──► Lambda 함수 ──► DynamoDB         │
│  DynamoDB Stream ──┤                                     │
│  CloudWatch 스케줄─┤                                     │
│  SQS 메시지 ───────┘                                     │
│  SNS 알림 ─────────┘                                     │
└──────────────────────────────────────────────────────────┘
```

### Lambda 트리거 예시

| 이벤트 소스           | 트리거 조건        | 사용 사례                  |
| --------------------- | ------------------ | -------------------------- |
| **API Gateway**       | HTTP 요청          | REST API 처리              |
| **S3**                | 파일 업로드/삭제   | 이미지 리사이징, 로그 처리 |
| **DynamoDB Streams**  | 테이블 변경        | 변경 데이터 캡처 (CDC)     |
| **CloudWatch Events** | 스케줄 (cron)      | 정기 배치 작업             |
| **SQS**               | 메시지 도착        | 비동기 작업 처리           |
| **SNS**               | 알림 발행          | 팬아웃 처리                |
| **Cognito**           | 사용자 가입/로그인 | 사용자 검증, 환영 이메일   |

### 이벤트 드리븐 vs 요청-응답

| 항목          | 요청-응답 (동기) | 이벤트 드리븐 (비동기)  |
| ------------- | ---------------- | ----------------------- |
| **결합도**    | 강결합           | 느슨한 결합             |
| **확장성**    | 호출자에 의존    | 독립적 확장             |
| **장애 전파** | 연쇄 장애 가능   | 격리됨                  |
| **디버깅**    | 쉬움 (추적 용이) | 어려움 (분산 추적 필요) |
| **응답 시간** | 즉시             | 비동기 (결과 나중에)    |

## 3. Cold Start 문제

> [!CONCEPT] Cold Start 문제와 해결
> **Cold Start**는 Lambda 함수가 처음 호출되거나 오랜 시간 미사용 후 호출될 때, 실행 환경을 새로 생성하느라 발생하는 지연 시간입니다. 일반적으로 100ms~수 초가 추가되며, VPC 연결 시 더 길어질 수 있습니다.

### 주요 용어

| 용어                        | 설명                                                                    |
| --------------------------- | ----------------------------------------------------------------------- |
| **Cold Start**              | Lambda 실행 환경을 새로 생성할 때 발생하는 초기 지연 시간               |
| **Warm Start**              | 이미 존재하는 실행 환경에서 즉시 함수를 실행하는 것                     |
| **Provisioned Concurrency** | 미리 실행 환경을 유지하여 Cold Start를 제거하는 기능                    |
| **SnapStart**               | Java Lambda의 초기화 스냅샷을 캐싱하여 Cold Start를 90% 감소시키는 기능 |

### Cold Start vs Warm Start

```
Cold Start (첫 호출 또는 장시간 미사용 후):
┌───────────────────────────────────────────────────────────────┐
│ 컨테이너 생성 │ 런타임 초기화 │ 핸들러 초기화 │ 함수 실행     │
│ (100ms+)      │ (100ms+)      │ (가변)        │ (실제)        │
└───────────────────────────────────────────────────────────────┘
├── Cold Start 지연 (100ms ~ 수 초) ────────────┤─── 실행 ──────┤

Warm Start (이미 실행 환경 존재):
┌─────────────┐
│  함수 실행  │  ← 즉시 실행 (수 ms)
└─────────────┘
```

### 런타임별 Cold Start 시간

| 런타임      | Cold Start | 비고                   |
| ----------- | ---------- | ---------------------- |
| **Python**  | 100~300ms  | 가벼움                 |
| **Node.js** | 100~300ms  | 가벼움                 |
| **Go**      | 50~100ms   | 컴파일 언어, 매우 빠름 |
| **Java**    | 500ms~3초+ | JVM 초기화 무거움      |
| **.NET**    | 300ms~1초  | CLR 초기화             |

### Cold Start 완화 전략

| 전략                        | 방법                     | 효과                        |
| --------------------------- | ------------------------ | --------------------------- |
| **Provisioned Concurrency** | 미리 실행 환경 유지      | Cold Start 제거 (비용 발생) |
| **SnapStart **(Java)        | 초기화 스냅샷 캐싱       | Java Cold Start 90% 감소    |
| **경량 런타임**             | Python/Node.js 사용      | 초기화 시간 최소화          |
| **패키지 최소화**           | 불필요한 의존성 제거     | 로딩 시간 감소              |
| **VPC 최적화**              | VPC 연결 최소화          | ENI 생성 시간 제거          |
| **Keep Warm**               | 주기적 호출 (CloudWatch) | 환경 유지 (비권장)          |

## 4. NoSQL 유형

> [!CONCEPT] NoSQL 데이터베이스 유형
> **NoSQL**(Not Only SQL)은 관계형 모델이 아닌 다양한 데이터 모델을 사용하는 데이터베이스의 총칭입니다. 유연한 스키마, 수평 확장성, 높은 처리량이 특징이며, 데이터 특성에 맞는 유형을 선택합니다.

### 주요 용어

| 용어            | 설명                                                          |
| --------------- | ------------------------------------------------------------- |
| **NoSQL**       | 고정 스키마 없이 유연한 데이터 모델을 지원하는 비관계형 DB    |
| **Key-Value**   | 키-값 쌍으로 데이터를 저장하는 가장 단순한 NoSQL 유형         |
| **Document DB** | JSON 문서 형태로 데이터를 저장하는 NoSQL 유형                 |
| **Graph DB**    | 노드와 엣지(관계)로 데이터를 저장하는 NoSQL 유형              |
| **BASE**        | Basically Available, Soft state, Eventually consistent의 약자 |

### NoSQL 4가지 유형

```
Key-Value:                    Document:
┌─────────┬──────────┐       ┌─────────────────────────┐
│  Key    │  Value   │       │ {                       │
├─────────┼──────────┤       │   "id": "user-1",       │
│ user:1  │ {json}   │       │   "name": "김철수",     │
│ session │ {data}   │       │   "orders": [           │
│ cache:x │ "value"  │       │     {"id": 1, ...}      │
└─────────┴──────────┘       │   ]                     │
예: DynamoDB, Redis          │ }                       │
                             └─────────────────────────┘
                              예: MongoDB, DocumentDB

Column-Family:                Graph:
┌────┬──────────────────┐    ┌───┐  친구  ┌───┐
│Row │ Col1 │ Col2 │Col3│    │ A │────────│ B │
├────┼──────┼──────┼────┤    └─┬─┘        └─┬─┘
│ r1 │ v1   │ v2   │    │      │  좋아요    │
│ r2 │ v3   │      │ v4 │      │            │
└────┴──────┴──────┴────┘    ┌─┴─┐        ┌─┴─┐
예: Cassandra, HBase         │ C │────────│ D │
                             └───┘ 팔로우 └───┘
                             예: Neptune, Neo4j
```

### NoSQL 유형별 비교

| 유형          | 데이터 모델 | 강점              | AWS 서비스            | 사용 사례            |
| ------------- | ----------- | ----------------- | --------------------- | -------------------- |
| **Key-Value** | 키-값 쌍    | 초고속 조회, 단순 | DynamoDB, ElastiCache | 세션, 캐시, 설정     |
| **Document**  | JSON 문서   | 유연한 스키마     | DocumentDB            | 카탈로그, CMS        |
| **Column**    | 열 패밀리   | 대규모 분석       | Keyspaces             | 시계열, IoT          |
| **Graph**     | 노드+엣지   | 관계 탐색         | Neptune               | SNS, 추천, 사기 탐지 |

### RDBMS vs NoSQL 선택 기준

| 기준            | RDBMS 선택        | NoSQL 선택        |
| --------------- | ----------------- | ----------------- |
| **데이터 구조** | 정형, 관계 복잡   | 비정형, 유연      |
| **일관성**      | 강한 일관성 필수  | 최종 일관성 허용  |
| **확장**        | 수직 확장 위주    | 수평 확장 필요    |
| **쿼리**        | 복잡한 JOIN, 집계 | 단순 키 기반 조회 |
| **트랜잭션**    | ACID 필수         | BASE 허용         |
| **예시**        | 금융, ERP, 주문   | 게임, IoT, 실시간 |

## 5. CAP 정리와 Eventually Consistent

> [!CONCEPT] CAP 정리와 최종 일관성
> **CAP 정리**는 분산 시스템에서 **일관성**(Consistency), **가용성**(Availability), **분할 내성**(Partition Tolerance) 세 가지를 동시에 만족할 수 없다는 이론입니다. 네트워크 분할은 불가피하므로, 실제로는 CP(일관성 우선) 또는 AP(가용성 우선) 중 선택합니다.

### 주요 용어

| 용어                                    | 설명                                                                       |
| --------------------------------------- | -------------------------------------------------------------------------- |
| **CAP 정리**                            | 분산 시스템에서 일관성(C), 가용성(A), 분할 내성(P) 중 2개만 동시 만족 가능 |
| **Consistency (일관성)**                | 모든 노드가 같은 시점에 같은 데이터를 보는 것                              |
| **Availability (가용성)**               | 모든 요청에 항상 응답하는 것                                               |
| **Partition Tolerance (분할 내성)**     | 네트워크 분할에도 시스템이 동작하는 것                                     |
| **Eventually Consistent (최종 일관성)** | 시간이 지나면 모든 노드가 일관된 상태에 도달하는 모델                      |

### CAP 삼각형

```
              Consistency (일관성)
              모든 노드가 같은 데이터
                    ╱╲
                   ╱  ╲
                  ╱    ╲
            CP   ╱      ╲   CA
          (선택)╱        ╲  (이론적)
               ╱    ❌    ╲
              ╱  3개 동시  ╲
             ╱   불가능     ╲
            ╱________________╲
  Availability              Partition Tolerance
  항상 응답 가능              네트워크 분할에도 동작
           ╲                 ╱
            ╲      AP       ╱
             ╲   (선택)    ╱
              ╲___________╱

실제 선택:
• CP: 일관성 우선 (금융) → 응답 지연 허용
• AP: 가용성 우선 (SNS) → 일시적 불일치 허용
```

### Eventually Consistent (최종 일관성)

```
시점 T1: 데이터 쓰기
┌──────┐ 쓰기  ┌──────┐
│Client│──────►│Node A│ = "v2" (최신)
└──────┘       └──────┘
               ┌──────┐
               │Node B│ = "v1" (구버전) ← 아직 복제 안 됨
               └──────┘
               ┌──────┐
               │Node C│ = "v1" (구버전) ← 아직 복제 안 됨
               └──────┘

시점 T2: 복제 완료 (수 ms ~ 수 초 후)
               ┌──────┐
               │Node A│ = "v2" ✅
               └──────┘
               ┌──────┐
               │Node B│ = "v2" ✅ (복제 완료)
               └──────┘
               ┌──────┐
               │Node C│ = "v2" ✅ (복제 완료)
               └──────┘

"결국에는(Eventually)" 모든 노드가 일관된 상태에 도달
```

## 6. DynamoDB 설계 원칙

> [!CONCEPT] DynamoDB 설계 원칙
> **DynamoDB**는 AWS의 완전 관리형 Key-Value/Document NoSQL 데이터베이스입니다. **파티션 키**(PK)와 선택적 **정렬 키**(SK)로 데이터를 식별하며, **Single Table Design**과 **Access Pattern 기반 설계**가 핵심입니다.

### 주요 용어

| 용어                              | 설명                                                          |
| --------------------------------- | ------------------------------------------------------------- |
| **DynamoDB**                      | AWS의 완전 관리형 Key-Value/Document NoSQL 데이터베이스       |
| **파티션 키 (Partition Key, PK)** | 데이터를 분산 저장하기 위한 기본 키                           |
| **정렬 키 (Sort Key, SK)**        | 파티션 내에서 데이터를 정렬하고 범위 쿼리를 가능하게 하는 키  |
| **Single Table Design**           | 여러 엔티티를 하나의 테이블에 저장하는 DynamoDB 설계 패턴     |
| **Access Pattern**                | 애플리케이션이 데이터에 접근하는 방식. DynamoDB 설계의 출발점 |

### DynamoDB 핵심 개념

```
테이블 구조:
┌─────────────────────────────────────────────────────────┐
│  Table: Orders                                          │
│                                                         │
│  Partition Key (PK)  │  Sort Key (SK)  │  Attributes    │
│  ────────────────────┼─────────────────┼────────────    │
│  USER#kim            │  ORDER#001      │  {상품, 금액}  │
│  USER#kim            │  ORDER#002      │  {상품, 금액}  │
│  USER#lee            │  ORDER#001      │  {상품, 금액}  │
│  USER#lee            │  ORDER#002      │  {상품, 금액}  │
│  USER#lee            │  ORDER#003      │  {상품, 금액}  │
└─────────────────────────────────────────────────────────┘

PK: 데이터 분산 결정 (파티션 선택)
SK: 파티션 내 정렬 및 범위 쿼리
```

### Single Table Design

```
RDBMS (여러 테이블):          DynamoDB (단일 테이블):
┌─────────┐                  ┌────────────────────────────┐
│  Users  │                  │  PK          │ SK          │
├─────────┤                  ├──────────────┼─────────────┤
│  Orders │                  │ USER#kim     │ PROFILE     │
├─────────┤                  │ USER#kim     │ ORDER#001   │
│  Items  │                  │ USER#kim     │ ORDER#002   │
└─────────┘                  │ ORDER#001    │ ITEM#A      │
3개 테이블 + JOIN            │ ORDER#001    │ ITEM#B      │
                             └────────────────────────────┘
                             1개 테이블, JOIN 없이 조회
```

### Access Pattern 기반 설계

| Access Pattern     | PK        | SK                                        | 쿼리 방식 |
| ------------------ | --------- | ----------------------------------------- | --------- |
| 사용자 프로필 조회 | USER#kim  | PROFILE                                   | GetItem   |
| 사용자의 모든 주문 | USER#kim  | ORDER# (begins_with)                      | Query     |
| 특정 주문의 상품들 | ORDER#001 | ITEM# (begins_with)                       | Query     |
| 최근 주문 10개     | USER#kim  | ORDER# (ScanIndexForward=false, Limit=10) | Query     |

### DynamoDB vs RDS 비교

| 항목         | DynamoDB           | RDS (MySQL)      |
| ------------ | ------------------ | ---------------- |
| **스키마**   | 유연 (Schemaless)  | 고정 (Schema)    |
| **확장**     | 자동 수평 확장     | 수직 확장 위주   |
| **쿼리**     | PK/SK 기반         | SQL (JOIN, 집계) |
| **트랜잭션** | 제한적 (25개 항목) | 완전한 ACID      |
| **비용**     | 요청당 과금        | 시간당 과금      |
| **지연**     | 한 자릿수 ms       | 수~수십 ms       |
| **관리**     | 완전 관리형        | 반관리형         |

## 7. Lambda 실행 모델

> [!CONCEPT] Lambda 실행 모델과 제약
> **Lambda**는 이벤트에 의해 트리거되어 함수를 실행하는 서비스입니다. 실행 환경은 AWS가 자동으로 생성·관리하며, 동시 요청 수에 따라 자동으로 확장됩니다. 실행 시간과 메모리에 제한이 있습니다.

### 주요 용어

| 용어                        | 설명                                                  |
| --------------------------- | ----------------------------------------------------- |
| **Lambda 실행 환경**        | 함수 코드를 실행하는 격리된 컨테이너. AWS가 자동 관리 |
| **INIT Phase**              | 실행 환경 생성 시 런타임과 함수를 초기화하는 단계     |
| **INVOKE Phase**            | 핸들러 함수가 실제로 실행되는 단계                    |
| **핸들러 (Handler)**        | Lambda가 이벤트를 받아 처리하는 진입점 함수           |
| **동시 실행 (Concurrency)** | 동시에 실행되는 Lambda 함수 인스턴스 수               |

### Lambda 실행 환경 수명 주기

```
┌─────────────────────────────────────────────────────────┐
│  Lambda 실행 환경 수명 주기                             │
│                                                         │
│  INIT Phase (Cold Start 시에만):                        │
│  ├── Extension init                                     │
│  ├── Runtime init                                       │
│  └── Function init (전역 변수, DB 연결 등)              │
│                                                         │
│  INVOKE Phase (매 호출마다):                            │
│  └── Handler 함수 실행                                  │
│                                                         │
│  SHUTDOWN Phase (환경 종료 시):                         │
│  └── 리소스 정리                                        │
│                                                         │
│  ⚡ 최적화 팁:                                           │
│  • DB 연결은 INIT에서 (핸들러 밖에서 생성)              │
│  • 핸들러는 가볍게 유지                                 │
│  • 환경 재사용 시 INIT 스킵 → Warm Start                │
└─────────────────────────────────────────────────────────┘
```

### Lambda 제한사항

| 항목               | 제한                         |
| ------------------ | ---------------------------- |
| **최대 실행 시간** | 15분 (900초)                 |
| **메모리**         | 128MB ~ 10,240MB             |
| **패키지 크기**    | 50MB (압축) / 250MB (비압축) |
| **동시 실행**      | 1,000 (기본, 증가 요청 가능) |
| **임시 스토리지**  | /tmp 512MB ~ 10GB            |
| **환경 변수**      | 4KB                          |
| **페이로드**       | 동기 6MB / 비동기 256KB      |

### Lambda + API Gateway + DynamoDB 아키텍처

```
┌──────┐     ┌───────────┐     ┌────────┐     ┌──────────┐
│Client│────►│API Gateway│────►│ Lambda │────►│DynamoDB  │
│      │     │           │     │        │     │          │
│      │◄────│  HTTPS    │◄────│  함수  │◄────│  테이블  │
└──────┘     └───────────┘     └────────┘     └──────────┘

특징:
• 완전 서버리스 (서버 관리 제로)
• 자동 확장 (수천 동시 요청 처리)
• 사용한 만큼만 과금
• 트래픽 없으면 비용 $0
```

| 개념                  | 한 줄 요약                                   |
| --------------------- | -------------------------------------------- |
| 서버리스              | 서버 관리 없이 코드 실행 (FaaS + BaaS)       |
