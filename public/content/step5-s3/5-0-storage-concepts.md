## 1. 스토리지 유형 개요

> [!CONCEPT] 세 가지 스토리지 유형
> 클라우드 스토리지는 데이터 접근 방식에 따라 **블록 스토리지**, **오브젝트 스토리지**, **파일 스토리지** 세 가지로 분류됩니다.  
> 각각의 특성이 다르므로 워크로드에 맞는 유형을 선택해야 합니다.
>
> **일상 비유로 이해하기:**
>
> - **블록 스토리지(EBS)** = 내 PC에 꽂은 SSD. 한 대의 컴퓨터에서만 사용하며, 빠르고 자유롭게 읽기/쓰기 가능
> - **오브젝트 스토리지(S3)** = 구글 드라이브/네이버 클라우드. 어디서든 URL로 파일을 업로드/다운로드. 폴더처럼 보이지만 실제로는 "주소(키)"로 파일을 찾는 구조
> - **파일 스토리지(EFS)** = 회사 공용 NAS(네트워크 드라이브). 여러 컴퓨터에서 동시에 같은 폴더에 접근 가능

### 핵심 용어 정리

| 용어                               | 설명                                                               | AWS 관련                   |
| ---------------------------------- | ------------------------------------------------------------------ | -------------------------- |
| **오브젝트 스토리지**              | 파일을 키-값 쌍의 객체로 저장. 계층 구조 없이 플랫하게 관리        | Amazon S3                  |
| **블록 스토리지**                  | 데이터를 고정 크기 블록으로 나누어 저장. OS가 파일시스템으로 포맷  | Amazon EBS                 |
| **파일 스토리지**                  | 디렉토리 계층 구조로 파일을 저장. 여러 서버가 동시 접근 가능       | Amazon EFS                 |
| **버킷 (Bucket)**                  | S3에서 객체를 담는 최상위 컨테이너. 전 세계적으로 고유한 이름 필요 | my-app-bucket-12345        |
| **객체 (Object)**                  | S3에 저장되는 파일 단위. 키(경로) + 값(데이터) + 메타데이터로 구성 | images/photo.jpg           |
| **내구성 (Durability)**            | 데이터가 손실되지 않을 확률. S3는 99.999999999% (11 nines)         | 연간 객체 손실 확률 거의 0 |
| **가용성 (Availability)**          | 서비스에 접근할 수 있는 시간 비율                                  | S3 Standard: 99.99%        |
| **스토리지 클래스**                | 접근 빈도와 비용에 따른 저장 등급                                  | Standard, IA, Glacier      |
| **CDN (Content Delivery Network)** | 전 세계 엣지 서버에 콘텐츠를 캐싱하여 빠르게 전달                  | Amazon CloudFront          |
| **TTL (Time To Live)**             | 캐시된 콘텐츠의 유효 시간. 만료 후 원본에서 다시 가져옴            | CloudFront 캐시 정책       |

### 세 가지 스토리지 유형 비교

```
블록 스토리지 (EBS)       오브젝트 스토리지 (S3)       파일 스토리지 (EFS)
┌───┬───┬───┬───┐          ┌─────────────────┐         ┌────────────────────┐
│ B │ B │ B │ B │          │  Object         │         │  /home/            │
│ l │ l │ l │ l │          │  ├── Data       │         │  ├── user1/        │
│ o │ o │ o │ o │          │  ├── Metadata   │         │  │   ├── file.txt  │
│ c │ c │ c │ c │          │  └── Key (ID)   │         │  │   └── img.png   │
│ k │ k │ k │ k │          │                 │         │  └── user2/        │
│ 1 │ 2 │ 3 │ 4 │          │  Flat namespace │         │                    │
└───┴───┴───┴───┘          └─────────────────┘         │  계층적 구조       │
고정 크기 블록 단위          키-값 기반 접근           └────────────────────┘
                                                        디렉토리/파일 구조
```

### 상세 비교표

| 항목           | 블록 (EBS)     | 오브젝트 (S3)      | 파일 (EFS)       |
| -------------- | -------------- | ------------------ | ---------------- |
| **접근 방식**  | 블록 단위 I/O  | HTTP API (GET/PUT) | NFS/SMB 프로토콜 |
| **구조**       | 고정 크기 블록 | 플랫 네임스페이스  | 계층적 디렉토리  |
| **수정**       | 부분 수정 가능 | 전체 교체만 가능   | 부분 수정 가능   |
| **성능**       | 매우 높음 (ms) | 보통 (100ms~)      | 높음 (ms)        |
| **확장성**     | 제한적 (16TB)  | 무제한             | 자동 확장        |
| **연결**       | 단일 EC2       | 어디서든 접근      | 다수 EC2 동시    |
| **비용**       | 중간           | 저렴               | 높음             |
| **AWS 서비스** | EBS            | S3                 | EFS              |

## 2. 블록 스토리지 (Amazon EBS)

> [!CONCEPT] 블록 스토리지의 특성
> **블록 스토리지**는 데이터를 고정 크기의 블록으로 나누어 저장합니다.  
>  파일 시스템을 직접 구성할 수 있어 데이터베이스, OS 부팅 볼륨 등 고성능 I/O가 필요한 워크로드에 적합합니다.

### 주요 용어

| 용어                              | 설명                                                             |
| --------------------------------- | ---------------------------------------------------------------- |
| **블록 스토리지 (Block Storage)** | 데이터를 고정 크기 블록으로 나누어 저장하는 방식                 |
| **파일 시스템 (File System)**     | OS가 블록 스토리지 위에 구성하는 데이터 관리 구조 (ext4, xfs 등) |
| **볼륨 (Volume)**                 | EC2에 연결된 하나의 EBS 스토리지 단위                            |

### EBS 특성

```
┌──────────────────────────────────────────────────────┐
│  EC2 인스턴스                                        │
│  ┌───────────────────────────────────────┐           │
│  │  운영체제 (Linux/Windows)             │           │
│  │  ┌─────────────────────────────────┐  │           │
│  │  │  파일 시스템 (ext4, xfs)        │  │           │
│  │  │  /dev/xvda (루트 볼륨)          │  │           │
│  │  │  /dev/xvdb (추가 볼륨)          │  │           │
│  │  └─────────────────────────────────┘  │           │
│  └───────────────────────────────────────┘           │
│           │              │                           │
│     ┌─────┴─────┐  ┌─────┴─────┐                     │
│     │ EBS Vol 1 │  │ EBS Vol 2 │  ← 네트워크 연결    │
│     │ (gp3, 8GB)│  │(gp3,100GB)│                     │
│     └───────────┘  └───────────┘                     │
└──────────────────────────────────────────────────────┘
```

### 사용 사례

| 사용 사례        | EBS 타입 | 이유                 |
| ---------------- | -------- | -------------------- |
| OS 부팅 볼륨     | gp3      | 범용 성능, 비용 효율 |
| MySQL/PostgreSQL | io2      | 높은 IOPS 필요       |
| 로그 저장        | st1      | 순차 읽기, 비용 저렴 |
| 백업 아카이브    | sc1      | 접근 빈도 낮음       |

## 3. 오브젝트 스토리지 (Amazon S3)

> [!CONCEPT] 오브젝트 스토리지와 S3
> **오브젝트 스토리지**는 데이터를 객체(Object) 단위로 저장합니다.  
> 각 객체는 데이터 본체, 메타데이터, 고유 키로 구성됩니다.  
> HTTP API로 접근하며, 사실상 무제한 확장이 가능합니다.

### 주요 용어

| 용어                       | 설명                                                        |
| -------------------------- | ----------------------------------------------------------- |
| **버킷 (Bucket)**          | S3에서 객체를 담는 최상위 컨테이너. 전 세계 고유 이름 필요  |
| **객체 (Object)**          | S3에 저장되는 파일 단위. 키 + 데이터 + 메타데이터로 구성    |
| **키 (Key)**               | 버킷 내에서 객체를 식별하는 고유 경로 (예: images/logo.png) |
| **메타데이터 (Metadata)**  | 객체에 대한 부가 정보 (Content-Type, 크기, 수정일 등)       |
| **버전 관리 (Versioning)** | 같은 키의 객체를 여러 버전으로 보관하는 기능                |

### S3 객체 구조

```
S3 버킷: my-app-bucket
│
├── images/
│   ├── logo.png          ← 객체 (Object)
│   │   ├── Data: 바이너리 데이터 (이미지 본체)
│   │   ├── Key: images/logo.png
│   │   ├── Metadata:
│   │   │   ├── Content-Type: image/png
│   │   │   ├── Content-Length: 45678
│   │   │   ├── Last-Modified: 2024-01-15
│   │   │   └── x-amz-meta-author: kim
│   │   └── Version ID: v1 (버전 관리 시)
│   │
│   └── banner.jpg
│
├── documents/
│   └── report.pdf
│
└── backups/
    └── db-2024-01-15.sql.gz
```

### S3 핵심 특성

| 특성               | 값                         |
| ------------------ | -------------------------- |
| **내구성**         | 99.999999999% (11 nines)   |
| **가용성**         | 99.99% (Standard)          |
| **객체 최대 크기** | 5TB                        |
| **버킷당 객체 수** | 무제한                     |
| **접근 방식**      | HTTPS REST API             |
| **일관성**         | 강력한 읽기 후 쓰기 일관성 |

> [!TIP]
> **S3에는 실제 "폴더"가 없습니다.**
>
> `images/profile/photo.jpg`는 하나의 키(문자열)일 뿐이며, `images/`와 `profile/`이라는 폴더가 존재하는 것이 아닙니다.  
> AWS 콘솔에서 폴더처럼 보이는 것은 슬래시(`/`)를 기준으로 UI가 계층 구조를 표현해주는 것뿐입니다.
>
> ```
> 실제 저장 구조 (플랫):
>   "images/profile/photo.jpg"  ← 이것이 키(Key) 전체
>   "images/profile/avatar.png"
>   "documents/report.pdf"
>
> 콘솔에서 보이는 구조 (UI가 만든 가상 폴더):
>   📁 images/
>      📁 profile/
>         📄 photo.jpg
>         📄 avatar.png
>   📁 documents/
>      📄 report.pdf
> ```
>
> 이 개념을 이해하면 버킷 정책에서 `public/*` 같은 prefix 기반 접근 제어가 왜 동작하는지 명확해집니다.

### 사용 사례

| 사용 사례         | 설명                        |
| ----------------- | --------------------------- |
| 정적 웹 호스팅    | HTML, CSS, JS, 이미지 서빙  |
| 미디어 저장       | 사진, 동영상, 음악 파일     |
| 백업/아카이브     | DB 덤프, 로그 아카이브      |
| 데이터 레이크     | 빅데이터 분석용 원시 데이터 |
| 애플리케이션 자산 | 사용자 업로드 파일          |

## 4. 파일 스토리지 (Amazon EFS)
> [!CONCEPT] 파일 스토리지와 EFS
> **파일 스토리지**는 계층적 디렉토리 구조로 데이터를 저장하며, NFS 프로토콜을 통해 여러 인스턴스가 동시에 접근할 수 있습니다.  
> 공유 파일 시스템이 필요한 워크로드에 적합합니다.

### 주요 용어

| 용어                          | 설명                                                    |
| ----------------------------- | ------------------------------------------------------- |
| **EFS (Elastic File System)** | AWS의 관리형 NFS 파일 스토리지. 다중 EC2 동시 접근 가능 |
| **NFS (Network File System)** | 네트워크를 통해 파일을 공유하는 프로토콜                |
| **Mount Target**              | EFS를 VPC의 각 AZ에서 접근하기 위한 네트워크 엔드포인트 |

### EFS 아키텍처

```
┌─────────────────────────────────────────────────────┐
│  VPC                                                │
│                                                     │
│  ┌───── AZ-a ──────┐    ┌───── AZ-b ──────┐         │
│  │                 │    │                 │         │
│  │  EC2-A ──┐      │    │  EC2-C ──┐      │         │
│  │  EC2-B ──┼──► Mount  │  EC2-D ──┼──► Mount       │
│  │          │   Target  │          │   Target       │
│  └──────────┼──────┘    └──────────┼──────┘         │
│             │                      │                │
│             └──────── EFS ─────────┘                │
│                    (공유 파일 시스템)               │
│                    자동 확장, 다중 AZ               │
└─────────────────────────────────────────────────────┘
```

### EFS vs EBS 비교

| 항목          | EFS              | EBS                |
| ------------- | ---------------- | ------------------ |
| **동시 접근** | 수천 개 EC2      | 1개 EC2 (io2 제외) |
| **크기 조정** | 자동 확장/축소   | 수동 조정          |
| **AZ**        | 다중 AZ          | 단일 AZ            |
| **프로토콜**  | NFS v4.1         | 블록 디바이스      |
| **비용**      | GB당 높음        | GB당 낮음          |
| **성능**      | 처리량 모드 선택 | IOPS 프로비저닝    |

## 5. 내구성과 가용성

> [!CONCEPT] 내구성과 가용성의 차이
> **내구성**(Durability)은 데이터가 손실되지 않을 확률이고, **가용성**(Availability)은 서비스에 접근할 수 있는 시간 비율입니다.  
> 둘은 다른 개념이며, 높은 내구성이 높은 가용성을 보장하지 않습니다.
>
> **일상 비유:**
> - **내구성** = "금고에 넣은 돈이 사라지지 않을 확률" → S3는 거의 100% (99.999999999%)
> - **가용성** = "금고 문이 열리는 시간 비율" → 연간 52분 정도는 문이 안 열릴 수 있음
>
> 데이터는 안전하지만(내구성 높음), 서비스가 일시적으로 응답하지 않을 수는 있습니다(가용성 한계).

### 주요 용어

| 용어                         | 설명                                                |
| ---------------------------- | --------------------------------------------------- |
| **내구성 (Durability)**      | 데이터가 손실되지 않을 확률. S3 Standard는 11 nines |
| **가용성 (Availability)**    | 서비스에 접근할 수 있는 시간 비율                   |
| **11 nines (99.999999999%)** | 1000만 객체 중 1만년에 1개 손실 수준의 내구성       |

### 내구성 vs 가용성 비교

| 항목            | 내구성 (Durability)             | 가용성 (Availability)      |
| --------------- | ------------------------------- | -------------------------- |
| **질문**        | "데이터가 사라지지 않을까?"     | "지금 접근할 수 있을까?"   |
| **측정**        | 데이터 손실 확률                | 서비스 접근 가능 시간 비율 |
| **S3 Standard** | 99.999999999% (11 nines)        | 99.99% (4 nines)           |
| **의미**        | 1000만 객체 중 1만년에 1개 손실 | 연간 약 52분 다운타임      |

### AWS 스토리지 서비스별 내구성/가용성

| 서비스         | 내구성   | 가용성 | 복제 방식        |
| -------------- | -------- | ------ | ---------------- |
| S3 Standard    | 11 nines | 99.99% | 3개 AZ 자동 복제 |
| S3 One Zone-IA | 11 nines | 99.5%  | 1개 AZ 내 복제   |
| EBS            | 99.999%  | 99.99% | AZ 내 복제       |
| EFS            | 11 nines | 99.99% | 다중 AZ 복제     |

## 6. 스토리지 클래스 (Hot/Warm/Cold)

> [!CONCEPT] 스토리지 클래스와 데이터 온도
> 데이터는 접근 빈도에 따라 **Hot**(자주 접근), **Warm**(가끔 접근), **Cold**(거의 접근 안 함)로 분류됩니다.  
> 접근 빈도가 낮을수록 저렴한 스토리지 클래스를 사용하여 비용을 최적화합니다.

### 주요 용어

| 용어                                  | 설명                                                  |
| ------------------------------------- | ----------------------------------------------------- |
| **스토리지 클래스 (Storage Class)**   | 접근 빈도와 비용에 따른 S3 저장 등급                  |
| **Hot / Warm / Cold**                 | 데이터 접근 빈도에 따른 온도 분류                     |
| **Glacier**                           | 아카이브용 초저비용 스토리지 클래스. 검색에 시간 소요 |
| **수명 주기 정책 (Lifecycle Policy)** | 시간 경과에 따라 스토리지 클래스를 자동 전환하는 규칙 |

### S3 스토리지 클래스 비교

| 클래스                   |    온도    | 저장 비용 | 검색 비용 | 최소 보관 | 사용 사례              |
| ------------------------ | :--------: | --------- | --------- | --------- | ---------------------- |
| **Standard**             |   🔥 Hot   | 높음      | 없음      | 없음      | 자주 접근하는 데이터   |
| **Intelligent-Tiering**  | 🔥/❄️ 자동 | 중간      | 없음      | 없음      | 접근 패턴 불확실       |
| **Standard-IA**          |  🌡️ Warm   | 낮음      | 있음      | 30일      | 월 1~2회 접근          |
| **One Zone-IA**          |  🌡️ Warm   | 더 낮음   | 있음      | 30일      | 재생성 가능 데이터     |
| **Glacier Instant**      |  ❄️ Cold   | 매우 낮음 | 높음      | 90일      | 분기 1회 접근          |
| **Glacier Flexible**     |  ❄️ Cold   | 극히 낮음 | 높음      | 90일      | 연 1~2회 (수분~수시간) |
| **Glacier Deep Archive** | 🧊 Frozen  | 최저      | 매우 높음 | 180일     | 7~10년 보관 (12시간)   |

### 수명 주기 정책 (Lifecycle Policy)

```
   데이터 생성
       │
       ▼ (0일)
┌──────────────┐
│  S3 Standard │  ← 활발히 사용
└──────┬───────┘
       │ 30일 후 자동 전환
       ▼
┌─────────────┐
│ Standard-IA │  ← 가끔 접근
└──────┬──────┘
       │ 90일 후 자동 전환
       ▼
┌─────────────┐
│   Glacier   │  ← 거의 접근 안 함
└──────┬──────┘
       │ 365일 후 자동 삭제
       ▼
    🗑️ 삭제
```

## 7. S3 보안과 접근 제어

> [!CONCEPT] S3 접근 제어의 3가지 계층
> S3의 접근 제어는 **Block Public Access**(버킷 레벨 차단), **버킷 정책**(JSON 정책 문서), **IAM 정책**(사용자/역할 레벨)의 3계층으로 구성됩니다.  
> 이 중 하나라도 "거부"하면 접근이 차단됩니다.
>
> **일상 비유 — 아파트 보안:**
> - **Block Public Access** = 아파트 단지 정문 잠금장치. 이것이 잠기면 어떤 방 열쇠를 가져도 들어올 수 없음
> - **버킷 정책** = 각 세대의 현관문 규칙. "택배기사(Principal: *)는 현관 앞(public/)에만 물건을 놓을 수 있다"
> - **IAM 정책** = 거주자 신분증. "김철수(IAM User)는 501호(특정 버킷)에 출입할 수 있다"

### 주요 용어

| 용어 | 설명 |
| ---- | ---- |
| **Block Public Access** | 버킷/계정 레벨에서 퍼블릭 접근을 일괄 차단하는 안전장치 |
| **버킷 정책 (Bucket Policy)** | 버킷에 연결하는 JSON 기반 접근 제어 정책. "누가, 무엇을, 어떤 조건에서" 허용/거부 |
| **ACL (Access Control List)** | 레거시 접근 제어 방식. 2023년 이후 기본 비활성화, 사용 비권장 |
| **Presigned URL** | 일정 시간만 유효한 임시 접근 URL. 인증 없이 특정 객체에 접근 허용 |
| **OAC (Origin Access Control)** | CloudFront에서 S3에 접근할 때 사용하는 인증 방식 |

### 접근 제어 계층 구조

```
외부 요청 → [Block Public Access] → [버킷 정책] → [IAM 정책] → 허용/거부
                    │                     │              │
                    │ 전부 차단 설정이면  │              │
                    └── ❌ 즉시 거부      │              │
                                          │              │
                              Principal, Action,    사용자/역할에
                              Resource, Condition   부여된 권한 확인
                              으로 세밀하게 제어
```

### Block Public Access

버킷 생성 시 **기본 활성화**되며, 의도하지 않은 퍼블릭 노출을 방지합니다.

| 설정 | 효과 |
| ---- | ---- |
| Block new ACLs | 새 ACL로 공개 설정하는 것을 차단 |
| Block any ACLs | 기존 포함 모든 ACL 공개를 무시 |
| Block new policies | 새 버킷 정책으로 공개 설정 차단 |
| Block any policies | 모든 정책의 공개 설정 차단 |

> [!WARNING]
> 정적 웹 호스팅 등 의도적 공개가 필요한 경우에만 해제하세요. 해제 후에도 버킷 정책이 없으면 공개되지 않습니다.

### 버킷 정책 vs ACL vs IAM 정책

| 항목 | 버킷 정책 | ACL | IAM 정책 |
| ---- | --------- | --- | -------- |
| 형식 | JSON | 미리 정의된 권한 | JSON |
| 적용 대상 | 버킷/객체 경로 | 개별 객체/버킷 | IAM 사용자/역할 |
| 외부 접근 | ✅ (Principal: *) | ✅ | ❌ (내부만) |
| 조건부 제어 | IP, 시간, VPC 등 | ❌ | 서비스, 리전 등 |
| 권장 | ✅ AWS 권장 | ❌ 레거시 | ✅ 내부 사용자용 |

## 8. S3 버전 관리 (Versioning)

> [!CONCEPT] 버전 관리란?
> **버전 관리**를 활성화하면 같은 키(파일명)로 객체를 업로드할 때마다 새 버전이 생성되고, 이전 버전도 보존됩니다.  
> 실수로 덮어쓰거나 삭제해도 이전 상태로 복원할 수 있습니다.

### 주요 용어

| 용어 | 설명 |
| ---- | ---- |
| **버전 ID (Version ID)** | 각 버전을 식별하는 고유 문자열. 버전 관리 활성화 전 객체는 `null` |
| **Delete Marker** | 삭제 시 추가되는 표시. 실제 데이터는 남아있고 마커만 추가됨 |
| **현재 버전 (Current)** | 가장 최근 업로드된 버전. Object URL로 접근 시 이 버전이 반환됨 |
| **비현재 버전 (Noncurrent)** | 이전 버전들. Version ID를 지정해야 접근 가능 |

### 버전 관리 상태

| 상태 | 동작 |
| ---- | ---- |
| **비활성화 (Disabled)** | 덮어쓰면 이전 데이터 영구 삭제. 복원 불가 |
| **활성화 (Enabled)** | 덮어쓰면 새 버전 생성, 이전 버전 보존 |
| **일시 중지 (Suspended)** | 새 버전을 만들지 않지만 기존 버전은 유지 |

### 동작 흐름

```
버전 관리 OFF:
  upload v1 → upload v2 → v1 영구 삭제됨 (복원 불가)

버전 관리 ON:
  upload v1 → upload v2 → v1, v2 모두 보존
                            │
                            ├── GET /file → v2 반환 (현재 버전)
                            └── GET /file?versionId=v1 → v1 반환

삭제 시:
  DELETE /file → Delete Marker 추가 (데이터 유지)
                  │
                  ├── GET /file → 404 (Delete Marker가 가림)
                  └── Delete Marker 삭제 → v2 복원!
```

> [!NOTE]
> 💸 모든 버전이 저장 용량에 포함되어 과금됩니다. 불필요한 이전 버전은 수명 주기 규칙으로 자동 삭제하여 비용을 관리합니다.

## 9. S3 암호화

> [!CONCEPT] S3 서버 측 암호화 (Server-Side Encryption)
> S3는 객체를 저장할 때 자동으로 암호화하고, 읽을 때 자동으로 복호화합니다.  
> 2023년 1월부터 모든 새 객체에 **SSE-S3가 기본 적용**됩니다.

### 암호화 방식 비교

| 방식 | 키 관리 | 비용 | 적합한 경우 |
| ---- | ------- | ---- | ----------- |
| **SSE-S3** | AWS가 자동 관리 | 무료 | 대부분의 경우 (기본값) |
| **SSE-KMS** | AWS KMS 키 사용 | KMS 요청당 과금 | 키 감사, 교차 계정 접근 제어 필요 시 |
| **SSE-C** | 고객이 키 제공 | 무료 | 자체 키 관리 시스템이 있는 경우 |
| **CSE** | 클라이언트에서 암호화 | 무료 | 업로드 전에 이미 암호화 필요 |

### 암호화 동작

```
SSE-S3 (기본):
  [클라이언트] → PUT 객체 → [S3] → 자동 암호화 후 저장
  [클라이언트] ← GET 응답 ← [S3] ← 자동 복호화 후 반환

  사용자가 신경 쓸 것: 없음 (완전 투명)
```

> [!TIP]
> 대부분의 워크로드에서는 SSE-S3(기본값)로 충분합니다.  
> 금융/의료 등 규정 준수가 필요하거나 키 감사 추적이 필요한 경우에만 SSE-KMS를 사용합니다.

## 10. IAM 기초 (사용자, 정책, 역할)

> [!CONCEPT] IAM이란?
> **IAM**(Identity and Access Management)은 AWS 리소스에 대한 접근을 안전하게 제어하는 서비스입니다.  
> "누가(Identity) 무엇을(Action) 어디에(Resource) 할 수 있는지"를 정의합니다.
>
> **일상 비유 — 회사 출입 시스템:**
> - **IAM 사용자** = 사원증. 이름과 사번이 있고, 이걸로 출입문을 통과
> - **IAM 정책** = 출입 권한 규칙. "개발팀은 3층 서버실만 출입 가능"
> - **IAM 역할** = 임시 방문증. EC2 같은 서비스에 "오늘만 S3 창고에 들어가도 됩니다"라고 발급
> - **Access Key** = 카드키. 사원증 없이도 문을 열 수 있는 물리적 열쇠 (분실하면 위험!)
>
> Step 2에서 EC2에 IAM Role을 연결한 경험이 있다면, 여기서는 S3 접근에 동일한 개념을 적용합니다.

### 주요 용어

| 용어 | 설명 |
| ---- | ---- |
| **IAM 사용자 (User)** | AWS에 접근하는 개별 계정. 사람 또는 프로그램용 |
| **IAM 정책 (Policy)** | 허용/거부할 작업을 JSON으로 정의한 문서 |
| **IAM 역할 (Role)** | EC2, Lambda 등 AWS 서비스에 권한을 부여하는 방식. Access Key 불필요 |
| **Access Key** | 프로그래밍 방식(CLI, SDK)으로 AWS에 접근하기 위한 인증 키 쌍 |
| **최소 권한 원칙** | 필요한 작업에만 필요한 리소스에만 권한을 부여하는 보안 원칙 |
| **ARN (Amazon Resource Name)** | AWS 리소스를 고유하게 식별하는 문자열 (예: `arn:aws:s3:::my-bucket/*`) |

### IAM 사용자 vs IAM 역할

| 항목 | IAM 사용자 (User) | IAM 역할 (Role) |
| ---- | ------------------ | ---------------- |
| **인증 방식** | Access Key (영구적) | 임시 자격 증명 (자동 갱신) |
| **사용 주체** | 사람, 로컬 CLI/SDK | AWS 서비스 (EC2, Lambda, ECS) |
| **키 유출 위험** | 있음 (파일로 존재) | 없음 (키 자체가 없음) |
| **로테이션** | 수동 (90일 권장) | 자동 (수 시간마다 갱신) |
| **적합한 환경** | 로컬 개발 | 서버 배포 (EC2, ECS, EKS) |

### IAM 정책 구조 (JSON)

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Action": ["s3:PutObject", "s3:GetObject"],
      "Resource": "arn:aws:s3:::my-bucket/*"
    }
  ]
}
```

| 필드 | 설명 | 예시 |
| ---- | ---- | ---- |
| **Effect** | 허용 또는 거부 | `Allow`, `Deny` |
| **Action** | 허용할 AWS API 작업 | `s3:PutObject`, `s3:GetObject` |
| **Resource** | 대상 리소스의 ARN | `arn:aws:s3:::my-bucket/*` |
| **Principal** | 정책 적용 대상 (버킷 정책에서 사용) | `*` (모든 사용자), 특정 ARN |

### 로컬 개발 → EC2 배포 시 인증 전환

```
[로컬 개발]
  ~/.aws/credentials의 Access Key → SDK가 자동 읽음

[EC2 배포]
  EC2에 IAM Role 연결 → SDK가 Instance Metadata에서 자동 획득
  → 코드 변경 없음, Access Key 불필요

같은 코드가 환경에 따라 다른 인증 방식을 자동으로 사용합니다.
```

> [!TIP]
> 5-2 실습에서 S3 전용 IAM 사용자를 만들고(로컬 개발), EC2에 IAM Role을 연결하여(배포) 동일한 코드가 두 환경에서 동작하는 것을 직접 체험합니다.

## 11. AWS CLI 기초

> [!CONCEPT] AWS CLI란?
> **AWS CLI**(Command Line Interface)는 터미널에서 AWS 서비스를 제어하는 명령줄 도구입니다.  
> 콘솔(웹)에서 할 수 있는 거의 모든 작업을 명령어로 자동화할 수 있습니다.
>
> **왜 CLI를 쓰는가?**
> - 콘솔에서 파일을 하나씩 업로드하면 수십 개 파일을 올리는 데 시간이 오래 걸림
> - CLI의 `aws s3 sync`는 폴더 전체를 한 번에 동기화 — 빌드 결과물 배포에 필수
> - 스크립트로 만들면 반복 작업을 자동화할 수 있음 (Step 9 CI/CD의 기초)
>
> Step 2에서 `aws` 명령어를 사용한 적이 있다면, 여기서는 S3 전용 명령어를 학습합니다.

### 주요 용어

| 용어 | 설명 |
| ---- | ---- |
| **AWS CLI** | AWS 서비스를 터미널에서 제어하는 명령줄 도구 |
| **프로파일 (Profile)** | CLI에 저장된 자격 증명과 설정의 이름 붙은 집합 (default, s3-dev 등) |
| **`~/.aws/credentials`** | Access Key가 저장되는 파일 (프로젝트 외부, Git에 포함되지 않음) |
| **`~/.aws/config`** | 리전, 출력 형식 등 설정이 저장되는 파일 |

### S3 관련 주요 CLI 명령어

| 명령어 | 설명 | 예시 |
| ------ | ---- | ---- |
| `aws s3 ls` | 버킷 목록 또는 버킷 내 객체 목록 조회 | `aws s3 ls s3://my-bucket/` |
| `aws s3 cp` | 단일 파일 복사 (로컬↔S3, S3↔S3) | `aws s3 cp file.txt s3://my-bucket/` |
| `aws s3 sync` | 폴더 동기화 (변경된 파일만 업로드/삭제) | `aws s3 sync dist/ s3://my-bucket/ --delete` |
| `aws s3 rm` | 파일 삭제 | `aws s3 rm s3://my-bucket/old.txt` |
| `aws s3 mb` | 버킷 생성 | `aws s3 mb s3://new-bucket` |
| `aws s3 rb` | 버킷 삭제 | `aws s3 rb s3://old-bucket --force` |

### `aws s3 sync` vs `aws s3 cp`

| 항목 | `sync` | `cp` |
| ---- | ------ | ---- |
| **대상** | 폴더 전체 | 단일 파일 |
| **동작** | 변경된 파일만 업로드 (비교 후 전송) | 항상 전송 |
| **삭제 옵션** | `--delete` (원본에 없는 파일 삭제) | 삭제 기능 없음 |
| **사용 시점** | 빌드 결과물 배포 (`dist/` → S3) | 개별 파일 업로드 |

```
aws s3 sync dist/ s3://my-bucket/ --delete

동작:
1. 로컬 dist/와 S3 버킷의 파일 목록을 비교
2. 새로 추가되거나 변경된 파일만 업로드
3. --delete: S3에는 있지만 로컬에 없는 파일 삭제 (이전 배포 잔여 파일 정리)
```

> [!TIP]
> 정적 사이트 배포에는 `aws s3 sync`가 적합합니다.  
> 변경된 파일만 전송하므로 빠르고, `--delete`로 이전 배포의 잔여 파일도 정리됩니다.

### 프로파일 사용

```bash
# 특정 프로파일로 명령 실행
aws s3 ls --profile s3-dev

# 환경변수로 프로파일 지정 (이후 --profile 생략 가능)
export AWS_PROFILE=s3-dev
aws s3 ls
```

## 12. S3 SDK 인증과 파일 업로드 개발

> [!CONCEPT] AWS SDK 자격 증명 체인 (Credential Provider Chain)
> Spring, Node.js 등에서 AWS SDK를 사용할 때, 코드에 Access Key를 명시하지 않아도 SDK가 자동으로 자격 증명을 찾습니다.  
> 환경에 따라 적절한 방식이 자동 선택됩니다.

### SDK 자격 증명 탐색 순서

| 우선순위 | 방식 | 적합한 환경 |
| -------- | ---- | ----------- |
| 1 | 환경변수 (AWS_ACCESS_KEY_ID) | CI/CD 파이프라인 |
| 2 | Java 시스템 속성 | 특수한 경우 |
| 3 | ~/.aws/credentials 파일 | 로컬 개발 |
| 4 | EC2 IAM Role (Instance Metadata) | EC2 배포 |
| 5 | ECS Task Role / EKS IRSA | 컨테이너 배포 |

> [!WARNING]
> **application.properties에 Access Key를 넣지 마세요.**  
> Git에 push되면 자동 스캐너에 의해 수 분 내에 탈취됩니다.  
> AWS가 유출을 감지해 경고하지만, 이미 수백 달러의 비용이 발생할 수 있습니다.

### 환경별 권장 인증 방식

```
┌──────────────────────────────────────────────────────────────┐
│  로컬 개발                                                   │
│  ~/.aws/credentials 파일 (프로젝트 외부, Git 무관)           │
│  → SDK가 자동으로 읽음, 코드 변경 불필요                     │
├──────────────────────────────────────────────────────────────┤
│  EC2 배포                                                    │
│  IAM Role을 인스턴스에 연결                                  │
│  → Access Key 불필요, 자동 로테이션, AWS 권장                │
├──────────────────────────────────────────────────────────────┤
│  컨테이너 (ECS/EKS)                                          │
│  Task Role / IRSA                                            │
│  → EC2와 동일 원리, 컨테이너별 최소 권한 부여                │
├──────────────────────────────────────────────────────────────┤
│  CI/CD (GitHub Actions 등)                                   │
│  GitHub Secrets → 환경변수로 주입                            │
│  → 평문 저장 안 됨, 팀원 간 키 공유 불필요                   │
└──────────────────────────────────────────────────────────────┘
```

### 파일 업로드 방식 비교

| 방식 | 흐름 | 장점 | 단점 | 적합한 경우 |
| ---- | ---- | ---- | ---- | ----------- |
| 서버 경유 | 클라이언트 → 서버 → S3 | 구현 간단, 서버에서 검증/가공 가능 | 서버 메모리/대역폭 소모 | 소용량, 서버 검증 필요 |
| Presigned URL | 서버가 URL 발급 → 클라이언트가 S3에 직접 업로드 | 서버 부하 최소 | 구현 복잡, Content-Type 일치 필수 | 대용량, 이미지/동영상 |

### 파일 업로드 개발 시 주의사항

| 항목 | 설명 |
| ---- | ---- |
| 파일명 | 원본 파일명 대신 UUID 사용 (충돌 방지, 보안, URL 인코딩 문제 방지) |
| 크기 제한 | Spring 기본 1MB, 필요 시 max-file-size 설정 변경 |
| Content-Type | 업로드 시 올바른 MIME 타입 지정 (브라우저 표시에 영향) |
| 경로 설계 | prefix로 용도 분리 (public/, private/, archive/) |
| 삭제 정책 | 업로드만 하고 삭제 안 하면 비용 누적 — 수명 주기 규칙 활용 |
| CORS | 브라우저에서 직접 S3로 업로드(Presigned URL) 시 CORS 설정 필수 |
| 파일 타입 검증 | 확장자뿐 아니라 실제 바이너리(Magic Bytes)를 확인하여 위장 파일 차단 |
| 업로드 후 URL 반환 | public/ 경로면 S3 URL 직접 반환, private/ 경로면 Presigned URL 발급 |

### 실무에서 흔히 겪는 실수

| 실수 | 증상 | 해결 |
| ---- | ---- | ---- |
| application.properties에 Access Key 넣고 Git push | AWS에서 경고 메일, 키 자동 비활성화 | 키 즉시 삭제 + credentials 파일 방식으로 전환 |
| admin 키를 팀 전체가 공유 | 누가 뭘 했는지 추적 불가, 퇴사 시 키 교체 필요 | 팀원별 개인 IAM 사용자 생성 |
| S3 버킷을 완전 공개(Block Public Access 해제 + 정책 없음) | 전체 버킷 데이터 인터넷 노출 | 버킷 정책으로 특정 prefix만 공개 |
| 파일 업로드 크기 제한 미설정 | 수 GB 파일 업로드 시 서버 OOM | max-file-size 설정 + Presigned URL 전환 |
| 원본 파일명 그대로 S3 키로 사용 | 한글/특수문자로 URL 깨짐, 파일명 충돌 | UUID 기반 키 생성, 원본 파일명은 DB에 별도 저장 |
| 업로드만 하고 삭제 안 함 | 월말 S3 비용 예상보다 높음 | 수명 주기 규칙으로 오래된 파일 자동 삭제/전환 |

### Access Key 보안 원칙

| 원칙 | 설명 |
| ---- | ---- |
| 최소 권한 | 필요한 버킷의 필요한 작업만 허용하는 커스텀 정책 사용 |
| 용도별 분리 | 서비스/용도마다 별도 IAM 사용자 생성 (admin 키 공용 사용 금지) |
| Root 키 금지 | Root 계정의 Access Key는 절대 생성하지 않음 |
| 정기 로테이션 | Access Key는 90일마다 교체 권장 |
| 팀 공유 금지 | 팀원마다 개인 IAM 사용자, 퇴사 시 해당 키만 삭제 |

### 자격 증명 방식별 장단점

| 방식 | 장점 | 단점 | 위험도 |
| ---- | ---- | ---- | ------ |
| application.properties에 하드코딩 | 가장 간단, 바로 동작 | Git push 시 키 유출, 스캐너가 수 분 내 탈취 | 🔴 매우 높음 |
| application-local.properties + .gitignore | 프로젝트별 명시적 관리 | .gitignore 누락 시 유출, 파일이 프로젝트 내부에 남음 | 🟡 중간 |
| 환경변수 직접 설정 | 코드/파일에 안 남음 | 터미널마다 재설정 필요, 팀원 간 공유 어려움 | 🟢 낮음 |
| ~/.aws/credentials | Git에 올라갈 위험 없음, 여러 프로젝트 공유 | PC에 키 파일이 남아있음 | 🟢 낮음 |
| EC2 IAM Role | 키 유출 불가, 자동 로테이션, AWS 권장 | Role 설정 필요, 로컬에서 사용 불가 | ✅ 가장 안전 |
| GitHub Secrets → 환경변수 | 팀원 간 키 공유 불필요, 암호화 저장 | 로컬 디버깅 시 별도 설정 필요 | ✅ 가장 안전 |

> [!TIP]
> **실무 요약:**
> - 로컬 개발: ~/.aws/credentials (용도별 전용 IAM 사용자의 키)
> - 서버 배포: IAM Role (EC2/ECS/EKS 모두)
> - CI/CD: GitHub Secrets 또는 AWS Secrets Manager
> - **절대 하지 말 것**: 코드/설정 파일에 키 하드코딩, Root 키 생성, admin 키 공유

## 13. CORS (Cross-Origin Resource Sharing)

> [!CONCEPT] CORS란?
> **CORS**는 웹 브라우저가 현재 페이지의 도메인(Origin)과 다른 도메인으로 요청을 보낼 때 적용되는 보안 메커니즘입니다.  
> 브라우저의 **Same-Origin Policy**(동일 출처 정책)에 의해 기본적으로 교차 도메인 요청이 차단되며, 서버가 CORS 헤더를 통해 특정 도메인의 접근을 명시적으로 허용해야 통신이 가능합니다.  
>
> **일상 비유 — 학교 출입:**
> - 브라우저 = 경비실. "이 학교(Origin) 학생만 출입 허용"이 기본 규칙
> - CORS 설정 = 허용 명단. "옆 학교(다른 Origin) 학생도 들어와도 됩니다"라고 명단에 추가
> - 명단에 없는 학교에서 온 학생 → 경비실(브라우저)이 차단
>
> **왜 알아야 하는가?**  
> 프론트엔드(`localhost:5173`)에서 S3(`amazonaws.com`)로 파일을 직접 업로드할 때, CORS 설정이 없으면 브라우저가 요청을 차단합니다.  
> 5-1, 5-2 실습에서 이 설정을 직접 구성합니다.

### Origin이란?

Origin(출처)은 **프로토콜 + 도메인 + 포트**의 조합입니다.  
하나라도 다르면 다른 Origin으로 간주됩니다.

| URL | Origin |
| --- | ------ |
| `http://localhost:5173` | http + localhost + 5173 |
| `http://localhost:3000` | http + localhost + 3000 (**포트 다름 → 다른 Origin**) |
| `https://my-bucket.s3.amazonaws.com` | https + s3.amazonaws.com + 443 |
| `https://www.mydomain.com` | https + www.mydomain.com + 443 |

### 왜 CORS가 필요한가?

```
CORS가 없다면:

  [악성 웹사이트] evil.com
       │
       │ JavaScript로 사용자의 브라우저를 이용
       ▼
  [은행 API] bank.com/api/transfer
       │
       └── 사용자 모르게 송금 요청 → 💀 보안 사고

CORS가 있으면:

  [악성 웹사이트] evil.com
       │
       │ JavaScript로 요청 시도
       ▼
  [은행 API] bank.com
       │
       └── CORS 헤더에 evil.com이 없음 → ❌ 브라우저가 차단
```

Same-Origin Policy는 이런 공격을 원천 차단합니다.  
하지만 정상적인 교차 도메인 통신(예: 프론트엔드에서 S3로 파일 업로드)도 차단하므로, 서버에서 CORS 설정으로 허용할 도메인을 명시합니다.

### CORS 동작 방식: Preflight 요청

브라우저는 실제 요청을 보내기 전에 **Preflight(사전 확인)** 요청을 먼저 보냅니다.

```
1. [브라우저] → OPTIONS /upload (Preflight 요청)
   "PUT 메서드를 써도 되나요? Origin은 localhost:5173입니다"

2. [S3 서버] ← 200 OK + CORS 응답 헤더
   "Access-Control-Allow-Origin: http://localhost:5173"
   "Access-Control-Allow-Methods: GET, PUT, POST"

3. [브라우저] 허용 확인 → 실제 PUT 요청 전송
   (Preflight 결과는 MaxAgeSeconds 동안 캐시하여 매번 보내지 않음)
```

> [!NOTE]
> 단순 GET 요청(Simple Request)은 Preflight 없이 바로 전송됩니다.  
> PUT, DELETE, 또는 커스텀 헤더가 포함된 요청만 Preflight가 발생합니다.

### S3 CORS 설정의 주요 항목

| 항목 | 설명 | 예시 |
| ---- | ---- | ---- |
| **AllowedOrigins** | 허용할 도메인 목록. `*`는 전체 허용 (프로덕션 비권장) | `["http://localhost:5173", "https://www.mydomain.com"]` |
| **AllowedMethods** | 허용할 HTTP 메서드 | `["GET", "PUT", "POST", "DELETE"]` |
| **AllowedHeaders** | 허용할 요청 헤더. `*`는 전체 허용 | `["*"]` 또는 `["Content-Type", "Authorization"]` |
| **ExposeHeaders** | 브라우저에서 읽을 수 있도록 노출할 응답 헤더 | `["ETag", "x-amz-request-id"]` |
| **MaxAgeSeconds** | Preflight 결과 캐시 시간 (초) | `3600` (1시간) |

### 실무 CORS 설정 패턴

**개발 환경 (로컬):**

```json
[
  {
    "AllowedHeaders": ["*"],
    "AllowedMethods": ["GET", "PUT", "POST", "DELETE"],
    "AllowedOrigins": ["http://localhost:5173", "http://localhost:3000"],
    "ExposeHeaders": ["ETag"],
    "MaxAgeSeconds": 3600
  }
]
```

**프로덕션 환경:**

```json
[
  {
    "AllowedHeaders": ["Content-Type", "Authorization", "x-amz-content-sha256"],
    "AllowedMethods": ["GET", "PUT"],
    "AllowedOrigins": ["https://www.mydomain.com"],
    "ExposeHeaders": ["ETag"],
    "MaxAgeSeconds": 86400
  }
]
```

> [!WARNING]
> `AllowedOrigins: ["*"]`는 모든 도메인에서 접근을 허용합니다.  
> 개발 테스트에서는 편리하지만, 프로덕션에서는 반드시 실제 도메인만 명시하세요.

### CORS 에러 트러블슈팅

브라우저 콘솔에 다음과 같은 에러가 나타나면 CORS 설정을 확인하세요:

| 에러 메시지 | 원인 | 해결 |
| ----------- | ---- | ---- |
| `Access to XMLHttpRequest has been blocked by CORS policy` | S3에 CORS 미설정 또는 Origin 미등록 | AllowedOrigins에 프론트엔드 도메인 추가 |
| `Method PUT is not allowed by Access-Control-Allow-Methods` | AllowedMethods에 PUT 미포함 | AllowedMethods에 필요한 메서드 추가 |
| `Request header field authorization is not allowed` | AllowedHeaders에 해당 헤더 미포함 | AllowedHeaders에 `"*"` 또는 해당 헤더 추가 |
| CORS 설정 변경 후에도 에러 지속 | 브라우저 Preflight 캐시 | 브라우저 캐시 삭제 또는 시크릿 창에서 테스트 |

## 14. CDN (Content Delivery Network) 개념

> [!CONCEPT] CDN의 동작 원리
> **CDN**은 전 세계에 분산된 엣지 서버에 콘텐츠를 캐싱하여, 사용자와 가장 가까운 서버에서 콘텐츠를 제공하는 네트워크입니다.  
> AWS CloudFront가 대표적인 CDN 서비스입니다.
>
> **일상 비유 — 편의점 유통:**
> - CDN 없이 = 모든 손님이 서울 본사 창고까지 가서 물건을 가져감 (느림)
> - CDN 사용 = 전국 편의점(엣지)에 인기 상품을 미리 배치. 손님은 가장 가까운 편의점에서 바로 구매 (빠름)
> - 편의점에 재고가 없으면(캐시 미스) 본사에서 가져와서 진열(캐싱). 다음 손님은 바로 구매 가능(캐시 히트)

### 주요 용어

| 용어                               | 설명                                                           |
| ---------------------------------- | -------------------------------------------------------------- |
| **CDN (Content Delivery Network)** | 전 세계 엣지 서버에 콘텐츠를 캐싱하여 빠르게 전달하는 네트워크 |
| **Origin**                         | CDN이 원본 콘텐츠를 가져오는 서버 (S3, ALB, EC2 등)            |
| **Edge Location**                  | 사용자와 가까운 곳에 위치한 캐시 서버                          |
| **Cache Hit / Miss**               | 엣지에 캐시가 있으면 Hit(빠름), 없으면 Miss(Origin 조회)       |
| **TTL (Time To Live)**             | 캐시된 콘텐츠의 유효 시간. 만료 후 Origin에서 재조회           |
| **Invalidation**                   | 캐시를 강제로 삭제하는 작업 (배포 시 사용)                     |

### CDN 동작 원리

```
CDN 없이:
┌────────┐                    ┌───────────┐
│ 한국   │ ───── 200ms ────►  │ 미국 서버 │
│ 사용자 │                    │ (Origin)  │
└────────┘                    └───────────┘

CDN 사용:
┌────────┐        ┌─────────────┐          ┌───────────┐
│ 한국   │ ─5ms─► │ 서울 Edge   │ ◄─캐싱── │ 미국 서버 │
│ 사용자 │        │ (캐시 히트) │          │ (Origin)  │
└────────┘        └─────────────┘          └───────────┘

첫 요청: Edge → Origin → 응답 + 캐싱
이후 요청: Edge에서 직접 응답 (캐시 히트)
```

### CloudFront 핵심 개념

| 용어              | 설명                               |
| ----------------- | ---------------------------------- |
| **Origin**        | 원본 콘텐츠 서버 (S3, ALB, EC2 등) |
| **Edge Location** | 전 세계 600+ 캐시 서버             |
| **Distribution**  | CloudFront 배포 설정 단위          |
| **TTL**           | 캐시 유지 시간 (Time To Live)      |
| **Cache Hit**     | 엣지에서 직접 응답 (빠름)          |
| **Cache Miss**    | Origin에서 가져옴 (느림)           |
| **Invalidation**  | 캐시 강제 삭제 (배포 시)           |

### CDN 적용 효과

| 항목            | CDN 미적용     | CDN 적용         |
| --------------- | -------------- | ---------------- |
| **지연 시간**   | 200~500ms      | 5~50ms           |
| **Origin 부하** | 모든 요청 처리 | 캐시 미스만 처리 |
| **대역폭 비용** | 높음           | 낮음 (엣지 캐싱) |
| **DDoS 방어**   | 직접 노출      | 엣지에서 흡수    |
| **HTTPS**       | 직접 설정      | ACM 무료 인증서  |

## 15. 캐싱 전략

> [!CONCEPT] 캐싱 전략과 패턴
> **캐싱**은 자주 접근하는 데이터를 빠른 저장소에 임시 보관하여 응답 속도를 높이는 전략입니다.  
> 캐시 적중률(Hit Rate)을 높이는 것이 핵심이며, 데이터 일관성과의 트레이드오프를 고려해야 합니다.

### 주요 용어

| 용어                       | 설명                                                    |
| -------------------------- | ------------------------------------------------------- |
| **캐싱 (Caching)**         | 자주 접근하는 데이터를 빠른 저장소에 임시 보관하는 전략 |
| **Cache-Aside**            | 앱이 캐시 확인 → 미스 시 DB 조회 → 캐시 저장하는 패턴   |
| **Write-Through**          | 쓰기 시 캐시와 DB에 동시 저장하는 패턴                  |
| **캐시 적중률 (Hit Rate)** | 전체 요청 중 캐시에서 응답한 비율. 높을수록 효율적      |

### 캐싱 계층 구조

```
사용자 요청 흐름:

          ┌──────────┐   ┌────────────┐   ┌──────────┐   ┌──────────┐   ┌────┐
          │ 브라우저 │──►│    CDN     │──►│  App     │──►│  Redis   │──►│ DB │
          │  캐시    │   │(CloudFront)│   │  Server  │   │  (캐시)  │   │    │
          └──────────┘   └────────────┘   └──────────┘   └──────────┘   └────┘
            L1 캐시         L2 캐시         L3 처리        L4 캐시      L5 원본

응답 속도:   <1ms           5~50ms         10~100ms         1~5ms       10~100ms
```

### 캐시 전략 패턴

| 전략              | 동작                                         | 장점             | 단점                 |
| ----------------- | -------------------------------------------- | ---------------- | -------------------- |
| **Cache-Aside**   | 앱이 캐시 확인 → 미스 시 DB 조회 → 캐시 저장 | 구현 간단, 유연  | 첫 요청 느림         |
| **Read-Through**  | 캐시가 자동으로 DB 조회                      | 앱 코드 단순     | 캐시 라이브러리 의존 |
| **Write-Through** | 쓰기 시 캐시+DB 동시 저장                    | 항상 최신 데이터 | 쓰기 지연            |
| **Write-Behind**  | 쓰기 시 캐시만 → 나중에 DB 반영              | 쓰기 빠름        | 데이터 손실 위험     |

### Cache-Aside 패턴 흐름

```
읽기 요청:
1. 캐시 확인 (Redis)
   ├── Hit → 캐시에서 반환 (빠름)
   └── Miss → DB 조회 → 캐시에 저장 → 반환

쓰기 요청:
1. DB에 저장
2. 캐시 무효화 (삭제)
   └── 다음 읽기 시 최신 데이터로 캐시 갱신
```

## 16. S3 정적 웹 호스팅과 SPA

> [!CONCEPT] S3 정적 웹 호스팅이란?
> S3 버킷을 웹 서버처럼 사용하여 HTML, CSS, JS 파일을 HTTP 엔드포인트로 서빙하는 기능입니다.  
> 별도 서버(EC2, Nginx) 없이 정적 사이트를 배포할 수 있으며, CloudFront를 연동하면 HTTPS와 CDN까지 적용됩니다.

### 주요 용어

| 용어 | 설명 |
| ---- | ---- |
| **정적 웹 호스팅 (Static Website Hosting)** | S3 버킷에 저장된 파일을 웹 서버처럼 HTTP로 서빙하는 기능 |
| **정적 사이트 (Static Site)** | 서버 측 처리 없이 브라우저에서 HTML/CSS/JS만으로 동작하는 사이트 |
| **SPA (Single Page Application)** | 하나의 HTML 파일을 로드한 뒤, JavaScript가 페이지 전환을 처리하는 앱 (Vue, React 등) |
| **클라이언트 사이드 라우팅** | URL 경로를 서버가 아닌 브라우저의 JavaScript(라우터)가 해석하는 방식 |
| **Index Document** | `/` 요청 시 반환할 기본 파일 (보통 `index.html`) |
| **Error Document** | 404 발생 시 반환할 파일. SPA에서는 `index.html`로 설정하여 라우터가 처리하도록 함 |
| **S3 웹사이트 엔드포인트** | 정적 호스팅 활성화 시 생성되는 HTTP URL (형식: `bucket.s3-website.region.amazonaws.com`) |

### 정적 사이트 vs 동적 사이트

| 항목 | 정적 사이트 | 동적 사이트 |
| ---- | ----------- | ----------- |
| **서버 측 처리** | 없음 (파일 그대로 서빙) | 있음 (DB 조회, 로직 실행) |
| **구성** | HTML, CSS, JS, 이미지 | Spring, Node.js 등 서버 앱 |
| **호스팅** | S3 + CloudFront | EC2, ECS, Lambda 등 |
| **빌드** | `npm run build` → 정적 파일 생성 | 서버 프로세스 실행 필요 |
| **예시** | Vue/React SPA, 문서 사이트 | REST API, 관리자 페이지 |

### SPA 라우팅이 특별한 이유

```
[일반 웹사이트] 각 URL에 실제 파일이 존재
  /index.html  → index.html 파일 반환
  /about.html  → about.html 파일 반환
  /contact.html → contact.html 파일 반환

[SPA] 모든 URL을 index.html 하나로 처리
  /           → index.html 반환 → JS가 메인 페이지 렌더링
  /about      → index.html 반환 → JS가 about 페이지 렌더링
  /users/123  → index.html 반환 → JS가 users/123 페이지 렌더링

문제: S3에는 /about 파일이 없음 → 404 반환
해결: Error Document를 index.html로 설정 → 404 대신 index.html 반환 → JS 라우터가 처리
```

> [!TIP]
> SPA 배포 시 에러 페이지 설정이 빠지면, 사용자가 `/about`을 직접 입력하거나 새로고침할 때 404 에러가 발생합니다.  
> CloudFront에서는 Custom Error Response(403/404 → index.html, 200)로 같은 문제를 해결합니다.

## 17. HTTPS와 TLS 인증서 기초

> [!CONCEPT] HTTPS란?
> **HTTPS**는 HTTP에 TLS(Transport Layer Security) 암호화를 적용한 프로토콜입니다.  
> 브라우저와 서버 사이의 통신을 암호화하여 데이터 도청과 변조를 방지합니다.  
> 현대 웹에서는 HTTPS가 사실상 필수입니다.

### 주요 용어

| 용어 | 설명 |
| ---- | ---- |
| **HTTPS** | HTTP + TLS. 암호화된 안전한 통신 프로토콜 |
| **TLS (Transport Layer Security)** | 통신을 암호화하는 보안 프로토콜. SSL의 후속 표준 |
| **TLS 인증서 (Certificate)** | 서버의 신원을 증명하고 암호화 키를 제공하는 디지털 문서 |
| **ACM (AWS Certificate Manager)** | AWS에서 TLS 인증서를 무료로 발급·관리하는 서비스 |
| **자물쇠 아이콘 (🔒)** | 브라우저 주소창에서 HTTPS 연결을 나타내는 표시 |

### HTTP vs HTTPS

| 항목 | HTTP | HTTPS |
| ---- | ---- | ----- |
| **암호화** | ❌ 평문 전송 | ✅ TLS 암호화 |
| **도청 방지** | ❌ 네트워크에서 내용 열람 가능 | ✅ 암호화되어 열람 불가 |
| **브라우저 표시** | ⚠️ "주의 요함" 경고 | 🔒 자물쇠 아이콘 |
| **SEO** | 불이익 | 가산점 |
| **포트** | 80 | 443 |

### CloudFront에서의 HTTPS

```
S3 정적 웹 호스팅만 사용:
  HTTP만 지원 → 브라우저에서 "안전하지 않음" 경고

CloudFront 연동 시:
  *.cloudfront.net 인증서 자동 적용 → HTTPS 즉시 사용 가능
  커스텀 도메인: ACM에서 무료 인증서 발급 → CloudFront에 연결
```

> [!TIP]
> CloudFront의 `*.cloudfront.net` 도메인은 별도 인증서 설정 없이 HTTPS가 바로 동작합니다.  
> 자체 도메인(예: `www.mysite.com`)을 사용하려면 ACM에서 인증서를 발급받아야 합니다.
